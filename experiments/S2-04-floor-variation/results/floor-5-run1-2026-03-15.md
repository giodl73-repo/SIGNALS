## Competitive Brief: Signal Plugin

### The Insight

Signal has no direct competitor. The risk isn't disruption — it's irrelevance. Every adjacent player occupies either the **build-side** (AI coding tools) or the **PM-side** (roadmap tools). The pre-commit evidence layer for developers is uncontested. But uncontested spaces don't mean high demand. The primary threat is the team that never reaches for evidence before committing.

---

### 1. The Primary Competitor: Inertia — Threat: HIGH

**"Just build it and see."**

What makes inertia structurally sticky:
- No visible failure mode upfront — you can't cite the bug you didn't find until you built the wrong thing
- Ship culture compounds — every sprint without Signal is retrospective evidence that Signal isn't needed
- Low activation energy wins — Signal requires knowing which skill to invoke; inertia requires nothing
- Cognitive load argument — 9 namespaces looks expensive before teams see the payoff

Signal's answer must be: **zero activation cost on first use** and **one artifact in one hour that would have saved a sprint**.

---

### 2. The Whitespace

No competitor owns the **pre-commit evidence layer for developers**:
- **Repo-native** — artifacts live in version control alongside the code they inform
- **Simulation-based** — structured hand-compilation, persona walkthroughs, and hypothesis investigation *before* production code is written
- **Developer-owned** — a skill set invoked in the toolchain, not a PM dashboard or SaaS subscription
- **Lifecycle-spanning** — from "should we build this?" to "will customers adopt this?" without leaving the repo

**Key findings:**

- **GitHub Copilot** (verified: github.com/features/copilot) — helps write code and review PRs, not simulate flows or run persona walkthroughs. Risk: if Copilot adds `/plan` or `/validate` skills producing Signal-like artifacts, it competes for early adopters.

- **Cursor** (verified: cursor.com/docs) — "best way to build software with AI." Planning means "plan what to code," not "decide whether to build." Adjacent, not direct.

- **Productboard** (verified: productboard.com) — Productboard Spark directly addresses PM-side competitive research and feedback synthesis — Signal's `/scout:` and `/listen:` from the PM chair. Coexist if PM-owned vs dev-native, compete directly if Productboard expands to dev artifacts.

- **Notion AI** (verified: notion.com/product/ai) — research + workspace automation, no structured simulation namespaces or repo-native artifact storage. Complementary.

- **Raw Claude Code** — THIS is Signal's existential competitor. Every Signal skill could be run as a freeform prompt. Signal's answer: accumulated artifacts are different from one-off chat responses. Structure compounds. Raw prompting doesn't produce a `TOPICS.md`.

---

### 5. --mode risk: The Cost of Building the Wrong Thing

Three observable failure modes, each preventable:

| Failure Mode | Recovery Cost | Signal Prevention |
|--------------|---------------|-------------------|
| Wrong architecture, right feature — state bug found in production instead of design | 2-4 sprint replanning cycles + refactor | One `/trace:state` + `/trace:contract` run |
| Right build, wrong adoption — feature ships but personas never reach the new flow | Post-ship redesign, usually abandoned | One `/listen:adoption` before design locks |
| Solved the local problem, not the user problem — spec gap surfaces post-ship | Full re-scoping cycle | Two `/scout:` + one `/prove:interview` pre-spec |

**Risk quantification**: One sprint of unvalidated work costs N engineer-weeks. Signal's evidence pipeline for a medium-complexity feature takes 2–4 hours. Expected value is positive at a 10% rework-avoidance rate for any team with >4 engineers shipping regularly.
