### Bug: `ROUND` is out of scope at Step 4

**Evidence** (lines 326, 343, 344, 373, 384):
```bash
$(cat "$LAST_SCORECARD" | head -100)    # line 326
rounds: $ROUND                           # line 343
score: $LAST_SCORE                       # line 344
skill['quest_status'] = 'GOLDEN (Round $ROUND, score $LAST_SCORE)'  # line 373
echo "Score: $LAST_SCORE, Rounds: $ROUND, ..."  # line 384
```
`ROUND` is the loop variable from `for ROUND in $(seq 1 $MAX_ROUNDS)`. After the loop ends, `ROUND` holds the last value from `seq` (i.e., `$MAX_ROUNDS`), not the round where convergence actually happened. If the loop breaks early at round 3, `ROUND` correctly reflects 3 â€” but only because bash `for` leaves the variable at its last assigned value when `break` exits. This is fragile: if the loop runs to completion without converging, `ROUND` is 20, which is accurate but coincidental. The safe fix is to capture into a dedicated variable (e.g., `CONVERGED_ROUND=$ROUND`) at the `break` site.

### Bug: `$SIMPLIFIED_PASS` is shell output used inside a Python string literal

**Evidence** (lines 318â€“321):
```bash
GOLDEN_PROMPT="You are finalizing a quest run for Signal skill: $SKILL_ID

Simplification result: $SIMPLIFIED_PASS
Use the SIMPLIFIED prompt body if simplification passed..."
```
`$SIMPLIFIED_PASS` can contain parentheses: `PASS (40% reduction)` or `FAIL (essential criteria lost -- use original)`. These are safe inside a double-quoted bash string being passed to `claude_ask`, but the `%` character is benign here. The real risk is `PARSE_FAILED (use original)` â€” the LLM receives ambiguous instructions since the rubric check line says "use original" in two different failure modes (`FAIL` and `PARSE_FAILED`) without distinguishing what "original" refers to (winning variation vs `$LAST_VARIATIONS` file). This isn't a shell crash, but it silently sends the LLM a misleading decision branch.

### Bug: `head -80` on rubric files may truncate mid-criterion

**Evidence** (lines 122, 147, 228, 252, 333):
```bash
$(cat "$RUBRIC_FILE" | head -80)
$(cat "$RUBRIC_FILE" | head -60)
$(cat "$RUBRIC_FILE" | head -80)
```
Used in both the VARIATE and SCORE prompts. A rubric with 6â€“9 criteria and frontmatter easily exceeds 80 lines. Truncating mid-criterion means the scorer evaluates against an incomplete rubric while the writer generates to a complete one â€” producing incoherent round results. The same pattern applies to `head -100` on scorecards (lines 231, 327) and `head -150` on variations (line 331). The rubric is the ground truth; it should never be truncated.

### Bug: Python regex for JSON block is too narrow

**Evidence** (lines 175, 185, 197):
```python
m = re.search(r'\`\`\`json\s*({[^}]+})\s*\`\`\`', txt, re.DOTALL)
```
The pattern `{[^}]+}` matches the first `{` to the first `}`. If any value in the JSON contains a nested object or an escaped brace, the match fails and `0`/`false` is returned silently. More critically, the pattern anchors on the final ` ``` ` but allows arbitrary whitespace before it â€” if the LLM outputs a trailing newline inside the fence, the `[^}]+` stops at `}` before the newline and the close-fence check never fires. The safer pattern is `\`\`\`json\s*(\{.*?\})\s*\`\`\`` with `re.DOTALL`, or using a JSON decoder with `json.loads` on everything between the fences.

### Bug: `set -e` + `|| echo "0"` interaction in subshell

**Evidence** (lines 9, 181, 192, 204):
```bash
set -e
...
TOP_SCORE=$(python -c "..." 2>/dev/null || echo "0")
```
`set -e` causes the script to exit on any non-zero return code. The `|| echo "0"` guard handles the python failure case, but `2>/dev/null` suppresses the python traceback silently. If `signal.skills.yaml` is malformed or `$SCORECARD_FILE` doesn't exist at parse time, python exits non-zero, the `echo "0"` fires, and the loop continues with `TOP_SCORE=0` â€” treating a parse failure as "score 0, keep looping" rather than halting. This is especially dangerous if an early `claude_ask` call fails: `set -e` would exit then, but only if the return code propagates through the command substitution correctly (it does not â€” command substitution swallows exit codes).

### Bug: Temp file name collision if called concurrently

**Evidence** (line 31):
```bash
tmp_prompt="/tmp/claude-ask-$$.txt"
```
The comment above says "P-01 concurrent by default" (line 6). `$$` is the PID of the parent shell. All `claude_ask` calls within one `quest-run-one.sh` invocation share the same PID and therefore the same temp filename. The function writes, invokes claude, then deletes â€” serial within a single run this is fine. But if two parallel invocations of `quest-run-one.sh` are running simultaneously (which the "concurrent by default" comment implies), they will share `/tmp/claude-ask-<PID>.txt` if the PID happens to collide. Using `mktemp` (already used correctly at line 250) is the consistent fix.
