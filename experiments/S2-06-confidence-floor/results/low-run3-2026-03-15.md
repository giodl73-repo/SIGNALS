Here are the findings:

---

## F-01 — `SPEC_FILE` is assigned but never used

`SPEC_FILE` is computed at line 16 (`signal-scout-$TODAY.md`) with the skill hardcoded as `scout`, not `$SKILL_ID`. It is never read or referenced anywhere in the script. Either remove it or fix the name and wire it into the rubric prompt as a third input alongside the trace.

---

## F-02 — `$ROUND` is undefined outside the loop (Step 4 and Step 5)

After the `for` loop, `$ROUND` holds the last value from `seq` — which is `$MAX_ROUNDS` (20) if the loop ran to completion without breaking, not the actual round where convergence occurred. The golden frontmatter and the `quest_status` string will say "Round 20" even when convergence happened at round 3. Fix: save the convergence round to a separate variable (`CONVERGED_ROUND`) inside the loop before `break`, then use that variable.

---

## F-03 — Score comparison uses string equality, not integer comparison

```bash
if [ "$NEW_PATTERNS" -gt "0" ]; then
```
This uses `-gt` which is fine, but:
```bash
if [ "$ALL_ESSENTIAL" = "true" ] && [ "$NEW_PATTERNS" = "0" ]; then
```
`$TOP_SCORE` and `$NEW_PATTERNS` come from Python stdout which could have trailing whitespace or newlines. Python's `print()` adds a newline; shell captures it in `$()` which strips trailing newlines — this is safe. But if Python fails and the `|| echo "0"` fallback fires, the string is clean. Low risk, but `TOP_SCORE` is never compared numerically so the score threshold (`>= 80`) is never enforced by the script itself — the loop only exits on dual convergence, not on score.

---

## F-04 — `set -e` interacts badly with the `|| echo` fallback pattern

`set -e` exits on any non-zero return. The `|| echo "..."` patterns suppress failures from Python correctly, but `claude_ask` failures are **not** suppressed — a failed `claude --print` call will abort the entire script mid-loop with no cleanup and no partial-results summary. The QU5 block does handle this (`|| { echo ...; cp ... }`) but the rubric, variate, and score calls do not. The most impactful case: if `claude --print` fails on round 8, all prior artifacts exist but the script exits silently with no golden and no yaml update.

---

## F-05 — Truncation of rubric and variations in prompts is aggressive and inconsistent

The rubric is truncated to `head -80` lines when fed into the variate and score prompts, but the initial rubric is written by Claude and could easily exceed 80 lines for a complex skill. A rubric with 6 criteria plus frontmatter and commentary hits 80 lines quickly. The variations are truncated to `head -200` for scoring and `head -100` for the golden step. These are arbitrary line limits, not character limits. A rubric that gets cut mid-criterion will produce incorrect scoring in the next round. Use character-based slicing (already done in the rubric prompt: `${TRACE_CONTENT:0:3000}`) or read the full file and let Claude's context window handle it.

---

## F-06 — QU5 step uses `$LAST_VARIATIONS` which is the full variations file, not the winning variation

The QU5 prompt says "Winning variation (the complete golden prompt)" but passes the entire variations file (all 5 variations). Claude must infer which one won from the scorecard context. This is ambiguous — Claude may simplify the wrong variation. The scorecard identifies the top-ranked variation; extracting just the winner before passing it to QU5 would make the prompt unambiguous.

---

## F-07 — Python inline scripts use shell variable interpolation inside Python string literals

In Step 2:
```bash
python -c "
...
    if s['id'] == '$SKILL_ID':
..."
```
If `$SKILL_ID` contains a single quote (e.g., `o'reilly-skill`), this breaks Python syntax. Same issue in Step 5 with `$SKILL_ID`, `$TODAY`, `$ROUND`, `$LAST_SCORE`. Use environment variables passed via `-c` arguments or write the Python to a temp file and pass values as `sys.argv` — the QU5 block already does this correctly with `sys.argv`. Apply that same pattern consistently.

---

## F-08 — The plateau reset logic resets on rubric evolution even when score is high

When `NEW_PATTERNS > 0`, `PLATEAU_COUNT` resets to 0 and the rubric evolves. This means a skill that scores 95 every round but keeps discovering minor new patterns will never converge — it will run all 20 rounds. The convergence condition should also check `TOP_SCORE >= 80` (or some threshold) so that a high-scoring skill with diminishing new patterns can exit early. Consider: plateau if `TOP_SCORE >= 80 AND NEW_PATTERNS <= 1` rather than strictly `NEW_PATTERNS == 0`.

---

## F-09 — `LAST_VARIATIONS` and `LAST_SCORECARD` are empty strings if the loop never executes

`MAX_ROUNDS=20` so the loop always runs at least once, but `seq 1 0` would produce nothing. More practically: if `$MAX_ROUNDS` were ever set to 0 (e.g., for a dry-run), `LAST_VARIATIONS` and `LAST_SCORECARD` remain empty strings, causing the QU5 and golden steps to `cat` an empty path and produce garbage or errors. A guard after the loop (`if [ -z "$LAST_SCORECARD" ]; then echo "No rounds completed"; exit 1; fi`) would make the failure explicit.

---

## F-10 — `signal.skills.yaml` is overwritten with `yaml.dump` which loses comments and key ordering

`yaml.safe_load` + `yaml.dump` with `sort_keys=False` preserves key insertion order but **drops all YAML comments**. If `signal.skills.yaml` has inline comments (e.g., `# promoted 2026-03-14`), they are silently lost on every `quest-run-one` run. Use a line-based sed/regex patch for the three specific fields (`golden:`, `rubric:`, `quest_status:`) instead of a full round-trip through PyYAML.
