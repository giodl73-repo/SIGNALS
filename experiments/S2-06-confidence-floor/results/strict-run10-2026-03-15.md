## Findings

### Bug — `ROUND` leaks from loop, undefined on empty iteration

**Evidence:** Line 344 uses `$ROUND` in the golden prompt frontmatter, and line 384 echoes `Rounds: $ROUND`. If `MAX_ROUNDS=0` (or the loop never enters), `ROUND` is never set and expands to empty string. Same issue at line 373: `skill['quest_status'] = 'GOLDEN (Round $ROUND, score $LAST_SCORE)'` — the Python string substitution happens at shell interpolation time, so an empty `$ROUND` silently writes a malformed string.

**Fix:** Initialize `ROUND=0` before the loop.

### Bug — Plateau logic resets on rubric evolution but doesn't reset on score regression

**Evidence (lines 211–241):**
```bash
if [ "$ALL_ESSENTIAL" = "true" ] && [ "$NEW_PATTERNS" = "0" ]; then
    PLATEAU_COUNT=$((PLATEAU_COUNT + 1))
    ...
else
    PLATEAU_COUNT=0
    if [ "$NEW_PATTERNS" -gt "0" ]; then
```

The `else` resets `PLATEAU_COUNT=0` whenever either condition fails — that's correct. But `$TOP_SCORE` is compared to nothing; there's no regression check. A round that scores 45 (below essential) resets the plateau counter the same way as one that finds new patterns. Both cases are `else`. This is only a logic concern, not a crash bug, but a score regression should arguably *not* reset plateau the same way as genuine discovery.

### Bug — Python inline script expands shell variables before Python runs; `$SKILL_ID` injection risk

**Evidence (lines 54–62):**
```bash
SKILL_DESC=$(python -c "
import yaml
with open('$REPO/signal.skills.yaml') as f:
    data = yaml.safe_load(f)
for s in data.get('skills', []):
    if s['id'] == '$SKILL_ID':
```

`$SKILL_ID` is shell-interpolated into the Python source before Python runs. If `SKILL_ID` contains a single quote (`'`), the Python string literal breaks. Example: `$SKILL_ID = "scout-it's"` → `if s['id'] == 'scout-it's':` — syntax error, silently falls through to `|| echo "Skill not found in registry"`. Step 5's Python block (lines 361–380) has the same pattern.

**Fix:** Pass `SKILL_ID` as an environment variable or `sys.argv` argument rather than interpolating into source.

### Bug — `RUBRIC_FILE` `head -80` truncates the rubric during scoring and evolution

**Evidence (lines 146–148):**
```bash
Rubric (v$RUBRIC_VERSION):
$(cat "$RUBRIC_FILE" | head -80)
```
And lines 122–123 (variate), 228–229 (evolve). Eighty lines is enough for a short rubric but the rubric itself is AI-generated and could grow past 80 lines after evolution, silently dropping criteria from the scoring prompt. The trace artifact is also hard-truncated at 3000 chars (line 77) and 800 chars (line 153) with no warning if content was cut.

**Fix:** Remove the `head` caps or replace with a deliberate, documented line limit constant. At minimum, log the actual line count so truncation is visible.

### Bug — `LAST_VARIATIONS` and `LAST_SCORECARD` used after loop without checking they were set

**Evidence (lines 106–107, 251, 329):**
```bash
LAST_VARIATIONS=""
LAST_SCORECARD=""
```
These are initialized empty. If the loop exits after round 1 without setting them (hypothetically, if `seq 1 $MAX_ROUNDS` produces nothing), the QU5 step at line 251 does `cat "$LAST_VARIATIONS"` on an empty string — which becomes `cat ""`, printing stdin or erroring depending on shell.

In practice `MAX_ROUNDS=20` prevents this, but the initialization pattern sets up a latent risk.

### Design Issue — Duplicate JSON parsing across three separate Python invocations

**Evidence (lines 172–204):** `top_score`, `all_essential_pass`, and `new_patterns` are each parsed by a separate `python -c` subprocess that opens and regex-searches the same file three times:
```bash
TOP_SCORE=$(python -c "... open('$SCORECARD_FILE').read() ...")
ALL_ESSENTIAL=$(python -c "... open('$SCORECARD_FILE').read() ...")
NEW_PATTERNS=$(python -c "... open('$SCORECARD_FILE').read() ...")
```

Single parse, three print calls (or a JSON output split on newlines) would be more reliable and 3× cheaper.

### Design Issue — `set -e` + `|| echo` fallback pattern is fragile

**Evidence (lines 9, 62, 181, 204):** `set -e` causes the script to abort on any unhandled non-zero exit. The Python subshells use `2>/dev/null || echo "..."` to suppress errors and provide fallbacks — that works. But `claude_ask` (line 33) pipes directly to a file with no fallback:
```bash
claude --print < "$tmp_prompt" > "$out_file"
```
If `claude` exits non-zero (rate limit, network error, etc.), `set -e` kills the whole script, leaving a partial `$out_file` that the next step will try to `cat`. The temp file `$tmp_prompt` is also leaked on failure because `rm -f` runs after the `local rc` capture — but `set -e` would have already exited before reaching `rm`.

**Fix:** Wrap the `claude` call in an explicit error handler, or use `set -E` + a trap to clean up on ERR.
