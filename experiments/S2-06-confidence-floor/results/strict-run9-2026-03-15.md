## Review: `tools/quest-run-one.sh`

### Bugs

**B-01: `ROUND` leaks out-of-loop scope when used after the loop**

```bash
for ROUND in $(seq 1 $MAX_ROUNDS); do
  ...
done
# ...
echo "Score: $LAST_SCORE, Rounds: $ROUND, Rubric: v$RUBRIC_VERSION"
```

`ROUND` holds its last value from the `for` loop in bash, so if the loop breaks early via `break` it works correctly — but if it runs to `MAX_ROUNDS` without convergence, `ROUND` will equal `20`, which is correct by accident, not by design. The `GOLDEN_PROMPT` heredoc also uses `$ROUND` directly (line 343: `rounds: $ROUND`). No bug if loop runs to exhaustion, but the intent is fragile. A dedicated `FINAL_ROUND` variable set before `break` and after the loop would make it explicit.

---

**B-02: `TOP_SCORE` comparison is string-based, not numeric**

```bash
if [ "$ALL_ESSENTIAL" = "true" ] && [ "$NEW_PATTERNS" = "0" ]; then
```

`NEW_PATTERNS` is compared with `= "0"` (string equality) which works, but `TOP_SCORE` is never compared numerically — the golden threshold check (`composite >= 80`) from the rubric prompt is delegated entirely to Claude's JSON output. If Claude hallucinates `"top_score": "85"` (a string) instead of `85`, the `python -c` block at line 172 calls `d.get('top_score', 0)` and `print()` on it — `print("85")` still produces `"85"` in stdout, which later gets stored as `LAST_SCORE` and interpolated into the YAML as a string. Not a crash, but the YAML field `quest_status: 'GOLDEN (Round 3, score 85)'` would silently accept any garbage value.

---

**B-03: `SPEC_FILE` is set but never used**

```bash
SPEC_FILE="$REPO/simulations/draft/specs/signal-scout-$TODAY.md"
```

`SPEC_FILE` is set at line 16 with a hardcoded `signal-scout-` prefix regardless of `SKILL_ID`, and is never referenced again in the script. The rubric prompt includes `$SKILL_DESC` and `$TRACE_CONTENT` but not `$SPEC_FILE`. This is either dead code or a missing feature.

---

**B-04: Plateau reset logic discards a partial convergence signal**

```bash
if [ "$ALL_ESSENTIAL" = "true" ] && [ "$NEW_PATTERNS" = "0" ]; then
    PLATEAU_COUNT=$((PLATEAU_COUNT + 1))
else
    PLATEAU_COUNT=0
```

`PLATEAU_COUNT` is reset to `0` any time either condition fails — including the case where `ALL_ESSENTIAL` is already `true` but a new pattern was found. That means round N=`true/1`, N+1=`true/0` increments to 1, but round N+2=`true/1` resets to 0, even though essential criteria have been passing continuously. A score that's oscillating due to rubric evolution could spin forever. Consider only resetting on `ALL_ESSENTIAL = false`.

### Findings (Non-Bug)

**F-01: Rubric and scorecard context is truncated to `head -80` / `head -200`**

```bash
$(cat "$RUBRIC_FILE" | head -80)
$(cat "$VARIATIONS_FILE" | head -200)
$(cat "$SCORECARD_FILE" | head -100)
```

These truncation limits appear across Steps A, B, and the evolve prompt. If a rubric grows through multiple evolution cycles (v2, v3...) the later sections — which contain the new aspirational criteria added by `EVOLVE_PROMPT` — are exactly what gets truncated. The rubric evolution pass may generate criteria that the score pass never sees.

---

**F-02: QU5 uses `LAST_VARIATIONS` but passes full file path, not the winning variation**

```bash
python -c "
...
variations = open(sys.argv[3]).read()
...
{variations[:3000]}
" "$SKILL_ID" "$RUBRIC_VERSION" "$LAST_VARIATIONS" "$RUBRIC_FILE" "$QU5_PROMPT_FILE"
```

`LAST_VARIATIONS` contains all 5 variations (V-01 through V-05), not the winner. The QU5 prompt says "Winning variation (the complete golden prompt)" but then receives all variations. Claude must infer which is the winner from context — it has the scorecard rank in memory only if the final round scorecard was included, which it is not here.

---

**F-03: `set -e` combined with `|| echo "..."` suppression creates silent failures**

```bash
SKILL_DESC=$(python -c "..." 2>/dev/null || echo "Skill not found in registry")
TOP_SCORE=$(python -c "..." 2>/dev/null || echo "0")
```

`set -e` is active, but every python invocation suppresses both stderr and failure via `2>/dev/null || echo fallback`. A misconfigured Python (missing `yaml`, wrong path) will silently produce `"Skill not found in registry"` or `"0"` and the loop will proceed to generate artifacts based on empty/wrong inputs. The rubric will be generated against `"Skill not found in registry"` as its skill description with no error surfaced.

---

**F-04: Temp file in QU5 uses `mktemp`, rest of script uses hardcoded `/tmp/claude-ask-$$.txt`**

```bash
# in claude_ask():
tmp_prompt="/tmp/claude-ask-$$.txt"

# in QU5:
QU5_PROMPT_FILE=$(mktemp /tmp/qu5-prompt-XXXXXX.txt)
```

The comment in `claude_ask` explicitly says `mktemp may return Windows paths in relay context` as the reason for not using it — but QU5 uses `mktemp` anyway. This inconsistency either means the comment is wrong (mktemp works fine) or QU5 has a latent bug on Windows.

---

**F-05: `yaml.dump` will reorder keys despite `sort_keys=False`**

```bash
yaml.dump(data, f, default_flow_style=False, allow_unicode=True, sort_keys=False)
```

`sort_keys=False` preserves insertion order for dicts loaded from YAML, but `yaml.safe_load` returns plain Python dicts. In Python 3.7+ dicts maintain insertion order, so this is fine — but it will still reformat multi-line strings, flow scalars, and comments. Any comments in `signal.skills.yaml` will be silently dropped on write. If the file has inline comments (common in YAML skill registries), they will be lost after the first successful `quest-run-one` run.
