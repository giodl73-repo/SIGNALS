## Findings

### Bug: `set -e` breaks plateau logic silently

**Evidence** (line 9, 222):
```bash
set -e
...
if [ "$NEW_PATTERNS" -gt "0" ]; then
```
`[ "$NEW_PATTERNS" -gt "0" ]` evaluates to false (exit 1) when `NEW_PATTERNS=0`, which under `set -e` would terminate the script instead of falling through. The plateau reset on line 220 never executes for the non-evolving path. Fix: use `[ "$NEW_PATTERNS" -gt "0" ] && ...` or wrap the if/else in `|| true`.

### Bug: `$ROUND` is undefined after `break`

**Evidence** (lines 217, 327, 344, 373, 384):
```bash
break
...
rounds: $ROUND
...
echo "Score: $LAST_SCORE, Rounds: $ROUND
```
`$ROUND` is a loop variable. When the loop completes naturally (hits `$MAX_ROUNDS` without `break`), `$ROUND` still holds the last value — but this is fragile because it's set by `seq` implicitly. More critically, `$ROUND` is only available inside the loop scope for `for` in bash, so outside the loop it *does* persist, but reading it as "rounds completed" vs "rounds run" is ambiguous. If you break at round 8, `$ROUND` is 8. If the loop exhausts, `$ROUND` is 20, which misleads the golden frontmatter. Fix: capture `FINAL_ROUND=$ROUND` just before `break` and at the natural exit, and use `FINAL_ROUND` in output.

### Bug: Python inline script uses shell variable expansion inside a string — command injection risk

**Evidence** (lines 54-62):
```python
python -c "
...
    if s['id'] == '$SKILL_ID':
```
`$SKILL_ID` is shell-expanded inside the Python string before Python sees it. If `SKILL_ID` contains a single quote or backslash (e.g., `foo'bar`), the Python syntax breaks. Fix: pass `SKILL_ID` as an argument: `python -c "..." "$SKILL_ID"` and use `sys.argv[1]` inside, as is already done in the QU5 block (line 253-256). The QU5 block uses `sys.argv` correctly; the Step 2 and Step 5 blocks don't.

### Bug: `SPEC_FILE` is defined but never used

**Evidence** (line 16):

```bash
SPEC_FILE="$REPO/simulations/draft/specs/signal-scout-$TODAY.md"
```
The path is hardcoded to `signal-scout-` regardless of `$SKILL_ID`, and `$SPEC_FILE` is never referenced again in the script. The comment in line 5 says inputs include a spec, but it's wired as dead code.

### Bug: rubric `head -80` truncation may cut mid-criterion, producing malformed context

**Evidence** (lines 122, 147, 228, 332):
```bash
$(cat "$RUBRIC_FILE" | head -80)
```
A rubric with 6 criteria could easily exceed 80 lines. Truncating at exactly 80 lines mid-criterion sends a malformed partial rubric to Claude. The scoring prompt then works from an incomplete rubric, which undermines criterion coverage. Fix: either remove the `head` limit (rubrics are short) or use a character limit like `${RUBRIC_CONTENT:0:4000}` for consistency with how trace content is handled.

### Finding: QU5 reads `$LAST_VARIATIONS` but that's the full variations file, not the winning variation

**Evidence** (lines 251, 265):
```bash
VARS_CONTENT=$(cat "$LAST_VARIATIONS" | head -100)
...
{variations[:3000]}
```
The QU5 pass asks Claude to simplify "the winning variation" but actually sends the entire variations file (V-01 through V-05). Claude must infer which is the winner from the scorecard. The scorecard is not passed to QU5. Fix: add `$(cat "$LAST_SCORECARD" | head -60)` to the QU5 prompt so Claude can identify the winner, or extract and pass only the top-ranked variation.

### Finding: golden prompt passes `$ROUND` from shell but `ROUND` may be post-loop stale in string interpolation

**Evidence** (lines 327, 344):
```bash
$(cat "$LAST_SCORECARD" | head -100)
...
rounds: $ROUND
```
These are inside a bash here-string-style variable, expanded at assignment time. This is fine as long as `$ROUND` still holds its loop value. This is safe in bash but worth noting that the variable is being used outside the construct where it's semantically meaningful.

### Finding: parallel `mkdir -p` calls throughout — minor but redundant

**Evidence** (lines 69, 114, 142, 316):
```bash
mkdir -p "$QUEST_DIR/rubrics"
mkdir -p "$QUEST_DIR/variations"
mkdir -p "$QUEST_DIR/scorecards"
mkdir -p "$QUEST_DIR/golden"
```
These are called inside the loop (lines 114, 142) on every round. `mkdir -p` is idempotent so it's not harmful, but moving the four `mkdir -p` calls to a single setup block before the loop avoids ~40 redundant syscalls for a 20-round run.
