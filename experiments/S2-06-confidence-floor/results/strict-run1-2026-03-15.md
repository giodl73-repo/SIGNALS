## Review: `tools/quest-run-one.sh`

---

### F-01 — Bug: `SPEC_FILE` hardcodes `signal-scout`

**Evidence:**
```bash
SPEC_FILE="$REPO/simulations/draft/specs/signal-scout-$TODAY.md"
```

`signal-scout` is a literal string. For any skill other than a scout variant (e.g., `listen-adoption`, `flow-throttle`), this resolves to the wrong file. Should be `signal-$SKILL_ID-$TODAY.md`.

---

### F-02 — Bug: `SPEC_FILE` is declared but never used

**Evidence:** `SPEC_FILE` is set on line 19 and never referenced again in the script. If it was meant to be passed to the rubric prompt (alongside trace and skill description), it is silently absent. Dead code at best, a missing input at worst.

---

### F-03 — Bug: Silent empty `SKILL_DESC` when skill is absent

**Evidence:**
```bash
SKILL_DESC=$(python -c "
...
for s in data.get('skills', []):
    if s['id'] == '$SKILL_ID':
        print(s.get('description', 'No description found'))
        break
" 2>/dev/null || echo "Skill not found in registry")
```

The `|| echo "Skill not found in registry"` guard only fires on a non-zero Python exit. If the skill ID is not in the YAML, Python exits 0 with no output, leaving `SKILL_DESC` empty. The rubric is then generated with a blank skill description — no warning, no abort.

---

### F-04 — Issue: `TRACE_CONTENT` truncated arbitrarily at 3000 chars

**Evidence:**
```bash
${TRACE_CONTENT:0:3000}
```

The trace is described in the comment as "ground truth," yet it is silently truncated mid-content to 3000 characters. This can cut off critical expected-output detail. The prompt is already written to a temp file via `claude_ask`, so the ARG_MAX concern that motivated this approach does not apply here — the full content can be included.

---

### F-05 — Issue: Shell injection in inline Python

**Evidence:**
```bash
python -c "
...
    if s['id'] == '$SKILL_ID':
```

`$SKILL_ID` is interpolated directly into the Python source string. A value containing `'` (single quote) would break parsing; one containing `\n` or `$(...)` could produce unexpected behavior. While skill IDs are currently kebab-case, this is a fragile pattern. Safer: pass `SKILL_ID` as an environment variable and read it inside Python with `os.environ`.

---

### F-06 — Minor: `ls` used for file discovery

**Evidence:**
```bash
TRACE_FILE=$(ls "$TRACE_DIR"/plugin-${SKILL_ID}-*.md 2>/dev/null | head -1 || true)
```

`ls` output is not safe to parse when filenames contain spaces or special characters. A glob expansion is more robust:
```bash
files=("$TRACE_DIR"/plugin-${SKILL_ID}-*.md)
TRACE_FILE=$([[ -e "${files[0]}" ]] && echo "${files[0]}" || true)
```

---

### F-07 — Minor: `MAX_ROUNDS` declared, never referenced

**Evidence:**
```bash
MAX_ROUNDS=20  # experimental -- let the rubric discover
```

The variable is declared but not referenced in the visible script. If the loop-control logic is in a later section not shown, this is fine — but if the script is complete as-is, it is dead code.
