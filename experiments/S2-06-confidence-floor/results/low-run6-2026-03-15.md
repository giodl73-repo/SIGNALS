## Findings

### F-01 · SPEC_FILE defined but never used — hardcoded to "scout"
`SPEC_FILE` is assigned on line 16 but referenced nowhere in the script. It is also hardcoded to `signal-scout-$TODAY.md` regardless of `$SKILL_ID`, so it would be wrong for every skill except scout. Dead variable with a latent naming bug.

---

### F-02 · VARS_CONTENT and RUBRIC_CONTENT are dead assignments
Lines 251–252 assign `VARS_CONTENT` and `RUBRIC_CONTENT` via `cat | head`, but the QU5 Python script ignores both — it reads the files directly via `sys.argv[3]` and `sys.argv[4]`. These two lines do nothing and add confusing noise.

---

### F-03 · Shell injection via SKILL_ID in inline Python
Three Python `-c` blocks interpolate `$SKILL_ID` directly into Python source code (Steps 2, 5, and the YAML update). A `SKILL_ID` containing a quote or backslash would produce malformed Python. A malicious value like `'; import os; os.system("rm -rf /"); x='` would execute arbitrary commands. Input should be passed as a `sys.argv` argument, not interpolated into source.

---

### F-04 · Regex `{[^}]+}` breaks if new_patterns strings contain `}`
The scorecard JSON parser uses `re.search(r'```json\s*({[^}]+})\s*```', ...)` in three places. The pattern stops at the first `}` it finds, so any pattern string containing a closing brace — `"use {topic} templating"` — would truncate the match and cause `json.loads` to fail silently, returning 0 for score and false for convergence. A closed-brace-aware approach (e.g. `json` module's decoder or `re.DOTALL` with balanced-brace logic) is needed.

---

### F-05 · yaml.dump strips all comments from signal.skills.yaml
Step 5 round-trips the YAML through PyYAML's `safe_load` + `dump`. PyYAML does not preserve comments. Every comment in `signal.skills.yaml` is destroyed permanently on the first successful run. Use `ruamel.yaml` for comment-preserving round-trips, or write only the changed keys via targeted regex/line replacement.

---

### F-06 · Rubric silently truncated to 80 lines in multiple prompts
The rubric is truncated to 80 lines in the score prompt (Step B), the evolve prompt, and the golden prompt. A rubric with 8–10 criteria plus explanation text easily exceeds 80 lines. Criteria at the bottom are silently dropped — the model scores against an incomplete rubric without knowing it. The truncation limit should be much higher, or the rubric should be structured so the criterion list always fits first.

---

### F-07 · Variations truncated to 200 lines in the golden prompt
The golden step reads `head -150` of the final variations file, which contains 5 complete prompt bodies. At ~30–60 lines per variation, the last 2–3 variations may be partially or fully cut. The model is asked to "copy it verbatim" but may not have seen the full winning variation. The golden prompt should receive the full winning variation, not a truncated dump of all five.

---

### F-08 · Useless use of cat (UUOC) throughout
`cat "$FILE" | head -N` appears six times (lines 122, 147, 228, 252, 324, 333). The correct form is `head -N "$FILE"`. No functional bug, but eliminates a needless subprocess on each call.

---

### F-09 · Scorecard JSON parsed three separate times
`TOP_SCORE`, `ALL_ESSENTIAL`, and `NEW_PATTERNS` each invoke a separate Python subprocess to read and parse the same scorecard file. All three values should be extracted in a single parse and printed as space-separated tokens, then read with `read TOP_SCORE ALL_ESSENTIAL NEW_PATTERNS`.

---

### F-10 · ls used for file discovery — fragile with special characters
Line 42 uses `ls "$TRACE_DIR"/plugin-${SKILL_ID}-*.md 2>/dev/null | head -1` to find the trace file. `ls` is unreliable for programmatic use when filenames contain spaces or newlines. Bash glob expansion is safer: `TRACE_FILE=$(echo "$TRACE_DIR"/plugin-${SKILL_ID}-*.md | head -c -1)` or a `for f in ...; do ...; break; done` pattern.

---

### F-11 · No recovery if claude fails mid-run — set -e will abort
`set -e` causes the script to abort on any non-zero exit. If `claude --print` fails once (network error, rate limit, context overflow) during a 20-round loop, the entire run aborts with no partial results committed. A retry wrapper around `claude_ask` (e.g., 3 attempts with a brief pause) would make long runs much more resilient without complicating the design.

---

### F-12 · QU5 mktemp path not guarded against Python failure
If the `python -c` block at lines 253–284 fails (syntax error, missing file), `$QU5_PROMPT_FILE` exists but is empty or absent. `claude --print < "$QU5_PROMPT_FILE"` then either reads nothing or errors. The fallback `||` correctly catches the `claude` failure, but Python's failure is silent and produces a misleading empty-prompt call first. Python's exit code should be checked before invoking `claude`.

---

### F-13 · Inconsistent and undocumented truncation limits
The script uses four different truncation limits with no stated rationale: 80 lines (rubric in score/evolve/golden), 100 lines (scorecard in evolve/golden), 150 lines (variations in golden), 200 lines (variations in score). These arbitrary limits affect model behavior differently in each step and will be invisible sources of quality variance.
