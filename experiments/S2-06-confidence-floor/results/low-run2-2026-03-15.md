Here are the findings, organized by severity:

## Bugs

**B-01: `ROUND` variable out of scope at golden prompt step**
The golden prompt (Step 4) and the YAML update (Step 5) reference `$ROUND` after the `for` loop ends. In bash, `for` loop variables persist after the loop, but only if the loop executed at least one iteration. If `MAX_ROUNDS=0` or the loop body is never entered, `$ROUND` is unset, producing empty values in the frontmatter and the `quest_status` string.

**B-02: `set -e` causes silent loop abandonment on `claude_ask` failure**
`set -e` is active. If `claude --print` returns a non-zero exit code for any reason (rate limit, network issue, malformed input), the script exits immediately mid-loop — including mid-round after variations are written but before scoring. The partial state is left on disk with no diagnostic message. The intent seems to be fault-tolerant artifact writing; `set -e` contradicts that.

**B-03: Regex for JSON parsing is fragile**
The regex `r'\`\`\`json\s*({[^}]+})\s*\`\`\`'` uses `[^}]+` which fails if any string value inside the JSON contains a `}` character. It also fails on multi-line JSON (the `re.DOTALL` flag is applied to the outer match but `[^}]` already excludes `\n` by default without DOTALL). A better approach: extract everything between the fences, then `json.loads()` on the full block.

**B-04: `SIMPLIFIED_PASS` contains spaces, breaking shell test**
`print(f'PASS ({pct}% reduction)')` produces a string with spaces and parentheses. This is later compared as `"$SIMPLIFIED_PASS"` in the golden prompt prompt string — it's interpolated into a prompt body, so it's not a comparison bug, but if it were used in a `[ ]` test it would break. Not critical now, but fragile as the script grows.

**B-05: `SKILL_DESC` silently swallows YAML parse errors**
The `2>/dev/null || echo "Skill not found in registry"` suppresses all Python errors including YAML syntax errors, import errors (if `pyyaml` isn't installed), and file-not-found. The fallback value `"Skill not found in registry"` gets embedded in the rubric prompt and produces a nonsense rubric with no diagnostic.

## Logic Findings

**L-01: Plateau convergence resets on rubric evolution, but essential-pass is not re-checked**
When `NEW_PATTERNS > 0`, `PLATEAU_COUNT` resets to 0 (correct). But the rubric just added aspirational criteria — it's possible the new rubric now has stricter essential criteria that the existing variations don't pass. The next round will catch this, but there's no indication to the user that the rubric changed and previous scores are stale.

**L-02: `head -80` on rubric truncates it for the scoring step**
The rubric is truncated to 80 lines in both the VARIATE and SCORE prompts. A rubric with 5 essential + 3 recommended + 2 aspirational criteria plus preamble and weight table will often exceed 80 lines. Claude receives an incomplete rubric and scores against phantom criteria.

**L-03: `LAST_VARIATIONS` holds the last round's file, not the best-scoring round's file**
After convergence or max rounds, `LAST_VARIATIONS` is the most recent round, not necessarily the round that produced the highest score. If score regresses in the final round (which can happen after rubric evolution), the golden prompt is built from a lower-scoring variation.

**L-04: `head -100` on variations passed to scorer**
Same issue as L-02. 5 complete skill body variations at 100 lines each would be 500 lines minimum. The scorer receives an incomplete view and cannot score V-03 through V-05 at all.

**L-05: QU5 receives `head -100` of variations, same truncation**
`VARS_CONTENT=$(cat "$LAST_VARIATIONS" | head -100)` — this variable is computed but then not actually used; the file path `sys.argv[3]` is passed to Python which opens it with `[:3000]` slice. The `VARS_CONTENT` variable is dead code.

**L-06: `SPEC_FILE` is set but never read**
`SPEC_FILE="$REPO/simulations/draft/specs/signal-scout-$TODAY.md"` — this variable is defined at the top but never used anywhere in the script. The path also hardcodes `signal-scout-` regardless of `SKILL_ID`, so it would be wrong for non-scout skills anyway.

## Security / Robustness

**S-01: `SKILL_ID` is interpolated into Python heredocs without sanitization**
`if s['id'] == '$SKILL_ID':` — if `SKILL_ID` contains a single quote, it breaks out of the Python string literal. A value like `scout'; import os; os.system("rm -rf /")  #` would execute arbitrary Python. The variable comes from a command-line argument, making this a command injection vector in relay contexts.

**S-02: `/tmp/claude-ask-$$.txt` uses PID, not a cryptographically random name**
PID-based temp files are susceptible to TOCTOU race conditions and PID prediction attacks. `mktemp` is already used correctly in the QU5 step — the same pattern should apply in `claude_ask`.

---

## Style / Minor

**M-01: `RUBRIC_PROMPT` embeds YAML frontmatter with literal `---` delimiters**
The frontmatter block in `RUBRIC_PROMPT` uses `---` which, when the prompt is stored in a bash variable and re-used, could be misread as YAML document separators if any tooling later processes the file as YAML.

**M-02: `cat "$FILE" | head -N` is a useless use of cat**
`head -N "$FILE"` is the idiomatic form. Appears throughout the script.

**M-03: `RUBRIC_FILE` in Step 4 YAML update uses shell `${##*/}` path manipulation**
`Path('$RUBRIC_FILE').name` does the same thing in Python (which is already running). The mix of shell path manipulation and Python pathlib in adjacent lines is inconsistent.
