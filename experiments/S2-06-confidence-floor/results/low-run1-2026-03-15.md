Here are the findings, grouped by severity:

---

## Bugs

**B-01: `$ROUND` used outside loop scope (Step 4 golden prompt)**
After the loop, `$ROUND` holds the last value from `seq`, which is `$MAX_ROUNDS` even if dual convergence happened at round 3. The golden frontmatter will say `rounds: 20` instead of the actual convergence round. Fix: capture the break round in a separate variable (`FINAL_ROUND`).

**B-02: `$ROUND` also used in Step 5 YAML update**
Same variable leaked from loop. `quest_status: 'GOLDEN (Round 20, score ...)'` will be wrong whenever early convergence occurs.

**B-03: `LAST_VARIATIONS` is unset if loop never executes**
`MAX_ROUNDS=20` means this can't happen normally, but if someone sets `MAX_ROUNDS=0`, QU5 and the golden step will fail on `cat "$LAST_VARIATIONS"` with an empty path. No guard exists.

**B-04: Python inline script in Step 5 uses shell-interpolated single-quoted strings**
```python
skill['golden'] = 'simulations/quest/golden/${SKILL_ID}-golden-$TODAY.md'
```
This is inside a `python -c "..."` double-quoted block, so `$SKILL_ID` and `$TODAY` will expand, but the single quotes around the value are Python string delimiters — if either variable contains a single quote, the Python syntax breaks. Low risk with current inputs, but it's a fragility.

**B-05: Regex in `parse result` snippets doesn't handle multi-line JSON**
The regex `r'\`\`\`json\s*({[^}]+})\s*\`\`\`'` uses `[^}]+` which stops at the first `}`. If the LLM returns a JSON block where `new_patterns` contains objects (instead of strings), or if Claude adds any trailing fields after the array closes, the parse silently returns 0. The prompt constrains the shape but the parser is fragile against any deviation.

**B-06: `set -e` + `|| echo "..."` interaction in Step 2**
The Python snippet uses `2>/dev/null || echo "Skill not found in registry"`. Under `set -e`, the `||` suppresses the exit, which is intentional. However, if `yaml` is not importable (missing module), the error is silently swallowed and `$SKILL_DESC` gets the fallback string. There's no warning that the YAML load failed for a reason other than "skill not found."

## Logic Issues

**L-01: Plateau convergence requires `all_essential_pass AND no new patterns` simultaneously**
A variation could score 95 with all essentials passing, but if Claude finds one new pattern, plateau resets to 0. This means a genuinely golden skill can spin through many extra rounds. The two conditions should be treated somewhat independently: plateau on score stability, not just pattern absence.

**L-02: Rubric evolves only when new patterns found, but score plateau check is independent**
If `ALL_ESSENTIAL=false` but `NEW_PATTERNS=0`, plateau resets and no rubric evolution happens. The loop will keep generating variations with the same rubric that's already not producing essential passes — no mechanism to force a rubric re-evaluation on persistent essential failure.

**L-03: `head -80` / `head -100` truncation of rubric/variations in prompts**
The rubric is passed to the variate and score prompts with `head -80`. A rubric with 7 criteria and verbose pass conditions can easily exceed 80 lines. Truncated rubric → LLM scores against partial criteria → flawed scores → wrong convergence signal. Should use `wc -l` or a character budget, not a fixed line count.

**L-04: `LAST_SCORE` is a string from Python stdout, compared with `-ge` never called**
`LAST_SCORE` is used in the golden frontmatter but never actually checked against the golden threshold (composite >= 80, all essential). The loop can break at plateau even if `LAST_SCORE` is 55. There's no guard preventing a below-threshold golden from being written.

## Design Issues

**D-01: `SPEC_FILE` is set but never used**
Line 16 sets `SPEC_FILE` pointing to `signal-scout-$TODAY.md` (hardcoded "scout", not `$SKILL_ID`). It is never read or passed to any prompt. Either remove it or fix the path and use it in the rubric generation prompt.

**D-02: QU5 passes file paths to Python but also pre-reads them into variables that go unused**
`VARS_CONTENT` and `RUBRIC_CONTENT` are set via `cat ... | head -N` but never used — the Python script opens the files directly via `sys.argv`. The pre-reads are dead code.

**D-03: Inconsistent truncation strategy — inline heredoc vs `head` vs slice**
Rubric prompt uses `${TRACE_CONTENT:0:3000}` (character slice). Variate/score prompts use `cat file | head -N` (line slice). QU5 prompt uses `variations[:3000]` (character slice inside Python f-string). Three different truncation strategies make the effective context window unpredictable.

**D-04: `signal.skills.yaml` is rewritten with `yaml.dump` which loses comments and key order**
`yaml.dump` with `sort_keys=False` preserves insertion order but destroys all YAML comments. If `signal.skills.yaml` has inline comments documenting skills, one quest run wipes them. Consider a line-oriented patch (regex replace) or `ruamel.yaml` for round-trip preservation.

**D-05: No deduplication guard on rubric file names**
If the script is run twice on the same day for the same skill, the initial rubric (`${SKILL_ID}-rubric-$TODAY.md`) is silently overwritten. A re-run mid-day clobbers the prior rubric, losing version history for that day.

## Minor / Style

**M-01: `cat file | head -N` — useless use of cat**
`head -N file` is equivalent and avoids the extra pipe. Appears multiple times in the loop.

**M-02: Three separate Python invocations to parse the same JSON block**
`TOP_SCORE`, `ALL_ESSENTIAL`, and `NEW_PATTERNS` each spawn a Python process that opens and parses the same scorecard file. One invocation could emit all three values.

**M-03: No timeout on `claude --print` calls**
A hung Claude invocation will block the loop indefinitely. With `MAX_ROUNDS=20` and 2 calls per round plus rubric evolution calls, a single hung call kills the entire run silently.

**M-04: Temp file uses PID (`$$`) but not round number**
If two instances of the script run concurrently for different skills, their temp files won't collide (PID differs). But the comment says "P-01 concurrent by default" — under concurrent execution the `/tmp/claude-ask-$$.txt` pattern is safe, but `$QU5_PROMPT_FILE` from `mktemp` follows a different naming convention. Minor inconsistency.
