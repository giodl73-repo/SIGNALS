## F-01 — `SPEC_FILE` declared but never used, with hardcoded skill name

**Line 16:**
```bash
SPEC_FILE="$REPO/simulations/draft/specs/signal-scout-$TODAY.md"
```

`SPEC_FILE` is never referenced again in the script. It is dead code. Compounding the problem, the path hardcodes `signal-scout-` regardless of `$SKILL_ID`, so it would produce the wrong path even if it were used.

## F-02 — Temp file leaks when `claude --print` fails under `set -e`

**Line 9:** `set -e`

**Lines 33–35:**
```bash
claude --print < "$tmp_prompt" > "$out_file"
local rc=$?
rm -f "$tmp_prompt"
```

With `set -e` active, a non-zero exit from `claude --print` causes bash to exit immediately at line 33. The `rm -f "$tmp_prompt"` on line 35 is never reached, leaving `/tmp/claude-ask-$$.txt` behind. The `local rc=$?` pattern correctly captures the code for the success path, but gives a false sense of protection against leaks.

**Lines 251–257:**

## F-04 — Trace content truncated to two different limits with no comment

**Line 77 (rubric generation):** `${TRACE_CONTENT:0:3000}`

**Line 153 (scoring loop):** `${TRACE_CONTENT:0:800}`

The scoring step gives Claude 73% less trace context than the rubric step, with no comment explaining why. For a file described as "hand-compiled expected output — ground truth," silently cutting to 800 characters risks incomplete scoring.

## F-05 — `head -80` on rubric files may cut criteria mid-document

**Lines 122 and 147:**
```bash
$(cat "$RUBRIC_FILE" | head -80)
```

A rubric with 6–10 criteria plus frontmatter and weights can easily exceed 80 lines. Cutting at an arbitrary line count may deliver a partial last criterion to the prompt with no truncation marker, producing ambiguous inputs to the scoring and evolve steps.

---

## F-06 — `$SKILL_ID` interpolated directly into Python inline scripts

**Lines 58–59:**
```python
for s in data.get('skills', []):
    if s['id'] == '$SKILL_ID':
```

`$SKILL_ID` is bash-interpolated inside a double-quoted Python `-c` string. A skill ID containing a single quote (e.g., `o'clock-test`) would produce a Python syntax error. More broadly, this pattern is a shell-injection surface for any caller that doesn't control `$1`.

The same pattern appears in Step 5 (line 370) and in the `quest_status` string (line 373).

## F-07 — `yaml.dump` overwrites the original YAML formatting

**Lines 376–377:**
```python
with open(yaml_file, 'w') as f:
    yaml.dump(data, f, default_flow_style=False, allow_unicode=True, sort_keys=False)
```

`yaml.dump` normalizes the file: it removes inline notation, trailing comments, blank-line separators, and multi-line string styles. Any hand-formatted structure in `signal.skills.yaml` is silently rewritten on every quest run.
