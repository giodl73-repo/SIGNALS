Here are the findings:

---

**F-01 — `set -e` + `|| true` creates silent failures**

`set -e` is active, but `|| true` on the `ls` pipe in Step 1 masks any unexpected failure. More critically, `set -e` means any unguarded command failure (e.g., `claude --print` returning non-zero) will silently abort the entire run mid-loop with no cleanup. The golden is never written, but the script exits 0 or non-zero without explaining what round it died on.

**F-02 — `ROUND` leaks out of loop with wrong value on max-rounds exit**

`ROUND` is set by `seq`/`for` and used after the loop in the golden and summary (`rounds: $ROUND`). If the loop exhausts all 20 rounds without converging, `$ROUND` equals 20, which is correct. But if the loop hits the `break` early, `$ROUND` holds the convergence round — correct. The *silent bug* is: if `claude_ask` fails mid-round and `set -e` fires, `$ROUND` may hold the partial round number, giving misleading metadata in the golden frontmatter.

**F-03 — Rubric truncated at 80 lines throughout the loop**

`cat "$RUBRIC_FILE" | head -80` is used in every subsequent prompt (variate, score, evolve). For a rubric that evolves to v5+ with 6-8 criteria each having detailed pass conditions, 80 lines is not a safe upper bound. A truncated rubric passed to the scoring step means Claude may not see all criteria, causing phantom pass/fail readings.

**F-04 — Scorecard parsed three separate times with identical regex**

`TOP_SCORE`, `ALL_ESSENTIAL`, and `NEW_PATTERNS` each open and re-parse the scorecard file with the same regex. One Python invocation extracting all three fields would be cleaner and avoid the triple file-read + three process forks per round (60 extra processes over 20 rounds).

**F-05 — `SPEC_FILE` is assigned but never used**

`SPEC_FILE` is set on line 16 to `signal-scout-$TODAY.md` — hardcoded to "scout" regardless of `$SKILL_ID` — and never referenced anywhere in the script. Either it was intended for the rubric or variate prompts and was dropped, or it should be removed.

**F-06 — Python inline scripts inherit shell variables via string interpolation, not arguments**

`$REPO`, `$SKILL_ID`, `$TODAY`, `$ROUND`, `$LAST_SCORE`, `$RUBRIC_VERSION` are all string-interpolated directly into `-c` Python strings. A skill ID or description containing a single quote (`'`) will break the Python syntax entirely and produce a silent `2>/dev/null` suppression. The QU5 step already uses `sys.argv` properly — that pattern should apply to all Python blocks.

**F-07 — `head -80 / head -100 / head -200` truncate without signal**

Claude receives truncated content with no indication that truncation occurred. If the rubric or variations file is large, the model may score against an incomplete rubric or generate golden output that doesn't account for later criteria. A comment noting the truncation threshold — or a line count check — would make this explicit.

**F-08 — Plateau counter resets on *any* deviation, not just backward movement**

If round N has `all_essential=true, new_patterns=0` (plateau +1), then round N+1 produces one new pattern from a marginal variation, PLATEAU_COUNT resets to 0. This means a skill that is nearly converged but keeps generating one minor pattern each round will loop to MAX_ROUNDS. The reset condition should distinguish "new structural patterns" from "minor wording variations" — or use a rolling window rather than strict reset.

**F-09 — `$LAST_VARIATIONS` and `$LAST_SCORECARD` unset on loop skip**

If the loop body exits on round 1 before writing the files (e.g., `claude_ask` returns non-zero and `set -e` fires before the assignment), the golden step reads `$LAST_VARIATIONS` and `$LAST_SCORECARD` as empty strings. `cat ""` will attempt to read stdin and hang. No guard exists for the empty-variable case.

**F-10 — QU5 falls back to copying `$LAST_VARIATIONS` (all 5 variations), not the winner**

On QU5 failure, `cp "$LAST_VARIATIONS" "$SIMPLIFIED_FILE"` copies the entire variations file (all V-01 through V-05). The golden step then does `cat "$SIMPLIFIED_FILE" | head -80` and asks Claude to extract the winning variation from a mixed-variation document. This is underdefined — there's no marker telling the golden step which variation won.

**F-11 — `signal.skills.yaml` overwritten with `yaml.dump` destroying comments and field order**

`yaml.safe_load` + `yaml.dump(..., sort_keys=False)` will drop all YAML comments (e.g., inline docs, section separators) and may reformat multi-line strings. `sort_keys=False` preserves key order per-mapping but doesn't preserve file-level structure. A targeted line-replacement approach (sed or ruamel.yaml) would be safer for a hand-maintained config file.

**F-12 — No lock / re-entrancy guard**

Two concurrent invocations for the same `$SKILL_ID` on the same `$TODAY` will write to the same `RUBRIC_FILE`, `VARIATIONS_FILE`, and `SCORECARD_FILE` paths. The comment says "P-01 concurrent by default" but that refers to running different skills concurrently — within a single skill there's no guard. A `.lock` file or per-run UUID suffix would prevent collision.
