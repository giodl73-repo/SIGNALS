## Findings

### BUG-01: `SPEC_FILE` hardcodes `scout` regardless of `$SKILL_ID`

**Line 16:**
```bash
SPEC_FILE="$REPO/simulations/draft/specs/signal-scout-$TODAY.md"
```
The namespace segment is hardcoded as `scout`. Running with `draft-proposal` or `flow-lifecycle` will point at the wrong spec. The path should derive from `$SKILL_ID` (e.g., `signal-${SKILL_ID}-$TODAY.md`).

---

### BUG-02: `SPEC_FILE` is defined but never used

`SPEC_FILE` is set at line 16 and never referenced again in the script. The rubric prompt (line 71) passes `$SKILL_DESC` and `$TRACE_CONTENT` but not spec content. Either the spec should be loaded and injected into the rubric prompt, or the variable should be removed.

---

### BUG-03: `set -e` makes `claude_ask` error handling dead code

**Lines 9, 33–36:**
```bash
set -e
...
claude --print < "$tmp_prompt" > "$out_file"
local rc=$?
rm -f "$tmp_prompt"
return $rc
```
With `set -e` active, if `claude --print` exits non-zero, bash exits the script before `local rc=$?` executes. `rm -f "$tmp_prompt"` and `return $rc` are unreachable on failure. The temp file leaks and the intended error propagation via `return $rc` never fires.

---

### BUG-04: Temp file leaks on failure — no EXIT trap

Related to BUG-03. No `trap` is registered for cleanup:
```bash
tmp_prompt="/tmp/claude-ask-$$.txt"
```
A failed `claude --print` call or SIGINT leaves `/tmp/claude-ask-$$.txt` on disk. A `trap 'rm -f "$tmp_prompt"' EXIT` inside the function (or a global cleanup trap) would fix this.

---

### BUG-05: Shell injection via `$SKILL_ID` in Python inline code

**Lines 56, 59:**
```python
with open('$REPO/signal.skills.yaml') as f:
...
    if s['id'] == '$SKILL_ID':
```
`$SKILL_ID` and `$REPO` are interpolated directly into the Python source string. A skill ID containing a single quote (e.g., `foo'bar`) would produce a Python syntax error or inject arbitrary Python. This same pattern recurs in the Step 5 Python block (lines 365–373). Use `sys.argv` args or env variables instead — the QU5 block at line 253 already demonstrates the correct pattern.

### BUG-06: Three separate Python invocations parse the same JSON block

**Lines 172–204:**
```python
# First invocation:
txt = open('$SCORECARD_FILE').read()
m = re.search(...)
print(d.get('top_score', 0))
# Second:
...print('true' if d.get('all_essential_pass', False) else 'false')
# Third:
...print(len(patterns))
```
The same file is opened and the same regex is applied three times in a row. A single Python invocation returning all three values (or writing a temp JSON) would be cleaner and avoids triple parse failures producing mismatched `TOP_SCORE=0 / ALL_ESSENTIAL=false / NEW_PATTERNS=0` even when the JSON is valid.

### FINDING-01: Hard `head` limits silently truncate prompts with no model-side indication

Multiple locations:
```bash
$(cat "$RUBRIC_FILE" | head -80)      # line 122, 147, 228, 333
$(cat "$SCORECARD_FILE" | head -100)  # line 149, 231
${TRACE_CONTENT:0:3000}               # line 77
${TRACE_CONTENT:0:800}                # line 153
```
The model receives truncated content with no "...truncated..." marker. A rubric cut at line 80 mid-criterion silently omits criteria from scoring. Consider appending `[TRUNCATED]` or increasing limits, and using `tail -n +1` with a higher line count instead of a fixed 80/100.

### FINDING-02: `ls` piped to shell variable is fragile for filenames with spaces

**Line 42:**
```bash
TRACE_FILE=$(ls "$TRACE_DIR"/plugin-${SKILL_ID}-*.md 2>/dev/null | head -1 || true)
```
Filenames containing spaces will cause the variable to hold a broken path. `glob` expansion via bash is safer: `TRACE_FILE=$(echo "$TRACE_DIR"/plugin-${SKILL_ID}-*.md | awk '{print $1}')` or simply use a bash array with `read -r`.

### FINDING-03: `ROUND` is used after the loop but may reflect the final iteration even on `break`

**Lines 109, 343, 373, 384:**
```bash
for ROUND in $(seq 1 $MAX_ROUNDS); do
  ...
  break   # on dual convergence
done
...
rounds: $ROUND
quest_status: 'GOLDEN (Round $ROUND, score $LAST_SCORE)'
```
When `break` fires, `$ROUND` holds the correct convergence round. But if the loop exhausts all 20 rounds without converging, `$ROUND` is 20 and there is no warning that golden was never achieved — it writes `status: GOLDEN` regardless. A post-loop check (`if [ "$LAST_SCORE" ... ]` or tracking a convergence flag) would catch the max-rounds-without-convergence case.

### FINDING-04: `LAST_VARIATIONS` points to the last variation file, not the winning one

**Lines 208, 251, 325–326:**
```bash
LAST_VARIATIONS="$VARIATIONS_FILE"   # updated every round
...
VARS_CONTENT=$(cat "$LAST_VARIATIONS" | head -100)
...
Winning variation (the complete golden prompt):
{variations[:3000]}
```
The QU5 prompt calls `$LAST_VARIATIONS` the "winning variation" but it is simply the most-recent round's file. The actual best-scoring variation is identified in the scorecard text but never extracted into its own variable. QU5 may simplify the wrong prompt if the winning variation was from an earlier round.
