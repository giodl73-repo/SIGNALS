## Bugs

**B-01 тАФ Shell injection via `SKILL_ID` in Python `-c` strings**
`SKILL_ID` is interpolated directly into Python inline scripts (`if s['id'] == '$SKILL_ID'`). A single quote in the skill ID (e.g., `my-skill'foo`) breaks the Python syntax. The `2>/dev/null` suppressor silently swallows the error, and the fallback `echo "Skill not found"` hides the root cause.

**B-02 тАФ `VARS_CONTENT` and `RUBRIC_CONTENT` are dead variables**
Lines 251тАУ252 do `$(cat ... | head -100/60)` into variables that are never referenced. The QU5 Python script reads those files directly via `sys.argv`. Two subprocess forks for nothing.

**B-03 тАФ Scorecard parsed three times for the same JSON block**
`TOP_SCORE`, `ALL_ESSENTIAL`, and `NEW_PATTERNS` each spawn a separate Python process to open and regex-search the same file for the same JSON block. Should be one parse, three prints.

**B-04 тАФ Variations truncated to 200 lines in score prompt**
`$(cat "$VARIATIONS_FILE" | head -200)` in `SCORE_PROMPT`. Five complete prompt bodies can easily exceed 200 lines. Variations 4 and 5 get silently truncated mid-content, so the scoring LLM never sees them тАФ but the scorecard still claims to evaluate all five.

**B-05 тАФ `mktemp` used in QU5 step despite comment warning against it**
`claude_ask` avoids `mktemp` with an explicit comment: "mktemp may return Windows paths in relay context." The QU5 section ignores this and calls `mktemp /tmp/qu5-prompt-XXXXXX.txt` anyway. Inconsistent; same relay-context problem applies.

## Logic / Behavioral Issues

**L-01 тАФ Convergence doesn't check score >= 80**
The dual-convergence condition (`ALL_ESSENTIAL=true` AND `NEW_PATTERNS=0`) can fire with a score of 45. There is no check against the stated golden threshold. A run that stabilizes at "all essential pass, no new patterns, score=58" will write a golden file with score 58 and mark the skill GOLDEN.

**L-02 тАФ No escape from persistent `ALL_ESSENTIAL=false`**
If essential criteria never pass, `PLATEAU_COUNT` resets every round and the loop always runs to `MAX_ROUNDS=20`. There's no early-exit branch for "stuck at essential failure after N rounds." The rubric will keep evolving (adding aspirational criteria) without fixing the core problem.

**L-03 тАФ `RUBRIC_FILE` truncated to 80 lines in scoring and variate prompts**
`$(cat "$RUBRIC_FILE" | head -80)` is used in both `VARIATE_PROMPT` and `SCORE_PROMPT`. A fully populated rubric (6тАУ10 criteria with pass conditions + scoring formula) easily exceeds 80 lines. The LLM scores against an incomplete rubric; the scoring formula and aspirational criteria may be cut.

**L-04 тАФ YAML dump destroys comments in `signal.skills.yaml`**
`yaml.dump()` does not preserve comments. Every run strips all `#` comments from the file permanently. If `signal.skills.yaml` has inline annotations or section headers, they're gone after the first skill completes.

**L-05 тАФ `LAST_SCORE` is a raw string, not validated as integer**
`LAST_SCORE` is printed to YAML as `'GOLDEN (Round $ROUND, score $LAST_SCORE)'`. If the LLM returns `"85.5"`, `""`, or `"N/A"` as `top_score`, the YAML gets garbage. There's also no integer coercion before the echo report on line 384.

## Dead Code / Misleading

**D-01 тАФ `SPEC_FILE` declared but never used**
`SPEC_FILE="$REPO/simulations/draft/specs/signal-scout-$TODAY.md"` is set on line 16 and never referenced. It also hardcodes `signal-scout-` instead of `$SKILL_ID`, so it would be wrong even if it were used.

**D-02 тАФ Step 5 Python has no error fallback under `set -e`**
The YAML update in Step 5 has no `|| echo` guard. If it fails (file not found, YAML parse error, etc.), `set -e` exits the script without printing the cause тАФ after the golden file is already written, the caller sees a non-zero exit and may retry or discard a valid golden.

**D-03 тАФ Useless uses of `cat` in pipe chains**
`$(cat "$FILE" | head -N)` appears throughout. `$(head -N "$FILE")` is cleaner and avoids the extra subprocess.
