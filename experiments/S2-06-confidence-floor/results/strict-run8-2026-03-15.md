## F-01 â€” SPEC_FILE hardcodes `signal-scout-` prefix (Bug)

```bash
SPEC_FILE="$REPO/simulations/draft/specs/signal-scout-$TODAY.md"
```

The script is parameterized by `$SKILL_ID` (e.g., `flow-conversation`, `trace-state`), but this path always resolves to a scout spec. Any non-scout skill would reference a nonexistent or wrong file. The fix is `signal-${SKILL_ID}-$TODAY.md` or, if the spec is namespace-scoped, parse the namespace out of `$SKILL_ID`.

## F-02 â€” Shell injection in Python inline via `$SKILL_ID` (Bug / Security)

```python
for s in data.get('skills', []):
    if s['id'] == '$SKILL_ID':
```

`$SKILL_ID` is interpolated directly into Python source code inside a double-quoted shell heredoc. A skill ID containing a single quote (e.g., `scout-it's`) would produce a Python syntax error. A more adversarial value could break logic silently. Fix: pass the value as an environment variable or positional arg to Python and read it with `os.environ` / `sys.argv`.

## F-03 â€” Temp file leaks on `claude` failure under `set -e` (Bug)

```bash
set -e
...
claude --print < "$tmp_prompt" > "$out_file"
local rc=$?
rm -f "$tmp_prompt"
return $rc
```

With `set -e` active, a nonzero exit from `claude --print` causes immediate script termination before `rm -f "$tmp_prompt"` runs. The temp file is orphaned. The `local rc=$?` / `return $rc` pattern is dead code in this path â€” it never executes. Fix: use a `trap` to remove the temp file on exit:

```bash
trap 'rm -f "$tmp_prompt"' EXIT
```

## F-04 â€” `ls` used for glob-based file selection (Fragile)

```bash
TRACE_FILE=$(ls "$TRACE_DIR"/plugin-${SKILL_ID}-*.md 2>/dev/null | head -1 || true)
```

`ls` is fragile for files with spaces, newlines, or special characters in names. The `head -1` also silently discards all but the first match with no warning when multiple traces exist. Prefer:

```bash
files=("$TRACE_DIR"/plugin-${SKILL_ID}-*.md)
TRACE_FILE=$([[ -f "${files[0]}" ]] && echo "${files[0]}" || echo "")
```

## F-05 â€” Silent trace truncation with no diagnostic (Quality)

```bash
${TRACE_CONTENT:0:3000}
```

Trace content is silently cut at 3000 characters inside the rubric prompt. A long trace file drops everything beyond that limit with no warning in the output, no indication in the generated rubric, and no log line. At minimum, log a warning:

```bash
[[ ${#TRACE_CONTENT} -gt 3000 ]] && echo "WARN: trace truncated (${#TRACE_CONTENT} -> 3000 chars)"
```

```bash

If later steps (variate, score, golden) consume them, no issue. If not, they are dead code â€” and `SPEC_FILE` carries the scout-hardcoded bug from F-01 regardless of whether it is used.
