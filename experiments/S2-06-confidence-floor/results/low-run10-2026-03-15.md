Here are the findings:

---

**F-01: `set -e` without per-command protection on `claude_ask` calls**
`set -e` is active, but `claude --print` can fail silently or return a non-zero exit code on model errors, empty output, or rate limits. The `rc` is captured but `set -e` means any failure in the middle of the loop kills the entire run — no retry, no fallback, and all prior rounds' artifacts are orphaned with no summary written.

**F-02: `SPEC_FILE` is declared but never used**
`SPEC_FILE="$REPO/simulations/draft/specs/signal-scout-$TODAY.md"` is also hardcoded to `signal-scout-` regardless of `SKILL_ID`. It's dead code that would be wrong if it were used.

**F-03: `$ROUND` is used after the loop but may be unset**
If `MAX_ROUNDS=0` or the loop body never executes, `$ROUND` is empty. This affects the golden prompt frontmatter, the YAML update string, and the final echo. The loop uses `seq` which would produce no output for `0`, leaving `ROUND` as whatever `seq`'s last iteration set it — but with `MAX_ROUNDS=20` this is currently safe. Still fragile.

**F-04: Truncation of rubric/variations to fixed line counts is load-bearing and silent**
`head -80`, `head -100`, `head -150`, `head -200` are scattered throughout prompts. If a rubric or variation file is longer than the cutoff, criteria or variation bodies are silently dropped — the LLM scores against an incomplete rubric without any warning. This is especially dangerous for evolved rubrics that grow with each version.

**F-05: Three separate Python invocations parse the same JSON block**
`TOP_SCORE`, `ALL_ESSENTIAL`, and `NEW_PATTERNS` each open and regex-parse the same scorecard file in separate subshells. This is three process spawns per round that could be one. More importantly, if the file is written between invocations (race condition in relay context) results could be inconsistent.

**F-06: `LAST_VARIATIONS` holds the last round's file, not the winning round's file**
The golden step uses `LAST_VARIATIONS` and `LAST_SCORECARD`, which are the files from the final round — not necessarily the round with the best score. If the loop exits via `MAX_ROUNDS` (not dual convergence), the last round may not be the best round. There's no tracking of `BEST_SCORE_ROUND` or `BEST_VARIATIONS_FILE`.

**F-07: Plateau counter resets on any non-convergence, including new-patterns-only rounds**
The convergence condition requires `all_essential_pass AND new_patterns==0`. If essential passes but one new pattern is found, `PLATEAU_COUNT` resets to 0. This means a skill that perpetually surfaces minor patterns (but stays above 80) can never converge — it will run all 20 rounds regardless.

**F-08: Shell injection risk in the Python inline script (Step 2)**
`SKILL_ID` is interpolated directly into a Python `-c` string: `if s['id'] == '$SKILL_ID'`. If `SKILL_ID` contains a single quote (e.g., `foo'bar`), the Python code breaks with a syntax error. In the relay context this is a correctness risk; in a broader context it's an injection surface.

**F-09: `QU5_PROMPT_FILE` temp file leaked on Python script failure**
The `QU5_PROMPT_FILE=$(mktemp ...)` is cleaned up only after the `claude --print` call. But if the `python -c` block on lines 253–284 fails (e.g., `LAST_VARIATIONS` is empty), the temp file path is set but potentially empty — and the `rm -f` on line 290 may not be reached if `set -e` triggers. The `claude_ask` helper correctly handles this, but the QU5 block uses a manual pattern without the same cleanup guarantee.

**F-10: `VARS_CONTENT` and `RUBRIC_CONTENT` are computed but never used**
Lines 251–252 define `VARS_CONTENT` and `RUBRIC_CONTENT` as bash variables, but the Python script on lines 253–284 reads the files directly via `open(sys.argv[3])` and `open(sys.argv[4])`. The two bash variables are dead code.

**F-11: `signal.skills.yaml` is overwritten via `yaml.dump` — loses comments and ordering**
`yaml.dump(..., sort_keys=False)` preserves key order but `ruamel.yaml` is not used, so any inline comments in the YAML file are silently dropped on write. If `signal.skills.yaml` has comments (likely for a developer-facing config), they are permanently lost on first quest completion.

**F-12: No validation that `LAST_SCORE >= 80` before writing golden**
The script writes a golden artifact regardless of final score. If the loop exits at `MAX_ROUNDS` with a score of 62, the golden file is stamped `status: GOLDEN` and the YAML is updated — without any warning that the threshold was never met. A score gate or a `status: PARTIAL_GOLDEN` distinction would prevent false positives in the registry.
