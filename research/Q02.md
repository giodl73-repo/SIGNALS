# Q02 — Common Skill Lifecycle Validation

**Priority**: P0
**Status**: Open
**Owner**: giodl
**Resolves**: Consistent lifecycle pattern across all simulate plugin namespaces
**Blocks**: plugin-plan.md (lifecycle section)

## Question

Does a common 4-phase lifecycle (setup → execute → findings → amend) help users understand and navigate skills across namespaces, or does it add unnecessary complexity?

## Hypothesis

Skills across all 8 namespaces follow a similar pattern: gather context, do the work, present results, iterate. Formalizing this as a common lifecycle could:
- Make skills predictable (learn once, apply everywhere)
- Enable `/program:plan` to orchestrate stages consistently
- Give users clear expectations for what each phase produces

But it could also:
- Feel bureaucratic for simple one-shot skills
- Confuse users who just want to run a skill and get results
- Add friction to the "zero barrier to entry" principle

## Proposed Common Lifecycle

### The 4 Phases

| Phase | What | Output |
|-------|------|--------|
| **Setup** | Gather context, select roles, confirm scope | Ready state (what we're testing and who's testing it) |
| **Execute** | Run the simulation/analysis | Raw output (traces, reviews, scores, findings) |
| **Findings** | Synthesize results into actionable format | Summary with priorities (P1/P2/P3) |
| **Amend** | User reviews, disputes, or accepts; iterate if needed | Final artifact (accepted or revised) |

### How It Manifests Per Namespace

---

#### `/scout:` — PM proves the idea

| Phase | What Happens | Example (`/scout:competitors`) |
|-------|-------------|-------------------------------|
| Setup | Identify the space, select competitive dimensions, confirm scope | "Analyzing CRM market, focusing on: pricing, AI features, extensibility" |
| Execute | Research competitors across dimensions using stock PM + Strategy roles | Competitor matrix, feature gaps, white space analysis |
| Findings | Synthesize into positioning recommendation with P1/P2/P3 priorities | "P1: We have no AI copilot story. P2: Pricing undercuts by 30%. P3: Missing mobile-first" |
| Amend | PM reviews, challenges assumptions, adds context ("we actually do have an AI story, it's just not shipped") | Final competitive brief with PM annotations |

**Lightweight version**: For quick scouts, Setup auto-detects from repo context, Execute + Findings merge into one step. Amend is optional.

---

#### `/draft:` — Author writes the design

| Phase | What Happens | Example (`/draft:spec`) |
|-------|-------------|------------------------|
| Setup | Gather intent, identify audience, select structure template | "Writing a spec for plugin auth. Audience: dev team. Template: ADR" |
| Execute | Author the document section by section using Architect role | Full spec draft with sections, diagrams, trade-offs |
| Findings | Self-review: highlight open questions, flag assumptions, identify gaps | "3 open questions flagged, 2 assumptions need validation" |
| Amend | Author reviews, fills gaps, resolves open questions | Final spec ready for `/review:design` |

**Lightweight version**: For simple proposals, Setup is one question ("what are you proposing?"), Execute produces the draft, Findings/Amend merge into "review your draft."

---

#### `/review:` — Team validates the design

| Phase | What Happens | Example (`/review:design`) |
|-------|-------------|---------------------------|
| Setup | Identify what's being reviewed, auto-select expert roles from content | "Reviewing auth spec. Selected: Security Architect, API Designer, Data Modeler" |
| Execute | Each role reviews independently, scores against criteria | 3-5 expert reviews with scores and specific findings |
| Findings | Synthesize across reviewers: consensus issues, split opinions, severity ranking | "2 P1 blocking issues (both reviewers agree), 3 P2 suggestions, 1 split opinion on caching" |
| Amend | Author addresses findings, re-runs specific reviewers on changes | Revised artifact with resolution notes |

**Lightweight version**: For quick code reviews, Setup auto-selects disciplines, Execute + Findings produce a single review document.

---

#### `/flow:` — Domain dev simulates the process

| Phase | What Happens | Example (`/flow:lifecycle`) |
|-------|-------------|----------------------------|
| Setup | Identify the business process, define start/end states, select domain roles | "Simulating Lead-to-Opportunity. Start: new lead. End: won opportunity. Roles: Sales, CRM Admin" |
| Execute | Walk through each step of the process, trace decisions, branching, exceptions | Step-by-step trace with decision points, data changes, role handoffs |
| Findings | Identify bottlenecks, missing steps, edge cases, exception paths not covered | "P1: No handler for duplicate lead detection. P2: Approval step has no timeout. P3: Notification missing for reassignment" |
| Amend | Domain expert reviews, adds business rules ("actually, duplicates are handled by a plugin"), re-simulates affected steps | Final process trace with validated business rules |

**Lightweight version**: For simple triggers (`/flow:trigger`), Setup is "which record type?", Execute traces the trigger chain, Findings lists what fires.

---

#### `/trace:` — System dev simulates the platform

| Phase | What Happens | Example (`/trace:request`) |
|-------|-------------|---------------------------|
| Setup | Identify the request, define entry point and expected endpoint, select system context | "Tracing POST /api/accounts. Entry: API gateway. Expected: Dataverse record created. Context: Commerce" |
| Execute | Hand-compile the request through each service boundary, middleware, transformation | Layer-by-layer trace: gateway → auth → routing → business logic → data layer → response |
| Findings | Identify failure points, latency risks, contract mismatches, missing error handling | "P1: No retry on Dataverse timeout. P2: Auth token not forwarded to downstream. P3: Response missing pagination headers" |
| Amend | System dev reviews, corrects assumptions ("auth is actually handled by APIM, not the gateway"), re-traces affected layers | Final system trace with validated architecture |

**Lightweight version**: For targeted traces (`/trace:permissions`), Setup is "who and what action?", Execute traces the RBAC chain, Findings lists gaps.

---

#### `/prove:` — Prove what you believe

| Phase | What Happens | Example (`/prove:hypothesis` → experiments → `/prove:synthesize`) |
|-------|-------------|----------------------------------------------------------------|
| Setup | State hypothesis and falsification criteria | "We believe customers want real-time sync. We'd change our mind if: <5% use it, or latency >2s makes it unusable" |
| Execute | Run selected experiments (websearch, intelligence, prototype, analysis, interview) | Evidence from each experiment type |
| Findings | `/prove:synthesize` — merge all evidence into answer-first format with confidence levels | "Answer: Yes, but only for <100 record sets. Evidence strength: moderate. 3 principles extracted" |
| Amend | Researcher reviews synthesis, challenges weak evidence, optionally `/prove:publish` | Final investigation report or published paper |

**Note**: `/prove:` already has an explicit multi-step lifecycle. The 4-phase model maps naturally — Setup = hypothesis, Execute = experiments, Findings = synthesize, Amend = publish/review.

---

#### `/listen:` — Team simulates post-ship feedback

| Phase | What Happens | Example (`/listen:feedback`) |
|-------|-------------|------------------------------|
| Setup | Identify the shipped feature, select customer personas, define feedback dimensions | "Simulating reactions to new dashboard. 12 personas. Dimensions: usability, value, friction" |
| Execute | Each persona evaluates the feature from their perspective | 12 persona reactions with scores and verbatim feedback |
| Findings | Synthesize across personas: adoption blockers, delight moments, friction patterns | "P1: 3 personas can't find the feature. P2: Power users want export. P3: Mobile layout broken" |
| Amend | Team reviews, triages findings into backlog, re-simulates after proposed fixes | Triage document with accept/defer/reject for each finding |

**Lightweight version**: For quick adoption checks (`/listen:adoption`), Setup auto-selects personas, Execute + Findings merge.

---

#### `/program:` — Lead orchestrates all

| Phase | What Happens | Example (`/program:plan`) |
|-------|-------------|--------------------------|
| Setup | Define the initiative, select which namespaces/skills to include, set gates | "Planning new feature: AI copilot for CRM. Include: scout, draft, review, trace, listen" |
| Execute | Generate staged plan with skill sequence, gates, and dependencies | program.yaml with stages, skills, gates |
| Findings | Validate the plan: are stages in right order? Are gates achievable? Missing skills? | "Warning: No `/flow:` skills included — business process simulation skipped" |
| Amend | Lead reviews plan, adjusts stages, adds/removes skills, confirms gates | Final program.yaml ready for execution |

## Experiments

- a: Unified lifecycle test — present the 4-phase model to 10 mixed-role personas (PM, dev, architect, maker), ask if it helps or hinders → expected: preference data + qualitative feedback | method: design analysis
- b: Per-namespace lifecycle test — test each namespace's lifecycle independently with role-appropriate personas → expected: per-namespace accept/reject + improvement suggestions | method: design analysis

## Test Protocol

### Experiment (a) — Unified Test

For each persona:
1. Present the 4-phase lifecycle model (setup → execute → findings → amend)
2. Show how it applies to 2-3 different namespaces
3. Ask: "Does knowing this pattern help you predict how other skills will work?"
4. Ask: "Does this feel like helpful structure or unnecessary bureaucracy?"
5. Ask: "Would you skip any phases? Which ones?"
6. Score: usefulness (1-5), predictability (1-5), complexity (1-5, lower is better)

### Experiment (b) — Per-Namespace Test

For each namespace, test with 3 role-appropriate personas:
1. Present the namespace's lifecycle (all 4 phases with examples)
2. Walk through a concrete scenario using the lifecycle
3. Ask: "Does this match how you'd actually use this?"
4. Ask: "What would you change about this flow?"
5. Ask: "Is the lightweight version enough, or do you want the full version?"
6. Score: natural fit (1-5), would-use-as-described (1-5)
