## Evidence Notes on Key Decisions

**C-25 V-02 and V-04 FAIL**: Both retain "Reference a role from ROLES-LOADED in DRI / Owner where a role is known" after the mandatory CHECK. C-25 explicitly names "where a role is known" as a prohibited softening qualifier. The HALT-AND-CONFIRM checklist item 3 in V-04 ("Confirm DRI / Owner column will be present — all values from ROLE-NAME LOCK") uses strict language, but the post-produce instruction reintroduces the qualifier, failing the criterion's anti-hedging requirement.

**C-28 V-01 FAIL**: "Verify before writing each DRI / Owner value" satisfies per-cell for DRI/Owner. Key Roles in V-01 states "treat it as a first-class enforcement site, not an optional best-effort column" — strong language but no "verify before next row" directive. C-28 requires both columns. FAIL.

**C-31 V-02 FAIL (intentional)**: V-02's design axis is C-32 isolation; it preserves advisory register at all BEFORE-PRODUCE sites to separate the C-32 contribution from C-31. ROLE-NAME LOCK: "re-read the ROLES-LOADED block. Confirm..." Rebuttal: "re-read the four-field form definition — confirm each label is ready." Advisory throughout. FAIL on C-31 by design.

**C-31 V-03 PASS**: V-03 has exactly one BEFORE-PRODUCE directive (ROLE-NAME LOCK). "STOP. [...checklist...] DO NOT emit the ROLE-NAME LOCK until all four items above are confirmed." STOP = standalone interruption ✓, four-item checklist ✓, prohibitive gate ✓. The Rebuttal is prose-form, so no second BEFORE-PRODUCE site exists to fail. All existing directives use HALT-AND-CONFIRM. PASS.

**C-32 V-01/V-03 FAIL**: Neither variation has named vocabulary-lock blocks (MECHANISM-TYPE VOCABULARY, TRIGGER-TYPE VOCABULARY, CAT-N-CITATION-SCHEMA). V-01 mentions "Type closed vocabulary: Channel / Informal Role / Recurring Cadence / Shared Artifact" inline but no block declaration. No governed-table context to apply per-row checks to.

## Excellence Signals (V-05)

**Signal 1 — Activation-layer forward-reference in ROLE-NAME LOCK**

V-05's ROLE-NAME LOCK block annotates per-granularity enforcement at each governed site inline:

```
(2) Operating Rhythm Table DRI/Owner column
    [per-cell directive applies: verify before each next row, not after table is complete]
(3) Committee Charter Membership and Decides fields
    [per-role-name directive applies: verify each name before appending the next]
(5) Headcount table Key Roles column
    [per-cell directive applies: verify before each next row, not after table is complete]

Sites 2 and 5 carry per-cell row-by-row directives in their section instructions.
Site 3 carries per-role-name directives in the Membership and Decides field instructions.
These are the activation layer for this lock at those sites.
```

The lock becomes a navigation hub: it not only declares the complete role set but points to the activation mechanism type at each site. A model reading the lock knows before reaching any section which granularity enforcement level applies. Prior rounds declared the lock as a reference but left activation-layer discovery implicit.

**Signal 2 — Schema back-reference inside per-row directive**

V-05's Anti-Pattern Watch per-row check includes: "Re-read the CAT-N-CITATION-SCHEMA before verifying each row's prefix." The VERIFY block extends this: "Re-read FORMAT CONTRACT DISQUALIFY-IF and the CAT-N-CITATION-SCHEMA before producing each cell." This marries C-26's back-reference pattern (read schema before producing) with C-32's per-row pattern (check each row before the next) inside a single per-row instruction. The schema is not read once before the table — it is re-read per-row. Prior rounds separated schema declaration (C-15), back-reference (C-26), and per-row check (C-32) as independent mechanisms; V-05 integrates all three at the per-row granularity level.
