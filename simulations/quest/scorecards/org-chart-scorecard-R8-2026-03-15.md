### V-03 — Enforcement Block Injection

**What changed:** V-03/R7 conversational prose preserved verbatim. Three labeled `[ENFORCEMENT — ...]...[END ENFORCEMENT]` blocks injected at the three compliance points.

The key evaluation question for C-25: do isolated labeled enforcement blocks satisfy the "imperative-mode directive" requirement, or does C-25 require inline prose integration?

The rubric pass condition: *"At least one DO NOT / MUST / REQUIRED directive is present for each of the three behaviors."* The enforcement blocks are the directives — not the surrounding prose — and each block contains explicit imperative language:

- Guard block: `DO NOT EMIT...`, `DO NOT begin...`, `REQUIRED: the count N...MUST match`
- Register block: `MUST produce...`, `DO NOT skip`, `DO NOT proceed...`, `REQUIRED: the register MUST appear as a physically distinct labeled block`
- Citations block: `MUST open each...`, `DO NOT write...`, `DO NOT use...`, `REQUIRED: element names MUST be copied exactly`

The rubric evaluator "checks C-25 by locating the skill's guard enforcement, register construction, and citation format directives" — these blocks ARE those directives and they use imperative mode.

The risk: "by convention" still appears in the surrounding prose. However, the enforcement blocks constitute the operative directives; the conversational prose is explanatory context. Under variation pressure, the enforcement blocks may resist drift less than integrated imperative prose (V-02), but that is a reliability concern, not a rubric failure under the PASS condition as written.

| C-01 through C-24 | All prior criteria | PASS | V-03/R7 base preserved verbatim; no structural changes |
| C-25 | Compliance directives imperative mode | PASS | Each enforcement block has at least one DO NOT / MUST / REQUIRED; blocks ARE the directives for each of the three behaviors |

### V-04 — Targeted Imperative + Two-Step Count Protocol

**What changed:** V-01/R8 targeted imperative injection PLUS an explicit two-step count-verification protocol for the guard:

> **Step 1 — Count:** Count the numbered items in Case for Staying Flat.
> **Step 2 — Verify:** If count < 2, write missing reasons first. Once count >= 2, substitute N and emit separator.
> **ENFORCEMENT — GUARD:** DO NOT EMIT until count-check in Step 2 passes (count >= 2). REQUIRED: N MUST equal actual numbered reasons — verify before substituting.

### Ranking

All five score 100.0 by rubric. Differentiation is by reliability profile under variation pressure:

1. **V-04** (recommended production) — Targeted imperative + two-step count protocol. Lowest failure probability for C-23 (the criterion most likely to fail silently via count mismatch). Minimal diff from V-03/R7 preserves V-03's readability while adding imperative compliance and the verification circuit.
2. **V-05** — Full imperative + MUST-PRECEDE gate. Highest C-21 reliability (register physical distinctness is the most common C-21 degradation vector). Best choice if C-21 register completeness is the primary deployment concern.
3. **V-02** — Full imperative conversion. Cleanest, no legacy conversational framing. Best for long-term maintainability and forks; every directive resists drift uniformly.
4. **V-01** — Targeted injection. Minimal change from V-03/R7, clean rationale for each injection. C-23 confidence lower than V-04 (no two-step protocol).
5. **V-03** — Enforcement block injection. Passes C-25 by rubric letter; retains "by convention" in surrounding prose, which is the exact pattern V-03/R7 failed on. Dual-layer framing is the most fragile design under future variation.

### Excellence Signals

**From V-04 (top production design):**

1. **Enforcement labeling pattern works** — `**ENFORCEMENT — [BEHAVIOR]:**` as an inline labeled paragraph (not a separate block) provides the imperative directive C-25 requires while remaining part of the running section. Clean, localized, auditable.
2. **Two-step count-verify-substitute circuit** — explicit Step 1 (count) → Step 2 (verify >= 2 before substituting) removes the premature-guard failure mode without restructuring the surrounding prose. This pattern generalizes to any criterion requiring a guard with a printed count.

**From V-05:**

3. **MUST-PRECEDE gate with anti-inline assertion** — `"Inline taxonomy notes inside other sections do NOT satisfy this requirement — the register is a standalone artifact"` explicitly closes the ambiguity about what qualifies as a "distinct block." This pattern directly names and forecloses the failure mode it guards against, making the criterion's pass condition explicit in the prompt.

**From V-03 (diagnostic value):**

4. **Block-isolated imperative satisfies rubric letter but not spirit** — C-25 passes by rubric when enforcement blocks supply the required imperative language, even when surrounding prose remains conversational. This sets the minimum: one imperative directive per behavior is the floor, not the ceiling. Integrated imperative (V-01/V-02) is the more durable design.

**Pattern across all five:**

5. **C-25 is a directive-layer criterion, not an output-layer criterion** — the criterion fires on how the skill's enforcement language is framed, not on whether the output is correct. This means a skill can produce perfect output via conversational framing (V-03/R7) while failing C-25. R8 confirms that adding imperative directives at the three compliance points is the minimal sufficient change to earn C-25 without any structural change.
