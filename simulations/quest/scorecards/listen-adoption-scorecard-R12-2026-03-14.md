### Excellence Signals — R13 Extraction Candidates

**Signal 1 — Co-constraint column header (V-01, V-05)**

V-01's `Incumbent Dependency: [workflow step this role currently relies on THE INCUMBENT to perform]` column header enforces two criteria simultaneously at the point of generation: C-01 (all 16 roles assigned) and C-26 (displacement framing grounded in THE INCUMBENT). A model filling TABLE 1 cannot complete the Incumbent Dependency column without naming THE INCUMBENT (it appears in the header preamble before the rows) and cannot assign an archetype without grounding it in a named dependency. This is the table-column analogue of C-27/C-28 (field-label-encodes-constraint) elevated to a multi-criterion co-constraint: one column header enforces two independent rubric requirements simultaneously rather than one. Candidate: C-37 — Incumbent Dependency column header co-encodes archetype-assignment and displacement-grounding constraints at TABLE 1 generation time.

**Signal 2 — Row-level typed prefix extends header-level type contract to point of generation (V-02, V-04, V-05)**

C-36 requires the TABLE 3 header to declare rows as typed structural slot keys. V-02's SLOT-KEY: prefix on each row label extends this from header level (read once at table entry) to row level (re-asserted at each cell generation). The distinguisher from C-36: the header declares the type contract once; the SLOT-KEY: prefix re-asserts it at every row individually, closing the gap where a model reads the header preamble and then ignores it during individual row completion. Same elevation class as C-36 but at one structural level lower — the label-encodes-constraint pattern applied at the row rather than the container. Candidate: C-38 — TABLE 3 row labels carry SLOT-KEY: prefix that re-asserts typed slot contract at point of per-row generation, independent of whether the header preamble is retained.

**Signal 3 — Per-element subsection decomposition prevents content merging in multi-element sections (V-03, V-04, V-05)**

C-03 requires all three chasm elements (incumbent defense, bridge condition, crossing signal) to be present. The failure mode C-03 doesn't prevent is producing all three elements at uneven depth within a single prose block. V-03's CHASM-A / CHASM-B / CHASM-C subsection headers make each element its own named structural slot requiring discrete content — a model cannot complete PHASE 3 without producing a separate CHASM-A section, a separate CHASM-B section, and a separate CHASM-C section. This is architecturally different from a preamble instruction requiring three elements in one block. Candidate: C-39 — PHASE 3 CHASM is structurally decomposed into CHASM-A (incumbent defense), CHASM-B (bridge condition), CHASM-C (crossing signal) as named subsection headers, making per-element omission structurally incompatible with completing the phase.

### Open Gap: C-19

C-19 PARTIAL persists for the same structural reason across all five: answer-form questions produce high compliance rates at gates H, I, J; no gate fires; no CORRECTION BLOCK appears inline. C-32 provides the proof-of-availability substitute but does not satisfy C-19's proof-by-execution requirement. The gap cannot be closed by rubric redesign without either (a) deliberately weakening gate compliance to ensure at least one correction fires, or (b) introducing an alternative proof-by-declaration path for C-19 itself analogous to how C-32 resolved the C-19 paradox. R13 option: convert C-19 from a proof-by-execution criterion to a proof-by-availability criterion (aligning with C-32), or introduce a synthetic gate that fires on a structural condition guaranteed to produce a correction block regardless of topic quality.
