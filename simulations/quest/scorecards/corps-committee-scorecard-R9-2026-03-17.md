Each `B*-COMMIT` line includes explicit `B* → B* availability: [what is now available and for which downstream consumer]` declarations. This converts COMMIT from a checkpoint into a hand-off: the model knows what output it is committing AND which blocks may consume it. **3. Dual-pass dissent architecture with structural-layer separation**

### New Patterns Not Yet in Rubric

**Pattern A — Block availability hand-off declaration.** The pattern of appending `B* → B* availability: [what becomes consumable and for whom]` to every COMMIT line converts the block system from a series of checkpoints into a typed dependency graph. Each block's output is named, and each downstream consumer is identified at commitment time. This is stronger than the BLOCK DEPENDENCY MAP alone (which declares what blocks *intend* to consume) because it confirms at execution time what actually became available. A rubric criterion could test whether availability declarations are present on every COMMIT line, not just in the opening map.

**Pattern B — Two-column trace confirmation (manifest at gate + final confirmation at output).** V-05 verifies token presence twice: once at the Phase 1→2 gate (TOKEN TRACE MANIFEST, pre-consumption confirmation) and once at the end of Phase 2 (TOKEN TRACE FINAL CONFIRMATION, post-consumption confirmation). The two-check architecture catches tokens that pass the gate but are dropped during Phase 2 execution. A criterion could require the closing confirmation to be a structural table that cross-references the Phase 1→2 manifest, making the full audit chain a two-node verification rather than a single gate.
