These patterns exceed the current 40-criterion ceiling and are candidates for R11 criteria:

**From V-05:**

**ES-1: Named preamble block architecture** — The preamble is organized as 4 structurally labeled, numbered blocks (BLOCK 1: PRIMARY ANALYTICAL FRAMEWORK / BLOCK 2: OUTPUT SCHEMA / BLOCK 3: GATE TOKEN PROTOCOL / BLOCK 4: ROLE SEQUENCE AND CONTROL-TRANSFER PROTOCOL). Each block is a discrete declaration unit that can be referenced by name. This makes the preamble structurally inspectable and each block functionally addressable before reading any instruction. The BLOCK structure signals that the preamble itself is a first-class structural layer, not prose setup.

**ES-2: Role sequence and control-transfer protocol co-declared in a single named block** — BLOCK 4 combines the ROLE SEQUENCE and the transition rules ("occupy exactly one role at a time… every role transition requires a named control-transfer declaration… missing declaration = role boundary violation") into a single pre-instruction block. This makes the control-transfer protocol a preamble declaration rather than per-role embedded instruction — any agent can verify the protocol before reading ROLE: PREDICTOR.

**ES-3: Compressed posture statement in framework declaration** — "Posture: hypothesize from dictionary before scanning; verify against evidence during scanning; resolve predictions during synthesis." A single sentence co-located with the framework declaration gives the agent its behavioral orientation without a separate mandate section. Compressed from 3 sentences to one by using "Posture:" as a label.

**From V-04:**

**ES-4: Anti-pattern posture contrast embedded in framework declaration** — "Agent posture this skill requires: hypothesize from dictionary, then verify against evidence, then resolve. Not: scan, then label." The explicit anti-pattern ("Not: scan, then label.") embedded in the framework declaration makes the wrong posture identifiable by name, not just the right posture. This is a stronger constraint than a positive statement alone — it closes off the default LLM behavior path explicitly.

**ES-5: Three-table hypothesis loop explicitly named in TABLE G purpose** — "Purpose: Explicit closure of every TABLE A prediction. Completes the three-table hypothesis loop: TABLE A -> TABLE B -> TABLE G." Naming the chain in the table purpose makes the TABLE G's architectural role self-documenting — an agent reading the schema sees the full loop architecture before reading any role instruction.

**ES-6: SYNTHESIZER mandate as loop closure declaration** — "SYNTHESIZER mandate: Close the hypothesis loop opened by PREDICTOR and threaded through SCANNER." The synthesis role is explicitly framed as the closer of a loop opened in PREDICTOR, making hypothesis testing the structural contract of the entire three-role sequence rather than an end-of-flow reporting step.

**From V-03:**

**ES-7: PHASE 0 as framework-declaration phase** — Declaring a dedicated PHASE 0 for framework and schema declaration, with its own exit condition ("PHASE 0 COMPLETE -- framework and schema declared -- entering PHASE 1"), extends lifecycle ceremony to the setup phase itself. PHASE 0 exit is a prerequisite for PHASE 1 entry — framework declaration becomes a gated prerequisite, not ambient context. Exit conditions include destination phase ("entering PHASE N"), making phase transitions traceable by destination.
