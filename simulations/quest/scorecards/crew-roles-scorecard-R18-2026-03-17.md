## Excellence Signals from V-05

**Signal 1 — Two-phase anchor validation chain eliminates cascade revision**: V-04 and V-05 formalize the insight that Phase 6.1 should be a *confirmation* gate, not an *error-detection* gate. The FRAME FILL PRE-ANCHOR CHECK fires in Phase 4 before Phase 5 (planning table) is built, so any label omission is caught when only one FRAME FILL block is written rather than after an entire planning table and inertia-advocate file are committed. The Phase 6.1 EARLY INERTIA ANCHOR CHECK explicitly notes "if Phase 4 check PASS was correctly emitted, this is confirmation" — the two-gate language reframes the execution contract.

**Signal 2 — Operational non-generic definition with negative-example list makes C-37 enforceable**: V-02/V-04/V-05 add the clause "a phrase is non-generic if and only if it contains at least one noun or verb specific to this domain and cannot appear unchanged in a different domain's INERTIA BASELINE" plus a concrete negative list ("easy to use", "fast", "reliable", "well-known", "familiar", "low cost", "widely adopted", "stable", "flexible"). This closes the LLM self-assertion loophole where any phrase can pass the "non-generic" gate without enforcement. The NON-GENERIC CHECK is then referenced by name in Phase 7 INERTIA ANCHOR CHECK, creating a chain of custody from definition to final confirmation.

**Signal 3 — LENS AUDIT 6th item + CROSS-ROLE SPREAD CHECK converts INERTIA BASELINE from local anchor to registry-wide influence signal**: V-03 and V-05 add a 6th LENS AUDIT item requiring each non-inertia role to declare whether its `lens.verify` references an INERTIA BASELINE S or G term, and a Phase 7 CROSS-ROLE SPREAD CHECK that requires at least one non-inertia role to pass. This transforms INERTIA BASELINE from a single-file anchor (only the inertia-advocate reads from it) into a registry-shaping artifact. The CROSS-ROLE SPREAD CHECK emits a count and list, making the spread quantifiable.
