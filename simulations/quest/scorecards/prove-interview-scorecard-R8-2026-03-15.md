## Key Discriminators

**C-30 (new v8 — Prerequisite completion confirmed)** was the main fault line:
- V-01/V-04/V-05: PASS — explicit gate structures, master gate tables, inline "prerequisite check: FULLY SATISFIED — proceeding" before each dependent mechanism
- V-02: FAIL — Schema Contract Rule addresses C-29 elegantly, but C-30 is absent; moment label section deploys C-25 bridge with no prerequisite confirmation that C-13 is fully satisfied
- V-03: FAIL — Conversational framing sequences prerequisites correctly but provides no explicit gate confirmation; the guidance-level structure "before you write any interview dialogue, establish..." is authorial sequencing, not structural enforcement

**C-28 and C-29**: All five pass. The v8 variations successfully universalized these criteria — all five declare a complete three-label decision system with decision rules, and all five produce at least one instantiated schema example row.

**V-03 partial losses (C-19, C-20, C-22)** confirm the hypothesis: conversational framing is load-bearing for explicit completion structure. When guidance language replaces gate language, sequence justification statements, completion checkboxes, and dedicated vocabulary signature fields become authorial rather than structural — the prompt asks but does not enforce.

## Excellence Signals (V-05 as primary reference, shared with V-01/V-04)

1. **Master prerequisite gate table maps prerequisites to dependent mechanisms**: Section 0-D's four-row table (`Prerequisite | Dependent Mechanism | Status`) concentrates all C-30 checks in a single auditable location. No dependent mechanism can deploy without passing through this table. V-01/V-04 achieve the same effect through distributed inline checks; V-05 additionally provides the table as a single-source confirmation.

2. **Section-level status checks embedded within the declaring section itself**: V-05's C-22 STATUS CHECK (in 0-A), C-28 STATUS (in 0-B), and C-13 STATUS (in 0-C) create completion signals immediately after each prerequisite is produced — before the master gate aggregates them. This catches incomplete preparation at the source rather than only at the aggregation point.

3. **"What 'Not Applicable' looks like" column in the decision taxonomy schema**: V-05's Section 0-B adds a column for the negative condition alongside the decision rule — making it explicit when each label should NOT be used, not just when it should. This closes a gap in C-28 enforcement: the label system is governed, not just available.

4. **Inertia Profile as named structural baseline before subject testimony (V-04)**: Positioning the current-state workflow as a structural evidence artifact — the "competitor" all subjects respond to — gives SURPRISING/CONFIRMING/INCONCLUSIVE labels a shared referent independent of any individual subject's prior knowledge. Every label is now measured against a concrete baseline, not just against that subject's register entry.

## New Patterns (Candidates for v9 Criteria)

**Pattern A — Section-level prerequisite status signal**: The declaring section should confirm its own completion inline (e.g., `C-22 STATUS: FULLY SATISFIED — proceeding`) rather than only at a downstream gate. This creates double-layer verification and surfaces incompleteness at the source. Outputs that declare prerequisites in one section but confirm completion only at a distant gate (or not at all) leave a gap between declaration and activation.

**Pattern B — Current-state baseline profiled as a named structural artifact**: Before any subject speaks, the workflow team currently uses without the feature should be profiled as a shared reference artifact — workflow, cost, risk, and why it persists. This anchors every label, objection, and evidence item against a concrete named baseline rather than against abstract prior knowledge. Outputs that profile current state only via subject prior knowledge (rather than as a structural artifact) allow the baseline to vary by subject, weakening label comparability across interviews.
