## V-01 — Role Sequence: Domain Experts First
**Single axis.** Domain expert finding blocks precede stock discipline blocks in BLOCK 2 (BLOCK 1.5 lists Domain rows first, BLOCK 2 emits them first). The C-52 inertia note and C-53 SEALED dual-naming follow R18 V-05 inline prose form exactly.

**Hypothesis**: Specialist gaps surface before generalist noise — domain P1s appear at the top of the review where they are structurally prominent, not buried in finding block #6.

---

## V-02 — Output Format: Unified Master Finding Table
**Single axis.** All reviewers emit findings into one `Section | Sev | Finding | Reviewer | Origin` master table instead of per-reviewer heading blocks. F-01/F-22 coverage enforcement fires on rows rather than blocks.

**Hypothesis**: Consensus detection becomes mechanical — matching `Section + Finding` values across distinct Reviewer rows in the same table are visually detectable without a separate analysis pass.

---

## V-03 — Phrasing Register: Formal Modal Contracts
**Single axis.** All enforcement uses `SHALL / SHALL NOT / is non-conformant`. No imperative `Halt.` directives. F-IDs are present as named labels embedded in modal clauses. The C-52 note becomes a SCOPE EVOLUTION subsection with explicit `SHALL NOT revert` language.

**Hypothesis**: Modal obligations are contractually read rather than imperatively encountered — a modal clause cannot be "skipped" the way a `Halt.` bullet can be.

---

## V-04 — Role Sequence + Inertia Framing: Domain-First + SCOPE EVOLUTION Block Header
**Two axes.** Combines V-01 domain-first sequence with the inertia note elevated from inline prose to a named `SCOPE EVOLUTION` section header in BLOCK 2.7, expressed as a 4-column table row.

**Hypothesis**: A named section header is detectable as absent; inline prose is not. The two axes are independent — domain-first changes review order; the SCOPE EVOLUTION block header changes how C-52's inertia note is anchored structurally.

---

## V-05 — Output Format + Lifecycle Emphasis + Inertia Framing: Master Table + PHASE Markers + INERTIA REGISTER
**Three axes.** Master finding table (V-02 format) + named PHASE markers at every block boundary + INERTIA REGISTER as a 4-column structural table (`F-Code | Prior Form | Current Scope | Why Both Non-Conformant`) in BLOCK 2.7 before the enforcement section.

**Hypothesis**: Maximum structural density — PHASE markers make lifecycle boundaries scannable; INERTIA REGISTER captures the C-52 requirement as verifiable table cells rather than prose, applying the same structural-column discipline that C-10 applied to findings and C-13 applied to expert selection.
