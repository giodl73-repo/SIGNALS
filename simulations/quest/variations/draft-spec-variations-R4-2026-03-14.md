### New pattern candidates for v5

**V-05 — IG-IDs as parallel finding surface**: sequential IG-IDs create a named finding namespace alongside R-IDs. Each gets an elimination-path, a risk signal from the NS column (feasibility < 70 or blocking compliance → AMPLIFIED), and a dedicated TYPE E finding type. Candidate C-20: *"Per-IG-ID elimination-path analysis with AMPLIFIED risk signal when NS column shows feasibility < 70 or any blocking compliance requirement."*

**All / V-04 explicit — Token validity rule**: gate token declared without its prerequisite evidence (contradiction scan block + scan completion statement) is invalid. Upgrades C-17 from "gate artifact must appear" to "gate artifact must prove its prerequisites."

**V-03 — `[SCAN REQUIRED]` as field-level cue**: placing the marker directly before the field requiring inspection (not at phase level) creates field-level active inspection guards — more granular than phase-level protocol rules, more readable than formal REQUIRES declarations.
