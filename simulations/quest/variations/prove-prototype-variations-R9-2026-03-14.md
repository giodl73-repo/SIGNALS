| Variation | Axis | Design Hypothesis |
|-----------|------|-------------------|
| V-01 | **Inertia framing** (C-29) — single | Adding explicit "must outperform B-00" language to CALIBRATE's header and Phase 4 instruction satisfies C-29 without structural change. FRAMER no longer names the competitor. |

| V-05 | **All three + phrasing register** — combined | V-03 (CALIBRATOR owns naming) + V-02 (clean table) + register shift ("status-quo competitor" throughout) + Phase 9 explicitly names the inertia null as the primary counter-interpretation candidate. Ceiling design for this round. |

---

## Key Design Decisions

**C-29 boundary finding (V-01 vs V-03)**
The live question is whether C-29 passes when the competitor is *named* in DESIGN but *framed as status-quo* in CALIBRATE (V-01/V-04), or whether the naming must also reside in CALIBRATE (V-03/V-05). V-01 and V-03 test this binary.

**C-30 table form (V-02 vs V-05)**
R8 V-05 used a multi-row table (Value row + Delta rows). V-02/V-04/V-05 use a single-row value table + adjacent labeled delta block. The question is whether the merged format or the separated format produces cleaner column-gap auditability.

**Phase 9 null seeding (V-05 only)**
V-05 seeds the inertia null in Phase 4's CALIBRATE output block and requires Phase 9 to address it explicitly by name. This is the strongest version of C-29's downstream benefit — the null counter-interpretation is not constructed retroactively in CLOSE but named before any build activity.
