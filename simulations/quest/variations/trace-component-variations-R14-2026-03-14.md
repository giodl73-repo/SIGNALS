# trace-component — Round 14 Variations

**Variation axes:** Role sequence (V-01), Phrasing register (V-02), Lifecycle emphasis (V-03), then combined (V-04, V-05).

All five carry **C-43 + C-44 + C-45** in every manifest close-line placeholder. The structural obligation for all five:

| Required property | Where it lives in placeholder |
|---|---|
| C-45 — count declaration as leading phrase | First token: "N=2 OBLIGATIONS" / "Two requirements, one instruction:" / authored count label |
| C-44 — numbered constraint enumeration | (1)/(2) or OBL-1/OBL-2 — each constraint independently addressable |
| C-43 — entanglement declaration | "Neither satisfied without the other" / "Mutual dependency: neither independently satisfies" |

---

## Variation summary

**V-01 · Role sequence** — Schema architect is explicitly tasked with deriving and justifying each of the three placeholder properties (Property A = count, Property B = numbered, Property C = entanglement) from abstract failure-mode reasoning. The manifests then invoke the authored placeholder form via `*[Apply your TRAVERSAL-SCHEMA Area 3 placeholder... That instruction carries Property A, B, C. Write the close-line now.]*`. Hypothesis: self-authored causal justification of all three properties makes compliance a logical consequence of the model's prior output. Expected ~160–162/164.

**V-02 · Phrasing register** — Charter-based (no schema architect). Clinical bracket notation replaces natural language: `*[DUAL-CONSTRAINT BLOCK | N=2 OBLIGATIONS — (OBL-1) content: ...; (OBL-2) position: ... Mutual dependency: OBL-1 satisfied alone does not satisfy this block; OBL-2 satisfied alone does not satisfy this block; both must be independently and simultaneously satisfied.]*`. Tests whether a formal/technical notational register for the same three properties produces different compliance characteristics. Expected ~156–158/164.

**V-03 · Lifecycle emphasis** — Each manifest opens with a **PHASE CONTRACT** that forward-declares the dual-constraint obligation (N=2, two numbered items, entanglement) *before* the manifest body. The close-line placeholder then references the contract: `*[Two requirements, one instruction — as declared in this manifest's Phase Contract: (1) content...; (2) position... Neither satisfied without the other.]*`. Tests whether the entanglement constraint being active from phase-open (not just at close) strengthens compliance. Expected ~158–160/164.

**V-04 · Role sequence + Phrasing register (combined)** — Schema architect designs the DUAL-CONSTRAINT BLOCK placeholder in clinical OBL-1/OBL-2 bracket notation from first principles (three causal questions: count label, labeled items, mutual-dependency). C-41 targeted (Question 1–3 causal chain). C-42 partially targeted (abstract questions, no copyable text). Expected ~160–164/164.

**V-05 · All three axes + Inertia framing** — Unified null-hypothesis epistemology: the inert-hop rule (Area 2) and the post-manifest null (Area 3) are structurally parallel instances of the same epistemic principle. Area 3 asks the model to derive the three placeholder properties *from the null-hypothesis framework* — satisfying only OBL-1 is the same error as an undocumented active hop (an unsupported compliance claim). Phase Contracts (lifecycle) + clinical bracket notation (register) + architect authorship (role sequence). Maximum R14 coverage. Expected ~162–164/164.

---

**Key R14 advance over R13:** R13 V-03/V-05 established the three-property form but only as prompt-provided or schema-invoked language. R14 V-01/V-04/V-05 test whether the model *authoring the justification* for each property — rather than executing a provided form — produces more durable compliance. R14 V-02 isolates whether the notational register of C-45/C-44/C-43 matters (OBL-label clinical form vs. natural language). R14 V-03 tests whether front-loading the entanglement constraint changes compliance by making it active throughout the phase lifecycle rather than only at the close-line position.
