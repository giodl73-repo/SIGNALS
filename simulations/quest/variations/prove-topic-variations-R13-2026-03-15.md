## R13 — prove-topic — Five Variations Summary

**New criteria targeted**: C-35 (ROLE C gate-structural ownership of `incumbent_threat_locked`) and C-36 (SESSION INVARIANT D phrasing register) on top of the full C-01–C-34 stack.

---

### V-01 — Role Sequence (ROLE C First)

ROLE C executes before ROLE A and ROLE B. GATE S clearance sequence becomes `incumbent_threat_locked` (ROLE C) → `gate_s_cleared` (ROLE B) → `invariant_registry_confirmed` (ROLE A). SESSION INVARIANT D is the 4th invariant. INCUMBENT CHECK blocks at each stage cite `incumbent_threat_locked` as structural origin explicitly.

**Axis hypothesis**: Placing ROLE C first makes the C-35 dependency chain visible before hypothesis work begins — you cannot load a scout artifact meaningfully until you have named what it must help displace. Retroactive ROLE C feels bolted on; ROLE C first feels load-bearing.

---

### V-02 — Output Format (Table-Dominant)

SESSION INVARIANTS as a 4-row table (D leads). ROLE definitions as a 3-row ownership table with GATE S step column. GATE S as a 3-row clearance table. INCUMBENT CHECK at each stage as a mini-table (ITEM | SUMMARY | TEMPLATE APPLIED & VERDICT). Handoff as a full table with FIELD | VALUE | DERIVATION | INDEPENDENCE CHAIN columns. Prose reserved for non-structural narration only.

**Axis hypothesis**: Table format makes each criterion independently scannable — C-35 is visible as a ROLE TABLE row, C-36 as an INVARIANTS TABLE row — and makes `schema_compliance_confirmed` trivially checkable by row count.

---

### V-03 — Lifecycle Emphasis (Gate-Boundary Explicit)

GATE S becomes Stage 0 with its own numbered ENTRY CONDITIONS and named EXIT SIGNAL (`gate_open`). Each of Stages 1–5 has numbered ENTRY CONDITIONS (including SESSION INVARIANT D template echo) and a named EXIT SIGNAL (`hypothesis_locked`, `websearch_complete`, etc.). No stage body begins without confirmed preconditions. Stage transitions are never silent.

**Axis hypothesis**: Campaigns break down at transitions. Explicit numbered entry checklists prevent GATE S skip and s2_ce_verdict blank-advancement. SESSION INVARIANT D echoed at each ENTRY CONDITIONS block means the phrasing constraint is visible at point-of-use, not just at CAMPAIGN OPEN.

---

### V-04 — Phrasing Register + Inertia Framing (SESSION INVARIANT D Primary)

SESSION INVARIANT D leads the invariant registration block, before A, B, C. CAMPAIGN OPEN opens with INCUMBENT ANCHOR (strength/vulnerability) before any invariant is registered. Each INCUMBENT CHECK block opens with `Template (SESSION INVARIANT D, Stage N — verbatim, fill bracketed slots only):` followed by the template exactly as declared. The format-error label is stated at the declaration site.

**Axis hypothesis**: Printing the exact template verbatim at the point of use — not just referencing it — makes paraphrase prevention legible. If a practitioner is about to write a differently-worded question, the template sitting right above it makes the violation self-evident before it happens.

---

### V-05 — Combined (All Four Axes)

ROLE C first + table format for invariants/roles + explicit Stage 0 and per-stage entry/exit signals + SESSION INVARIANT D leading with verbatim template echo in each stage's ENTRY CONDITIONS. The four axes amplify each other: ROLE C first makes the gate chain legible, table format makes criteria scannable, lifecycle gates make transitions explicit, SESSION INVARIANT D echo makes phrasing enforcement point-of-use.

**Axis hypothesis**: Maximal completeness-visibility design — a reviewer can confirm C-35 by scanning the ROLE TABLE, C-36 by scanning row D of the INVARIANTS TABLE, GATE S three-field clearance from the clearance table, and per-stage D inheritance from the ENTRY CONDITIONS checklist. All structural obligations are locatable without reading prose.
