### Key structural additions per variant

**V-01:** New `C11` constraint requiring Arc Position column on every LIFECYCLE EPOCH MAP epoch row. GATE A/B/D updated to check C11. EPOCH-CLUSTER ANALYSIS now produces role-grounded remediation: "Gate boundary structural-role problem" not "QUALIFY epoch problem."

**V-02:** `C11` carries a synergy requirement: remediation implication IS REQUIRED to reference Arc Position structural role, not epoch name alone. IS-NOT/IS contrast in remediation: "IS NOT an APPROVAL epoch-name finding; IS a Gate boundary structural-role finding."

**V-03:** `C11` adapted for step-block format — Arc Position expressed as `[Arc: Entry boundary]` tags on LIFECYCLE EPOCH MAP rows and ECL step entries. Tests C-51 format-neutrality without a dedicated column.

**V-04:** New `C12` constraint covering both LIFECYCLE EPOCH MAPs. DOMAIN DEPENDENCY DECLARATION adds Arc Position cross-domain note (structural roles ARE the comparison axis for C-52). EPOCH-CLUSTER ANALYSIS names source-domain Arc Position AND target-domain Arc Position at the Finance→Sales handoff boundary.

**V-05:** New `C13` constraint covering all three LIFECYCLE EPOCH MAPs. Both BRIDGE TABLEs (F→S and S→CS) use named row IDs for C-47 citation precision. EPOCH-CLUSTER ANALYSIS characterizes BOTH handoff boundaries with Arc Position vocabulary, enabling the diagnostic: a Finance Approval-event defect + Sales Entry-boundary defect at the same handoff = one handoff contract failure, not two independent domain defects.
