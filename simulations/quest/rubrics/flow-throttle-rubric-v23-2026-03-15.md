# Rubric: flow-throttle

Evaluate simulated throughput analysis across rate-limited systems. The output must trace bottleneck locations, rate limit triggers, backpressure propagation, and user-visible behavior across throttle tiers for a given request volume.

---

## Criteria

### Essential (60 points total — must all pass)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-01 | **Bottleneck Localization** | correctness | essential | Output names at least one specific system boundary where the first rate limit is hit (e.g., connector tier, gateway, backend API) and states the limit type (requests/min, concurrent calls, payload size). |
| C-02 | **Backpressure Propagation Trace** | correctness | essential | Output traces how throttling at the identified bottleneck propagates to at least one upstream or downstream component — shows the causal chain, not just a list of affected components. |
| C-03 | **User Experience at Throttle Tier** | coverage | essential | Output describes observable user-facing behavior at each distinct throttle tier reached (e.g., 429 response shown to user, queue delay, silent retry, flow run failure with error message). At least two tiers described if multiple tiers exist. |
| C-04 | **Unprotected Burst Path Identification** | correctness | essential | Output identifies at least one path where burst traffic bypasses throttle controls (e.g., a trigger that fires N flows in parallel with no concurrency cap, an action with no retry policy). |
| C-05 | **Rate Limit Specificity** | depth | essential | At least one rate limit is cited with a concrete numeric threshold (e.g., "600 requests/min per connection", "5 concurrent flow runs per environment") rather than described only in abstract terms. |

---

### Recommended (30 points total — output is better with these)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-06 | **Retry-After Handling Gap** | coverage | recommended | Output explicitly checks whether the simulated flow/connector handles `Retry-After` headers or equivalent backoff signals from throttled APIs. Flags missing handling as a finding. |
| C-07 | **Cascading Throttle Failure Scenario** | depth | recommended | Output constructs or identifies a specific scenario where throttling at one tier triggers a second throttle event at a different tier — describes the compounding effect on throughput or error rate. |
| C-08 | **Throttle Tier Matrix** | format | recommended | Output includes a structured mapping (table, list, or equivalent) of request volume ranges to throttle behavior, showing which tier activates at which volume and what the resulting behavior is. |

---

### Aspirational (10 points total — raise the bar once essential/recommended are stable)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-09 | **Mitigation Prescriptions** | depth | aspirational | For each identified bottleneck or unprotected path, output proposes a concrete remediation (e.g., batching strategy, concurrency limit config, exponential backoff policy, queue buffering pattern) — not generic advice. |
| C-10 | **Quantified Impact at Scale** | depth | aspirational | Output provides numeric estimates of degradation at a specific load spike (e.g., "at 10× normal volume, 40% of requests are queued for >30s, 8% fail with 429") using the stated rate limits as the basis for calculation. |
| C-11 | **Per-hop Causal Mechanism Column in Propagation Trace** | format | aspirational | The backpressure propagation trace is expressed as a multi-row table or equivalent structure where each row covers one hop (From component → To component → Causal mechanism) and explicitly names the mechanism at that hop. A narrative paragraph listing affected components does not pass — the mechanism at each hop must appear as a discrete, attributed field. |
| C-12 | **Step-by-step Arithmetic with Named Limit Citations** | depth | aspirational | The quantified impact section (C-10) shows step-by-step arithmetic that explicitly names each numeric limit used in the calculation — the chain from "limit L at component C → at N× load → Y% degradation" is written out and auditable, not asserted as a conclusion. |
| C-13 | **Upfront Component-limit Inventory** | coverage | aspirational | Before propagation or UX analysis begins, output provides an explicit inventory table or list of all named system components with their numeric rate limits. Components where the actual limit is unknown must be flagged with a platform default or an explicit unresolved note — not silently omitted. |
| C-14 | **Volume-multiplier Tier Matrix with Error Rate Column** | format | aspirational | The throttle tier matrix (C-08) is structured with fixed volume multiplier rows (1×, 3×, 5×, 10× of normal load) and includes an estimated error rate or degradation column per row, enabling direct visual comparison of how throttle behavior changes as load scales. A matrix without fixed multiplier rows or without a degradation column does not pass. |
| C-15 | **Full Output Table Contract Fidelity** | format | aspirational | All required analytical tables (component-limit inventory, propagation trace, tier matrix, mitigation table) appear in the output with their declared column structure intact. No required table is replaced by a prose equivalent, a summary paragraph, or a partial list. An output that produces three of four required tables does not pass. Pre-committed schema declarations are the primary mechanism — prose substitution of any schema-declared table fails this criterion regardless of analytical quality. |
| C-16 | **Cross-table Referential Consistency** | depth | aspirational | The propagation trace references only components that appear in the component-limit inventory — no phantom components introduced mid-analysis. The tier matrix arithmetic citations trace to named inventory rows — each arithmetic step names a specific component and its numeric limit from the inventory. An output where the propagation trace introduces unlisted components, or where arithmetic asserts limits without citing their inventory source, fails this criterion. |
| C-17 | **Named Status Taxonomy for Unknown Limits** | format | aspirational | The component-limit inventory uses a named status taxonomy token (e.g., `confirmed`, `estimated`, `?-unresolved`) for each row rather than describing uncertainty in prose inline. Each component's limit row has an explicit status designation — a component with an unknown limit receives the unresolved token, not a prose explanation such as "limit unclear" or "may vary." Informal inline descriptions of uncertainty do not pass. The taxonomy form enables downstream arithmetic gates to proceed with explicit assumption statements rather than halting or silently guessing. |
| C-18 | **Retroactive Structural Correction Gate** | format | aspirational | The analysis includes at least one explicit review gate — a "before proceeding to [next phase/section]" checkpoint — that names specific structural deficiencies requiring correction and mandates rewriting the deficient section before continuing. Generic "check your work" instructions do not pass. The gate must name the target deficiency (e.g., "any hop with a generic or absent mechanism must be rewritten with a specific technical cause before Phase 3 begins") and treat correction as blocking. This mechanism is complementary to upfront schema mandates — it catches partial compliance that occurs under token pressure when upfront constraints alone are insufficient. |
| C-19 | **Per-table At-header Anti-substitution Placement** | format | aspirational | Each required analytical table has its ANTI-SUBSTITUTION or prose-prohibition notice placed immediately after the table's column header row — at the exact production point of that table. A single global schema declaration at the prompt header, a phase-level warning before the table begins, or a final verification sweep after all tables are complete does not pass. The inline placement fires at the moment the model produces each table and is the only mechanism that survives token pressure uniformly across all tables including those produced late in the output. An output that places inline enforcement on early tables but defers later tables to a global or post-hoc check fails this criterion even if C-15 passes. |
| C-20 | **Named Failing-token Vocabulary in Correction Gates** | format | aspirational | The retroactive correction gate (C-18) enumerates a specific set of failing token strings or patterns — actual text fragments that constitute an insufficient mechanism — rather than only describing deficiency in structural terms. A gate that says "any vague mechanism must be rewritten" passes C-18 but fails C-20 unless it also names exemplar tokens (e.g., "also throttled", "cascade occurs", "propagates to", "downstream effect", "flow is affected", "rate limiting applies"). The vocabulary makes the gate auditable: an evaluator can scan the output for the named tokens to verify whether the gate would have fired. An enumerated failing-token list is required; structural descriptions alone are not sufficient. |
| C-21 | **Structurally Separated Consistency Auditor** | depth | aspirational | The cross-table consistency check (C-16) is performed by an entity — a named role, a dedicated audit phase, or an explicit verification step — whose sole mandate is verification and who has no analytical output in the tables being audited. Self-referential phantom flagging does not pass: when the same role that built the propagation trace or arithmetic section also verifies its own cross-table references, the model can rationalize its own phantoms without detection. The auditor entity must be structurally incapable of having introduced the components or arithmetic it is auditing — role separation is the mechanism, not the quality of the verification instructions. An output that achieves C-16 through self-verification fails C-21 regardless of whether the final content is consistent. |
| C-22 | **Per-token Binary Attestation with Failure Localization** | format | aspirational | The failing-token scan (C-20) uses per-token checkbox items (`- [ ]`) that each require an explicit individual disposition: `[CLEAR]` if the token is absent from the output, or `[FAIL: hop N — rewritten below]` naming the specific hop or location if the token is present. A holistic "all tokens reviewed" statement does not pass even when token vocabulary is enumerated (C-20). The attestation format requires that each named token's disposition is independently recorded and that failure annotation names the location, enabling targeted rewrite rather than whole-section regeneration. Unresolved checkbox items — any `- [ ]` without a disposition — constitute gate-not-passed. |
| C-23 | **Structural Incapability Preamble for Consistency Auditor** | depth | aspirational | The structural auditor (C-21) opens its role or phase with an explicit statement of structural facts establishing why it is incapable of having produced the tables it is auditing — framed as fact, not instruction (e.g., "Role 4 has no output in Tables 1–3. It did not produce the component limits, the propagation hops, or the tier arithmetic. It cannot rationalize its own prior choices because it made none."). This preamble precedes the verification mandate. Without it, a structurally separated role can still rationalize phantom entries by appealing to intent or partial knowledge — the incapability statement forecloses that path by establishing, as a declared structural fact, that the auditor has no authorial stake in the content it is auditing. A role that is declared structurally separate (C-21) but does not open with an incapability statement passes C-21 but fails C-23. |
| C-24 | **Stub-row-proof At-header Gate Placement** | format | aspirational | When a table schema includes pre-filled example or stub rows as part of its declaration, the anti-substitution enforcement notice appears between the column header separator row (`|---|---|`) and the first stub row — not after the stub rows. Pre-filled stub rows in a schema declaration are structural scaffolding, not production output; deferring the enforcement notice to after them creates the same gap between header and gate as any other deferred placement. A table whose schema includes stub rows but places the enforcement notice after those rows fails this criterion even if all other stub-free tables meet C-19. This criterion is stricter than C-19: a prompt can pass C-19 (no gap on tables without stubs) while failing C-24 (gap introduced by stubs in the last or heaviest table). |
| C-25 | **Auditor-mandate Scope Restriction to Flags and Verdict** | depth | aspirational | The structurally separated auditor role (C-21) has its mandate affirmatively scoped in its role definition to producing flags and verdicts only — no root-cause explanations, fix proposals, diagnostic analysis, or supplementary output of any kind. The restriction must appear in the role definition itself (e.g., "Role mandate: flags and verdict only"), not be inferred from the auditor's actual output. A role that achieves C-21 (no analytical rows in the four tables) but whose mandate does not explicitly prohibit non-flag output fails C-25 — residual analytical authority reintroduces authorial stake under token pressure even when the auditor happens to self-limit in a given run. |
| C-26 | **Structural-fact Label on Incapability Preamble** | format | aspirational | The incapability preamble (C-23) carries an explicit label — "Structural fact" or equivalent — that identifies it as a statement of system state, not an instruction, and that appears before the verification mandate (e.g., "Structural fact (this section precedes the mandate and is not an instruction)"). Without the label, incapability language can be reinterpreted as a soft guideline under token pressure — the model may discount "I made no choices" as an orientation remark rather than a declared fact. The label forecloses that reinterpretation by classifying the preamble as a structural datum, not a directive. An unlabeled preamble with correct incapability language passes C-23 but fails C-26. |
| C-27 | **Role-scoped Output Suspension in Correction Gate** | format | aspirational | The retroactive correction gate's blocking condition is stated as a role-scoped output suspension — "Role X produces no further output while any item is unresolved" — rather than a phase-boundary gate — "Phase N does not begin." Phase-boundary language allows the model to insert inter-phase output (clarifications, summaries, bridging analysis) while nominally respecting the gate; role-scoped suspension forecloses that gap by blocking all output from the correcting entity until every item reaches a resolved disposition. A prompt with C-18 (blocking gate) and C-22 (per-token attestation) that uses phase-boundary language passes both but fails C-27. |
| C-28 | **Validity-Predicate Instruction Form** | format | aspirational | All instructional statements governing table production and role output are expressed as validity predicates — "TABLE X is valid when [condition]", "Role Y output is valid when [condition]" — rather than as imperative directives — "Fill TABLE X", "Role Y must [action]". The predicate form makes compliance a binary evaluable property of the output: either the output satisfies the stated condition or it does not. An imperative directive may or may not have been honored; a predicate condition is self-auditable against the output at evaluation time without reconstructing the model's intent. A prompt that achieves all prior structural criteria through imperative instructions passes those criteria but fails C-28 unless the instruction surface consistently uses validity-predicate form. Partial predicate adoption — predicates on some tables, imperatives on others — does not pass. |
| C-29 | **Gate-cleared Predicate Form for Blocking Gate Invariants** | format | aspirational | The blocking gate invariant (C-18) is expressed as a cleared-when predicate — "GATE N is cleared when and only when every [item] has received [disposition]" — rather than as an imperative do-not-proceed directive — "Do not proceed until all items are resolved." The predicate form makes gate state a declared property of the output that can be verified by inspection without interpreting whether a directive was honored: the gate is either cleared or not cleared by the state of the attested items. A prompt with a blocking gate (C-18), per-token attestation (C-22), and role-scoped suspension (C-27) passes all three but fails C-29 if the gate invariant uses imperative rather than predicate language. The distinction is auditor-accessible: imperatives encode intent; predicates encode state. Only predicate form is self-auditable at evaluation time. |
| C-30 | **Pre-committed Audit Contract Before Analytical Roles** | depth | aspirational | Role 4's complete specification — incapability preamble (C-23), structural-fact label (C-26), scope restriction mandate (C-25), all verification checks (C-16), and verdict format — is declared as a named, standalone PRE-COMMITTED AUDIT CONTRACT in a dedicated section before any analytical role begins. At Role 4's execution point, it references the pre-committed contract rather than receiving its specification inline. Pre-commitment temporally fixes the audit contract before the content it will audit is produced, preventing the contract terms from being adjusted — consciously or under token pressure — to accommodate the analytical output. A prompt where Role 4's specification appears inline at its execution point passes C-21, C-23, C-25, and C-26 on content grounds but fails C-30 unless the specification was declared as a named contract before the analytical roles. Pre-commitment is a temporal property of the declaration, not a structural property of its content. |
| C-31 | **Negative Undisposed-item Invariant in Gate Predicate** | format | aspirational | The cleared-when gate predicate (C-29) includes an explicit negative invariant naming the failure state in addition to the positive cleared condition — "GATE N is cleared when and only when [positive condition] — no `- [ ]` item may remain undisposed" or equivalent. The positive predicate alone declares what cleared looks like; the negative invariant makes the not-cleared failure state a directly declared constraint rather than an implication that requires logical negation of the positive. An evaluator can determine gate-not-cleared by scanning for undisposed checkbox items without applying any inference step. A gate that satisfies C-29 using only the positive cleared-when form passes C-29 but fails C-31 unless the failure state is also explicitly stated as a negative constraint. Partial adoption — negative invariant on some gates but not others — does not pass. |
| C-32 | **Spatially Co-located Rewrite Mandate** | format | aspirational | The correction gate's rewrite instruction prescribes the output location of rewrites relative to their failure annotations — "rewrite flagged sections below their `[FAIL]` annotations" or equivalent — rather than permitting rewrites in a separate correction section, at end of output, or in a later phase. Spatial co-location keeps correction output immediately adjacent to its triggering annotation, making the rewrite auditable by linear scan: the `[FAIL]` annotation and the corrected content form a contiguous block at the failure's location in the output stream. A correction gate that satisfies C-22 (per-token binary attestation with failure localization) but does not prescribe the rewrite's location relative to the `[FAIL]` annotation passes C-22 but fails C-32. The rewrite-below-annotation form is the only spatial prescription that keeps correction evidence and corrected content co-located for every token simultaneously, without the evaluator needing to match failure annotations to a separate correction section. |
| C-33 | **Enumerated-list Block Form for Gate Failure States** | format | aspirational | When a gate has multiple failure conditions, the negative invariant uses a dedicated enumerated-list block — "GATE N is not cleared when any of the following failure states are present: (1) [state], (2) [state], …" — rather than inline append to the positive predicate. The inline form ("cleared when X — no Y may remain, no Z cell may be blank") satisfies C-31 for a single condition but conflates positive and negative predicates in a single sentence, making the failure-state list difficult to extend and hard to scan independently. The separate enumerated block treats each failure state as a discrete list item, is independently scannable without parsing the positive predicate, and accommodates additional failure predicates (such as C-32 spatial co-location) without rewriting the cleared-when sentence. A gate with a single inline negative append passes C-31 but fails C-33 when the gate governs multiple subsidiary criteria. Partial adoption — enumerated block on some gates but inline append on others — does not pass. |
| C-34 | **Gate Negative Block Subsumes All Subsidiary Failure Predicates** | depth | aspirational | When a gate governs multiple downstream behavioral requirements — item disposition (C-31) and spatial rewrite location (C-32) — the gate's not-cleared block enumerates all subsidiary failure predicates in a single compound list, so gate state is derivable from the block alone without reference to scattered criterion definitions. A gate whose not-cleared block covers undisposed items (C-31) but omits the co-location failure (C-32) passes both C-31 and C-32 individually yet requires cross-reference to determine complete gate state — the block is incomplete as a self-contained gate invariant. Compound coverage requires each subsidiary failure predicate to appear as a discrete item in the not-cleared enumeration: e.g., "(1) any `- [ ]` item remains without a disposition, or (2) any `[FAIL]` annotation does not have its corrected content appearing immediately below it." A gate satisfying C-31 and C-32 but listing only one failure state in its not-cleared block fails C-34. |

## Scoring

```
composite = (essential_pass / 5 * 60)
          + (recommended_pass / 3 * 30)
          + (aspirational_pass / 26 * 10)
```

**Golden threshold**: all 5 essential criteria pass **AND** composite >= 80.

| Band | Composite | Meaning |
|------|-----------|---------|
| Gold | >= 80, all essential pass | Ready for golden file |
| Silver | >= 65, all essential pass | Usable, missing depth |
| Bronze | >= 50, >= 4 essential pass | Partial — gap in core coverage |
| Fail | < 50 or any essential fail | Not useful — regenerate |

## Evaluator Notes

- **C-01 vs C-05**: C-01 requires naming a boundary; C-05 requires a number. A response can pass C-01 without passing C-05 (named boundary but no numeric threshold).
- **C-02 causal chain**: A flat list of "affected components" does not pass C-02. There must be a directional flow showing how throttle at point A causes effect at point B.
- **C-04 burst path**: The burst path must be structurally unprotected — if the output describes a path that is throttled but at a higher limit, that does not count. Look for missing caps, missing retry, or misconfigured concurrency.
- **C-07 cascading**: Two independent throttles on the same request do not constitute a cascade. The second throttle must be triggered *by* the first.
- **C-10 quantified impact**: Estimates are acceptable; the key is that the calculation is grounded in the specific limits cited in C-05, not generic percentages.
- **C-11 vs C-02**: C-02 passes when a causal chain is present in any form. C-11 requires the stronger structural form — a table or row-per-hop structure with mechanism as a discrete column or field. A narrative that satisfies C-02 does not automatically satisfy C-11.
- **C-12 vs C-10**: C-10 passes when the estimate is grounded in stated limits. C-12 requires the arithmetic to be written out step by step with each limit named inline — auditable, not implicit.
- **C-13 vs C-05**: C-05 requires at least one numeric threshold anywhere in the output. C-13 requires a comprehensive upfront inventory covering all named components, with unknowns explicitly flagged.
- **C-14 vs C-08**: C-08 passes with any structured volume-to-behavior mapping. C-14 requires fixed multiplier rows (1×/3×/5×/10×) and a degradation or error-rate column — a free-form tier summary does not pass.
- **C-15 vs C-11/C-13/C-14**: C-11, C-13, and C-14 each evaluate a specific table's internal structure. C-15 evaluates whether *all four* required tables are present and structurally intact. An output can pass C-11, C-13, and C-14 individually while failing C-15 if it omits one of the four tables entirely.
- **C-16 vs C-12**: C-12 requires step-by-step arithmetic with named limits. C-16 requires that the named limits can be traced back to a specific row in the component-limit inventory — the citation must be cross-table, not just internally consistent within the arithmetic section.
- **C-17 vs C-13**: C-13 passes when unknowns are explicitly flagged in any form (including prose notes). C-17 requires the stronger form — a named status token in a designated taxonomy column. "Limit unclear — estimated at ~500/min" satisfies C-13 but not C-17. Only a typed token (`confirmed`, `estimated`, `?-unresolved` or equivalent vocabulary) in a status field satisfies C-17.
- **C-18 vs C-02**: C-02 requires that a causal chain is present in the final output. C-18 requires that the prompt or analysis structure contains an explicit blocking gate that forces correction of structural gaps mid-execution — a gate that is triggered by deficiency, not merely by phase transition. An output that happens to contain a good causal chain satisfies C-02 regardless of how it was produced; C-18 is only satisfied when a named correction gate is present and would have fired if the mechanism were absent.
- **C-19 vs C-15**: C-15 evaluates whether all four required tables are present and structurally intact. C-19 evaluates *where* each table's enforcement notice is placed. An output can pass C-15 (all tables present) while failing C-19 if the ANTI-SUBSTITUTION enforcement is declared globally at the prompt header or swept in a post-hoc verification step rather than appearing inline immediately after each table's column header row. The distinction is production-point enforcement vs. perimeter enforcement.
- **C-20 vs C-18**: C-18 requires a blocking correction gate that names the structural deficiency. C-20 requires the gate to also enumerate specific failing token strings — actual text fragments the model can match against its own output. A gate that fires on "vague mechanism" passes C-18 but fails C-20 unless it names exemplar tokens ("also throttled", "cascade occurs", "propagates to", etc.). Named tokens make the gate machine-auditable; structural descriptions alone leave triggering conditions ambiguous.
- **C-21 vs C-16**: C-16 evaluates whether the output's cross-table references are consistent (no phantoms, citations trace to rows). C-21 evaluates whether the consistency check was performed by an entity with no analytical stake in the tables being audited. An output that achieves cross-table consistency through self-verification — the same analytical role that built the propagation trace also checks it — passes C-16 but fails C-21. Role separation is the mechanism: the auditor entity must have no output in the tables it checks.
- **C-22 vs C-20**: C-20 requires the correction gate to enumerate specific failing token strings. C-22 requires the stronger form — per-token checkbox items where each token receives an individual disposition (`[CLEAR]` or `[FAIL: hop N — rewritten below]`) rather than a collective attestation. A gate with a named token list but a holistic pass/fail verdict passes C-20 but fails C-22. The checkbox format makes each token's disposition independently verifiable and forces failure annotation to name the location.
- **C-23 vs C-21**: C-21 requires that the consistency auditor is structurally separated — a distinct role or phase with no analytical output in the tables being checked. C-23 requires the stronger form — the auditor opens with a structural incapability statement framed as fact, not instruction, establishing why it cannot have authored the content it is auditing. A structurally separated role that proceeds directly to verification without the incapability preamble passes C-21 but fails C-23. The preamble closes the rationalization path that role separation alone leaves open.
- **C-24 vs C-19**: C-19 requires each table's enforcement notice to be placed immediately after the column header separator row with no gap. C-24 applies the same requirement specifically to tables that include pre-filled stub rows in their schema declaration. A schema that inserts stub rows before the enforcement notice creates the same production-point gap as any deferred placement — the stub rows are scaffolding, not output. An output can pass C-19 on all stub-free tables while failing C-24 on a single stub-bearing table. C-24 is the strictest form of at-header placement and subsumes C-19 for tables that contain stubs.
- **C-25 vs C-21**: C-21 requires the consistency auditor to have no analytical output in the tables being audited — role separation is the structural fact. C-25 requires the stronger form — the auditor's mandate affirmatively states the restriction to flags and verdict only. A role with no actual analytical rows in the tables passes C-21 on output evidence; C-25 requires the mandate to pre-declare the restriction, preventing scope creep in future runs. An auditor role that self-limits in a given run but whose mandate does not explicitly prohibit non-flag output passes C-21 but fails C-25.
- **C-26 vs C-23**: C-23 requires the structural auditor to open with an incapability statement framed as fact. C-26 requires the stronger form — the preamble carries an explicit label ("Structural fact" or equivalent) that classifies it as a system-state datum, not an instruction, and that marker appears before the verification mandate. The label prevents the model from treating the incapability statement as a soft orientation remark under token pressure. An unlabeled preamble with correct incapability language passes C-23 but fails C-26.
- **C-27 vs C-18/C-22**: C-18 requires a blocking correction gate; C-22 requires per-token binary attestation within that gate. C-27 requires the gate's blocking condition to be scoped to the correcting role's entire output — not just the next phase boundary. Phase-boundary language ("Phase N does not begin") allows inter-phase output while nominally respecting the gate; role-scoped language ("Role X produces no further output while any item is unresolved") forecloses that gap. A prompt with C-18 and C-22 that uses phase-boundary blocking passes both but fails C-27.
- **C-28 vs C-18/C-19**: C-18 and C-19 evaluate whether correction gates and anti-substitution notices are present and correctly placed — they are agnostic to instruction surface form. C-28 evaluates whether the instructions governing table production are expressed as validity predicates rather than imperatives. A prompt can satisfy C-18 and C-19 using imperative language while failing C-28. Partial adoption does not pass C-28 — the predicate form must be consistent across all table contracts and role output specifications; a prompt that uses predicates for TABLE 1 and imperatives for TABLE 3 fails C-28 even if every individual criterion is otherwise met.
- **C-29 vs C-18/C-27**: C-18 requires a blocking gate; C-27 requires the gate to use role-scoped suspension language. C-29 requires the gate invariant to use predicate form — "GATE N is cleared when and only when…" rather than "Role X produces no further output until…". A prompt that satisfies C-27 (role-scoped language) but frames the invariant as a directive rather than a cleared-when predicate passes C-27 but fails C-29. The distinction is state-declarative vs. instruction-declarative: the predicate form lets an evaluator determine gate status by inspection of the output alone, without interpreting whether a directive was obeyed.
- **C-30 vs C-21/C-25/C-26**: C-21, C-25, and C-26 evaluate structural properties of the auditor role's content — separation, mandate scope, preamble labeling. C-30 evaluates a temporal property: whether the complete specification was declared before any analytical role ran. A prompt where Role 4's specification satisfies C-21, C-25, and C-26 on content grounds but is declared inline at Role 4's execution point passes those criteria but fails C-30. Pre-commitment is the only mechanism that prevents the audit specification from being implicitly calibrated to the analytical content it is meant to audit — role separation (C-21) ensures no analytical stake after the fact; pre-commitment ensures the specification was fixed before the fact.
- **C-31 vs C-29**: C-29 requires the gate to use cleared-when predicate form for the positive condition. C-31 requires the gate to additionally state the negative failure-state invariant explicitly — "no `- [ ]` item may remain undisposed" — rather than leaving the not-cleared condition as an implication of the positive predicate's negation. A gate that satisfies C-29 with a positive cleared-when predicate passes C-29 but fails C-31 unless the failure mode is also directly named. The two forms are complementary: C-29 declares cleared state, C-31 declares not-cleared state. Both must be present for the gate's full state space to be declared without inference.
- **C-32 vs C-22**: C-22 requires per-token binary attestation where failure annotation names the location (hop or section). C-32 requires the stronger form — the rewrite instruction prescribes the spatial location of the correction output relative to the failure annotation: immediately below it in the output stream. A correction gate that satisfies C-22 (the failure annotation names where the deficiency is) but whose rewrite instruction permits the correction to appear elsewhere — in a later phase, in a dedicated correction section, or at end of output — passes C-22 but fails C-32. Spatial co-location makes the correction auditable by linear inspection without reference matching: annotation and corrected content are always adjacent.
- **C-33 vs C-31**: C-31 requires any explicit negative invariant on the gate predicate — inline append form satisfies it. C-33 requires the enumerated-list block form when a gate governs multiple failure conditions. A gate with a single inline negative append ("cleared when X — no Y may remain") satisfies C-31 but fails C-33 when the gate must also cover a second failure condition (e.g., spatial co-location from C-32). The separate block form decouples the failure-state enumeration from the positive predicate sentence, making each failure state independently scannable and the list extensible without rewriting the cleared-when predicate.
- **C-34 vs C-31/C-32**: C-31 and C-32 evaluate whether each subsidiary failure state is declared anywhere in or near the gate. C-34 evaluates whether ALL subsidiary failure states are co-located in the not-cleared block as a compound enumerated list, making the block a self-contained declaration of gate state. A gate can satisfy C-31 (undisposed items declared) and C-32 (spatial co-location prescribed in the rewrite instruction) while failing C-34 if the spatial failure is declared outside the not-cleared block — requiring cross-reference to determine complete gate state. C-34 requires the not-cleared block to be the single source of truth for all gate failure conditions.
