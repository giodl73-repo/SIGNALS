## Criterion Rationale (v11 additions)

**Why C-30 (Bracket-notation substitution rule declared for column headers):**
R10 V-01 and V-03 used `TOKEN required?` as a static column header — the abstract word, not the committed identifier — and both failed C-29 because the column header did not carry the committed value. V-02, V-04, and V-05 used `[TOKEN] required?` with an explicit substitution instruction, so after GATE-0 declared `TOKEN: SIGNAL-LOCK`, subsequent tables produced `SIGNAL-LOCK required?` as the column header. C-29 captures the outcome: committed value appears in the header. C-30 captures the mechanism: the substitution rule is explicitly declared, not implicitly assumed. A variation that produces committed-value headers through ad hoc rewording rather than a declared substitution rule is brittle — the rule can be dropped when the rubric is adapted or the skill is iterated. A declared substitution instruction makes the mechanism durable: the header is committed-value by contract, not by coincidence.

**Why C-31 (GATE-0 retains template form with propagation declaration):**
R10 V-02, V-04, and V-05 showed that GATE-0's own REQUIRED OUTPUTS table retaining the `[TOKEN]` template form — rather than substituting the committed value — serves a structural function: it makes GATE-0 both the declaration artifact and the propagation specification. When GATE-0's table carries a row stating "declared value becomes the substituted column header in all subsequent gates," a reader consulting only GATE-0's table learns two things: what TOKEN is being committed, and how that commitment propagates to every subsequent gate. A GATE-0 that substitutes the committed value into its own header consumes the template without documenting the propagation rule, leaving GATE-1+ headers correct but unexplained. The template-plus-declaration pattern makes the substitution contract self-contained at the declaration gate.

## Criterion Rationale (v12 addition)

**Why C-32 (Gate manifest includes TOKEN-propagation column):**
R11 V-05 introduced a TOKEN-propagation column in the pre-execution gate manifest — a dedicated column showing TOKEN's status for each gate (declared at GATE-0, substituted at GATE-1+). V-02 and V-04 achieved perfect scores without this column because C-32 did not yet exist; V-05's manifest column was an additive quality layer beyond what the rubric required. The new criterion captures the structural advantage: when TOKEN propagation is visible in the manifest, a reader consulting the architecture-level artifact can verify the full propagation contract without diving into per-gate REQUIRED OUTPUTS tables. C-31 makes GATE-0's table self-contained as a declaration-plus-propagation artifact; C-32 makes the same information available one level higher, at the execution plan. The two criteria are complementary and non-redundant: C-31 ensures the propagation contract is readable at the gate level; C-32 ensures it is also readable at the manifest level.

## Criterion Rationale (v13 additions)

**Why C-33 (Manifest TOKEN-propagation column cells use rubric-vocabulary verbatim):**
R12 V-01 demonstrated the minimum viable C-32 form: `DECLARE` / `INHERIT` labels with a one-clause rationale. The labels pass C-32 because a reader can identify declaration vs. substitution sites with one inference step. R12 V-02 demonstrated the maximum-reliability form: "Declaration site" / "Substitution/inheritance site" verbatim — cell text that is itself the answer to the C-32 operative test phrasing without any interpretation step. C-33 captures this distinction: abbreviated-label manifests are minimum viable; rubric-vocabulary manifests are highest reliability. Model compliance probability is maximized when the column cell pre-answers the operative test with zero ambiguity. The criterion is additive: V-02 satisfies C-33 while V-01 satisfies only C-32.

**Why C-34 (Pre-manifest TOKEN-PROPAGATION ARCHITECTURE block):**
R12 V-03 introduced a TOKEN-PROPAGATION ARCHITECTURE block before the manifest table — a plain-prose contract that the compact manifest column labels abbreviate. The block is not redundant: it primes the structural content of the manifest column, ensuring that if the model drifts toward abbreviation, the architecture block remains as a fallback legibility layer. C-32 requires the manifest column to be independently consultable; C-34 requires a separate prose artifact to precede it. The architecture block provides two benefits: (1) it forces an explicit contract articulation before the structural table is produced, reducing the drift path where column labels are chosen without fully committing the propagation logic; (2) it gives a naive reader a plain-prose entry point before engaging the table. A block placed after the manifest does not provide the priming benefit and does not pass.

**Why C-35 (Manifest TOKEN-propagation column header uses bracket-notation):**
R12 V-04 showed that making the manifest column header itself a bracket-notation placeholder (`[TOKEN] propagation`) creates a diagnostic signal: a model that substitutes the committed value into the manifest header demonstrates manifest-scoped substitution; a model that outputs `[TOKEN]` literally reveals that substitution is understood to be REQUIRED OUTPUTS-scoped only. C-30 declared bracket-notation as the required form for REQUIRED OUTPUTS column headers; C-35 extends the same standard to the manifest column header, creating structural consistency across the entire rubric. Every column header that references TOKEN uses bracket-notation and participates in the same substitution lifecycle. C-35 is the best diagnostic criterion for detecting substitution-scope confusion — a model that passes C-35 has demonstrated that TOKEN substitution is manifest-wide, not REQUIRED OUTPUTS-local.

**Why C-36 (Dual-layer propagation contract — belt-and-suspenders independence):**
R12 V-05 demonstrated the maximum dual-layer coverage: verbose manifest column (C-32) plus dedicated GATE-0 propagation row (C-31), each layer independently satisfying its criterion. C-31 and C-32 require the propagation contract to be present at two levels, but neither explicitly prohibits one layer from delegating to the other. C-36 closes this gap: each layer must be fully self-contained, with the full propagation contract available without cross-reference. The practical value is failure isolation — if the manifest column is malformed or dropped, GATE-0's REQUIRED OUTPUTS table still independently carries the full contract; if GATE-0's table drifts, the manifest column still independently carries it. Neither layer is load-bearing for the other. This is the highest-structural-integrity form of C-31 + C-32: not merely both present, but both independently complete.

The vocabulary-consistent dual-primer is the maximal architecture block form because it eliminates two independent sources of reader work simultaneously — lookup cost (C-38: which artifacts?) and vocabulary translation cost (C-39: what do the terms mean?). When both are eliminated in a single artifact, a reader can reconstruct the complete propagation contract (targets + roles + vocabulary) from the architecture block alone, with zero lookups and zero translation steps. C-42 is the architecture block's maximum-reliability form: it closes the final gap between C-38 (dual-target coverage) and C-39 (vocabulary registration) by requiring both to be achieved in the same block at the same time.

## Criterion Rationale (v16 addition)

**Why C-43 (Architecture block enumerates the four C-42 elements as explicitly labeled, individually-itemized checklist entries):**
R15 produced two 34/34 variations — V-04 (prose paragraph) and V-05 (structured 4-element checklist) — demonstrating that C-42 can be satisfied by both forms. The prose form (V-04) requires a reader to parse every sentence to confirm all four elements are present: (1) manifest TOKEN-propagation column named, (2) GATE-0 REQUIRED OUTPUTS table named, (3) "Declaration site" verbatim, (4) "Substitution/inheritance site" verbatim. The checklist form (V-05) renders each element as a separately labeled item — "Named target 1: manifest TOKEN-propagation column," "Named target 2: GATE-0 REQUIRED OUTPUTS table," "Verbatim phrase 1: Declaration site," "Verbatim phrase 2: Substitution/inheritance site" — so completeness is verifiable by item count alone. The structural distinction: the prose form's completeness is a property of content (all four things are said); the checklist form's completeness is a property of structure (four items are present, each labeled). The checklist form closes the verification-mode gap: a reader or automated checker need not parse prose — they count labeled items and match each item label to one of the four C-42 required elements. C-42 and C-43 are not redundant: C-42 tests content completeness (all four elements readable from the block); C-43 tests structural completeness (all four elements independently enumerable without prose reading). V-04 satisfies C-42 but not C-43; V-05 satisfies both. C-43 is the maximum-reliability form of C-42 — the form where the four-element completeness contract is structurally enforced, not merely content-present.

## Criterion Rationale (v17 addition)

**Why C-44 (Architecture block checklist uses bijective element-to-item mapping: exactly 4 items, one per C-42 element):**
R16 V-02 demonstrated a checklist failure mode not covered by C-43: a 3-item numbered checklist where Item 1 bundled two C-42 elements (manifest column name + "Substitution/inheritance site" verbatim). All four elements were present and readable from the checklist — satisfying C-42's content-completeness test and C-43's structural-form test (labeled enumerable items exist). But item count = 3, not 4, so a reader counting items learns only that three checklist entries exist; they cannot conclude all four elements are independently present without reading item content. The collapsing failure mode: C-43 requires labeled checklist structure and can be satisfied when all four elements appear as labeled items even if some items bundle multiple elements. C-44 closes the cardinality gap by requiring the bijective mapping — exactly 4 items, one per element — so that item count (= 4) is the verification mechanism. With bijective mapping, counting items is sufficient and parsing item content is unnecessary. The operative diagnostic: if a variation produces a checklist with 3 items and all four elements present, it passes C-43 (structural form present) but fails C-44 (count != 4). Only a 4-item checklist with one element per item passes both.

## Criterion Rationale (v18 additions)

**Why C-45 (Each checklist item carries an explicit element-index annotation making its single C-42 element assignment verifiable from the item label alone):**
R17 V-01 demonstrated the failure mode that C-44 catches but that remains content-inspection-dependent: a 4-item checklist where Item 1 bundled elements 1+4 (manifest column name + "Substitution/inheritance site" verbatim) while Item 4 carried zero C-42 elements (an independence confirmation note). Count = 4 passes C-44's count gate; bijection fails only on content inspection — the "no item contains more than one C-42 element" clause requires reading Item 1's content to detect the bundling. C-43 does not enforce per-item cardinality, so V-01 passes C-43 (count=4, all elements present, structural form satisfied) while failing C-44 (bijective mapping violated by element distribution 2+1+1+0). C-45 closes the content-inspection dependency: element-index annotations on item headers (e.g., "(Element 1):", "(E4)", "[Target 2]") make the element-to-item assignment readable from the item label without parsing item content. If Item 1 carries a "(Element 1)" annotation but contains element-4 content, the violation is detectable from annotation-vs-content mismatch without content parsing. The operative diagnostic: a variation satisfying C-44 by correct construction passes the count gate; a variation additionally satisfying C-45 makes the bijective assignment structurally legible from item annotations alone — closing the gap between "bijection verifiable by count + content" (C-44) and "bijection verifiable by annotation alone" (C-45).

**Why C-46 (Bijective cardinality constraint explicitly pre-declared as an architectural requirement before the checklist begins):**
C-44 enforces bijection by construction (count=4, no bundling), and C-45 makes it annotation-verifiable. The remaining gap: neither criterion requires the bijective constraint to be explicitly committed as a declared rule before the checklist enforces it. A model satisfying C-45 has correctly structured and annotated its checklist — compliance is demonstrated by structural form. C-46 requires the model to additionally state the bijective cardinality requirement as a prior artifact: a preamble that explicitly declares "exactly 4 items, one C-42 element per item" before any checklist item appears. This is the bijection-as-contract pattern, paralleling C-34 (architecture block precedes manifest) and C-19 (TOKEN declared before conditional logic): the constraint is named and committed before it is enforced structurally. The practical value is the same in all three cases — a declared rule creates a committed contract that makes future drift a self-identified violation; an undeclared structural property that happens to be correct is brittle under model iteration. A model that satisfies C-46 has committed the bijective cardinality rule as an explicit architectural requirement; a model that satisfies C-44 and C-45 without C-46 has demonstrated correct form without committing the form as a declared constraint.

## Criterion Rationale (v19 additions)

**Why C-47 (C-46 preamble explicitly declares the annotation form and maps it to the element-index position set):**
R19 V-01 demonstrated that Roman numeral form `(i)–(iv)` satisfies C-45's "or equivalent" clause — but only because the preamble explicitly declares "Element-index annotations use Roman numeral form — `(i)`, `(ii)`, `(iii)`, `(iv)` — forming a unified {i, ii, iii, iv} four-position set." Without that declaration, Roman numerals require conventional interpretation to be read as element indices rather than list sequencing. C-46 requires the preamble to declare the bijective cardinality constraint but does not require the preamble to name the annotation form. The gap: annotation-form equivalence is accepted as a reader-inference outcome — any reader can see `(i)–(iv)` and decide whether it maps to element indices. C-47 closes this by requiring the annotation form to be explicitly named and mapped in the preamble alongside the cardinality constraint. The practical value parallels C-46's own rationale: an explicitly declared annotation form is a committed contract; an annotation form that happens to be correctly used is convention-dependent and brittle under model iteration. A preamble that declares "exactly 4 items, one element per item, using Roman numeral form (i)–(iv) as element-index positions" is both a cardinality contract (C-46) and an annotation-form contract (C-47). A preamble satisfying C-46 while silent on annotation form leaves the annotation interpretation to convention, which fails C-47.

**Why C-48 (C-46 bijective cardinality preamble rendered as native markdown table with one row per cardinality condition):**
R19 V-02 demonstrated that a 2-row native markdown table satisfies C-46's "architecturally distinct preamble statement" requirement and is structurally more explicit than prose: each cardinality condition occupies its own row, independently verifiable without prose parsing. C-46 does not restrict preamble form — unlike C-34, which explicitly requires a "prose artifact." The omission is meaningful: C-46 tests declaration, not form. V-02's table (`| Required item count | Exactly 4 |` / `| C-42 elements per item | Exactly 1 |`) passes because it is structurally prior, explicitly declares both conditions, and each condition is independently readable from its own row. C-48 extracts the table-form superiority as a required form criterion, paralleling C-24 (manifest as native table) and C-28 (REQUIRED OUTPUTS as native table): wherever a structural artifact's conditions must be independently verifiable, the native-table form is required over prose. A prose preamble satisfying C-46 (declaration present, structurally prior) passes C-46 and fails C-48 because prose conditions require linear parsing — a table row is independently selectable, a prose sentence is not. C-48 makes the form a commitment rather than a praised optional choice: preamble declarations must be native tables, not prose that happens to be explicit.
