## Purpose

Score a `flow-dataflow` skill output: a data lineage trace covering source, transformation stages,
and destination for a Finance, Operations, or Commerce pipeline scenario.

---

## Essential Criteria (weight: 60 pts total)

| ID | Criterion | Category | Pass Condition |
|----|-----------|----------|----------------|
| C-01 | **Data lineage chain** | correctness | Every in-scope item has an unbroken source => stage(s) => destination chain. No item appears at a destination without an identified origin. |
| C-02 | **Boundary error handling** | correctness | Each boundary is annotated: either the mechanism (retry, dead-letter, rollback) or an explicit "no handling" flag. Silence is not acceptable. |
| C-03 | **Data loss point identification** | coverage | At least one concrete loss point is named per pipeline stage. Generic "errors may occur" does not qualify. |
| C-04 | **Schema state at each stage** | correctness | Schema at every stage entry/exit is described or diffed. A stage that changes field names or types with no schema note fails. |

---

## Recommended Criteria (weight: 30 pts total)

| ID | Criterion | Category | Pass Condition |
|----|-----------|----------|----------------|
| C-05 | **Latency characterization** | depth | Every stage has a latency annotation: value, range, or characterization (real-time / batch / daily). |
| C-06 | **Stale data windows** | depth | At least one stale window quantified. Failure-mode staleness addressed separately from normal-operation staleness. |
| C-07 | **Domain framing** | format | Domain-specific entity names appear in the lineage chain. Purely generic trace fails. |

## Aspirational Criteria (weight: 10 pts total) — **40 criteria in v15**

Weight per criterion: 10 / 40 = **0.25 pts**. PARTIAL = 50% of criterion weight (0.125 pts).

| ID | Criterion | Category | Pass Condition |
|----|-----------|----------|----------------|
| C-08 | **Recovery prescriptions** | depth | Every "no handling" annotation from C-02 and every loss point from C-03 has a paired, specific recovery suggestion. |
| C-09 | **Pattern trade-off analysis** | behavior | Named alternative pattern with at least two quantified or qualified trade-off dimensions. |
| C-10 | **Pre-trace domain context gate** | format | A DOMAIN CONTEXT section appears before the first stage block, locking entity names, downstream consumer, and business cadence. At least two of those terms reappear verbatim in subsequent lineage or stale-window content. |
| C-11 | **Interleaved boundary gates** | behavior | A boundary check block appears between every consecutive stage pair — inline, not post-hoc. A trailing BOUNDARY SUMMARY alone does not satisfy this criterion. |
| C-12 | **Domain entity exposure per boundary** | format | Every boundary check that satisfies C-11 names a domain entity from the scenario vocabulary. "data" or "records" alone does not pass. |
| C-13 | **Pre-declared staleness contract** | depth | The stale threshold for at least one entity is declared as an explicit contract in DOMAIN CONTEXT before Stage 1 is written. The stale analysis derives its rows by evaluating actual elapsed time against this threshold. A stale window asserted or synthesized after the stages does not pass. |
| C-14 | **Additive elapsed time calculation** | depth | Latency annotations across stages and boundaries are accumulated into an explicit end-to-end elapsed total. This total appears in or immediately before the stale analysis, enabling direct numeric comparison against C-13's threshold. Per-stage annotations alone with no cumulative total do not pass. |
| C-15 | **Incumbent or manual-process baseline** | behavior | A named baseline section describes the incumbent or manual process this pipeline replaces. The baseline must surface at least one vocabulary term that reappears in C-10 DOMAIN CONTEXT, and must be referenced in C-08 (recovery prescriptions) or C-09 (trade-off analysis) to close the comparison loop. A baseline not connected to C-08 or C-09 does not pass. |
| C-16 | **Cross-role reference chain** | behavior | Each role or section explicitly cites a named output from at least one prior role or section — not a generic back-reference ("as established above") but a named citation (e.g., "Role 1's DOMAIN CONTEXT", "INCUMBENT BASELINE from Role 2"). A section that introduces vocabulary or analysis without attributing it to a prior named source does not pass. |
| C-17 | **Immutability declaration on pre-committed values** | depth | At least one pre-committed contract value (such as the staleness threshold from C-13) is accompanied by an explicit prohibition on post-trace revision. The prompt text must state that the value cannot be changed after a specified point (e.g., "This threshold is fixed. You may not revise it after Stage 1."). A request for early declaration alone — as required by C-13 — does not satisfy this criterion. |
| C-18 | **Incremental boundary elapsed computation** | behavior | The cumulative elapsed total required by C-14 is computed as a named field within each BOUNDARY CHECK block, incrementally updated at every boundary as a structural side effect of writing that block. A cumulative total that appears only in a post-stage summary section does not pass, even if numerically correct. |
| C-19 | **Machine-verifiable citation target convention** | format | A formal citation convention — a consistent prefix, schema, or required field (e.g., `ARTIFACT:` prefix, `R-xx` IDs, `Citing:` row) — is declared and applied uniformly to every named handoff target in the variation. Every citation made by any subsequent role or section uses this convention, making references verifiable by token match rather than prose judgment. A variation that names handoff targets using inconsistent or ad hoc labels does not pass. C-16 checks whether citations are named; C-19 checks whether citation targets are created via a machine-parseable convention applied without exception. |
| C-20 | **Stage-write progression gate** | behavior | The prompt contains an explicit write-order prohibition: Stage N+1 may not be written until the boundary block between Stage N and Stage N+1 is fully complete with all required fields present. The prohibition must be stated as a sequencing constraint that names the gate condition (e.g., "Do not write Stage N+1 until this block is complete"). C-11 requires interleaved boundary block placement; C-20 requires an explicit deferral prohibition that makes the boundary block a structural prerequisite for stage advancement. A prompt that instructs boundary block placement without a write-order prohibition does not pass. |
| C-21 | **Binary freshness verdict per boundary** | depth | Each BOUNDARY CHECK block contains a required declarative verdict field (FRESH / STALE or equivalent binary) derived by comparing the cumulative elapsed total against the pre-declared threshold from C-13. C-14 requires the elapsed total to exist; C-18 requires it to be computed incrementally; C-21 requires that each boundary produce an explicit pass/fail verdict, converting the elapsed computation into a decision gate. A boundary block that records elapsed time without a verdict field does not pass. A verdict that appears only in the final stale analysis does not satisfy this criterion. |
| C-22 | **Formal pre-declared registry table** | format | A registry table — mapping every artifact to a machine-parseable token (e.g., `[A-xx]`) with at minimum an owner and description column — is produced before any role output begins. The table must enumerate all artifacts that will be cited downstream: a prefix declaration (C-19) or a flat label list satisfies token convention but does not satisfy this criterion. Every citation target named in any subsequent role must appear as a row in this table. A citation to a target not in the table does not pass. The table form makes citation targets first-class, enumerable, and column-verifiable independent of prose context. |
| C-23 | **Phase gate self-verification checklists** | format | Each phase gate artifact (e.g., BOUNDARY BLOCK CHECKLIST) contains a self-verification checklist with individually ticked or checkable items confirming completion of the prior structural requirements that gate progression. A phase gate artifact that narrates completion in prose without a structured checklist of individually verifiable items does not pass. |
| C-24 | **Upfront constraint section with inline callback syntax** | format | A STRUCTURAL CONSTRAINTS section appears before any role output and declares constraints using an inline callback syntax — each constraint specifies its scope of application inline (e.g., `[Apply SC-x at every boundary block]`). A constraint section that declares rules without binding them to structural application points via inline callback syntax does not pass. |
| C-25 | **Phase gate artifacts as first-class registry rows** | format | Every phase gate artifact (e.g., [A-05] BOUNDARY BLOCK CHECKLIST, [A-08] TRADE-OFF REVIEW) appears as a named row in the ARTIFACT REGISTRY with a token, owner, and description. Phase gate artifacts that exist in the output but are absent from the registry, or that are referenced only by prose description, do not pass. |
| C-26 | **Non-natural role ordering as citation stress test** | behavior | The role ordering declared in STRUCTURAL CONSTRAINTS includes at least one non-natural (non-sequential) assignment — a role at a later ordinal position that must cite an artifact produced by a role at an earlier non-adjacent position, creating a skip-level citation obligation. A variation where every role cites only the immediately preceding role (natural sequential ordering) does not satisfy this criterion. |
| C-27 | **Named recovery formula anchoring incumbent baseline** | depth | The ARTIFACT REGISTRY description field for the recovery prescriptions artifact explicitly names the recovery formula structure, anchoring it to the incumbent baseline artifact token (e.g., `Fall back to [A-01] if [condition]`). A registry row with a generic description or a recovery body that does not reference the incumbent baseline token by formula does not pass. |
| C-28 | **Named-column boundary block schema** | format | A BOUNDARY BLOCK SCHEMA standalone section appears before role outputs, declaring the required column headers (or bold label sequence for prose variants) using exact string tokens that also appear as compliance markers in REGISTER DECLARATION Part A. An evaluator can verify column completeness by token match against this schema without heuristic output inspection. A schema embedded inline in role instructions, or one whose field names differ from Part A marker strings, does not pass. |
| C-29 | **Trade-off analysis with baseline token citation** | behavior | The trade-off analysis artifact is declared as a required output in the ARTIFACT REGISTRY, and the prompt imposes an explicit citation obligation: the trade-off section must cite the incumbent baseline artifact token (e.g., `[A-01]`) and the domain context artifact token (e.g., `[A-02]`) by token. A trade-off section that discusses alternatives without citing these anchors by token does not pass. |
| C-30 | **Output register declaration with format-specific field-compliance markers** | format | REGISTER DECLARATION Part A lists, for each declared output field or column, a format-specific compliance marker — a token string (column header, bold label, or equivalent) that evaluators can match by string scan to verify field presence in output artifacts. A register declaration that names fields without declaring compliance markers, or where markers are implied rather than explicitly enumerated, does not pass. |
| C-31 | **Pre-declared boundary block schema section** | format | A BOUNDARY BLOCK SCHEMA section appears as a standalone named section before the first role output begins, declaring the complete field sequence for boundary blocks. This section must be self-contained and referenceable by name. An inline description of boundary block fields within role instructions — without a standalone named section — does not pass. |
| C-32 | **Register-specific compliance marker mapping per structural field** | format | REGISTER DECLARATION Part A provides a per-field compliance marker mapping: for every declared output field, a distinct compliance marker is assigned (column header token, bold label, or equivalent register-format token). The mapping is exhaustive — no declared field is left without a compliance marker. An evaluator can check compliance for any field by looking up its marker in Part A without inspecting output structure. |
| C-33 | **Register-invariant declaration for non-tabular output registers** | format | A REGISTER INVARIANTS section (or equivalent named section in REGISTER DECLARATION Part B) declares the complete set of required structural tokens for non-tabular output formats (e.g., all required bold labels). Every token in this set must appear at every output location where the register applies. A non-tabular register that uses structural tokens without a pre-declared invariant set does not pass. For tabular registers throughout, this criterion is vacuously satisfied. |
| C-34 | **Per-boundary incumbent equivalent field** | depth | Every boundary block contains an Incumbent Equivalent field (column or bold label) that identifies the specific incumbent step being replaced and its duration, cited by the incumbent baseline artifact token (e.g., `[A-01]: [named step + duration]`). A boundary block whose incumbent equivalent field contains only a generic reference ("see baseline") or omits the token citation does not pass. |
| C-35 | **INCUMBENT TOTAL as pre-trade-off closing artifact** | depth | An INCUMBENT TOTAL artifact appears immediately before the trade-off analysis section, collecting per-boundary incumbent equivalent values into an additive breakdown by role with a Grand Total row. The trade-off section must cite this artifact's token to make the comparison token-verifiable. A trade-off section that refers to incumbent costs without a dedicated INCUMBENT TOTAL artifact does not pass. |
| C-36 | **Baseline-first artifact ordering** | format | The ARTIFACT REGISTRY ordering and prompt structure place the incumbent baseline artifact (e.g., [A-01]) as the first artifact produced, before any stage or boundary output. An SC (e.g., SC-11) explicitly declares this ordering constraint. A prompt that allows the baseline to be written at any point, or that places baseline production after stage output begins, does not pass. |
| C-37 | **REGISTER DECLARATION Parts A/B as single-location authority** | format | REGISTER DECLARATION is structured as two named parts: Part A declares output format compliance markers; Part B declares structural invariants. Together, Parts A and B serve as the single authoritative reference for all format-compliance and field-completeness decisions. Compliance marker definitions or structural invariants stated elsewhere without being consolidated in Parts A/B do not pass. An evaluator resolves all format-mode disputes by consulting Parts A/B exclusively. |
| C-38 | **Skip-level citation requirement in non-natural role ordering** | behavior | When a non-natural role ordering is present (satisfying C-26), the prompt contains an explicit SC that imposes a skip-level citation obligation on the non-adjacent role: the later role must cite a specific named artifact from the earlier non-adjacent role by token. The SC must be declared in STRUCTURAL CONSTRAINTS and enforced via a Phase Gate 2 verification item. |
| C-39 | **Skip-level SC names governed role in body** | format | The skip-level citation SC (satisfying C-38) explicitly names the governed role — the role carrying the skip-level citation obligation — within the SC's body text. A skip-level SC that describes the obligation without naming the specific governed role does not pass. |
| C-40 | **Skip-level SC declares position distance in body** | format | The skip-level citation SC explicitly states the position distance of the skip — the number of ordinal positions separating the governed role from the role whose artifact must be cited (e.g., "two positions: Commerce pos 3, Finance pos 1"). A skip-level SC that names the roles without specifying position distance does not pass. |
| C-41 | **Phase Gate 2 skip-level verification item cites governing SC by identifier** | format | The Phase Gate 2 verification checklist contains at least one item that cites the skip-level citation SC by its identifier (e.g., `[SC-12]`) rather than by description. A Phase Gate 2 item that refers to skip-level citation obligations by prose description without naming the SC identifier does not pass. |
| C-42 | **C-37 + C-41 closed verification chain as co-occurrence requirement** | behavior | REGISTER DECLARATION includes a closed-chain paragraph that explicitly links Parts A/B (C-37) and the Phase Gate 2 SC citation (C-41) as a co-occurring navigable set. The closed-chain paragraph must name both the REGISTER DECLARATION authority and the Phase Gate 2 verification path, so an evaluator consulting the REGISTER DECLARATION can navigate to both artifacts. A REGISTER DECLARATION that names Parts A/B without referencing the Phase Gate 2 citation chain, or vice versa, does not pass. |
| C-43 | **All three skip-level SC attributes co-present in single SC block** | format | The skip-level citation SC must declare all three required attributes — governed role name (C-39), position distance (C-40), and the enforcement mechanism (citation obligation) — in a single SC block without distributing them across separate SC entries, inline notes, or supplemental sections. A skip-level obligation distributed across multiple locations does not pass even if all three attributes are individually present elsewhere. |
| C-44 | **SC-13 BASELINE CLOSURE as named upfront constraint with dual inline callbacks** | behavior | A named SC (e.g., SC-13 BASELINE CLOSURE) appears in STRUCTURAL CONSTRAINTS declaring an explicit dual citation obligation: the incumbent baseline artifact token must appear in both the recovery prescriptions artifact header and the trade-off analysis artifact header. Inline callbacks at both section headers enforce this obligation by token. C-15 baseline closure achieved through execution discipline — without a named SC and without inline callbacks at both headers — does not pass. |
| C-45 | **SC-14 FORMAT MODE DECLARATION with criterion substitution map** | format | For variations using a non-tabular output register, a named SC (e.g., SC-14 FORMAT MODE DECLARATION) appears in STRUCTURAL CONSTRAINTS containing a criterion substitution table that maps each affected aspirational criterion by its exact criterion ID (e.g., C-28, C-30, C-32) to its prose-mode equivalent. An evaluator determines compliance for each affected criterion by criterion-ID lookup in the substitution table, without inspecting output structure. An SC-14 that enumerates affected criteria by description rather than by criterion ID does not pass. For tabular variants (no non-tabular registers present), this criterion is vacuously satisfied. |
| C-46 | **SC-13 as named navigation entry in REGISTER DECLARATION closed-chain paragraph** | format | The REGISTER DECLARATION closed-chain paragraph must explicitly name SC-13 BASELINE CLOSURE as a navigation path, identifying it as the governing authority for baseline-closure failures. An SC-13 declared only in STRUCTURAL CONSTRAINTS without a corresponding closed-chain paragraph entry in REGISTER DECLARATION does not pass — baseline-closure failures must be discoverable from the register index, not only from a sequential scan of STRUCTURAL CONSTRAINTS. The closed-chain paragraph must be updated at SC-13 introduction time to add SC-13 as a named path alongside Parts A/B and SC-12. |
