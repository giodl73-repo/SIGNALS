**Two new criteria extracted from R9:**

**C-34 — Per-boundary incumbent equivalent column with token-cited baseline derivation** (`depth`)
Every boundary block carries an `Incumbent Equiv.` column (or declared conversational marker) citing the incumbent baseline by registry token with the named manual step and its duration. Cell form: `[token]: [step + duration]`. Omission or bare assertion without token citation fails. Closes the gap where C-15 requires the baseline to surface at recovery/trade-off time but never at each boundary checkpoint — with C-34, the pipeline-vs-manual latency delta is auditable at every gate.

**C-35 — INCUMBENT TOTAL as a required pre-trade-off closing artifact** (`behavior`)
The prompt mandates a named INCUMBENT TOTAL section produced immediately before TRADE-OFF ANALYSIS. It must sum all C-34 cells with additive breakdown by role, and its registry token must be cited in the TRADE-OFF ANALYSIS as the numeric comparison endpoint. A trade-off that cites the baseline artifact directly — or only prose — without a computed INCUMBENT TOTAL fails. Closes the gap where per-boundary equivalents accumulate with no required aggregation: C-35 converts the baseline comparison from narrative judgment to numeric, token-verifiable measurement.

## Purpose

Score a `flow-dataflow` skill output: a data lineage trace covering source, transformation stages,
and destination for a Finance, Operations, or Commerce pipeline scenario.

## Essential Criteria (weight: 60 pts total)

| ID | Criterion | Category | Pass Condition |
|----|-----------|----------|----------------|
| C-01 | **Data lineage chain** | correctness | Every in-scope item has an unbroken source => stage(s) => destination chain. No item appears at a destination without an identified origin. |
| C-02 | **Boundary error handling** | correctness | Each boundary is annotated: either the mechanism (retry, dead-letter, rollback) or an explicit "no handling" flag. Silence is not acceptable. |
| C-03 | **Data loss point identification** | coverage | At least one concrete loss point is named per pipeline stage. Generic "errors may occur" does not qualify. |
| C-04 | **Schema state at each stage** | correctness | Schema at every stage entry/exit is described or diffed. A stage that changes field names or types with no schema note fails. |

| C-08 | **Recovery prescriptions** | depth | Every "no handling" annotation from C-02 and every loss point from C-03 has a paired, specific recovery suggestion. |
| C-09 | **Pattern trade-off analysis** | behavior | Named alternative pattern with at least two quantified or qualified trade-off dimensions. |
| C-10 | **Pre-trace domain context gate** | format | A DOMAIN CONTEXT section appears before the first stage block, locking entity names, downstream consumer, and business cadence. At least two of those terms reappear verbatim in subsequent lineage or stale-window content. |

A request for early declaration alone — as required by C-13 — does not satisfy this criterion. |

Every citation made by any subsequent role or section uses this convention, making references verifiable by token match rather than prose judgment. A variation that names handoff targets using inconsistent or ad hoc labels does not pass. C-16 checks whether citations are named; C-19 checks whether citation targets are created via a machine-parseable convention applied without exception. |

The prohibition must be stated as a sequencing constraint that names the gate condition (e.g., "Do not write Stage N+1 until this block is complete"). C-11 requires interleaved boundary block placement; C-20 requires an explicit deferral prohibition that makes the boundary block a structural prerequisite for stage advancement. A prompt that instructs boundary block placement without a write-order prohibition does not pass. |

---

Two new criteria added. The progression:

- **C-34** closes the gap where C-15 requires an incumbent baseline to be referenced at recovery or trade-off time but imposes no per-boundary obligation. Without C-34, the pipeline-vs-manual comparison is deferred to the end of the trace; with C-34, every boundary block carries a token-cited incumbent step, making latency advantage (or disadvantage) visible at every checkpoint.
- **C-35** closes the gap where per-boundary incumbent equivalents (C-34) accumulate with no required aggregation endpoint. Without C-35, the closing trade-off section may assert pipeline superiority narratively without quantified support. With C-35, the prompt mandates a named INCUMBENT TOTAL artifact — with additive breakdown and registry token — that the trade-off section must cite, converting the baseline comparison from judgment-based to computation-verifiable.
