## Purpose

Score a `flow-dataflow` skill output: a data lineage trace covering source, transformation stages,
and destination for a Finance, Operations, or Commerce pipeline scenario.

## Essential Criteria (weight: 60 pts total)

| ID | Criterion | Category | Pass Condition |
|----|-----------|----------|----------------|
| C-01 | **Data lineage chain** | correctness | Every in-scope item has an unbroken source => stage(s) => destination chain. No item appears at a destination without an identified origin. |
| C-02 | **Boundary error handling** | correctness | Each boundary is annotated: either the mechanism (retry, dead-letter, rollback) or an explicit "no handling" flag. Silence is not acceptable. |
| C-03 | **Data loss point identification** | coverage | At least one concrete loss point is named per pipeline stage. Generic "errors may occur" does not qualify. |
| C-04 | **Schema state at each stage** | correctness | Schema at every stage entry/exit is described or diffed. A stage that changes field names or types with no schema note fails. |

| ID | Criterion | Category | Pass Condition |

| C-06 | **Stale data windows** | depth | At least one stale window quantified. Failure-mode staleness addressed separately from normal-operation staleness. |
| C-07 | **Domain framing** | format | Domain-specific entity names appear in the lineage chain. Purely generic trace fails. |

## Progression Notes

**C-39** closes the gap where C-38 requires the skip-level SC to name the required token but imposes no requirement on role scoping. Without C-39, an SC can name the token globally — applying to all roles — and a variation appears compliant while the constraint is structurally ambiguous about which role it targets. With C-39, the SC body must name the governed role explicitly, making the constraint role-targeted and its application scope verifiable without reading the full role sequence.

**C-40** closes the gap where C-38 requires a citation gap of two or more positions but the SC body is not required to state that distance. Without C-40, an evaluator must count role positions from the sequence to verify that the cited token is sufficiently far back. With C-40, the gap size is declared in the SC body — the skip distance is verifiable from the constraint declaration alone, independent of sequence length or role ordering.

**C-41** closes the gap where C-38 requires a Phase Gate 2 checklist item to verify the skip-level token but does not require the item to name the SC it enforces. Without C-41, a failed checklist item points to a missing token but not to the governing constraint — an evaluator must search the constraint section to determine which SC was violated. With C-41, the checklist item cites the SC number, creating a direct navigation path from failed output to governing declaration.

**C-42** closes the gap where C-37 and C-41 each independently close one failure-navigation path, but neither criterion requires the other to be present. Without C-42, a variation can satisfy C-37 (field-format failures self-locate) while leaving skip-level gate failures unannotated, or satisfy C-41 (skip-level failures self-locate) while leaving field-format failures distributed across BOUNDARY BLOCK SCHEMA and role instructions. With C-42, both paths must be closed simultaneously — the evaluator-search problem is eliminated for every class of failure mode, not just one.

**C-43** closes the gap where C-39, C-40, and C-41 together ensure each skip-level SC attribute is present somewhere in the variation, but no criterion requires all three to coexist in the same SC block. Without C-43, a prompt can satisfy all three by distributing the role boundary label to a footnote, the position distance to a role instruction callback, and the SC number only to the Phase Gate item — each criterion passes in isolation while the SC block itself is incomplete. With C-43, the SC block must be self-contained: all three attributes must be readable from the constraint declaration alone, making the skip-level specification verifiable at a single location.
