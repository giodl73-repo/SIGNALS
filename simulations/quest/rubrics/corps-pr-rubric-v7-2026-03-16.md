## Four new criteria from Round 6

**C-21 — Net direction column + derived Budget verdict** *(C-19 derivation gap)*
R6 V-01 added a `Net` column (WORSE/BETTER/NEUTRAL) per ledger row and required Budget verdict to explicitly cite which rows are WORSE/BETTER. Without it, the verdict is a standalone assertion; with it, it's a derivable consequence traceable row by row.

**C-22 — Dual-clause Phase C entry: C1 pre-flight checklist + C2 per-role gate** *(C-20 single-clause gap)*
R6 V-02 split the single entry condition into two independently auditable sub-conditions. C1 failure blocks the whole phase; C2 failure is locatable to the specific role that omitted the ledger citation. Conflating both into one clause satisfies C-20 but fails C-22.

**C-23 — Per-role IA READ RECEIPT output block before any findings** *(C-20 observability gap)*
R6 V-03 made the read prerequisite observable in the output, not just declared in the pipeline. A missing receipt is a visible structural gap; a pipeline violation requires reading the declaration to detect. C-23 requires per-role post-hoc verifiability.

**C-24 — Domain-Lens column in findings table** *(C-15 output-visibility gap)*
R6 V-03 made gate compliance visible as a named column per finding. Without it, the only evidence the gate was applied is finding quality itself. C-24 requires the gate result to appear in the output structure.

**C-23 — Each technical role section begins with a named IA READ RECEIPT output block before any findings** *(closes C-20 per-role observability gap)*

C-20 requires the pipeline entry condition to declare the IA read prerequisite. C-22 requires it to be a named dual-clause sub-condition. Both are satisfied by pipeline-level declarations — a reviewer verifies compliance by reading the declaration, not the output. R6 V-03 showed a different mechanism: each technical role section begins with a named IA READ RECEIPT block that is output (not declared), appearing before any finding is written. The receipt names the section read, restates the Budget verdict and Budget strength, identifies the most relevant cost row for this role's domain, and states the role's initial position on that row. Because the receipt is output, C-20 compliance is verifiable post-hoc per role: a reviewer can check each receipt block against Phase B content. A missing receipt is a visible output gap; detecting a pipeline violation requires reading the declaration. C-23 requires per-role observability: the read prerequisite must appear as an output artifact, not only as a pipeline declaration.

**C-24 — Per-role findings are output as a named-field table with an explicit Domain-Lens column showing gate compliance (Passed) per finding** *(closes C-15 output-visibility gap)*

C-15 requires the domain-lens validity check to be an explicit rewrite gate with a binary test and rewrite consequence. But C-15 does not require the gate result to be visible in the output — a run can apply the gate correctly and leave no structural trace of it. R6 V-03 showed a stronger pattern: findings are output as a table with a Domain-Lens column marking each finding as "Passed." This makes gate compliance independently auditable per finding: a reviewer can identify any finding missing its Domain-Lens entry without re-running the gate. Without the column, the only evidence the gate was applied is finding quality itself — there is no structural indicator. C-24 closes this: gate compliance must be observable as a named column in the output, not only inferrable from finding content.

## What changed from v6

**C-21 closes the C-19 derivation gap.** C-19 requires the IA's cost framing to use a named-row, named-column ledger with Budget verdict and Budget strength as terminal fields. R6 V-01 showed a tighter pattern: the ledger includes a third column for Net direction (WORSE / BETTER / NEUTRAL) per row, and the Budget verdict is declared as a direct derivation from those row directions -- it must name which rows are WORSE and which are BETTER and state whether adoption is justified given the balance. Without the Net column, the Budget verdict can draw conclusions not traceable to the individual cost rows; the verdict becomes a standalone assertion rather than a derived summary. C-21 requires the Net column to be present so each row's contribution to the verdict is independently auditable, and requires the Budget verdict to explicitly cite row directions rather than assert a conclusion.

**C-22 closes the C-20 single-clause gap.** C-20 requires the pipeline entry condition for the technical role phase to explicitly name IA phase completion and the read dependency. R6 V-02 showed a tighter pattern: the entry condition is split into two named sub-conditions -- C1 (pre-flight Phase B exit checklist, checked once before Phase C begins) and C2 (per-role IA read prerequisite, checked per role after each section is written). C1 and C2 have different verification mechanisms and different failure granularities: a C1 failure blocks the entire phase; a C2 failure is locatable to the specific role that omitted the ledger citation. A single entry condition that conflates both requirements satisfies C-20 but fails C-22.

**C-23 closes the C-20 per-role observability gap.** C-20 and C-22 are satisfied by pipeline-level declarations -- a reviewer verifies compliance by reading the declaration, not the output. R6 V-03 showed a different mechanism: each technical role section begins with a named IA READ RECEIPT output block before any finding is written. Because the receipt is output (not declared), C-20 compliance is verifiable post-hoc per role by inspecting the text: a missing receipt block is a visible structural gap, not a pipeline violation. C-23 requires this per-role observability: the read prerequisite must appear as an output artifact in each role section, not only as a declared architectural contract.

**C-24 closes the C-15 output-visibility gap.** C-15 requires the domain-lens gate to be an explicit binary test plus rewrite consequence. But C-15 does not require the gate result to be visible in the output -- a run can apply the gate correctly and leave no structural trace. R6 V-03 showed a stronger pattern: findings are output as a table with a Domain-Lens column marking each finding as "Passed." This makes gate compliance independently auditable per finding: a reviewer can verify each row without re-running the gate, and identify any finding missing its Domain-Lens entry. Without the column, the only evidence that the gate was applied is the quality of the findings themselves. C-24 requires gate compliance to be observable as a named field in the output structure.
