## Scoring Formula

```
composite = (essential_pass/5 * 60) + (recommended_pass/3 * 30) + (aspirational_pass/39 * 10)
```

Essential each = 12 pts | Recommended each = 10 pts | Aspirational each = 0.256 pts

## Essential Criteria (60 points total)

| ID | Text | Category | Weight | Pass Condition |
|----|------|----------|--------|----------------|
| C-01 | **Role-operation matrix** -- The output produces a matrix of roles against operations (Create, Read, Write, Delete, Append, AppendTo, Assign, Share at minimum). Every cell must be filled: Granted, Denied, Conditional, or Not Applicable. | correctness | essential | A complete matrix is present. Every cell has an explicit value. Listing roles without operations, or operations without roles, fails C-01. |
| C-02 | **FLS coverage with explicit null case** -- The output checks whether field-level security (column security profiles) is configured for sensitive fields, names which fields are covered and which are not, and explicitly states when no FLS is configured. Absent FLS on sensitive fields is flagged as a gap. | coverage | essential | Every sensitive field in scope has an explicit FLS status (Configured / Not Configured). A system with no FLS is explicitly noted, not silently omitted. Absent FLS on sensitive fields appears in the gap inventory. |
| C-03 | **Record scope per role** -- The output assigns a record access scope to each role: own-records-only, business unit, parent-child BU, org-wide, or team-scoped. | correctness | essential | Every role in the trace has an associated access scope. Ambiguous or unresolved scope is flagged as a gap, not silently omitted. |
| C-04 | **Privilege escalation path detection** -- The output identifies at least one privilege escalation path (e.g., team membership granting broader role, sharing rule bypassing FLS, environment-level admin override). | coverage | essential | At least one escalation path is named with the specific mechanism: which team, role, or rule creates the escalation and what elevated access results. |
| C-05 | **Security gap inventory** -- The output produces a named list of gaps: missing FLS on sensitive fields, team membership holes, sharing rule conflicts, or over-permissioned roles. | coverage | essential | Every identified gap has a name, a type, and a brief description. The inventory is a discrete section, not inline commentary. |

## Design Notes

**C-37 vs C-34 and C-36**: C-34 requires the preamble to contain a STRUCTURAL AXIS NON-INTERFERENCE DECLARATION with labeled axis rows (A-1/A-2/A-3). C-36 requires the CA terminal verdict to contain a TRI-DIMENSIONAL SELF-EVIDENCE ATTESTATION block. C-37 tests the labeling bridge: the attestation rows must use the A-N labels from the declaration as their primary identifiers, closing the preamble->verdict loop under a single label scheme.

**C-38 vs C-12 and C-35**: C-12 requires SHALL obligations to be stated. C-35 requires CA-1.9 to exist and be distinct from CA-1.4/CA-1.7. Neither tests whether the SHALL obligation that owns SE-4 (SHALL-D) carries a named extension sub-clause for the C-35 sub-criterion, or whether CA-1.9's preamble-anchor paragraph cites that extension sub-clause label. C-38 tests exactly this inheritance chain.

**C-39 vs C-22 and C-35**: C-22 requires phase labels in the output body to make execution order self-attestable. C-35 requires CA-1.9 to contain distinctness language. C-39 requires the phase-mandate block to name CA-1.9 and declare its non-overlap with CA-1.4/CA-1.7 -- making scope verifiable from the phase boundary without reading the CA-1.9 row.

**C-37, C-38, C-39 are independent**: C-37 requires C-34+C-36 (axis label cross-link). C-38 requires C-12+C-35 (SHALL extension chain). C-39 requires C-22+C-35 (phase-mandate distinctness echo). A variation can satisfy any subset. R13-V-05 satisfies C-38+C-39 but fails C-37.

**C-40 vs C-38**: C-38 requires SHALL-D (SE-side escalation-path obligation) to carry a named extension sub-clause, and CA-1.9 to cite it. C-40 mirrors this symmetrically on the CS-side: SHALL-F carries a named extension sub-clause (SHALL-F-EXT-CS2CS3), and CA-1.6 cites it. The two criteria together create bilateral sub-clause parity across SE and CS enforcement chains.

**C-41 vs C-34 and C-37**: C-34 requires the preamble to contain a STRUCTURAL AXIS DECLARATION with A-N labels. C-37 requires attestation rows to use those labels. C-41 tests whether the declaration block itself carries a binding rule mandating the A-N label usage -- converting C-37 from an external scoring condition into a self-declared preamble contract.

**C-42 vs C-39**: C-39 requires the Phase 3 mandate to name CA-1.9 and declare its distinctness from CA-1.4 and CA-1.7. C-42 extends this to a complete CA-1.N scope roster in the Phase 3 mandate, followed by a standalone non-overlap statement as a separate structural element below the roster.

**C-40, C-41, C-42 are independent**: C-40 requires C-38+C-30 (CS-side sub-clause parity). C-41 requires C-34+C-37 (self-enforcing preamble mandate). C-42 requires C-39+C-35 (roster-level non-overlap). R14 demonstrated each in isolation.

**C-43 vs C-42 and C-41**: C-42 requires the Phase 3 mandate to contain a CA-1.N scope roster plus a standalone non-overlap statement as a separate structural element below the roster.
