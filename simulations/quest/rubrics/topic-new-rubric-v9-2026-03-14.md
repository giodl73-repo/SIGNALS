**Gap map** — what each new criterion closes:

**C-29** closes the gap between gate-declarative and consequence-informative overviews. C-24 + C-26 ensure all gate conditions are declared and consumed before Phase 1. C-29 requires the overview to also carry a per-row consequence column ("If Skipped" or equivalent) so the cost of each bypass is visible at the same architectural moment the gate is encountered. V-04 was the first variation with consequence framing at the overview layer — "If skipped" as a named column alongside Phase | Purpose | Produces | Exit Gate.

**C-30** closes the gap between structural independence and semantic independence at enforcement points. C-28 requires two discrete checkbox items. C-30 requires the independence of those items to be labeled as an explicit instruction — "Check these two items independently / do not advance until both are checked separately" — so they cannot be rationalized as a sequential pair. V-03 demonstrated two directive layers: a lead label and a closing enforcement line.

**C-31** closes the gap between valid-vocabulary enforcement and high-visibility placement of the hardest failure mode. C-04 catches substitution after the fact. C-31 requires Priority to be the leftmost column so the field most likely to receive "high/medium/low" is encountered before namespace or skill values anchor any framing. V-05 demonstrated Priority | Namespace | Skill | Item Name | Owner Role | Stakeholder Ref.

**Full criterion definitions**:

**C-29 -- Pipeline overview table carries a per-row consequence column** (`structure`): Sharpens C-24. Requires the pipeline overview table to include a per-row consequence column alongside the Exit Gate column, stating the cost of bypassing each phase at the architectural declaration layer. A prompt satisfying C-24 via a gate-only overview table without a consequence column does not satisfy C-29.

**C-30 -- Gate independence stated as an explicit semantic instruction** (`structure`): Sharpens C-28. Requires the fill-in table exit gate to carry an explicit label naming the independence of its conditions at the enforcement point — e.g., "Check these two items independently" followed by "Do not advance until both are checked separately." Two discrete checkboxes satisfy C-28 but can still be rationalized as sequential; the explicit instruction closes that gap. A prompt satisfying C-28 without an independence instruction does not satisfy C-30.

**C-31 -- Priority column placed first in signal table column order** (`structure`): Sharpens C-04. Requires the Priority column to be the leftmost column in the signal planning table, placing the hardest-to-catch failure mode at the highest-visibility fill position before any other field can anchor contextual framing that might induce vocabulary substitution. A prompt satisfying C-04 with a non-leading priority column does not satisfy C-31.

**C-30 -- Gate independence stated as an explicit semantic instruction** (`structure`)
Sharpens C-28. C-28 requires the fill-in table exit gate to list column completeness as a separately stated condition from the minimum row count -- two discrete checkbox items. C-30 requires the independence of those conditions to be made explicit as a semantic instruction at the enforcement point itself -- for example: "Check these two items independently" and "Do not advance until both are checked separately." Two physically proximate checkboxes can still be treated as a sequential pair and rationalized past; an explicit independence instruction names the evaluation mode, transforming checklist proximity into a sequencing constraint that cannot be interpreted as "check one, then the other." V-03 demonstrated this as a bold "Check these two items independently" label preceding the checkboxes, followed by "Do not advance until both items are checked separately" as a closing enforcement statement -- two directive layers, not one. A prompt satisfying C-28 via two distinct checkbox items without an explicit independence instruction does not satisfy C-30.

**C-31 -- Priority column placed first in signal table column order** (`structure`)
Sharpens C-04. C-04 requires every signal priority value to be one of the canonical vocabulary terms: essential / recommended / optional. C-31 requires the Priority column to be placed as the leftmost column in the signal planning table -- making the hardest-to-catch failure mode the highest-visibility field. A model populating a signal table from left to right encounters the priority field before any other field is filled; namespace, skill, and role values cannot yet anchor contextual framing that might induce "high/medium/low" substitution. A priority column positioned after namespace and skill fields inherits context that makes substitution more likely; priority-first eliminates that context at fill time. V-05 demonstrated this as Priority | Namespace | Skill | Item Name | Owner Role | Stakeholder Ref column ordering. A prompt satisfying C-04 via valid-vocabulary enforcement with a right-positioned priority column does not satisfy C-31.
