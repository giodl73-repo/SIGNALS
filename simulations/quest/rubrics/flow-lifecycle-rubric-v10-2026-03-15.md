**v10 changelog**: Added C-31–C-33 from Round 9 excellence signals.

C-31 captures the dual-mechanism threshold operationalization pattern — the structure where the Decision Condition column header carries both C-29 mechanisms simultaneously rather than choosing one. V-01 and V-02 each implement one mechanism: V-01 enumerates threshold type categories ("dollar amount, day count, retry count, percentage threshold"); V-02 provides a quoted passing example (`"Order value > $25,000"`). V-03 applies both in the same header, revealing that the mechanisms are not substitutable but complementary: taxonomy without example leaves format ambiguous (a practitioner knows that "day count" qualifies but not how to express it), and example without taxonomy leaves category-boundary ambiguous (a practitioner can match the example's format but cannot judge whether a novel threshold type qualifies). Only the combination closes both gaps simultaneously. A rubric can pass C-29 with either mechanism alone, but C-31 requires both. A rubric that passes C-31 necessarily satisfies C-29.

C-32 captures the bidirectional exception-catalog gate pattern — the structure where sequential gates bracket both boundaries of the exception catalog, not just one. V-01 implements only the upstream gate (state trace → exception catalog); V-02 implements only the downstream gate (exception catalog → terminal declaration); V-03 implements both. C-30 requires coverage at ≥1 exception-catalog boundary, which V-01 and V-02 each satisfy. But C-30 leaves the unbounded direction open: V-01's downstream boundary is unguarded (exception flows could be updated after terminal declarations are partially authored), and V-02's upstream boundary is unguarded (exception flows could be drafted before the state trace is final). C-32 requires both boundaries to be gated, fully closing the exception catalog as a structural node. A rubric cannot pass C-32 without first passing C-30.

C-33 captures the pre-schema lifecycle inventory pattern — the Step 0 block that appears before the role registry to establish problem-space framing from named entry triggers before any schema section begins. V-04 and V-05 both add this block. The distinction from C-16 (phase entry triggers) and C-01 (state entry triggers) is temporal: those triggers appear within their respective schema sections, after roles and phases are already partially defined, while Step 0 enumerates the full set of lifecycle initiation conditions as a foundation that role scope, phase boundaries, and state definitions can all be anchored against. A practitioner authoring the role registry after Step 0 can verify that every trigger source is represented in the registry; a practitioner authoring the phase map can verify that every entry trigger routes to a named phase. Aspirational because the inventory requires domain-specific enumeration that may not be available at rubric-authoring time, but when present it substantially tightens the binding between initiation conditions and schema structure.
