**Key evaluator distinctions (v12 additions):**

- **C-37 vs C-35**: C-35 tests whether the bilateral gate mechanism is packaged as a self-contained named block (BOUNDARY PROTOCOL or equivalent) at each boundary, making gate coverage verifiable by counting section headers against the known phase-boundary count. C-37 tests whether a named PROTOCOL COUNT MANIFEST (or equivalent pre-commitment table) appears at Phase B entry, listing all boundary names, gate names, and gate states before any boundary is traversed inside Phase B. - **C-38 vs C-36**: C-36 requires the three-part inertia-contrast baseline to be established in a named section (STATUS QUO COST or equivalent) that precedes all per-entity analytical content in Phase A. | C-37 | **Protocol Count Manifest Dual-Surface Verification** | structure | aspirational | At Phase B entry, the output contains a named PROTOCOL COUNT MANIFEST (or equivalent label) -- a table with one row per phase boundary -- that pre-commits the boundary name, gate name, and gate state for every transition before any Phase B sub-section is written. This creates a second verification surface for C-35: gate coverage is testable by (1) counting BOUNDARY PROTOCOL section headers against known phase-boundary count (C-35 surface) AND (2) counting manifest rows against the same boundary count (C-37 surface). A boundary present in the header scan but absent from the manifest, or present in the manifest but missing a corresponding header, is a cross-document inconsistency detectable at Phase B entry without reading any phase body interior. An output that satisfies C-35 with N BOUNDARY PROTOCOL section headers but contains no pre-committed manifest table at Phase B entry PASSES C-35 and FAILS C-37. The dual-surface design means a discrepancy between header count and manifest row count is itself a structural failure signal, even when each surface alone appears complete. |
| C-38 | **Cost Ledger Table Machine-Countability** | behavior | aspirational | The named STATUS QUO COST section required by C-36 expresses the three-part inertia-contrast baseline as a table (COST LEDGER or equivalent) with exactly one row per part -- row (a) for the current schema condition, row (b) for the dependent business process, row (c) for the accumulation trajectory. Completeness is then verifiable by row count alone: a table with fewer than three filled rows has a visible structural gap without reading cell contents. Prose-form three-part framing satisfies C-36 but requires reading to confirm all three parts are present and distinct; the table format makes a missing part a count failure visible at a glance. An output that satisfies C-36 with a named pre-Q1 STATUS QUO COST section whose three-part structure is expressed in prose paragraphs or as a bulleted list without a table PASSES C-36 and FAILS C-38. The row-per-part table applies the same machine-countability principle to inertia-baseline completeness that C-35 applies to gate-block completeness and C-37 applies to manifest-row completeness. |
| C-39 | **Cross-Role Inertia Chain in Phase B** | behavior | aspirational | The Phase B domain section (B2 or equivalent) contains a three-part inertia-contrast example that explicitly names a cross-role causal chain: a shared infrastructure condition (schema state, migration step, or system dependency) whose disruption produces direct, named consequences in at least two distinct domain roles (e.g., Operations substrate condition → Commerce order-processing consequence AND Finance reconciliation consequence). C-30 is satisfied by any Phase B example distinct from the Phase A example. C-39 is satisfied only when the Phase B example links a single infrastructure failure to consequences in at least two domain roles, such that the Operations substrate condition must hold for both downstream role consequences to follow -- creating dependency evidence no single-role inertia section can produce. An output whose Phase B domain section names only one domain role's consequence -- even with a different step and numeric threshold than the Phase A example -- PASSES C-30 and FAILS C-39. The cross-role chain forces the analyst to reason from shared infrastructure state rather than per-role risk in isolation; a consequence named for only one role does not demonstrate the substrate dependency. |
