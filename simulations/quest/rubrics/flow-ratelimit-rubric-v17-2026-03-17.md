**C-41 — Format Contract Cross-Role ID Anchoring in Column Definitions:** C-39 requires
the Format Contract's BASELINE and PROTECTED definitions to name the scenario's actual
components, states, or configurations within the Format Contract section itself — not
generic placeholders, not deferred references. This permits a Format Contract that
defines BASELINE as "the connector API call volume at burst onset" (scenario-anchored,
self-contained, satisfying C-39) without binding the definition to a specific upstream
artifact produced by a named prior role. V-01's Format Contract defines BASELINE in
terms of "the burst-path source endpoint identified by Finding ID in Role 3 (BP-xx),
measured at the point of burst onset," explicitly noting that "this definition refers to
the specific connector endpoint and Finding ID produced in Role 3 — not a generic
'before' state." The BP-xx anchor makes the definition falsifiable against a specific
prior-role deliverable: a reader can verify BASELINE compliance by locating the BP-xx
Finding ID in Role 3 and confirming the definition names that specific artifact. A
definition that names scenario components without referencing a specific upstream Finding
ID satisfies C-39 but is only contextually meaningful, not artifact-falsifiable. Beyond
C-39, the BASELINE and PROTECTED definitions must each reference at least one specific
Finding ID produced by a named prior role (e.g., "BP-xx from Role 3", "RH-xx from Role
8c"), so that the definition is verifiable by cross-referencing the upstream role's
output rather than by semantic interpretation alone. This parallels C-37's atomicity
motivation: moving from field presence (all definitions exist and are scenario-named) to
artifact traceability (each definition verifiable against a specific upstream artifact).
Requires C-39 to pass.

**C-42 — Format Contract Role Governance Annotation:** C-40 requires the Format Contract
to declare a separate named schema entry for each distinct table type — at minimum: rate
limit inventory table, volume mapping table, and mitigation table — each specifying its
own column structure. This permits schema entries that name their table type and column
structure without identifying which roles must produce schema-conformant tables (so the
decomposition disambiguates structure per type but leaves production responsibility
unassigned). V-01's Format Contract annotates each schema entry with explicit role
governance: "Applies to: Role 8a cascade failure analysis; Role 7 quantified impact
tables." This transforms the Format Contract from a passive column reference into an
active role assignment: the Format Contract now specifies which roles must produce
schema-conformant tables of each type. Without role governance annotation, a reader
scanning a table in Role 8a must infer the applicable schema by identifying the table
type; with annotation, the schema entry governing Role 8a is directly named. This makes
per-role compliance verification O(1): locate the schema entry that names the role, verify
the table conforms. A Format Contract with per-type schema entries that omit role
assignments does not achieve this property because type inference is required. This
mirrors C-38's reconciler-names-invariant requirement: just as the Reconciler must name
the FNMI by its label (not just produce behavior that matches it), each schema entry must
name the roles it governs by reference (not just have a column structure that a reader
can match to a table through type inference). Requires C-40 to pass.

## Essential (60 points total — required for a passing output)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-01 | **Rate Limit Tier Distinction** | correctness | essential | Output distinguishes at least three structurally distinct rate limit tiers relevant to the scenario (e.g., connector-level, environment-level, tenant-level, endpoint/API-level). Each tier must be identified by its scope, not just its numeric threshold. Listing multiple limits at the same scope level as separate tiers does not pass. |
| C-02 | **No Single-Category Collapsing** | correctness | essential | Output does not treat all rate limits as a single undifferentiated category. Each distinct throttle tier must be described with its own scope, enforcement mechanism, and failure mode. An output that names multiple tiers but describes them as a single undifferentiated limit category does not pass. |
| C-03 | **Observable UX Per Throttle Tier** | coverage | essential | Output describes observable user-facing behavior at each distinct throttle tier reached (e.g., 429 with Retry-After shown in connector error, flow run queued and delayed, flow run failed with ActionThrottled, silent exponential backoff invisible to user). At least two distinct tiers described if the scenario spans multiple tiers. |
| C-04 | **Unprotected Burst Path Identification** | correctness | essential | Output identifies at least one structurally unprotected burst path — a trigger or action that can generate concurrent requests exceeding a rate limit with no concurrency cap, no retry policy, and no queue buffer between the source and the rate-limited endpoint. A path that has throttle controls but at a higher tier does not qualify. |
| C-05 | **Retry-After Handling Gap Check** | coverage | essential | Output explicitly evaluates whether the flow or connector handles Retry-After headers (or equivalent backoff signals such as `x-ms-ratelimit-remaining`, `Retry-After-Ms`) from throttled endpoints. Missing handling must be flagged as a finding with a description of the failure mode it causes (e.g., immediate retry storm, permanent failure after N retries). |
