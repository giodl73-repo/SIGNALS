## Four new criteria extracted from Round 18

**C-51 -- IA section carries explicit adversarial designation label**

C-11 required the Inertia Advocate to be sequenced first and structured as a reference object. R18 showed all five variations achieving C-11 -- and V-05's "Status-quo champion" framing revealed that the IA section's adversarial function can be declared at the label level, not only in the section's content. C-51 closes this gap: the IA section header carries an explicit adversarial function designation (e.g., "Status-quo champion [C-11]:") distinct from the role name -- a label that declares the section's structural purpose without requiring the reader to read section content. C-11 established that the IA must be sequenced first and output as a reference object; C-51 requires that its adversarial function be declared at the opening label of the section, making the pipeline role of the IA section auditable by label-reading alone.

**C-52 -- Pipeline sub-condition descriptions carry inline criterion cross-references**

C-22 required Phase C entry to be structured as two independently auditable sub-conditions (C1 and C2). R18 showed all five variations satisfying C-22, and the evidence revealed that the sub-condition description text carried inline criterion annotations -- e.g., "each producing an exact result token [C-26]" -- cross-referencing the criterion that governs the result token format. C-52 closes this gap: each sub-condition's pipeline declaration text carries a bracketed criterion annotation citing the criterion that governs its output form, making the governing criterion traceable from the pipeline declaration text without consulting the rubric. C-22 established that two named sub-conditions must exist; C-52 requires each sub-condition to declare which criterion governs its output form.

**C-53 -- Named output structural elements carry inline governing criterion annotations**

C-25 required Budget verdict to be expressed as a three-clause labeled formula, and C-28 required each clause on its own output line. R18 showed all five variations satisfying both, and the evidence revealed that each clause line carried an inline end-of-line annotation citing its governing criterion -- e.g., "<- B3 line 1 [C-25]" following "WORSE rows: [list]". C-53 closes this gap: named structural output elements carry bracketed governing criterion IDs as inline end-of-line annotations, so each element's compliance requirement is traceable by line-scan without consulting the rubric. At minimum, each Budget verdict clause (required by C-25/C-28) must carry its governing criterion annotation. C-25 and C-28 established the formula and line-separation requirements; C-53 requires the formula lines to annotate which criterion governs them.

**C-54 -- C2 RESULT validation instructions name both acceptable evidence forms for field (e) as explicit disjuncts**

C-32 required READ RECEIPT to explicitly cross-reference PRE-COMMITMENT for fields (d) and (e). R18 showed all five variations satisfying C-32, and the evidence revealed that the C2 RESULT validation instruction for field (e) named both acceptable forms explicitly: "valid: inline value OR cross-reference 'see PRE-COMMITMENT block: [row]'". C-54 closes this gap: when C-32 and C-39 are both active, the C2 RESULT validation instruction for field (e) names both acceptable evidence forms as explicit disjuncts -- inline value and named cross-reference to PRE-COMMITMENT block -- in the template text. A reviewer auditing field (e) can determine what counts as sufficient evidence from the validation instruction alone without consulting C-32 or C-39. C-32 established the cross-reference requirement; C-54 requires the template to make both valid forms of that cross-reference (inline and named reference) explicit in the C2 RESULT.

## Design Intent

The core tension this rubric guards against is **silent selection + generic output**: corps-pr's entire value is that reviewer routing is traceable to the diff and that each reviewer sounds like their role. A run that selects reviewers without showing why, or produces findings that any generic reviewer could write, fails two essential criteria independently -- making it impossible to pass golden without fixing both.

The IA thread (C-11 through C-54) enforces a second tension: **post-hoc rationalization vs. pre-committed position**. The PRE-COMMITMENT + READ RECEIPT + C2 RESULT chain exists to make it structurally impossible for a role to observe findings before committing its initial position on the IA's cost framing. Each version tightens one link in that chain.

The structural enforcement hierarchy (introduced in v12) partitions criteria across three independent axes -- pipeline exit condition, block-level, and table-column level -- so each gap closes at its natural structural depth without interference from adjacent criteria. C-44 (v13) closes the alignment gap between the declared hierarchy and the template. C-45 and C-46 (v14) close the declaration-elevation gap: C-43's prohibition must live in the pipeline declaration, and C-44's path elimination must be declared as an explicit negation. C-47 and C-48 (v15) close the named-entry gap: C-46's negation must be a named independent sub-entry, and the CLOSED PATHS block must cross-reference that sub-entry. C-49 and C-50 (v16) close the reverse-direction gap: the Eliminated sub-entry must carry a back-reference to CLOSED PATHS, and the pipeline must carry an explicit symmetry declaration confirming the loop is bidirectionally traversable.

C-51 through C-54 (v17) close the **annotation and designation layer**: the IA section's adversarial function must be declared at the label level (C-51); pipeline sub-condition descriptions must cite governing criteria inline (C-52); named output elements must carry governing criterion annotations for line-scan traceability (C-53); and C2 RESULT validation instructions must name both acceptable evidence forms to prevent false fails (C-54). Together these four criteria make the skill self-annotating: an auditor reading any named element in the output or pipeline text can identify which criterion governs it without consulting the rubric separately.

## Essential Criteria (60 pts total)

| ID | Text | Weight | Category | Pass Condition |
|----|------|--------|----------|----------------|
| C-01 | Reviewer selection is disclosed with a files-changed-to-role mapping | essential | transparency | Output contains a reviewer selection section that maps each changed file (or file group) to the org.yaml role selected to review it, with an explicit reason for each selection. A reviewer list with no file mapping is not acceptable. A role that appears in the output but whose selection cannot be traced to a specific file or files-changed signal is a hard fail. |
| C-02 | Every selected role produces its own findings section with P1/P2/P3 severity on each finding | essential | format | Output contains one findings section per selected role. Each finding within a section carries an explicit severity label: P1 (blocking), P2 (major), or P3 (minor). A role section that lists observations without severity labels fails. A run with fewer findings sections than selected reviewers fails -- every selected reviewer must produce output. |
| C-03 | A consensus analysis section synthesizes across all role findings | essential | coverage | Output contains a dedicated consensus analysis section that appears after all per-role findings. The section must: (a) identify where roles agree on a finding, (b) identify where roles disagree, and (c) surface the most critical finding across all roles. A run with per-role sections but no cross-role synthesis is a hard fail. A summary that only restates individual findings without identifying agreement or disagreement fails. |
| C-04 | Output ends with an explicit go/no-go recommendation with justification | essential | correctness | Output contains a go/no-go recommendation section that states one of: GO, NO-GO, or GO WITH CONDITIONS. The recommendation must name the primary reason (e.g., "NO-GO: P1 finding from security-architect requires resolution before merge"). A run that omits the recommendation or leaves it ambiguous ("the team should decide") fails. A run with a recommendation but no justification fails. |
| C-05 | Domain-lens gate includes binary test plus a rewrite consequence, not a naming contract | essential | correctness | The skill's domain-lens gate includes: (a) a binary test -- "would only this role raise this finding given their domain?" -- and (b) an explicit rewrite consequence -- "if the answer is NO, revise the finding to name a specific mechanism owned by this role's domain; if still NO after revision, drop the finding." A gate that enforces naming precision (file, line, function) without the binary test and drop consequence fails. A specificity contract does not substitute for the domain-lens test. |

## Scoring Formula

```
composite = (essential_pass / 5 * 60)
          + (recommended_pass / 3 * 30)
          + (aspirational_pass / 46 * 10)
```

**Golden threshold**: all 5 essential criteria pass AND composite >= 80.

*aspirational denominator increased from 42 to 46 in v17; each criterion is worth 10/46 ~= 0.217 pts; a perfect run still scores 100.*

PASS = full pts / PARTIAL = half pts / FAIL = 0
