---

**Gap logic per criterion:**

- **C-49** — C-15 requires a stakeholder-risk opener that enumerates role categories and risk exposure before signal planning begins. The Round 17 PASS+ cluster (V-03/V-05) went further: the opener explicitly named the document's override mission — "These are the defaults this skill overrides. Every block that follows exists to prevent these defaults from applying." C-15 is satisfied when stakeholders are enumerated and risks named before signal planning. C-49 is satisfied only when the opener frames the entire prompt as an inertia-override construct — making the document's purpose explicit to a model reading it for the first time, not merely its contents. A opener satisfying C-15 (stakeholder enumeration + risk exposure) but omitting the override-mission declaration leaves the document's purpose implicit: the model infers from the content that defaults are being overridden, but has no explicit forward frame for why the document exists. The override-mission declaration is to the prompt what C-09's commit gate is to the strategy: it names the decision the document serves.

- **C-50** — C-26 requires an explicit read-before-execute directive at the pipeline overview ("Read this entire table before executing Phase 1"). V-01 and V-02 added a prohibition clause: "Do not begin Phase 1 until you have read every row." C-26 converts the overview from passive reference to required read. But a directive that says "read this" operates in instructive register — it can be treated as advisory under token pressure, skipped as ambient context. C-50 closes this gap by requiring the prohibition form: "Do not begin Phase 1 until you have read every row." The prohibition converts the read from a recommendation into a structural gate — non-compliance is named as an error state, not just a missed suggestion. The same logic governs C-30 above C-28 (two conditions must be stated; C-30 adds the independence instruction) and C-26 above C-24 (overview exists; C-26 adds the read directive). C-50 is the prohibition-layer analog for C-26.

- **C-51** — C-30 requires an explicit independence instruction at the fill-in table exit gate ("Check these two items independently") followed by a closing enforcement statement ("Do not advance until both are checked as separate acts"). V-05 used full command register: "Stop. Do not advance until you have run BOTH of the following checks as separate acts." plus "Record both results." Two distinctions above C-30: (1) "Stop." opens in command register — a single-word imperative that halts execution before the instruction is read, rather than embedding the independence requirement in a clause the model may process as ambient; (2) "Record both results." adds an output obligation — the model must produce a record of both check outcomes before advancing, making the check auditable at the gate rather than performed silently. A gate satisfying C-30 (two conditions + independence instruction + closing enforcement) without "Stop." and without a recording obligation satisfies C-30 but not C-51.

- **C-52** — C-47 requires a concrete worked example of sequential-treatment failure output — a specific string or table state that sequential checking would accept but independent checking would reject. V-05 framed the example in second person: "Output you accepted: 'S-01 | Product Manager | | ' — SK-03 and SK-04 empty on all rows. This output is malformed." The second-person framing positions the malformed output as something the model itself produced and accepted, enabling self-recognition: the model can match its own actual output against the labeled example directly. First-person or third-person framing of the same example satisfies C-47 but requires the model to abstract from the example to its own output — to reason "does what I produced resemble what sequential treatment produces?" Second-person framing closes this abstraction gap by addressing the model as the agent that accepted the malformed output. C-52 is the framing-layer analog of C-47's example-layer requirement: C-47 requires an example exists; C-52 requires the example be addressed to the model as its own output.

- **C-53** — C-43 requires "This phase overrides IR-NN." at each phase header — a declarative third-person annotation. V-05 used command register: "Stop. You are overriding IR-NN:" — opening with "Stop." imperative, then second-person present progressive ("you are overriding") rather than declarative third-person ("this phase overrides"). The effect is structural: the declarative form is an ambient note; the command register form is a decision point. A model reading "This phase overrides IR-01" may process it as context-setting prose. A model reading "Stop. You are overriding IR-01:" must halt, register itself as the agent performing the override, and consciously proceed. C-53 is the command-register analog for phase-entry override directives, parallel to C-51 for independence check gates: both require "Stop." imperative at the structural moment where a critical decision should be consciously made rather than passively recorded.

- **C-54** — C-47 requires a concrete worked example of sequential-treatment failure output. V-03 and V-05 presented this as a labeled BEFORE/AFTER comparison — a table or structured block with a "sequential path (wrong)" row or column showing the output sequential checking accepted, and an "independent path (correct)" row or column showing the condition independent checking would have caught. C-47 is satisfied by any concrete output example (a specific string: "'3 rows, columns empty' reads as compliant — this is wrong"). C-54 requires the structural-contrast format: showing not only what sequential treatment accepted but also what independent checking would have rejected it on, side by side. The contrast table makes the failure-to-correct trajectory visible by column inspection — the model sees both the wrong outcome and the correct detection condition without having to hold both in narrative context simultaneously. C-52 and C-54 are independent sharpenings of C-47: a prompt can have the BEFORE/AFTER contrast table without second-person framing (satisfying C-54 but not C-52), or have second-person framing in a narrative without a BEFORE/AFTER table structure (satisfying C-52 but not C-54).

- **C-55** — C-48 requires a named PHASE CONSEQUENCE REGISTRY (PCR) with stable PCR-NN IDs; the pipeline overview's consequence column cites PCR-NN IDs rather than embedding prose directly. V-04 went further: PCR consequence entries cross-cite the relevant failure-example registry entry (FER-NN) in their downstream consequence text — "skip failure recognizable by FER-01 output inspection." C-48's PCR defines what consequence follows a phase skip; a consequence definition alone tells the model what breaks downstream. It does not tell the model how to recognize that the break has occurred. The FER-NN cross-citation closes this detection gap: a model reading a PCR downstream consequence entry has a structural pointer to the exact output-inspection artifact that makes the skip consequence recognizable at validation time. Without the cross-citation, consequence and failure-example are each present in the prompt but structurally disconnected — the model must cross-reference them by content inference. The cross-citation converts inference into a mechanical link: PCR-01 names the consequence; PCR-01's downstream field cites FER-01 where the detectable output state lives.

## v15 — two new criteria from R16 excellence signals

| ID | Name | Sharpens | Source |
|----|------|----------|--------|
| C-47 | Independence gate provides a concrete worked example of sequential-treatment failure output — a specific output string that sequential checking would accept but independent checking would reject — making the failure recognizable by output inspection rather than by reasoning about what sequential treatment produces | C-33 | R16 C-33 PASS+ cluster (V-01/V-02/V-03) |
| C-48 | Pipeline consequence column entries are backed by a named PHASE CONSEQUENCE REGISTRY (PCR-NN IDs); the pipeline overview cites PCR-NN IDs in its consequence column rather than embedding prose directly; consequence definitions live in one authoritative location and are auditable by reference rather than by content matching | C-29 | R16 V-02/V-05 PCR citation pattern |

**Aspirational denominator (v15)**: 38 → 40

---

**Gap logic — v15 criteria:**

- **C-47** — C-33 requires the independence gate to name the specific sequential-treatment failure mode (e.g., "Sequential checking produces row-count acceptance without column-completeness verification"). Naming a failure mode is necessary but not sufficient: the model can acknowledge the named pattern abstractly without having a recognizable output target to match against when checking its own work at the gate. C-47 closes the **failure-recognition gap**: a concrete worked example shows the model exactly what wrong-but-seemingly-valid output looks like at the moment of checking. The example — e.g., "'3 rows, columns empty' reads as compliant — this is wrong" — makes the failure detectable by output inspection rather than by abstract reasoning about what sequential treatment produces. A gate that names the failure mode (satisfying C-33) but provides only the abstract name and no concrete output example does not satisfy C-47.

- **C-48** — C-29 requires the pipeline overview to carry a consequence column, and C-32 requires those entries to use first-person framing. Both criteria are satisfied when consequence prose is embedded directly in the pipeline overview cells. But direct embedding creates a **consequence-drift channel**: when a consequence changes, it must be updated everywhere it appears — in the pipeline overview, in any phase narrative that references it, and in any review or scoring rubric that quotes it. Drift between display sites is invisible to structural inspection. C-48 closes this channel by requiring a named PHASE CONSEQUENCE REGISTRY (PCR-NN) where consequence definitions live in one authoritative location; the pipeline overview cites PCR-NN IDs rather than embedding prose. This makes consequence definitions maintainable by ID, auditable by reference, and immune to display-layer drift. C-48 is the pipeline-level analog of C-23 (schema rows carry IDs for gate citation) applied to the consequence column: a pipeline overview satisfying C-29 with embedded prose consequences does not satisfy C-48 unless the consequence column cites PCR-NN IDs backed by a registry.

## v14 — two new criteria from R15 excellence signals

| ID | Name | Sharpens | Source |
|----|------|----------|--------|
| C-45 | Phase bodies contain zero inline prose constraints; every constraint lives exclusively in a named schema row; phase text consists solely of schema-ID references, fill-in slots, and structural navigation | C-16, C-18, C-23, C-27 | V-01 PASS+ cluster |
| C-46 | INERTIA REGISTER includes a "Stakeholder Most Harmed" column naming the role category most vulnerable to each inertia default; per-phase override directives reproduce both IR-NN ID and the role name; exit gate verbatim reproductions include the role name alongside the full IR-NN text | C-43, C-44 | V-02 ES-1/ES-2/ES-3 |

**Aspirational denominator (v14)**: 36 → 38

---

**Gap logic — v14 criteria:**

- **C-45** — C-16 requires every constraint to be enforced by a structural element. C-18 requires schemas to exist with consequence columns. Neither forbids inline prose restatements of schema rules inside phase bodies. A prompt can satisfy C-16 and C-18 while still restating rules as prose within phase text — creating a dual-truth channel where schema and prose can diverge on revision without a mechanical signal. C-45 closes the **schema-drift channel**: phase bodies carry no constraint prose; the schema is the sole and exclusive truth. When schema and execution point share no prose, drift is detectable by structural inspection rather than content comparison.

- **C-46** — C-44 requires the full IR-NN text reproduced verbatim at exit gates. But IR-NN text typically describes an inertia *pattern* (what the default behavior is) without naming who bears the cost when the override is skipped. A model at the gate sees the inertia text but may not have the role-vulnerability mapping in active context. C-46 requires the INERTIA REGISTER to carry a "Stakeholder Most Harmed" column, and for that role name to travel with IR-NN citations at phase override directives and at exit gate reproductions. Closes the **role-abstraction axis**: inertia text is present but stakeholder vulnerability is abstract, so the model can evaluate the gate without knowing whose outcome depends on it.
