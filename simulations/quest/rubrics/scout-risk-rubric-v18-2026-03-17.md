# Scout-Risk Rubric — v18

## What This Rubric Evaluates

A scout-risk prompt is evaluated on its ability to produce a well-structured risk register
covering inertia risk, dimensional risks, and interdependencies. This rubric grades the
prompt text -- not a trace artifact.

---

## Essential Criteria (weight: 12 pts each, 60 pts total)

Fail any essential -> max composite 52 -> cannot reach golden.

| ID | Criterion | Category | Pass Condition |
|----|-----------|----------|----------------|
| C-01 | **Inertia risk is mandatory and first** | structure | The prompt explicitly requires an inertia risk entry as the first item in the register. It must be clearly labeled as inertia and the instruction must be non-optional -- "include if relevant" or "you may add" fails. Pass if the prompt states the inertia risk is required and must appear before dimensional risks. |
| C-02 | **Dimensional coverage enforced** | structure | The prompt requires risks across multiple dimensions -- at minimum three of: technical, market, compliance, dependency, timeline. Fail if fewer than 3 dimensions are represented. |
| C-03 | **Risk anatomy complete** | correctness | Every risk entry includes all three required fields: severity (HIGH/MEDIUM/LOW), likelihood (any meaningful expression of probability or triggering condition), and mitigation (a stated action or control). Fail if any risk is missing one or more fields. |
| C-04 | **Severity uses correct scale** | format | Severity values are drawn exclusively from {HIGH, MEDIUM, LOW}. No numeric scales, percentages, or invented labels. Fail if any severity value deviates from this vocabulary. |
| C-05 | **Mitigations are specific and actionable** | depth | Each mitigation describes a concrete action, owner class, or control mechanism -- not a generic hedge ("monitor closely", "be careful"). A passing mitigation names what to do or what to watch, not just that something should happen. Fail if two or more mitigations are generic non-actions. |

---

## Recommended Criteria (weight: 10 pts each, 30 pts total)

| ID | Criterion | Category | Pass Condition |
|----|-----------|----------|----------------|
| C-06 | **Risks are dimension-labeled** | format | Each risk entry is explicitly tagged with its dimension (e.g., "Technical", "Market", "Compliance", "Dependency", "Timeline"). Pass if >= 80% of risks carry a visible dimension label. |
| C-07 | **Likelihood is qualified beyond binary** | depth | Likelihood expressions go beyond "possible/unlikely" -- they specify a condition, scenario, or rough probability bracket (e.g., "high if team has no prior SDK experience", "~30% given current adoption curve"). Pass if >= half of risks meet this bar. |
| C-08 | **Risks are ordered by priority** | behavior | The register is ordered highest-to-lowest severity (after inertia), or explicitly states a prioritization logic. A randomly ordered flat list fails. Pass if the ordering is defensible by severity or compound impact. |

## Aspirational Criteria (weight: 2 pts PASS / 1 pt PARTIAL / 0 pts FAIL each, 94 pts max)

| ID | Criterion | Category | Pass Condition |
|----|-----------|----------|----------------|
| C-09 | **Risk interdependencies noted** | depth | At least one risk entry notes how it compounds with or unlocks another (e.g., "If dependency risk materializes, timeline risk escalates to HIGH"). Pass if at least one cross-risk dependency is surfaced explicitly. |
| C-10 | **AMEND behavior demonstrated** | behavior | When an AMEND directive is present in the prompt (focus on one dimension, adjust severity thresholds), the output correctly narrows scope or recalibrates without losing inertia or risk anatomy. Pass if AMEND is applied correctly and inertia risk is retained even in a narrowed register. |
| C-11 | **Full mitigation specificity (zero-generic)** | depth | Every mitigation in the register names a concrete action, owner class, or control mechanism -- no generic hedge language anywhere. Raises the C-05 bar from "fewer than 2 generic" to zero. Pass if a full scan of all mitigations finds no instance of "monitor closely," "keep an eye on," "revisit later," "consider alternatives," or equivalent non-actions. |
| C-12 | **All likelihoods are trigger-qualified** | depth | Every likelihood expression specifies a concrete triggering condition or scenario (not just "possible/unlikely/high/medium"). Raises the C-07 bar from ">= half" to all entries. Pass if every risk entry has a likelihood framed as a named condition or scenario rather than a bare probability label. |
| C-13 | **Interdependencies structured in a dedicated section** | depth | Risk interdependencies appear in a labeled section (not as inline footnotes or scattered notes), with at least two cross-risk links named. Raises the C-09 bar from "at least one" to "a structured section with >= 2 named dependencies." Pass if a dedicated interdependency section exists and names at least two compound-risk pairings. |
| C-14 | **Likelihoods use IF-THEN syntactic form** | depth | Every likelihood entry is written as "IF [named condition], THEN this risk activates" (or equivalent explicit IF-THEN phrasing). Bare probability labels ("high", "~30%", "possible") are absent from all entries. Raises the C-12 bar from "trigger-qualified" to "syntactically enforced IF-THEN form." Pass if every likelihood entry uses the IF...THEN construct -- no bare label survives a full scan. |
| C-15 | **Mitigation type declared for every entry** | depth | Every mitigation begins with or explicitly names its type class (e.g., Spike, Validate, Gate, Contract, Cut, Instrument, or domain-equivalent taxonomy). The type is a positive structural claim, not merely the absence of generic language -- a Spike requires a named unknown and time-box, a Gate requires a named criterion. Raises the C-11 bar from "zero-generic" to "positively typed." Pass if every mitigation carries a visible type label with its required sub-field contracts present in the prompt. |
| C-16 | **Interdependency count reaches specificity threshold** | depth | The dedicated interdependency section names at least 3 compound-risk pairings. The count functions as an indirect quality signal for the underlying register -- only a sufficiently specific risk register generates 3 distinct pairings naturally. Raises the C-13 bar from >= 2 to >= 3 named pairs. Pass if the interdependency section lists at least 3 explicitly named compound-risk pairs. |
| C-17 | **Interdependency pairs carry severity-transition labels** | depth | Every compound-risk pair in the interdependency section states not only which risks are linked but also the severity transition that activation triggers (e.g., "IF Risk A activates, Risk B escalates from MEDIUM to HIGH"). A bare compound statement ("A compounds B" or "A worsens B") without a named severity transition fails. Raises the C-16 bar from ">= 3 named pairs" to ">= 3 pairs each with an explicit FROM/TO severity label." Pass if every interdependency pair in the section names both the linked risk and its severity escalation. |
| C-18 | **Mitigation type portfolio covers >= 3 distinct classes** | depth | The risk register uses at least 3 distinct type class names (from: Spike, Validate, Gate, Contract, Cut, Instrument) across its entries. A register where every mitigation uses the same type class is type-monoculture -- the label is present (C-15 passes) but the taxonomy is not meaningfully applied. Raises the C-15 bar from "every entry has a type" to "the register's type distribution is substantively diverse." Pass if scanning all mitigation entries finds at least 3 distinct type class names in use. |
| C-19 | **Mitigation prohibition uses enumerated forbidden-phrase list** | depth | The prompt's mitigation quality gate names at least 3 specific forbidden hedge-phrases (e.g., "monitor closely", "keep an eye on", "revisit later", "consider alternatives") rather than describing the category conceptually. An enumerated list makes the prohibition testable without judgment; "avoid generic language" does not. Grounded in V-02's C-11 enforcement mechanism vs. V-01's conceptual gate. Raises C-11 from zero-generic-by-concept to zero-generic-by-enumeration. Pass if the prompt lists at least 3 specific forbidden phrases by name in its mitigation quality check. |
| C-20 | **Downstream count failure triggers upstream revision instruction** | depth | When a minimum-count gate (interdependency count, type diversity count) finds fewer items than required, the prompt instructs revision of the upstream section rather than accepting the gap or issuing a warning. "If you find fewer than 3 pairs, note the limitation" fails; "If you find fewer than 3 pairs, revise Step 2" passes. Grounded in V-03's back-pressure pattern: "If you cannot find any... revise Step 2." A prompt with no repair loop is self-reporting failure; a prompt with a repair loop is self-healing. Pass if at least one downstream minimum-count check includes an explicit instruction naming the upstream step to revise. |
| C-21 | **Repair-loop count matches minimum-count gate count** | depth | The prompt contains one repair loop for every minimum-count gate it introduces. A prompt with two gates (e.g., interdependency count + type diversity count) but only one repair loop leaves one gate without self-healing. Grounded in V-02's two-loop structure: a dedicated loop for interdependency count ("return to main register and add or refine entries") and a separate dedicated loop for type diversity ("return to main register and revise mitigations"). V-01, by contrast, has one gate without a corresponding second loop, which is why it fails C-18. Raises C-20 from "at least one repair loop" to "one repair loop per gate." Pass if the number of explicitly named repair instructions equals the number of minimum-count gates in the prompt. |
| C-22 | **Severity-transition columns are type-constrained at cell level** | depth | In the interdependency section, the From and To severity values are each governed by an explicit cell-level vocabulary rule restricting them to {HIGH, MEDIUM, LOW} -- not simply illustrated by prose examples or implied by context. Cell-level type constraints make violations mechanically detectable without prose interpretation; a prose FROM/TO label (sufficient for C-17) still allows arbitrary text in the cell. Grounded in V-02's interdependency table where each From and To column "must contain exactly one of HIGH, MEDIUM, LOW" as a stated column rule. Raises C-17 from "prose FROM/TO severity-transition label required" to "typed column constraint with explicit cell-level vocabulary." Pass if the interdependency section defines From and To as vocabulary-constrained fields with the {HIGH, MEDIUM, LOW} set named as their complete allowed value set. |
| C-23 | **Repair loops are explicitly named with unique labels** | depth | Every repair loop in the prompt is assigned a unique human-readable label (e.g., "Repair Loop A", "Repair Loop B", or equivalent named designators) that appears in the prompt text, making each loop independently addressable by name and the total count unambiguous without inference. A prompt where repair loops exist only as unlabeled prose instructions requires a reader to infer their boundaries and count them; a prompt with named labels makes the count trivially verifiable. Grounded in V-03's explicit "Repair Loop A / Repair Loop B" structure, where the scorecard notes "explicit labeling makes count unambiguous." Raises C-21 from "count match" to "named-and-independently-addressable match." Pass if every repair loop carries a unique label that appears verbatim in the prompt text. |
| C-24 | **Complete mitigation type taxonomy pre-defined in the prompt** | depth | The prompt defines all 6 mitigation type classes (Spike, Validate, Gate, Contract, Cut, Instrument) together with their required sub-fields as a reference taxonomy -- not just the types that happen to appear in the output. C-15 requires each output entry to carry its type and sub-field contracts; C-24 requires the prompt itself to provide a complete 6-class reference that the model draws from, preventing type omission at the source. A prompt that defines only the types it expects to use is subject to coverage gaps; a prompt with a complete reference taxonomy is closed under the type vocabulary. Grounded in V-02's complete definition: "Spike: Unknown/Time-box; Gate: Criterion; Validate: Assumption/Method; Contract: Party/Commitment; Cut: Element/Scope effect; Instrument: Metric/Alert threshold." Raises C-15 from "per-entry sub-field contracts present" to "full 6-class taxonomy pre-defined in one or more reference blocks." Pass if the prompt explicitly defines all 6 type classes with their sub-fields, regardless of which types appear in any given output. |
| C-25 | **Mitigation type diversity audit occupies a dedicated named step** | depth | The mitigation type diversity gate (C-18's >= 3 distinct class check and its repair loop) is isolated in its own dedicated named phase or step rather than embedded within the main risk-entry generation instructions. Isolation separates the generation concern (write risk entries) from the audit concern (check type distribution), making each step's purpose unambiguous and preventing the diversity check from being skipped when a model satisfies the generation step in one pass. Grounded in V-01's Phase 2b, which is a standalone phase solely for diversity auditing -- distinct from the phase that produces risk entries. Raises C-18 from "audit gate exists with repair loop" to "audit gate is isolated in a dedicated named step." Pass if the prompt assigns the mitigation type diversity audit its own step number or phase label, separate from the step that generates risk entries. |
| C-26 | **Mitigation sub-field values rendered inline in output text** | depth | C-15 requires each mitigation to carry its type class label with sub-field contracts. V-03's PARTIAL on C-15 reveals that defining the taxonomy and requiring a type label is insufficient if the prompt does not explicitly enforce that sub-field values appear in the output text. A mitigation reading "[Spike] -- conduct an API feasibility study" satisfies the type-label requirement but omits the sub-field values (Unknown=X, Time-box=Y) that give the type class its structural meaning. The escalation: the prompt's output format instruction requires each mitigation to render sub-field values inline -- "[Type-Class: sub-field=value, ...] -- action" -- not just the class name. Raises C-15 from "type class label with sub-field contracts present in the prompt" to "sub-field values rendered verbatim in each output mitigation entry." Pass if the prompt's format specification for mitigation entries requires sub-field values to appear in the output text, not merely requires the type class label. |
| C-27 | **Taxonomy reference block positioned before all generation phases** | depth | C-24 requires the prompt to define all 6 mitigation type classes with their sub-fields. V-03's PARTIAL on C-24 reveals that defining the complete taxonomy inside a generation phase (e.g., Phase 2) is structurally weaker than positioning it before Phase 1: when the taxonomy is embedded within a phase, the model may treat it as phase-local scope rather than a global contract available to all phases. A standalone reference block placed before Phase 1 -- visible before any generation begins -- eliminates phase-local taxonomy scoping and makes the full 6-class reference available to every subsequent phase simultaneously. Raises C-24 from "complete 6-class taxonomy pre-defined somewhere in the prompt" to "complete 6-class taxonomy defined in a standalone reference block that precedes Phase 1 (or the first generation step)." Pass if the taxonomy block appears before the first generation phase in the prompt's instruction sequence, not embedded within any numbered phase or step. |
| C-28 | **Inertia risk carries dedicated anatomy fields separate from dimensional risk anatomy** | depth | C-01 requires inertia to be mandatory and first; C-03 requires all risks to have severity, likelihood, and mitigation. V-01's scorecard evidence reveals a more structurally differentiated inertia anatomy: a dedicated "Inertia condition" field (replacing the generic likelihood field for the inertia entry specifically) and a "Status-quo description" field (capturing the cost or state that persists if no action is taken), alongside the standard mitigation. These inertia-specific fields encode structural knowledge about what makes an inertia risk distinct from a dimensional risk: its trigger is the absence of decision (inertia condition) and its harm is the perpetuation of the current state (status-quo description). A prompt that applies the same 3-field anatomy to inertia as to dimensional risks conflates entry types; a prompt with dedicated inertia fields preserves the semantic distinction. Raises C-03 from "all entries have severity/likelihood/mitigation" to "the inertia entry has inertia-specific anatomy with at minimum a named inertia-condition field and a status-quo-description field (or semantic equivalents) distinct from the generic likelihood/mitigation structure used for dimensional risks." Pass if the prompt specifies a distinct anatomy for the inertia risk entry that includes both an inertia-condition field and a status-quo-description field, separate from the generic likelihood/mitigation structure applied to dimensional risks. |
| C-29 | **Inertia anatomy includes a Decision Window field** | depth | C-28 requires the inertia entry to carry inertia-condition and status-quo-description fields distinct from dimensional risk anatomy. V-01's 6-field anatomy demonstrates a third inertia-specific field: a Decision Window -- a named deadline, forcing event, or expiry horizon after which the inertia condition becomes irreversible or more costly to exit. The Decision Window encodes the time dimension of inertia risk: the cost of the status quo is not merely a state but a trajectory, and the window names when the trajectory forecloses the option. Without a Decision Window, an inertia entry is structurally incomplete -- it names what persists and why, but not when the door closes. A prompt that includes only inertia-condition and status-quo-description has two of three inertia-specific fields; the full anatomy requires all three. Raises C-28 from "inertia-condition + status-quo-description present" to "inertia-condition + status-quo-description + decision-window all present as named fields in the inertia entry's specified anatomy." Pass if the prompt's inertia entry specification names a Decision Window field (or semantic equivalent naming a deadline or forcing event) alongside inertia-condition and status-quo-description. |
| C-30 | **Inertia entry occupies a dedicated table or sub-schema isolated from dimensional risks** | depth | C-01 requires inertia to be first; C-28 and C-29 require inertia to carry distinct anatomy fields. V-02's Table 1 / Table 2 separation demonstrates a stronger structural pattern: the inertia entry is not merely the first row in a unified register but is isolated in its own dedicated table or sub-schema with a column layout that reflects its distinct anatomy. When inertia and dimensional risks share a single table, column definitions must accommodate both anatomies simultaneously -- producing either under-specified inertia columns (likelihood != inertia-condition) or over-specified dimensional columns with unused fields. A prompt that assigns inertia its own schema eliminates this tension: the inertia table can define inertia-condition, status-quo-description, and decision-window as first-class columns, while the dimensional table defines likelihood and mitigation-type as its columns. Raises C-28/C-29 from "distinct anatomy fields required" to "distinct anatomy fields isolated in a separate schema or table that does not share column definitions with the dimensional risk register." Pass if the prompt specifies separate output structures (tables, schemas, or labeled sub-sections with independent field sets) for the inertia entry and the dimensional risk entries. |
| C-31 | **Dimensional risk register has a minimum row floor enforced as a count gate** | depth | C-02 requires dimensional coverage across at least 3 dimensions; C-03 requires all entries to carry complete anatomy. Neither criterion prevents a minimal register -- a prompt that produces exactly 3 entries (one per dimension) passes both. V-02's 7+ row specification demonstrates that a quantitative floor on the dimensional table is a distinct and necessary gate: it forces the register to be substantively populated rather than technically compliant. A register with 3 rows can satisfy dimensional coverage; a register with 7+ rows must exhibit risk decomposition, showing that the author interrogated each dimension for multiple distinct failure modes rather than producing one representative entry per label. The row floor is an indirect structural quality signal -- only a genuinely thorough risk analysis produces 7 or more distinct dimensional entries. The gate should be accompanied by a repair loop (per C-20/C-21) instructing the model to return to the dimensional register and add entries if the floor is not met. Raises C-02 from "at least 3 dimensions present" to "at least N rows in the dimensional register, where N >= 5 (or prompt-specified higher threshold), with a repair loop for underpopulation." Pass if the prompt specifies a minimum row count for the dimensional risk register (at least 5 rows) and includes an explicit repair instruction if the count is not met. |
| C-32 | **Interdependency table rows carry a dedicated Trigger Condition field** | depth | C-17 requires every interdependency pair to carry FROM/TO severity-transition labels; C-22 requires those labels to be vocabulary-constrained at cell level. Both criteria address what happens when a dependency fires. Neither addresses how the firing condition is represented structurally. V-02's Table 3 demonstrates a third typed column alongside From and To: a Trigger Condition field that names the specific event or state that activates the dependency chain (e.g., "IF API-rate-limit Risk materializes"). Without this field, the activation logic for each dependency lives in free prose -- interpretable but not structurally isolated or mechanically verifiable. A prompt that requires a Trigger Condition column makes the IF side of every interdependency pair a first-class schema element rather than an embedded phrase, enabling dependency coverage to be audited by column scan rather than prose interpretation. Raises C-17 from "severity-transition labels required per pair" to "severity-transition labels AND a dedicated Trigger Condition field per pair, specified as a named column or field in the interdependency output schema." Pass if the prompt's interdependency section or table specification includes a Trigger Condition (or semantic equivalent) as an explicitly named column or field, separate from the From and To severity fields. |
| C-33 | **Dimension column in the dimensional register is vocabulary-constrained at cell level** | depth | C-06 requires visible dimension labels on risk entries; C-02 requires at least 3 dimensions. Neither constrains what values the Dimension field may take -- a model can invent unlisted dimension labels ("Organizational", "Cultural") and satisfy both criteria. V-02's vocabulary-constrained Dimension column demonstrates that closing the dimension vocabulary is structurally distinct from requiring labels or counting unique values: a closed vocabulary makes cross-dimensional consistency verifiable by scan, not by semantic judgment. A prompt that defines the permissible dimension set (e.g., {Technical, Market, Compliance, Dependency, Timeline}) and requires cell values to be drawn from it eliminates invented dimension labels and makes "dimensional coverage" a mechanical check rather than an interpretive one. Raises C-06 from "visible dimension label >= 80%" to "Dimension column values restricted to a named closed vocabulary, with unlisted values prohibited." Pass if the prompt defines a closed set of permissible dimension values and specifies that every Dimension cell must draw from that set. |
| C-34 | **AMEND directive produces a structured header block naming active and suppressed dimensions** | behavior | C-10 requires the prompt to respond correctly to an AMEND directive. V-01 demonstrates a stronger AMEND pattern: the prompt not only narrows scope in response to AMEND, but produces a dedicated AMEND Header block at the start of output that explicitly names (a) which dimension(s) are active under this AMEND and (b) which dimension(s) are suppressed. A plain scope-narrowing (pass C-10) leaves the active/suppressed partition implicit; a structured header makes it explicit and verifiable at a glance without reading the full register. The header also establishes a reference point for the downstream repair loops: a dimension that appears in the suppressed list should not appear in any register entry, making suppression mechanically auditable. Raises C-10 from "AMEND scope correctly applied" to "AMEND produces a named header block with explicit active-dimension and suppressed-dimension lists." Pass if the prompt includes an AMEND-specific instruction that requires the output to include a structured header naming both the active and suppressed dimensions before the risk register begins. |
| C-35 | **AMEND header block is produced in an isolated pre-phase step** | behavior | C-34 requires the prompt to produce an AMEND header naming active and suppressed dimensions. V-01's Phase 0a demonstrates a stronger pattern: the AMEND header is produced in a dedicated phase that precedes all role activation -- it is not generated inline within the risk-entry phase or as a preamble to the dimensional register. Isolation before any role activates means the header is available as a static contract for all downstream phases: each phase can reference it to confirm scope without re-interpreting the AMEND directive. A header produced inside Phase 1 (the role-activation and entry-generation phase) is subject to scope-bleed -- the role may begin generating entries before the header is fully resolved. A pre-phase header block eliminates this risk by completing scope resolution before any generation begins. Raises C-34 from "AMEND produces a structured header somewhere in the output" to "AMEND header is produced in an isolated step that precedes all role activation and risk-entry generation." Pass if the prompt explicitly assigns AMEND header production to a pre-phase step that runs before any risk-entry generation phase. |
| C-36 | **Per-dimension HIGH coverage guaranteed by a named seed sub-phase before expansion** | depth | C-15 requires every mitigation entry to carry its type class; C-17 requires interdependency pairs to carry severity-transition labels. Neither addresses the severity distribution of the dimensional register itself. V-01 demonstrates a seed-before-expand pattern: a named sub-phase (e.g., Phase 1b) explicitly guarantees that at least one HIGH-severity risk is seeded for every active dimension before the expansion phase (Phase 1c) adds MEDIUM and LOW rows. Without a seed guarantee, the register may be fully populated but severity-top-heavy in some dimensions and sparse in others -- satisfying dimensional coverage (C-02) while failing to ensure that every dimension has at least one HIGH anchor. The seed guarantee is also a prerequisite for the audit step exclusion pattern (C-40): if Phase 1b guarantees per-dimension HIGH coverage, Phase 2b can safely exclude re-auditing it, referencing "which Phase 1b guarantees" as the exclusion authority. Raises C-07/C-12 from "likelihood expressions qualified" to "at least one HIGH entry per active dimension guaranteed by a named upstream phase before any expansion begins." Pass if the prompt contains a named phase or sub-phase that explicitly guarantees at least one HIGH-severity risk per active dimension, and this phase is ordered before the expansion phase that adds MEDIUM/LOW entries. |
| C-37 | **Schema block declares a named field list and all generation phases reference it** | depth | C-24 requires a complete mitigation type taxonomy pre-defined; C-27 requires it positioned before all generation phases. Neither addresses the dimensional register's column schema or whether generation phases reference it by name. V-01 demonstrates a schema-first pattern for the dimensional register: a named field list (e.g., Phase 0's column schema declaration) is defined before Phase 1, and every generation phase references that schema by name rather than re-defining or assuming it. When generation phases reference the schema by name (backward pointer), column drift is detectable -- any phase that deviates from the declared schema creates a named violation. A prompt without backward pointers allows each generation phase to implicitly assume a compatible schema, which is unverifiable without comparing phase-level output against the original declaration. Raises C-27 from "taxonomy block positioned before generation phases" to "named schema block declared before Phase 1, with all generation phases carrying an explicit back-reference to that schema by name." Pass if the prompt's generation phases include explicit back-references to the pre-declared schema by name, not only that a schema block is defined earlier. |
| C-38 | **Seed phase closes with an equality gate before expansion begins** | depth | C-36 requires a named seed sub-phase to guarantee at least one HIGH entry per active dimension before expansion. V-01's Phase 1b demonstrates a stronger closure pattern: the seed phase does not merely produce HIGH entries -- it closes with an equality gate that verifies the count of seeded HIGH rows equals the number of active dimensions before expansion begins. Without an equality gate, the seed phase is a generation instruction without a completion criterion: the model may proceed to Phase 1c having seeded 3 HIGH entries for 5 active dimensions, satisfying the "at least one HIGH per dimension" instruction by intention but not by verification. An equality gate (dimension-count = HIGH-row-count) makes the seed phase self-checking: it cannot be considered complete until the count relationship is verified. Raises C-36 from "seed step executes before expansion" to "seed step closes with an explicit equality gate (dimension-count = HIGH-row-count) that must be satisfied before expansion is permitted to begin." Pass if the prompt's seed phase includes an explicit count-equality check between the number of active dimensions and the number of seeded HIGH rows, and names this check as a completion condition for the seed phase. |
| C-39 | **Expansion phase header names seed as its starting state and restricts additions to MEDIUM/LOW** | depth | C-36 requires a seed sub-phase before expansion; C-38 requires the seed phase to close with an equality gate. Together, these ensure the dimensional register has at least one HIGH per dimension when expansion begins. V-01's Phase 1c demonstrates a complementary pattern in the expansion phase itself: the expansion phase header explicitly names the seed phase output as its starting state (e.g., "Starting from the Phase 1b seed register...") and restricts its additions to MEDIUM and LOW severity rows only -- prohibiting HIGH rows from being added during expansion. Without this restriction, the equality gate (C-38) is bypassed by the expansion phase: a model that adds HIGH rows during expansion inflates the HIGH count beyond the seed guarantee, potentially masking dimensional coverage gaps that the seed phase was supposed to close. The expansion restriction locks HIGH coverage at the seed guarantee level, preventing coverage drift during population. Raises C-36 from "seed executes before expansion" to "expansion phase header names seed as starting state and carries an explicit prohibition on HIGH rows during expansion, enforcing that HIGH coverage is set by the seed phase and not modified during expansion." Pass if the prompt's expansion phase includes a header that (a) names the seed phase output as its starting state and (b) explicitly prohibits adding HIGH-severity rows during expansion. |
| C-40 | **Audit step excludes upstream-guaranteed properties from repair scope** | depth | C-25 requires the type diversity audit to occupy a dedicated named step. V-02 demonstrates a stronger audit posture: an audit step that explicitly excludes from its repair scope properties already guaranteed by an upstream phase. An audit step with no exclusions implicitly treats all register properties as open for repair -- including properties that were guaranteed by an earlier phase (e.g., per-dimension HIGH coverage guaranteed by Phase 1b). Re-auditing guaranteed properties is not just redundant; it creates a repair surface where none should exist: a repair loop for an upstream-guaranteed property could silently overwrite the upstream guarantee. An audit step that names what it will not repair converts its posture from "scan and fix everything" to "confirm that upstream guarantees hold; repair only what is within this phase's scope." Raises C-25 from "audit gate is isolated in a dedicated named step" to "audit step explicitly names the properties that are outside its repair scope because they are guaranteed by an upstream phase, redefining its posture from all-repair to confirmation-only for those properties." Pass if the prompt's audit step includes an explicit exclusion clause naming at least one property and the upstream phase that guarantees it, with a stated prohibition on re-auditing or repairing that property. |
| C-41 | **Schema block enumerates all consuming phases by name (forward pointer)** | depth | C-37 requires generation phases to reference the pre-declared schema by name (backward pointer). V-03 demonstrates a bidirectional traceability pattern: the schema block itself also enumerates all phases that are expected to consume it, by name (forward pointer). A schema block with only backward pointers (phases reference the schema) allows new phases to be added without updating the schema block -- the schema does not know its consumers. A schema block with forward pointers (the schema names its consumers) is a closed contract: adding a new generation phase without updating the schema's consumer list creates a detectable inconsistency. Bidirectional traceability converts the schema from a passive reference into an active constraint on phase structure. Raises C-37 from "generation phases reference pre-declared schema by name (backward pointer)" to "schema block also enumerates all consuming phases by name (forward pointer), making coverage two-directionally verifiable." Pass if the prompt's schema block explicitly lists all generation phases that are expected to draw from it by name, in addition to those phases carrying backward references to the schema. |
| C-42 | **Expansion-phase prohibition on HIGH rows is named as a typed violation** | depth | C-39 requires the expansion phase to carry an explicit prohibition on adding HIGH-severity rows. V-05 demonstrates a stronger prohibition pattern: the prohibition is not stated as a plain instruction but is named as a typed violation (e.g., "format violation: adding a HIGH row during Phase 1c"). Naming a violation type converts a behavioral instruction ("do not add HIGH rows") into a rule-classifiable constraint ("adding a HIGH row is a format violation"). Rule-classifiable constraints can be audited by type at the conclusion of expansion, not just by scanning for HIGH rows -- a downstream checker can ask "did any format violations occur during Phase 1c?" rather than re-reading all rows. The typed violation also creates a vocabulary anchor for the audit step: Phase 2b can exclude "format violations that Phase 1c's prohibition already covers" by type name, rather than by describing the prohibition again in full. Raises C-39 from "explicit prohibition on HIGH rows in expansion" to "prohibition named as a violation type (e.g., 'format violation'), making non-compliance rule-classifiable rather than instruction-based."
