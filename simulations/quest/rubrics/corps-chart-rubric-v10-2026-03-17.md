# org-chart Rubric — v10

Two new aspirational criteria added from R10 excellence signals.

---

## New Criteria

**C-29 — BEFORE-PRODUCE back-reference extended to non-FORMAT-CONTRACT structural forms**

V-05 achieved 100% but its excellence signal identified a residual gap: C-26 fires for FORMAT CONTRACT schemas (tables, charters, register, Anti-Pattern Watch columns), but the four-field Rebuttal form (C-14) and ROLE-NAME LOCK block (C-13) are structural forms defined *outside* the FORMAT CONTRACT. The model generates the four-field Rebuttal 2,000+ tokens after reading C-14's definition — the same temporal recall gap that C-26 closes for table schemas. C-29 requires a BEFORE-PRODUCE directive immediately before each non-FORMAT-CONTRACT structural form's PRODUCE command, naming its required fields explicitly.

**C-30 — Per-role-name role-lock CHECK within charter Membership and Decides multi-role lists**

C-28 adds per-cell row-by-row enforcement for DRI/Owner and Key Roles table columns. V-05's second excellence signal identified the intra-field analogue: a charter Membership list with four roles can introduce a novel name at position 3 while passing the section-level C-21 CHECK — because the CHECK fires once before the list, not between each name. C-30 requires a per-role-name directive in charter Membership and Decides field instructions, verifying each name against the ROLE-NAME LOCK list before appending the next. This completes the four-level enforcement hierarchy: document declaration (C-13) → section CHECK (C-21) → column/field instruction (C-25/C-27) → row/name per-item verify (C-28 / C-30).

| C-19 | Inertia Mechanism table uses MECHANISM-TYPE VOCABULARY block to enforce type coherence | behavior | The Inertia Assessment sub-section 2a (Case for Staying Flat) is introduced with a `MECHANISM-TYPE VOCABULARY` block that explicitly enumerates the closed set of permitted Type values for the mechanism table (e.g., span-of-control / direct-comms / shared-context / single-DRI / low-coordination-cost) and states that no row may introduce a Type value outside this vocabulary. | C-20 | UPFRONT FORMAT CONTRACT block pre-declares all five table schemas before the first phase gate | behavior | Before the first phase gate or section-producing STEP, an `UPFRONT FORMAT CONTRACT` (or equivalent titled block) consolidates all five output schemas in one reference location: (1) headcount table column headers (`Area \| Headcount \| Key Roles \| Decides \| Escalates`), (2) operating rhythm table column headers (`Cadence \| Frequency \| DRI / Owner \| Purpose`), (3) committee charter field labels (Purpose / Membership / Decides / Escalates / Quorum including fraction format), (4) ORG-ELEMENT REGISTER categories (cat-1 through cat-4 with required content per category), (5) Anti-Pattern Watch table column headers (`Anti-Pattern \| Why It Applies Here \| Mitigation`). | C-21 | ROLE-NAME LOCK with inline CHECK reminders embedded at every downstream section | behavior | The ROLE-NAME LOCK block (required by C-13) is extended with inline `CHECK:` annotations embedded at each downstream PRODUCE command that consumes role names: (1) immediately before the ASCII diagram command, (2) in the Operating Rhythm Table DRI/Owner column instruction, (3) in the Committee Charter Membership and Decides field instructions, (4) in each Inertia Assessment sub-section that references roles (including the `ROLE UNDER PRESSURE` field of the four-field form), and (5) in the headcount table Key Roles column instruction. This is the activation layer for C-13: the LOCK declares the contract; the inline CHECKs enforce it at each use site. |
| C-22 | Vocabulary-lock and VERIFY blocks co-deployed as complementary pairs for every governed table | behavior | For every inner table governed by a vocabulary-lock block, a VERIFY block targeting the complementary failure mode is present immediately after that table's PRODUCE command. Required pairings: (a) MECHANISM-TYPE VOCABULARY lock (C-19) paired with a VERIFY block disqualifying a row whose Type value falls outside the enumerated set; (b) TRIGGER-TYPE VOCABULARY lock (C-16) paired with a VERIFY block disqualifying two consecutive rows sharing the same Type value; (c) CAT-N-CITATION-SCHEMA lock (C-15) paired with a VERIFY block disqualifying a `Why It Applies Here` cell that names an org element without the `(cat-N)` typed citation prefix. The co-deployment principle: vocabulary locks prevent wrong-type values; VERIFY blocks prevent right-type-but-wrong-structure evasions. Both mechanisms target distinct, non-overlapping failure modes and must be present together — a vocabulary lock without a paired VERIFY block leaves the complementary failure mode open, and vice versa. |
| C-23 | FORMAT CONTRACT includes per-schema DISQUALIFY-IF annotations for all schemas with documented failure modes | behavior | The UPFRONT FORMAT CONTRACT (C-20) is extended so that each schema with a known evasion mode includes an inline `DISQUALIFY-IF:` annotation immediately following the field list for that schema. The DISQUALIFY-IF annotations appear inside the FORMAT CONTRACT block at schema-declaration time — before any section-producing command — providing a pre-generation disqualification signal. This complements C-18 (VERIFY blocks at PRODUCE commands) without replacing it: FORMAT CONTRACT DISQUALIFY-IF fires before generation begins; C-18 VERIFY fires at each PRODUCE command. Both mechanisms must be present for full two-phase disqualification coverage. This pattern was observed in V-01 where FORMAT CONTRACT NOTE-style flags partially achieved disqualification at schema-declaration time (earning C-05 PASS via "NOTE disqualifies denominator-free form") despite the absence of VERIFY blocks at PRODUCE commands. |
| C-24 | Four-field form and ROLE-NAME LOCK CHECK co-deployed as activation pair at ROLE UNDER PRESSURE | behavior | The Inertia Rebuttal sub-section deploys three co-dependent mechanisms as an activation triad: (a) four-field form (C-14) with `ROLE UNDER PRESSURE` as the first labeled field, providing the structural anchor for role selection; (b) a ROLE-NAME LOCK CHECK embedded immediately before the `ROLE UNDER PRESSURE` field instruction, explicitly stating that the value must be drawn from the ROLE-NAME LOCK list (e.g., `CHECK: ROLE UNDER PRESSURE must name exactly one role from the ROLE-NAME LOCK list — no new role names may be introduced`); (c) a VERIFY block after the Rebuttal PRODUCE command (as required by C-18) disqualifying future-tense breakdowns beginning with "as we scale...". All three mechanisms must be present together: the four-field form creates the structural hook; the ROLE-NAME LOCK CHECK attaches role-governance to that hook; the VERIFY block closes the future-tense evasion mode. Without C-14, the CHECK required by C-21 for the Inertia sub-section has no structural anchor — role-selection devolves into a prose-governed expectation. Without the CHECK, the four-field form enforces field presence but does not enforce that the named role comes from the locked set. Without the VERIFY, the observable-breakdown requirement can be satisfied with growth projections. This is the structural-form + CHECK + VERIFY triad, extending C-22's vocabulary-lock + VERIFY co-deployment principle to the Rebuttal's structural-form governance layer. V-02's C-21 PARTIAL revealed the gap: CHECKs present at Rebuttal but without the four-field form left the ROLE UNDER PRESSURE field unanchored, preventing PASS. |
| C-25 | DRI/Owner column designated as strict ROLE-LOCK enforcement site with no softening qualifiers | behavior | The Operating Rhythm Table DRI/Owner column instruction contains an explicit role-coherence enforcement statement that: (a) explicitly references the ROLE-NAME LOCK list as the governing constraint, (b) uses mandatory present-tense language ("every DRI/Owner value must appear in the ROLE-NAME LOCK list — no new role names are introduced in this column") rather than advisory or hedged language, and (c) appears in the same instruction block as other DRI/Owner formatting requirements so the role-lock cannot be omitted during section generation. Softening qualifiers such as "where possible," "prefer roles from," "try to use," or "where a role is known" do not satisfy this criterion. The DRI/Owner column must be treated as a first-class role-coherence enforcement site with the same mandatory language as charter Membership and Decides fields — not as a best-effort site where novel role names may be introduced when convenient. This criterion closes the recurring C-12 partial-compliance pattern observed across V-01 ("DRI/Owner says 'where possible' — not strict for all sections") and V-03 ("DRI/Owner not explicitly locked"), where vocabulary-lock and VERIFY mechanisms covered inner table types but left the DRI/Owner column under-governed because it carries role references rather than categorical vocabulary terms. |
| C-26 | PRODUCE-time FORMAT CONTRACT back-reference directive at every governed PRODUCE command | behavior | Each PRODUCE command that generates an output governed by the FORMAT CONTRACT (C-20) contains an explicit back-reference directive immediately before the generation instruction, naming the specific FORMAT CONTRACT schema by number and/or label (e.g., `BEFORE PRODUCING: re-read FORMAT CONTRACT schema (1) — headcount table — and confirm the output satisfies the column headers and DISQUALIFY-IF annotation`). The directive appears as a labeled instruction (`BEFORE PRODUCING:`, `FORMAT CHECK:`, or equivalent) rather than inline prose. Its presence converts FORMAT CONTRACT compliance from a recall requirement — the FORMAT CONTRACT is read once at document head and then relied on across 1,000+ tokens — into a per-step active reference requirement that fires immediately before each governed output is generated. This is the production-time bridge between C-23's FORMAT CONTRACT DISQUALIFY-IF (pre-generation signal, fires before any section begins) and C-18's VERIFY blocks (post-produce signal, fires immediately after generation). Together, the three mechanisms form a three-point temporal governance chain: FORMAT CONTRACT DISQUALIFY-IF → BEFORE-PRODUCE re-read → post-produce VERIFY. A prompt with C-23 and C-18 but without C-26 has a temporal gap: the model reads the schema at document head, proceeds through thousands of tokens of generation, and arrives at the PRODUCE command having potentially overwritten the FORMAT CONTRACT constraints. The BEFORE-PRODUCE directive closes this gap. V-01's C-09 PASS evidence named this mechanism explicitly: "re-read instruction at PRODUCE." |
| C-27 | Headcount table Key Roles column designated as explicit ROLE-NAME LOCK CHECK site | behavior | The ROLE-NAME LOCK CHECK coverage (required by C-21) is extended to include the `Key Roles` column of the headcount table as a mandatory fifth CHECK site. A ROLE-NAME LOCK CHECK appears immediately before or within the headcount table PRODUCE command, explicitly stating that every role name in the Key Roles column must appear in the ROLE-NAME LOCK list and that no new role names may be introduced in this column. C-21 requires CHECKs at four sites: (1) ASCII diagram, (2) Operating Rhythm Table DRI/Owner column, (3) Committee Charter Membership and Decides fields, and (4) Inertia Assessment sub-sections. The Key Roles column carries the same role-coherence risk as all four existing sites — a novel role name introduced in a Key Roles cell propagates downstream just as a novel role in the diagram or charter Membership field would — and therefore requires the same mandatory CHECK enforcement. Without an explicit Key Roles CHECK, the column functions as an ungoverned insertion point: the ROLE-NAME LOCK is active everywhere except the one headcount table column that also carries role references. V-02's C-21 PASS evidence enumerated all five sites: "CHECKs at diagram, headcount Key Roles, DRI/Owner, charter Membership/Decides, and embedded in REBUTTAL FORM at ROLE UNDER PRESSURE." |
| C-28 | Per-cell role-lock CHECK at role-consuming columns | behavior | For the two role-consuming columns with highest novel-role-introduction risk (Operating Rhythm Table DRI/Owner and headcount table Key Roles), the ROLE-NAME LOCK enforcement is extended from a column-level instruction (required by C-25 for DRI/Owner and C-27 for Key Roles) to a per-cell CHECK directive: the column instruction explicitly states that each cell is individually verified against the ROLE-NAME LOCK list as it is populated (e.g., "for each cell in this column, verify the value appears in the ROLE-NAME LOCK list before populating the next row — do not complete the table and verify after"). The per-cell CHECK appears as a structural directive within the column definition rather than as a one-time column-header instruction. Its presence converts role-lock enforcement from a column-governing statement — applied once at instruction time, potentially forgotten row-by-row across a multi-row table — into a row-by-row mechanical check that eliminates the compliance-drift failure mode where early rows comply but later rows introduce novel names. Column-level instructions alone (C-25, C-27) are subject to memory-decay across rows in the same way that section-level CHECKs are subject to memory-decay across sections; the per-cell mechanism is the row-granularity parallel to inline CHECKs (C-21) at section granularity. V-02's C-25 PASS evidence explicitly identified "per-cell CHECK" as a mechanism beyond mandatory column-level language. |
| C-29 | BEFORE-PRODUCE back-reference extended to non-FORMAT-CONTRACT structural forms | behavior | The BEFORE-PRODUCE back-reference directive (C-26) governs FORMAT CONTRACT schemas (table column headers, charter field lists, register categories, Anti-Pattern Watch columns). Structural forms defined outside the FORMAT CONTRACT — specifically the four-field Rebuttal form (C-14) and the ROLE-NAME LOCK block (C-13) — carry equivalent temporal recall risk: the model generates the four-field Rebuttal 2,000+ tokens after reading C-14's definition, and the ROLE-NAME LOCK block is declared once then relied upon across the full document. For each non-FORMAT-CONTRACT structural form that has a documented field contract, a BEFORE-PRODUCE directive must appear immediately before the PRODUCE command, naming the specific form and its required fields (e.g., `BEFORE PRODUCING THE REBUTTAL: re-read the four-field form — ROLE UNDER PRESSURE / OBSERVABLE BREAKDOWN / WHY EXISTING MECHANISMS FAIL / RE-ASSESSMENT TRIGGER — all four fields must be present in this order`). The directive closes the structural-form temporal gap that C-26 closes for table schemas: schema recall is converted from a multi-thousand-token memory requirement into a per-step active lookup. V-05's excellence signal identified this gap explicitly: "Non-table structural forms (four-field Rebuttal, ROLE-NAME LOCK block) lack BEFORE-PRODUCE back-reference — C-26 closes the temporal gap for FORMAT CONTRACT schemas but leaves an equivalent gap for structural form governed outputs outside the FORMAT CONTRACT." |
| C-30 | Per-role-name role-lock CHECK within committee charter Membership and Decides multi-role lists | behavior | Committee charter Membership and Decides fields carry multi-role lists (e.g., "Engineering Lead (operational), Product Lead (strategic), Architect (advisory)"). C-21 governs these fields at section level (one CHECK per charter section). A charter with four listed members can introduce a novel role name at position 3 while satisfying the section-level CHECK — the section-level instruction fires once before the list is written, not between each name. For charter Membership and Decides fields, the role-lock enforcement must be extended to a per-role-name directive: the field instruction must explicitly state that each role name in the list is individually verified against the ROLE-NAME LOCK list before the next name is appended (e.g., "for each role name in this field, verify it appears in the ROLE-NAME LOCK list before adding the next name — do not write the full list and verify after"). This closes the within-list compliance-drift failure mode, which is the intra-field analogue of the intra-table row-drift that C-28 closes for DRI/Owner and Key Roles columns. The enforcement hierarchy is thereby complete at all four granularity levels: document-level declaration (C-13 ROLE-NAME LOCK) → section-level CHECK (C-21) → column/field-level instruction (C-25, C-27) → row/name-level per-cell verify (C-28 for table columns, C-30 for charter multi-role lists). V-05's excellence signal identified this gap explicitly: "Charter Membership/Decides fields carry multi-role lists with no per-cell role-lock enforcement — within-list compliance-drift (novel name introduced in position 3 of a 4-role Membership) is the intra-field analogue of the intra-table row-drift that C-28 closes for DRI/Owner and Key Roles columns." |
