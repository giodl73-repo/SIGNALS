# Rubric: quest-rubric (v8)

Evaluates output from the `quest-rubric` skill, which produces a scoring rubric for a named
Signal skill. The rubric must function as a working objective function: each criterion must be
independently checkable, the tiers must follow the structural contract, and the essential
criteria must be grounded in the target skill's actual non-negotiable behaviors — not generic
quality observations that would apply to any skill.

**v2 additions**: Three aspirational criteria extracted from R1 excellence signals — the inertia
test (V-05), the concrete discrimination example (V-04), and phase-gate enforcement (V-01).

**v3 additions**: Five aspirational criteria extracted from R2 excellence signals — calibration
pair lifecycle position (ES-2/V-03), sub-step output sequencing (ES-1/V-03), multi-level gate
architecture (ES-3/V-03), audit role separation (ES-4/V-05), and audit trail captures reasoning
(ES-5/V-05).

**v4 additions**: Four aspirational criteria extracted from R3 excellence signals — named-block
artifact architecture (ES-R3-1/V-02/V-04), gate-blocking vs retroactive-repair distinction
(ES-R3-2/V-02), cross-criterion audit consolidation (ES-R3-3/V-03/V-05), and structural
enforcement paired with behavioral anti-deferral instruction (ES-R3-4/V-01/V-04/V-05).

**v5 additions**: Three aspirational criteria extracted from R4 excellence signals — structural
verification phase as Auditor entry gate (ES-R4-1/V-05), three-level verification chain with
non-overlapping defect classes (ES-R4-2/V-05), and anti-deferral prohibition co-located at
sub-step level rather than phase header (ES-R4-3/V-05).

**v6 additions**: Three aspirational criteria extracted from R5 excellence signals — phase-level
workflow transitions marked by named section headings (ES-R5-1), cross-criterion verification
produces a named artifact with a forward-blocking gate (ES-R5-2), and scoring formula
denominators explicitly stated and matched to current criterion count (ES-R5-3).

**v7 additions**: Two aspirational criteria extracted from R6 excellence signals — mechanism
overview declares structural output format requirement alongside behavioral requirement
(ES-R6-1), and structural verification explicitly declares its phase-bounded coverage scope and
names the deferral destination for out-of-scope heading checks (ES-R6-2).

**v8 additions**: One aspirational criterion extracted from R7 excellence signal — pre-phase
output skeleton declared from MECHANISMS alone and cross-referenced in Structural Verification
(ES-R7-1). The skeleton step serves as an executable test of C-29's textual sufficiency claim
and creates a second independent source of truth for Structural Verification: the
instruction-derived heading patterns and the builder-declared skeleton are checked jointly,
making builder-commitment failures (declared but not produced) distinguishable from
heading-pattern misses (produced but wrong format).

## Criteria

### Essential (must all pass — without these the output is not useful)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-01 | **Field completeness** — Every criterion entry contains all five required fields: ID (C-NN), text with a bold label, weight (essential/recommended/aspirational), category (correctness/depth/format/coverage/behavior), and a pass condition. | format | essential | Count the fields present for each criterion. Pass if no criterion is missing any of the five fields. A criterion with "see above" or an empty pass condition is a fail. |
| C-02 | **Tier count compliance** — The rubric contains 3-5 essential criteria, 2-3 recommended criteria, and 1-2 aspirational criteria — no more, no fewer. | correctness | essential | Count criteria in each tier. Pass only if all three counts fall within the specified ranges. A rubric with 2 essential criteria or 4 recommended criteria fails immediately. |
| C-03 | **Essential criteria grounded in target skill** — Essential criteria cover behaviors that are specific and non-negotiable for the named target skill, not generic output-quality criteria that would apply to any skill. | correctness | essential | Read the essential criteria and ask: could these criteria be copy-pasted unchanged onto a rubric for a completely different skill? Pass if fewer than 2 of the essential criteria are purely generic (e.g., "output is well-formatted" with no connection to the skill contract). |
| C-04 | **Testable pass conditions** — Every pass condition names a specific, observable outcome that an evaluator can verify without subjective judgment. | correctness | essential | For each pass condition, ask: can two independent evaluators reach the same pass/fail verdict without discussion? Pass conditions containing only vague language ("good", "sufficient", "appropriate") with no concrete anchor are a fail. |
| C-05 | **Composite score formula and golden threshold present** — The rubric includes the composite scoring formula using the standard weights (essential 60, recommended 30, aspirational 10) and states the golden threshold condition. | format | essential | Check that the score section contains a formula matching the standard composite structure and a golden threshold statement. Missing formula or threshold is a fail. |

### Recommended (output is better with these)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-06 | **Category diversity** — Criteria span at least three distinct categories across the full rubric, avoiding clustering in a single dimension. | coverage | recommended | Count distinct categories used. Pass if >= 3 of the 5 canonical categories (correctness/depth/format/coverage/behavior) appear across all criteria. |
| C-07 | **Criteria ordered by severity within tiers** — Within the essential tier, criteria are listed from most structurally critical (output unusable if this fails) to least. Same ordering principle applied to recommended and aspirational tiers. | depth | recommended | Read the essential criteria in order. If swapping C-01 and C-04 would produce a more logical triage sequence, the ordering fails. Pass if the first essential criterion represents the single highest-stakes failure mode for the target skill. |
| C-08 | **Quick checklist present** — Output includes a checklist block (markdown checkboxes) summarizing all criteria IDs and their key assertion for rapid manual scoring. | format | recommended | A checklist section exists with one checkbox line per criterion referencing its ID and a short description. Pass if all criteria IDs appear in the checklist. |

### Aspirational (raise the bar once essential/recommended are stable)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-09 | **Criteria enable iteration** — When a criterion fails, its pass condition gives enough specificity that the output author knows exactly what to fix without running the skill again to ask. Each criterion's failure mode is self-describing. | behavior | aspirational | For each criterion, ask: if this criterion failed, does the pass condition text tell the author the minimum change needed to achieve a pass? Pass if >= 80% of criteria have self-describing failure modes (not just "does not pass"). |
| C-10 | **No redundancy or contradiction** — No two criteria overlap such that passing one guarantees passing the other, and no two criteria impose logically contradictory requirements on the output. | depth | aspirational | For each pair of criteria, ask: is there a rubric output that passes one but fails the other for a non-trivial reason? If two criteria always agree, they are redundant and one should be folded in. Pass if all criteria add unique signal with no fully-overlapping pair. |
| C-11 | **Discriminating pass conditions** — Each pass condition tests a behavior that a generic rubric ("the output is clear and comprehensive") would fail to catch. Distinct from C-04 (observability): a condition can be observable but still redundant with a vague rubric if it checks only structural presence rather than skill-specific contract fulfillment. | correctness | aspirational | For each pass condition, ask the inertia test: "Could a rubric that says 'the output is clear and comprehensive' still pass this condition?" If yes for any pass condition, that condition has not been sharpened enough. Pass if every pass condition names at least one count, named field, structural pattern, or enumeration that only the target skill's contract could supply — not shared structural minimums. |
| C-12 | **Calibration example present** — The rubric includes at least one concrete positive/negative pair showing the boundary between acceptable and failing output for the most critical essential criterion. The pair must name the skill — e.g., "generic: 'output is low quality' / grounded: 'the output has no formula so quest-golden cannot compute a composite score'" — so a new evaluator can self-calibrate without a training session. | behavior | aspirational | A calibration example block exists containing one labeled generic example (shows what NOT to write) and one labeled grounded example (shows what to write) for at least one essential criterion. Both examples must reference the target skill by name or reference its output contract. Pass if a reader unfamiliar with the rubric could use the pair to correctly classify two additional borderline cases. |
| C-13 | **Phase-gate enforcement present** — The rubric's structure includes at least one hard gate that blocks forward progress until a structural requirement is verified. A gate must be explicit (e.g., "DO NOT proceed to the next step until...") and must reference a checkable condition, not an aspiration. | behavior | aspirational | A gate statement is present using explicit blocking language ("DO NOT proceed", "stop here", "must complete before advancing"). The gate condition is checkable: it names a count, a field, or a structural property, not a quality judgment. Pass if violating the gate would result in a detectably incomplete output (the gated step is one that earlier steps feed). |
| C-14 | **Calibration pair produced at point of drafting** — The calibration pair required by C-12 is written while the pass condition is being drafted, not retrospectively after all criteria are sealed. Retrospective calibration pairs lose context and risk being rationalized from the final condition rather than derived alongside it. | behavior | aspirational | The rubric's workflow positions the calibration pair as a step immediately following pass-condition drafting for each criterion, before the criterion row is sealed. A workflow that permits all calibration pairs to be written at the end, after all criteria rows are finalized, does not pass. The ordering must be: draft pass condition → write calibration pair → seal criterion — not: draft all pass conditions → seal all criteria → write all calibration pairs. |
| C-15 | **Sub-step output sequencing makes omission detectable** — When the rubric requires checking mechanisms (inertia test, calibration pair), each mechanism's output appears as an independent artifact in the output stream before the criterion is recorded in the summary table. A builder who skips a check produces a table row with no preceding artifact — a structural gap detectable by inspection, not a quality judgment. | behavior | aspirational | For each mandatory check step (e.g., inertia test, calibration pair), the rubric's instructions require that step's output to appear in the output before the criterion is entered into the table (not embedded in the table cell or omitted silently). A rubric where all checking can be done mentally with only the final table as output does not satisfy this criterion. Pass if a reviewer scanning the output would see a detectable structural gap — not just a short table cell — wherever a required check was skipped. |
| C-16 | **Multi-level gate architecture** — The rubric enforces gating at two distinct granularities: a phase gate ensuring criterion-building begins from a sufficient structural foundation, and at least one per-criterion gate ensuring each criterion is complete before the next begins. A single gate at either level alone cannot enforce both structural completeness and per-criterion integrity simultaneously. | behavior | aspirational | The rubric contains at least two gates at different granularities: (1) one phase-level gate that blocks entry to criterion-writing until a structural precondition is met (e.g., minimum failure modes enumerated, minimum categories identified), and (2) at least one per-criterion gate that blocks finalization of an individual criterion until its required sub-steps are complete. A rubric with only a phase gate or only per-criterion gates does not pass. Both gates must use explicit blocking language with checkable conditions. |
| C-17 | **Audit role or phase separation** — The rubric's workflow separates the criterion-authoring concern (achieving coverage: do the criteria address the right behaviors?) from the criterion-auditing concern (achieving specificity: are the pass conditions discriminating?). When both concerns are handled in a single step, competing cognitive demands reduce the quality of each. Separation makes both the drafts and the audits independently reviewable. | depth | aspirational | The rubric either (a) designates distinct roles for authoring and auditing criteria (e.g., Author and Challenger), each producing its own output artifacts, or (b) structures the workflow into a drafting phase that produces criterion drafts and a separate review phase that produces a structured audit of those drafts. A rubric where a single step simultaneously writes each criterion and evaluates its specificity does not satisfy this criterion. The drafts and the audit must be separately readable in the output. |
| C-18 | **Audit trail captures reasoning, not only verdicts** — The audit phase produces a structured artifact (table or equivalent) recording not just whether each pass condition passed the inertia test, but which specific skill-contract element made it discriminating. Future rubric maintainers can read why each condition survived, enabling targeted revision without re-running the full evaluation. | depth | aspirational | The rubric requires the audit artifact to include a field capturing the discriminating element for each condition that passes the inertia test — e.g., a column named "Skill-Specific Element" or "Discriminating Anchor" that names the count, field, pattern, or enumeration that ties the condition to the target skill's contract. A rubric that requires only pass/fail verdicts from the audit, with no reasoning field, does not satisfy this criterion. Pass if a rubric maintainer reading the audit table could identify what to preserve if the pass condition were rewritten. |
| C-19 | **Named-block artifact architecture** — Mandatory check outputs (inertia test records, calibration pairs) are required as named markdown section blocks (e.g., `### INERTIA-RECORD [C-NN]`, `### CALIBRATION-PAIR [C-NN]`) rather than inline prose within the drafting flow. Named blocks create heading-level gaps detectable by section scan: a missing block produces a missing section heading, which a structural scan catches without reading content. Inline prose can be abbreviated silently; a named block either exists or it does not. Distinct from C-15 (which requires artifacts to appear before the criterion row): C-19 specifies the mechanism that makes C-15's structural detectability guarantee reliable rather than content-dependent. | behavior | aspirational | The rubric's instructions specify named section headings for each mandatory check output (e.g., `### INERTIA-RECORD [C-NN]`), not inline prose or table cells. A completeness verification step references these section headings by pattern (not by content inspection). Pass if a reviewer who only scans section headings — without reading paragraph content — would detect every instance where a mandatory check was skipped. A rubric where omission is only detectable by reading and judging content length or completeness does not satisfy this criterion. |
| C-20 | **Gate must block forward progress, not repair backward omissions** — A retroactive repair step ("if the block is missing, write it now") is not a gate. Gates prevent a builder from proceeding until a condition is satisfied; repair steps detect a past omission after criterion rows are already written and request correction. Retroactive repair preserves the overall structure of criterion-first, check-second — the exact sequencing failure that C-16's per-criterion gates are designed to prevent. A rubric that labels a repair step as a "completeness check" without blocking language does not satisfy C-16's per-criterion gate requirement, even if it catches all omissions eventually. Distinct from C-16 (which requires two gate levels): C-20 requires that the per-criterion gates in C-16 are designed as forward blocks, not backward corrections. | behavior | aspirational | For each gate in the rubric's workflow, ask: does this gate prevent a builder from writing the next criterion before the current criterion's checks are complete, or does it detect missing checks after all criteria are written and request retroactive addition? Pass if every per-criterion gate uses explicit forward-blocking language ("DO NOT proceed to C-NN+1 until...") and is positioned in the workflow before the criterion row is produced — not as a post-hoc audit of a completed criteria table. A rubric containing a retroactive repair step that doubles as its per-criterion gate does not pass. |
| C-21 | **Audit table consolidates all criteria into a single cross-criterion view** — The audit phase produces a single table where each row corresponds to one criterion, enabling a maintainer to scan all criteria simultaneously and detect systemic specificity failures (e.g., every criterion anchors to counts but no criterion anchors to named field patterns). Distributed per-criterion blocks with identical fields capture the same per-criterion data but prevent cross-criterion pattern analysis without manual aggregation. Distinct from C-18 (which requires the audit to capture reasoning): C-21 requires that reasoning to be consolidated in a form that surfaces patterns across criteria, not only records per-criterion verdicts. | depth | aspirational | The rubric's audit phase produces a single table containing all criteria as rows, with columns including at minimum: criterion ID, inertia test verdict (PASS/FAIL), and the discriminating element. The audit table must be producible as a contiguous block — not assembled by scanning multiple named blocks scattered through the output. Pass if a rubric maintainer reading only the audit table (without reading the criteria table or the intermediate named blocks) could identify which criteria share the same discriminating element type and which are under-specified. A rubric where the audit output is distributed across per-criterion sections, requiring aggregation to compare, does not satisfy this criterion. |
| C-22 | **Structural sequencing enforced by both ordering constraint and explicit anti-deferral instruction** — Structural ordering constraints (e.g., requiring named blocks in sequence: CALIBRATION-PAIR before CRITERION) are necessary but not sufficient to guarantee per-criterion production. A builder can satisfy block ordering by producing all CALIBRATION-PAIR blocks in a batch before all CRITERION blocks — satisfying the structural sequence while still deferring calibration pair authorship until after all pass conditions are finalized. The rubric must pair structural sequencing with explicit behavioral language that names the prohibited deferral pattern. Distinct from C-14 (which requires calibration pairs to be produced at drafting time): C-22 identifies the architectural gap — structural ordering alone permits batch-within-phase deferral, and closing the gap requires both the ordering constraint and the behavioral instruction. | behavior | aspirational | The rubric's per-criterion calibration pair instruction contains: (1) a structural sequencing requirement specifying the block ordering (calibration pair block precedes criterion block), AND (2) an explicit anti-deferral instruction naming the prohibited behavior (e.g., "Do not defer to after the [phase name] is complete" or "Write it now, while the pass condition is in working memory — not after all criteria are drafted"). A rubric with structural ordering only, or behavioral instruction only, does not satisfy this criterion. Pass if a builder reading only the calibration pair sub-step instruction could not rationalize batch-producing all calibration pairs at the end of the drafting phase while still following the written instructions. |
| C-23 | **Structural verification phase gates Auditor entry** — C-20 requires per-criterion gates to block forward progress rather than repair backward omissions. C-23 extends this principle one level up: the structural heading-scan (or equivalent structural completeness check) must itself be a hard gate that blocks the Auditor phase from starting. A structural check positioned after Auditor work has begun is a retroactive repair at phase granularity — the same category of failure C-20 prevents at criterion granularity. Positioning the structural check before Auditor work begins makes structural completeness a prerequisite for reasoning analysis: the Auditor role never encounters structurally incomplete criteria. Distinct from C-20 (which governs per-criterion forward gates): C-23 governs the inter-phase gate between structural verification and the Auditor role. | behavior | aspirational | The rubric's structural verification step (heading-scan or equivalent) carries explicit language blocking the Auditor phase from starting until all required structural patterns are confirmed present (e.g., "The Auditor phase cannot begin until all heading patterns are verified"). The structural check must be positioned in the workflow between the Author phase and the Auditor phase, not after the Auditor has already begun processing criteria. A structural check positioned as a trailing cleanup step or as an Auditor precondition without blocking language does not pass. Pass if a builder reading the workflow would treat structural completeness verification as the last mandatory checkpoint before entering the Auditor role. |
| C-24 | **Three-level verification chain covers non-overlapping defect classes** — C-16 requires two gate levels (phase and per-criterion). C-24 requires the full verification architecture to cover three distinct defect classes in sequence, each addressed by a mechanism the prior level cannot handle: (1) per-criterion forward gate at write time — prevents omission while the criterion is being constructed, before the row exists; (2) structural heading-scan post-phase — detects gate bypass after Author phase by pattern-matching section headings without reading content, catching omissions the forward gate failed to prevent; (3) consolidated audit table at Auditor time — surfaces specificity failures and cross-criterion patterns through reasoning analysis across all criteria simultaneously, catching quality failures that structural scanning cannot detect. A verification chain with only two levels, or with three levels that address overlapping failure modes, does not satisfy this criterion. The chain's value is that each level catches exactly what the preceding levels missed. | behavior | aspirational | The rubric's workflow contains three explicitly named verification mechanisms, each targeting a distinct defect class: (a) omission prevention at criterion write time using forward-blocking language per-criterion; (b) structural bypass detection post-Author-phase using heading or pattern scan without content evaluation; (c) cross-criterion specificity and pattern analysis in the Auditor role using a consolidated view of all criteria. Pass if a rubric maintainer reading the workflow can identify which mechanism catches each of the three defect classes and confirm that no two mechanisms address the same failure mode. A rubric where two levels address the same defect class (e.g., two omission-detection mechanisms at the same phase boundary) or where one defect class has no dedicated mechanism does not pass. |
| C-25 | **Anti-deferral prohibition co-located at the operative sub-step, not only in phase header** — C-22 requires an explicit anti-deferral behavioral instruction paired with structural sequencing. C-25 specifies where that instruction must appear: inside the sub-step body it governs (e.g., directly in the calibration pair sub-step instruction), not only in a phase preamble or overview header. Phase-level framing is read once at phase entry and is easily forgotten before the relevant sub-step is reached — especially in multi-sub-step phases where the preamble may be several paragraphs above the operative step. Sub-step-level placement re-exposes the prohibition at the exact moment when the deferral decision occurs. A rubric that satisfies C-22's structural+behavioral pairing requirement but places the behavioral instruction only in a phase header, with no repetition inside the sub-step body, does not satisfy C-25. Distinct from C-22 (which requires both structural ordering and behavioral prohibition): C-25 specifies that the behavioral prohibition must be positioned where it intercepts the decision. | behavior | aspirational | The anti-deferral instruction (e.g., "Do not produce all calibration pairs after all pass conditions are finalized") appears directly within the sub-step instruction that triggers the deferrable action — not only in a phase header, preamble, or general instruction block above. A rubric where the anti-deferral instruction appears exclusively in a wrapper or overview section, with only the structural ordering constraint present in the sub-step body, does not pass. Pass if a builder who begins a new criterion while skipping the phase-level preamble — reading only the per-criterion sub-step instructions — would still encounter the anti-deferral prohibition before completing that criterion's calibration pair. |
| C-26 | **Phase-level workflow transitions marked by named section headings** — C-19 requires per-criterion artifact outputs to appear as named markdown section blocks, making their absence detectable by heading scan without reading content. C-26 extends this principle one level up: each workflow phase transition must itself be marked by a named section heading (e.g., `### AUTHOR PHASE`, `### STRUCTURAL VERIFICATION`, `### AUDITOR PHASE`). Without named phase headings, inter-phase gate positioning — required by C-23 — can only be verified by reading instruction prose, not by structural scan. With named phase headings, the structural verification step's position between Author and Auditor phases is verifiable by checking only the section heading sequence, the same mechanism used to verify per-criterion block order in C-19. Distinct from C-19 (which governs per-criterion artifact blocks): C-26 governs the phase-level structural skeleton that makes inter-phase gate positions scannable rather than prose-readable. The R5 universal positional fix — repositioning Phase 7 between Author and Auditor phases — was both necessary and sufficient to satisfy C-23 across all variations simultaneously, confirming that phase boundary position is a structural property verifiable independently of instruction content. | behavior | aspirational | The workflow uses named section headings at each phase boundary — at minimum: a heading marking the start of the Author phase, a heading marking the Structural Verification phase, and a heading marking the start of the Auditor phase. The position of the structural verification step relative to Author and Auditor phases is verifiable by reading only the section headings in order, without reading instruction prose within any phase. A workflow that uses numbered steps (e.g., "Step 7: Structural Verification") within a single continuous prose or list section does not satisfy this criterion — steps within a phase are not phase headings. Pass if a reviewer scanning only section headings can confirm that structural verification appears between the Author phase heading and the Auditor phase heading. |
| C-27 | **Cross-criterion verification produces a named artifact with a forward-blocking gate** — C-15 requires each mandatory per-criterion check to emit an independent output artifact before the criterion is sealed, making omission detectable by structural gap. C-19 requires those artifacts to be named section blocks so their absence creates a missing heading detectable by scan. C-27 extends both principles to cross-criterion verification mechanisms — any step that operates across multiple criteria simultaneously (e.g., a redundancy check, a pairwise independence table): such a step must produce a named artifact (e.g., `### REDUNDANCY-CHECK-TABLE`) and must carry an explicit forward-blocking gate preventing the Auditor from advancing past it until its requirements are satisfied. A cross-criterion check performed as a mental step or embedded as observations within the consolidated audit table produces no independently scannable artifact; its omission is undetectable by heading scan. Distinct from C-10 (which requires that redundancy checking exists): C-27 requires that the redundancy checking mechanism, when present, uses the same structural architecture — named block, forward gate — as per-criterion checks governed by C-19 and C-20. | behavior | aspirational | Any cross-criterion verification mechanism in the workflow (redundancy check, pairwise independence verification, coverage analysis, or equivalent) produces a named section block (e.g., `### REDUNDANCY-CHECK-TABLE`) as a distinct output artifact in the output stream. The mechanism carries an explicit forward-blocking gate using language such as "DO NOT proceed to [next step] until all pairs show YES or NO-flagged pairs are resolved." A cross-criterion check that produces only inline observations within another artifact (e.g., a note at the bottom of the Audit Table), or that runs as a mental step with no artifact, does not satisfy this criterion. Pass if a reviewer scanning section headings would detect the omission of any cross-criterion check by the absence of its named block heading. |
| C-28 | **Scoring formula denominators explicitly annotated with the criterion ID range they cover** — The scoring formula uses a fixed denominator for each tier (5 for essential, 3 for recommended, N for aspirational). As aspirational criteria are added across rubric versions, the aspirational denominator must be updated to match the actual criterion count. A formula with a stale denominator silently miscalculates scores: a rubric with 20 aspirational criteria evaluated against denominator 17 appears to score higher than it should, masking coverage gaps. C-05 requires the formula to be present; C-28 requires the formula to self-document its denominators by embedding the criterion ID range each denominator covers — either as an inline annotation (e.g., `aspirational_pass / 20  /* C-09 through C-28 */`) or as a labeled definition block immediately adjacent to the formula. With this annotation, a maintainer verifying the formula can confirm the denominator by counting criteria in the ID range, without cross-referencing a separate note or remembering the baseline count from a prior version. Distinct from C-05 (which requires the formula to be present and correct): C-28 requires the formula to make its correctness self-verifiable. | format | aspirational | The scoring formula section contains, for the aspirational tier's denominator, an explicit annotation identifying the criterion ID range it covers (e.g., a comment, a parenthetical, or a labeled definition block stating "C-09 through C-NN" where NN matches the last aspirational criterion ID). A formula that states only a bare number (e.g., `/ 20`) with no identification of which criterion IDs that number covers does not satisfy this criterion, even if a note elsewhere in the document records the count informally. Pass if a rubric maintainer reading only the scoring formula section — without reading the criteria table or any separate notes — could verify that the aspirational denominator matches the actual count of aspirational criteria by counting criterion IDs in the stated range. |
| C-29 | **Mechanism overview declares structural output format requirement alongside behavioral requirement** — C-27 requires cross-criterion verification mechanisms to produce named section blocks; C-19 requires per-criterion check mechanisms to produce named section blocks. Both criteria govern what happens at execution time — when the builder reaches the operative step. C-29 requires that the same structural format requirement also appear in the opening mechanism overview (or equivalent orientation section), alongside the behavioral description of what the mechanism does. A mechanism overview that names the behavior ("pairwise redundancy check") without naming the required structural format ("under a named `###` section heading") withholds the structural contract until the builder reaches the operative step — after they may have already planned their output skeleton. When the format requirement appears in the overview, a builder reading only the orientation section knows which mechanisms produce named headings, can plan the output structure before beginning, and encounters the structural contract at both orientation time and execution time. Distinct from C-27 (which requires the named heading to exist in the output): C-29 requires the mechanism overview to declare that requirement, making the structural contract visible from first reading. | behavior | aspirational | The rubric's opening mechanism overview (or equivalent orientation section that lists and describes the workflow's key mechanisms before the phase instructions begin) names the structural output format — specifically, whether a named `###` section heading is required — for every mechanism governed by C-19 or C-27. A mechanism overview entry that describes only behavior ("the Auditor performs a pairwise redundancy check") without naming the format ("under a named `###` section heading") does not satisfy this criterion for that mechanism. Pass if a builder reading only the mechanism overview — without reading any phase-level instructions — could identify which mechanisms require named section headings and could therefore construct the structural heading skeleton of their output before beginning any phase. A rubric whose mechanism overview omits structural format requirements entirely, leaving them discoverable only in the operative step instructions, does not pass. |
| C-30 | **Structural Verification explicitly declares its phase-bounded coverage scope and names the deferral destination for out-of-scope checks** — C-24's three-level verification chain assigns each level a distinct defect class. C-23 requires Structural Verification to gate Auditor entry. Neither criterion addresses a silent gap: a Structural Verification step that lists required heading patterns without acknowledging its phase boundary implies comprehensive structural coverage. A builder completing the scan without knowing its scope might treat it as confirming all structural requirements — including Auditor Phase headings that have not yet been produced. C-30 requires the Structural Verification step to explicitly state: (1) which phase's output it covers (Author Phase), (2) which required heading patterns are out of scope because they belong to a later phase (e.g., `### REDUNDANCY-CHECK-TABLE` in the Auditor Phase), and (3) where each deferred check will be performed (e.g., "deferred to AUDITOR GATE entry"). This prevents builders from treating any single verification level as covering more than its phase boundary, and prevents silent scan-coverage gaps from accumulating as new phase-specific headings are added. Distinct from C-24 (which requires three levels with non-overlapping defect classes to exist): C-30 requires each level to make its scope boundary explicit in the instructions, so the chain's coverage is self-documenting rather than inferred from phase sequencing. | behavior | aspirational | The Structural Verification step's instructions explicitly state: (a) the phase whose output it covers (e.g., "covers Author Phase output"), (b) at least one required heading pattern that is out of scope at this step because it belongs to a later phase, named by heading pattern (e.g., "`### REDUNDANCY-CHECK-TABLE` — Auditor Phase, not yet produced"), and (c) the location where each out-of-scope check will be performed (e.g., "verified at AUDITOR GATE entry"). A Structural Verification step that lists only the heading patterns it checks, with no acknowledgment of out-of-scope patterns or their deferral destination, does not satisfy this criterion — even if the later checks are present elsewhere in the workflow. Pass if a builder reading only the Structural Verification instructions could identify both the scan's coverage boundary and where any deferred heading checks will occur, without reading any other section of the workflow. |
| C-31 | **Pre-phase output skeleton declared from MECHANISMS alone and cross-referenced in Structural Verification** — C-29 requires the mechanism overview to declare the structural format (named headings) each mechanism will produce, making the structural contract visible at orientation time. ## Scoring

```
composite = (essential_pass / 5 * 60)
          + (recommended_pass / 3 * 30)
          + (aspirational_pass / 23 * 10)  /* C-09 through C-31 */
```

**Note**: aspirational denominator is 23 (C-09 through C-31).

**Golden threshold**: all 5 essential criteria pass AND composite >= 80.

| Band | Score | Interpretation |
|------|-------|----------------|
| Golden | all essential + >= 80 | Ship-ready rubric — use as objective function |
| Acceptable | all essential + 70-79 | Minor gaps, usable with reviewer awareness |
| Marginal | all essential + < 70 | Coverage or ordering gaps, iterate before use |
| Fail | any essential fails | Not a valid objective function regardless of total score |
