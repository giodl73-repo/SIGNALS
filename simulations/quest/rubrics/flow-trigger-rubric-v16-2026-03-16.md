# Scoring Rubric — flow-trigger

Evaluate outputs from the `flow-trigger` skill: simulate which automations fire when a
record changes. For each field or event change, the output must list every trigger that
fires, the order they fire, their inputs/outputs, and any side effects. It must identify
trigger storms, missing triggers, and circular triggers. Stock role: Power Automate /
Copilot Studio domain expert.

---

## Composite Score Formula

```
composite = (essential_pass / N_essential * 60)
          + (recommended_pass / N_recommended * 30)
          + (aspirational_pass / N_aspirational * 10)
```

N_essential = 5 | N_recommended = 3 | N_aspirational = 37

**Point weights (raw):** Essential 10 | Recommended 5 | Aspirational 2.5 (PARTIAL = half)
**Max (raw):** 157.5

**Golden threshold**: all essential criteria pass AND composite >= 80.

## Essential Criteria (60 pts total — must all pass)

| ID   | Criterion | Category | Pass Condition |
|------|-----------|----------|----------------|
| C-01 | **Trigger enumeration** — For every field or event change in the scenario, the output lists every trigger that fires. No expected trigger is silently omitted. | correctness | All triggers that should fire for the given change appear as named entries. Zero silent omissions. |
| C-02 | **Execution order stated** — Triggers are presented in the sequence they fire (sequential, parallel, or chained), grounded in the platform's execution model — not in arbitrary or alphabetical order. | correctness | Output shows an ordered sequence or explicit parallel grouping consistent with platform execution semantics. A flat unordered list fails. |
| C-03 | **Inputs and outputs per trigger** — Each trigger entry includes: what data it receives (trigger input / payload) and what it produces or mutates (output, field write, notification, downstream call, or explicit "no output"). | depth | At least 80% of trigger entries carry explicit input and output/side-effect content. Generic placeholders ("does something") do not count. |
| C-04 | **Three anomaly class verdicts declared** — The output renders a named verdict for each of the three anomaly classes: trigger storm, missing trigger, and circular dependency. Each is explicitly labeled (e.g., "Storm: not detected"). An output that assesses fewer than all three classes, or lists findings without a per-class named verdict, fails. | coverage | All three anomaly verdicts appear in the output. Absence of any class's verdict is a fail even when the scenario is clean. |
| C-05 | **Platform grounding** — Trigger types, execution tiers, and flow categories use the platform's approved vocabulary (sync plugin step, async plugin step, instant flow, automated flow, scheduled flow, etc.). Claims about order, payload shape, and side effects are internally consistent with that platform's documented behavior. | correctness | At least one platform is named and all platform-specific claims are consistent with that platform. Contradictions with well-known platform behavior are a fail. |

## Recommended Criteria (30 pts total)

| ID   | Criterion | Category | Pass Condition |
|------|-----------|----------|----------------|
| C-06 | **Circular trigger analysis** — The output traces whether any trigger's side effects could re-fire a trigger in the same chain. It either identifies a cycle (with the loop path named) or explicitly confirms no cycles were found and states why. | depth | A circular-trigger analysis section is present. Bare "no cycles" with no reasoning is a weak pass; tracing the dependency graph earns full credit. |
| C-07 | **Conditional branch paths** — Where a trigger has conditional logic (filter conditions, run-after settings, switch branches), the output identifies which branch fires for the given inputs and why at least one alternative branch does not fire. | depth | For every trigger with visible conditional logic, the firing branch and at least one skipped branch are stated. Triggers with no conditional logic pass by default. |
| C-08 | **Anomaly remediation proposed** — For each detected anomaly (storm, missing trigger, cycle), the output proposes a concrete fix: debounce strategy, missing trigger registration, concurrency control, cycle-break condition, or re-sequencing recommendation. | behavior | Every detected anomaly has at least one actionable, named remediation step. Anomaly sections that describe problems without prescribing fixes do not earn credit. If no anomalies are detected, criterion passes vacuously. |

## Aspirational Criteria (10 composite pts | 2.5 raw pts per criterion)

| ID   | Criterion | Category | Pass Condition |
|------|-----------|----------|----------------|
| C-09 | **Execution tier and latency flags** — The output labels each trigger's execution tier (sync / async / scheduled) and flags any timing implications: async delay windows, throttle limits that could defer execution, the 24-hour re-trigger deferral window in Power Automate, or concurrency settings that serialize otherwise-parallel triggers. | depth | At least one timing or throttle consideration is surfaced per trigger chain. "Runs async" without any specifics is a weak pass. |
| C-10 | **Cascade completeness — side-effect writes spawn trigger rows** — When a trigger's side-effect field write carries downstream automation potential, the output adds the cascaded trigger as the next numbered entry in the sequence rather than leaving it as a dangling annotation. An output that annotates side effects but does not add downstream trigger rows for those with automation potential fails. |
| C-11 | **Candidate denominator declared** — Before the detailed trigger enumeration begins, the output states the full set of trigger candidates considered: a named list or explicit count of every trigger that was evaluated, including those that did not fire. This makes completeness auditable independent of the scorer's platform knowledge. | coverage | The output contains a pre-enumeration candidate set (list or count) that covers both firing and non-firing triggers. An output that only lists what fired, with no reference to what was considered and ruled out, fails. |
| C-12 | **Gap tokens for non-firing candidates** — When a candidate trigger is evaluated and does not fire for the given input, the output inserts an explicit gap token (e.g., `[NOT FIRED — {reason}]`) in the numbered sequence at the position where that trigger would have appeared. Silent omission of non-firing candidates is a structural gap; named disposal is a structural pass. | coverage | At least one non-firing candidate (where present in the scenario) is disposed of with an explicit inline token and a stated reason. An output that identifies candidates in a preamble but then silently drops them from the numbered sequence fails. If all candidates fire, criterion passes vacuously. |
| C-13 | **Anomaly verdict evidence citation** — Each anomaly verdict (storm, missing trigger, circular) cites the specific numbered trigger rows that support the finding. A clean verdict ("Storm: not detected") must cite the row range inspected. A positive finding must name the rows involved in the anomaly. Verdicts that assert without row-level citation are weak passes at best. | depth | Every anomaly verdict — clean or positive — references at least one numbered row or row range from the trigger enumeration section. A verdict section with no row citations fails. |
| C-14 | **Scope-closing event gate** — Before any candidate is named, the output declares a formal scope constraint — an event tuple specifying at minimum: entity type, operation (create/update/delete), and triggering field or event — that bounds which trigger candidates are eligible. A denominator (C-11) counts candidates; this gate defines what qualifies as a candidate. | coverage | A formal scope declaration (event entity, operation, field or event bounds) appears before candidate enumeration. A narrative preamble that does not constitute a named, structured gate fails. |
| C-15 | **Bilateral counterfactual per entry** — Every enumerated entry — firing and non-firing — carries a counterfactual that states the flip condition: firing entries declare what would cause the trigger to NOT fire; gap entries declare what would cause it to FIRE. This collapses C-07 (conditional branch paths) and C-12 (gap tokens) into a single unified per-entry obligation, eliminating the structural risk that one side of the pair is satisfied while the other is omitted. | depth | At least 80% of entries (firing and non-firing combined) carry an explicit counterfactual statement. An output where counterfactuals appear only in a summary section rather than inline per entry is a weak pass. An output where only one direction (flip-to-fire or flip-to-not-fire) is covered across the full enumeration fails. |
| C-16 | **Per-entry registration witness** — Every enumerated entry — firing and non-firing — carries a registration witness: a reference to the concrete platform artifact (plugin step registration, flow trigger configuration, solution layer, etc.) that registers the trigger. Non-firing entries carry `[UNWITNESSED]` when no registration artifact can be cited, making the gap explicit rather than silent. C-05 grounds the output at platform level; this anchors every individual row to a specific registration artifact. | coverage | At least 80% of entries carry an explicit registration witness or an `[UNWITNESSED]` token. An output that cites a platform globally but does not anchor individual rows to registration artifacts fails. |
| C-17 | **Per-entry execution order rule citation** — The output declares a named Execution Order Rule (EOR) table before enumeration, assigning a rule ID (e.g., EOR-01, EOR-02) to each ordering principle the platform enforces (sync-before-async, plugin-step-tier, etc.). Every firing entry then carries an inline `Positioned because: EOR-{NN}` citation referencing the specific rule that places it at its position. C-02 requires execution order to be stated globally; this criterion requires it to be auditable at the individual entry level. | correctness | A named EOR table appears before enumeration and at least 80% of firing entries carry an explicit EOR rule citation. An output that states a global ordering rationale without per-entry rule anchors is a weak pass. An output with no EOR table fails regardless of how well-ordered the entries appear. |
| C-18 | **Cascade depth budget with overflow entry** — Before enumeration, the output declares a CASCADE DEPTH BUDGET (a named maximum chain depth). Every cascade entry carries a `[Depth: N/MAX]` counter showing its position against the budget. When a chain exceeds the budget, a formal `[DEPTH EXCEEDED]` entry terminates it and the storm verdict explicitly checks whether any depth-exceeded entries exist (DE > 0). C-10 requires cascade chains to be traced end-to-end; this criterion requires a formal depth accounting that makes runaway chains structurally visible and storm-verdict-detectable rather than silently absorbed into the enumeration. | correctness | A named CASCADE DEPTH BUDGET appears before enumeration. At least one cascade entry (where present) carries a `[Depth: N/MAX]` counter. If any chain exceeds the budget, a `[DEPTH EXCEEDED]` entry is present and the storm verdict references it. An output that traces cascade chains without depth accounting fails. If no cascade chains are present, criterion passes vacuously. |
| C-19 | **Pre-enumeration exclusion log with verdict citation** — Before the candidate scan begins, the output produces a named EXCLUSION LOG: a dedicated sweep of all automation layers that records each layer considered and states whether it was included in or excluded from the candidate set, with a reason for every exclusion. Anomaly verdict blocks carry an `Exclusion log reference:` field that cites specific exclusion log entries, completing three-part accounting: (a) excluded layers, (b) in-scope non-firers, (c) firers. C-14 closes the scope gate on what qualifies as a candidate; this criterion documents what was swept and ruled out before that gate, and ensures verdicts anchor to the log. C-13 requires verdicts to cite enumeration rows; this criterion adds a second required citation artifact. | coverage | A named EXCLUSION LOG appears before candidate enumeration, covers at least two automation layers, and records a disposition (included / excluded + reason) for each. At least one anomaly verdict block carries an `Exclusion log reference:` field. An output whose verdicts cite only enumeration rows but omit exclusion log references fails. |
| C-20 | **Zero-tolerance closure counters** — After enumeration and verdicts, the output produces a named CLOSURE CHECK block containing a counter for each mandatory per-entry obligation (e.g., `EOR citations missing: {count} [must be 0]`, `Gap entries missing counterfactual: {count} [must be 0]`, `Entries missing registration witness: {count} [must be 0]`, `Depth-exceeded chains unresolved: {count} [must be 0]`). Each counter asserts a zero-tolerance condition. This makes the document self-auditing: completeness of per-entry obligations (C-15 through C-19) is verifiable at a glance by reading the closure block rather than rescanning every entry. An output that satisfies per-entry criteria but provides no closure accounting fails this criterion. | coverage | A named CLOSURE CHECK block appears after enumeration. It contains at least three named counters covering distinct per-entry obligations. Each counter carries a `[must be 0]` or equivalent zero-tolerance assertion. A closure block with narrative summary but no named counters is a weak pass. |
| C-21 | **Named role prohibition contracts** — In a multi-phase or multi-role output, each role or phase carries at least one explicitly named PROHIBITION statement — a declaration of what that role is structurally barred from doing (e.g., "Role 3 may not add new candidate entries; it may only classify existing rows"). Prohibitions are cited by name when a violation would otherwise be structurally possible. C-14 (scope-closing event gate) closes scope by event definition; this criterion closes scope by role-action definition, making it impossible for a later phase to silently expand the candidate set. An output that relies solely on gate tokens or phase ordering to enforce role boundaries, without explicit named prohibitions, is a weak pass. | correctness | At least one named PROHIBITION statement appears per role or phase in a multi-role output. Prohibitions must be stated as explicit declarations of barred actions, not implied by phase order. An output with a single role or single phase passes vacuously. |
| C-22 | **Uniform "all slots required" mandate** — The output declares a global template invariant, stated once and applied to every entry type (firing, non-firing, cascade, verdict): every named slot in every entry template is required; an empty named slot is a structural gap equivalent to a missing entry, not a silent "not applicable." This collapses the per-entry obligations of C-15, C-16, C-17, C-18, and C-19 into a single auditable structural rule, making partial omission detectable by slot-presence alone rather than by re-evaluating each criterion independently. An output that states individual per-entry obligations (counterfactual required, witness required, etc.) scattered across separate sections without a unifying global mandate is a weak pass. | correctness | A single named structural invariant appears in the output — before or alongside the entry templates — declaring that all named slots in all entry types are required and that empty named slots constitute structural gaps. The mandate must apply uniformly across at least three entry types (e.g., firing root, non-firing, cascade, verdict). |
| C-23 | **Pre-enumeration artifact lock — structural citeability for closure counters** — Before enumeration begins, each structural artifact that will be referenced by the CLOSURE CHECK (e.g., SCOPE GATE, EXCLUSION LOG, PROHIBITION CONTRACTS, EOR TABLE, CASCADE DEPTH BUDGET) is assigned a reference ID (ART-NN format or equivalent) in a named ARTIFACT MANIFEST. When artifact reference IDs are pre-declared, closure counters are formally verifiable — the manifest guarantees the referenced artifact exists, so the counter resolves by name rather than by scanning the document for the artifact's presence. C-20 requires a CLOSURE CHECK to exist; this criterion requires the artifacts counted in that check to have been locked as named manifest entries before enumeration begins. An output where prohibition contracts or other artifacts emerge during execution (rather than being locked before enumeration) fails because its closure counters reference artifacts whose existence is not guaranteed at CLOSURE CHECK time. | coverage | A named ARTIFACT MANIFEST (or equivalent pre-enumeration artifact registry) appears before the first enumeration entry. The CLOSURE CHECK references at least two artifact IDs or manifest-declared artifact names. An output where CLOSURE CHECK counters reference section headings or criteria labels rather than pre-declared artifact IDs is a weak pass. |
| C-24 | **Prohibition breach markers — named runtime anomaly signals in the output** — The output defines a formal named breach token (`[PROHIBITION BREACH: Role N \| {violated prohibition name}]` or equivalent) and inserts it inline — without interrupting enumeration flow — at any point where a role takes an action its PROHIBITION CONTRACT explicitly barred. C-21 requires PROHIBITION CONTRACTS to be declared; this criterion requires that violations of those contracts produce named, visible signals in the artifact itself rather than being detectable only through external rubric evaluation. Enforcement is asymmetric if compliance is structurally ensured (no breach = no marker) but violations are rubric-detectable only: this criterion closes that asymmetry. An output with no prohibitions defined (C-21 vacuous pass) passes vacuously. An output where C-21 passes but no breach marker protocol is defined is a weak pass. | correctness | The output defines at least one named breach token format before or alongside the prohibition contracts. In a complete output, breach tokens appear inline wherever a breach occurs — or the CLOSURE CHECK explicitly states "PROHIBITION BREACHES: 0 [must be 0]." An output that defines prohibition contracts (C-21 pass) but contains no breach marker protocol fails. |
| C-25 | **Named pre-enumeration lifecycle phase with typed exit conditions** — The output declares a named pre-enumeration lifecycle phase (e.g., "Phase 0 — Manifest Lock") with formally stated entry conditions and a typed list of exit conditions. This mechanism makes C-14 (scope-closing event gate) a structural obligation rather than a best-practice: the event tuple becomes one named exit condition among a set, so any output that begins enumeration without declaring scope has structurally failed the phase gate — not merely omitted a recommended practice. C-23 requires artifacts to be locked before enumeration; this criterion provides the lifecycle mechanism that enforces that requirement: the phase exit signal is the structural proof that locking occurred. An output that locks artifacts by role prohibition alone (without a lifecycle phase gate) or by format pre-population alone does not satisfy this criterion; the exit signal must be an explicit, declared lifecycle event. | coverage | A named pre-enumeration lifecycle phase appears with at least three explicitly listed exit conditions. At least one exit condition names scope declaration (event tuple, trigger scope, or equivalent). The phase carries an explicit exit signal statement (e.g., "Phase 0 EXIT SIGNAL: all exit conditions met — enumeration may begin"). Enumeration entries must not appear before the exit signal. A lifecycle section with narrative checklist but no formal exit signal statement is a weak pass. |
| C-26 | **INERTIA CONTRAST — embedded design rationale for structural mechanisms** — The output contains a named INERTIA CONTRAST section (or equivalent) that documents the structurally weaker alternative the current design improves upon: for at least two named structural mechanisms (e.g., named artifact IDs, breach tokens, Phase 0 gate, prohibition contracts), the section states (a) what the inertia-driven alternative looks like — the pattern that prior or simpler outputs would use, and (b) the specific failure mode that alternative produces — the observable error, omission, or undetectable violation that occurs when the stronger mechanism is absent. This section makes each structural choice's rationale traceable within the artifact itself without requiring access to the external rubric. C-23 and C-24 establish structural mechanisms for artifact citeability and breach visibility; this criterion ensures those mechanisms are accompanied by a rationale that survives rubric-version changes: a reader of the artifact can determine why each mechanism exists even when the rubric that motivated it is unavailable. An output that satisfies C-23 and C-24 but provides no embedded motivation is at risk of criterion-drift: future variations may silently weaken the mechanisms because the reason they exist is not co-located with the structure. | depth | A named INERTIA CONTRAST section (or equivalent rationale section) appears in the output. It names at least two structural mechanisms by name and for each states both (a) the weaker alternative pattern and (b) the specific failure mode the stronger mechanism prevents. The failure mode description must be concrete — a named or described error class, not a generic "this is worse." An output that motivates mechanisms through inline comments scattered across role mandates rather than a consolidated rationale section is a weak pass. |
| C-27 | **Computable Phase 0 exit signal via typed status aggregate** — Where C-25 requires a named exit signal, this criterion requires that signal to be *computable*: each exit condition entry in the Phase 0 registry carries a typed status value from a named enumeration (e.g., `Status: SATISFIED \| PENDING \| BLOCKED`), and the exit signal is derived from the aggregate of all status values ("5/5 SATISFIED — enumeration authorized") rather than stated as a free-text assertion. This advances C-25 by making exit signal trustworthiness independent of the author's assertion: an evaluator who doubts the exit signal can verify it by re-reading the status column, not by re-assessing each obligation's prose. A narrative Phase 0 checklist with no typed status column fails. |
| C-28 | **Per-obligation refutation conditions — string-detectable violation criteria** — Each named pre-enumeration obligation (scope gate declaration, artifact manifest lock, breach token protocol, EOR table, cascade budget) carries a formal refutation condition: an explicit, named statement of the form "This obligation is violated if [condition]" or equivalent, where [condition] is specific enough to be detected by string match or structural inspection against the output artifact — without requiring semantic re-evaluation against the rubric. | C-29 | **INERTIA CONTRAST as typed inline columns — cell-level rationale co-location** — Where C-26 requires a named INERTIA CONTRAST section covering at least two mechanisms, this criterion captures the advance where the contrast is embedded as typed column slots within the mechanism's own registry row, rather than in a separate section. This eliminates the navigation dependency between a mechanism row and its rationale section: a reader examining any single mechanism row immediately sees both the mechanism and the reason it exists, without section-jumping. C-26 requires that at least two mechanisms' rationale survive rubric-version changes; this criterion requires that survival to be structurally guaranteed at the cell level rather than the section level — rationale cannot be stripped from a mechanism by removing a section heading. An output that satisfies C-26 with a consolidated INERTIA CONTRAST section but does not embed contrast columns within each mechanism row is a weak pass. An output with no Phase 0 registry or equivalent mechanism table fails. | depth | The pre-enumeration registry (Phase 0 or equivalent) contains at least two named columns for contrast data ("Weaker alternative" and "Failure mode" or equivalent typed column names). At least two mechanism rows carry non-empty values in both contrast columns. The contrast columns are defined in the table header — not appended as inline prose annotations to individual rows. An output that achieves C-26 via a standalone INERTIA CONTRAST section but has no contrast columns in the mechanism registry is a weak pass. |
| C-30 | **Temporally-anchored prohibition activation events** — Where C-21 requires named role prohibition contracts, this criterion requires at least one prohibition to carry an explicit temporal activation anchor: a named lifecycle event after which the prohibition takes effect (e.g., "Domain Expert MAY NOT add candidates *after denominator declaration*"). A prohibition without a temporal anchor applies ambiguously across all phases — a role agent could comply during enumeration while violating the prohibition before it was active. A temporally-anchored prohibition creates a lifecycle-bound invariant: from the named activation event forward, the barred action produces a detectable protocol violation. This couples prohibition enforcement to the Phase 0 lifecycle gate structure (C-25): the activation event is a named Phase 0 exit condition or its immediately-following transition, so the prohibition's effective period is self-evident from the Phase 0 registry. The invariant most commonly targeted is denominator integrity (no candidate additions after the denominator is declared) or scope gate immutability (no scope modifications after the event tuple is locked), because these are the Phase 0 artifacts most vulnerable to silent expansion by a later-phase role. An output where all prohibitions are stated globally ("Role N may not do X" with no temporal scope) is a weak pass. An output whose prohibitions are lifecycle-anchored but reference events not named in the Phase 0 registry fails because the activation event is not self-verifiable. | correctness | At least one named prohibition carries an explicit temporal activation anchor referencing a named Phase 0 exit condition or lifecycle transition (e.g., "after denominator declaration," "once SCOPE GATE is locked," "following EXIT SIGNAL"). The prohibition's effective period is determinable by reading the Phase 0 registry without consulting external documentation. A multi-role output where prohibitions are stated as timeless role-level constraints, with no activation event, is a weak pass. An output with a single role or single phase passes vacuously. |
| C-31 | **Role-attributed artifact production mandate** — Where C-23 requires each pre-enumeration artifact to be locked in a named manifest before enumeration begins, this criterion requires the manifest to additionally name the producing role and production phase for each artifact entry. An artifact that is declared in the manifest but has no assigned producing role can be absent without identifying which role failed to produce it: the gap is anonymous and the accountability is unresolvable from the artifact alone. With a named producing role, a missing artifact is a named role deliverable failure — the CLOSURE CHECK can report "ART-02 (EXCLUSION LOG) — Role 1 (Auditor): ABSENT" rather than "ART-02: not found." This is the affirmative-obligation counterpart to C-21's prohibition contracts: C-21 names what roles may not do; this criterion names what roles must produce. The advance is that artifact accountability becomes role-traceable, enabling the CLOSURE CHECK to attribute production gaps to specific roles rather than treating them as anonymous structural omissions. An output where artifacts are declared in the manifest with IDs but no producing role is assigned is a weak pass. An output where producing roles are identified in prose commentary but not as a named manifest column or slot fails because the attribution is not structurally inspectable. | coverage | At least two artifacts in the pre-enumeration manifest carry a named producing role and production phase as explicit manifest fields (column, slot, or inline typed label). At least one CLOSURE CHECK counter or post-enumeration accounting statement references the producing role for an artifact (e.g., "ART-02 (EXCLUSION LOG) — Role 1 (Auditor), Phase 0: PRODUCED / ABSENT [must be PRODUCED]"). An output where the manifest lists artifact IDs with no role attribution fails. An output where role attribution appears only in a prose description of roles' responsibilities — not co-located with the manifest entry — is a weak pass. |
| C-32 | **Unified Phase 0 obligation registry — all per-row compliance metadata in a single named table** — Where C-25 through C-31 each require a specific metadata type per obligation (typed status, refutation condition, contrast data, activation event, producing role), this criterion captures the structural advance where all six metadata types appear as named columns within a single Phase 0 registry table, so that any obligation's full compliance picture is inspectable from a single row without cross-table navigation.
