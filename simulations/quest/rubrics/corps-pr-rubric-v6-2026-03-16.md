# corps-pr Rubric — v6 (2026-03-16)

**Two new criteria extracted from Round 5 excellence signals:**

**C-19 — IA cost framing uses a named-row, named-column ledger schema with Budget verdict and Budget strength fields** *(closes C-17 schema gap)*

V-01's cost ledger went beyond C-17's "cost terms" requirement. C-17 requires a quantified tradeoff between maintenance cost and adoption cost. V-01 expressed this as a 4-row × 2-column schema — named rows (Maintenance, Incident exposure, Integration cost, Regression risk), named columns (Current-State Cost, Adoption Cost), and terminal fields `Budget verdict` + `Budget strength`. A skill can satisfy C-17 by mentioning costs in prose and still leave the comparison unstructured. C-19 requires the ledger to be a named schema so the comparison is independently auditable row by row.

**C-20 — Technical role phase names IA phase completion as an explicit entry condition in the pipeline declaration** *(closes C-18 IA-read-prerequisite gap)*

V-02's pipeline declaration stated "Phase B exit met; roles read Phase B before writing" as the entry condition for Phase C. C-11 requires sequencing, C-14 requires per-role anchoring, C-18 requires phase gates — but none require the pipeline entry condition to explicitly declare the IA read prerequisite. A skill can pass all three prior criteria and still declare Phase C with a generic entry condition (e.g., "all files received"). C-20 closes this: the entry condition must name IA phase completion and the read dependency as a structural gate, not a sequencing convention.

```
composite = (essential_pass / 5 * 60)
          + (recommended_pass / 3 * 30)
          + (aspirational_pass / 12 * 10)
```

**Golden threshold**: all 5 essential criteria pass AND composite >= 80.

## What changed from v5

**C-19 closes the C-17 schema gap.** C-17 requires the Inertia Advocate's section to frame the status quo in cost terms -- a quantified tradeoff between maintenance cost and adoption cost. Round 5 (V-01) showed a variation that went further: the cost framing was expressed as a 4-row x 2-column named ledger, with row categories (Maintenance, Incident exposure, Integration cost, Regression risk) and column headers (Current-State Cost, Adoption Cost), plus explicit terminal fields for `Budget verdict` and `Budget strength`. A section can pass C-17 by mentioning cost figures in prose or as bullets and still leave the comparison unstructured. C-19 requires the ledger to be a named-schema artifact: each cost category is a named row, each perspective is a named column, and the IA's overall position is captured in named terminal fields. This makes the cost comparison independently auditable -- a reviewer can check each row independently rather than parsing prose to determine whether all cost dimensions were addressed.

**C-20 closes the C-18 IA-read-prerequisite gap.** C-18 requires the review to be structured as named phase gates with explicit entry and exit conditions. C-11 requires the IA section to be sequenced first and structured as a reference object. C-14 requires each technical role to anchor at least one finding against the IA verdict. Round 5 (V-02) revealed a gap in the intersection: even with phase gates in place, C-18 does not specify what the entry condition for the technical role phase must contain. V-02's pipeline declaration stated "Phase B exit met; roles read Phase B before writing" as the explicit entry condition for Phase C -- making IA completion a structural read prerequisite, not just a sequencing convention. C-20 requires this entry condition to be declared explicitly in the pipeline. A skill that sequences the IA first (C-11), gates phases (C-18), and has per-role anchors (C-14) can still fail C-20 if the pipeline entry condition for the technical role phase does not explicitly name IA phase completion and the read prerequisite.

**Scoring formula updated**: aspirational denominator 10 -> 12; each aspirational criterion worth ~0.83 pts; perfect run still 100.
