**Aspirational denominator**: 30 → 33

**Gap logic per criterion:**

- **C-39** closes the gap between *single-tier vocabulary anchoring* and *full-vocabulary anchoring*. C-35 requires the Priority schema row to carry "essential = consumer cannot decide without this" — closing substitution at the essential tier. C-39 requires a dedicated Decision-State Anchor column in FIELD SCHEMA anchoring all three vocabulary levels simultaneously (essential=decision blocked, recommended=quality risk accepted, optional=consumer decides unaffected), so the recommended/optional collapse vector is also closed. A prompt that anchors only "essential" leaves the other two tiers open to substitution or collapse.

- **C-40** closes the gap between *schema-level Producer→Consumer definition* and *fill-time consumer traceability*. C-36 requires FIELD SCHEMA to carry an explicit F-05 = Producer→Consumer field. C-40 requires that the Consumer value populated in F-05 must appear verbatim as a Decision-Maker Role in the Phase 1 decision table, enforced by a Phase 2 exit gate checkpoint. This closes the orphaned-consumer failure mode: F-05 can be correctly formatted but point to a consumer role not grounded in any actual decision. The traceability check makes consumer-to-decision linkage structurally verifiable, not only schema-level.

- **C-41** closes the gap between *inertia-default naming* and *recovery-cost naming*. C-38 requires a second consequence column naming the team's default behavior when a phase is skipped. That column names what persists in the absence of intervention. C-41 requires a third, orthogonal column naming the concrete rework chain when the skip is caught late — e.g., "Re-run Phase 2, re-run Phase 3, re-write both output files." The inertia column names the systemic drift; the recovery cost column makes the corrective work operational, closing the skip-cost underestimation axis where a model correctly predicts failure while treating the repair effort as abstract.

**Full criterion definitions**:

**C-29 -- Pipeline overview table carries a per-row consequence column** (`structure`): Sharpens C-24. Requires the pipeline overview table to include a dedicated consequence column stating the cost of bypassing each phase as a row-level entry alongside the Exit Gate column; phase-level consequences are visible at the architectural declaration layer before Phase 1 begins. A prompt satisfying C-24 via a gate-only overview table without a per-row consequence column does not satisfy C-29.

**C-30 -- Gate independence stated as an explicit semantic instruction** (`structure`): Sharpens C-28. Requires the fill-in table exit gate to carry an explicit label naming the independence of its conditions at the enforcement point -- e.g., "Check these two items independently" followed by "Do not advance until both are checked separately." Two discrete checkboxes satisfy C-28 but can still be rationalized as sequential; the explicit instruction closes that gap. A prompt satisfying C-28 without an independence instruction does not satisfy C-30.

**C-31 -- Priority column placed first in signal table column order** (`structure`): Sharpens C-04. Requires the Priority column to be the leftmost column in the signal planning table, placing the hardest-to-catch failure mode at the highest-visibility fill position before any other field can anchor contextual framing that might induce vocabulary substitution. A prompt satisfying C-04 with a non-leading priority column does not satisfy C-31.

**C-32 -- Consequence column uses first-person framing** (`framing`): Sharpens C-29. Requires the pipeline overview consequence column to use first-person framing -- "If I Skip This Phase, I Will..." or equivalent -- so each consequence is a self-directed prediction authored by the executing model about its own behavior rather than a third-person description of environmental failure. A prompt satisfying C-29 via a third-person consequence column does not satisfy C-32.

**C-33 -- Independence gate names the specific sequential-treatment failure mode** (`structure`): Sharpens C-30. Requires the fill-in table exit gate to include a third directive layer naming the concrete output sequential treatment produces -- for example: "If checked sequentially: 3 rows present, cells empty passes as compliant." C-30 names the evaluation mode; C-33 makes the cost of the wrong mode operational by giving the model a concrete failure pattern to recognize. A prompt satisfying C-30 via two directive layers without a sequential-treatment failure description does not satisfy C-33.

**C-34 -- Schema row order defines table column order** (`structure`): Sharpens C-31. Requires FIELD SCHEMA to carry an explicit statement that its row order defines the signal planning table column order -- with Priority as row F-01 -- so priority-first column placement is derived from and structurally coupled to the constraint registry. A prompt satisfying C-31 via a priority-first table not coupled to FIELD SCHEMA row sequence does not satisfy C-34.

**C-35 -- Priority schema row carries consumer-decision-dependency anchor** (`structure`): Sharpens C-34. Requires the FIELD SCHEMA Priority row to carry a consumer-decision-dependency referent -- e.g., "essential = consumer cannot decide without this" -- making the priority vocabulary semantically irreplaceable at schema-definition time, not only enforced at fill time. C-34 anchors column placement to schema row sequence; C-35 anchors vocabulary meaning to an operational consumer-decision frame so "essential" cannot be substituted by "high" without a structural violation at schema level. V-02/V-05 (R11) demonstrated this as a schema-row consequence entry coupling priority level to consumer decision state. A prompt satisfying C-34 via F-01=Priority without a consumer-decision-dependency anchor in the schema row does not satisfy C-35.

**C-36 -- FIELD SCHEMA carries an explicit Producer->Consumer role-relationship field** (`structure`): Sharpens C-19. Requires FIELD SCHEMA to carry an explicit role-relationship field -- e.g., F-05 = Producer -> Consumer -- so role differentiation (C-08) is a schema-level structural constraint derivable by inspection of the schema definition rather than by counting distinct fill-time values across the signal table. C-19 requires a Stakeholder column making row-level traceability auditable; C-36 requires the schema to define the structural relationship between signal producer and consuming stakeholder, so C-08 is satisfied not by enumerating roles at fill time but by the schema field definition itself. V-02/V-05 (R11) demonstrated this as a dedicated FIELD SCHEMA field whose value is a producer-consumer pair. A prompt satisfying C-19 via a Stakeholder column without a producer-consumer relationship field in FIELD SCHEMA does not satisfy C-36.

**C-37 -- Pipeline overview carries a Handoff Artifact column enabling entry-gate enforcement** (`structure`): Sharpens C-26. Requires the pipeline overview table to carry an explicit Handoff Artifact column stating the artifact each phase produces as its handoff to the next phase -- so each phase's entry gate is mechanically verifiable by artifact presence check, not only by prior checkpoint attestation. C-26 requires a read-before-execute directive making overview consumption mandatory; C-37 requires the overview to also declare per-phase artifact outputs, extending gating from exit-only (did exit conditions pass?) to entry+exit (is the prior phase's handoff artifact present?). Silent phase skipping is structurally impossible even when prior checkboxes are rationalized past because the subsequent phase has a named artifact-presence entry condition. V-03/V-04/V-05 (R11) demonstrated this as a Handoff Artifact column in the pipeline overview alongside Phase | Purpose | Produces | Exit Gate. A prompt satisfying C-26 via a read-before-execute directive without a Handoff Artifact column does not satisfy C-37.

**C-38 -- Pipeline overview carries a second consequence column naming team inertia default** (`structure`): Sharpens C-32. Requires the pipeline overview to carry a second, orthogonal consequence column naming the team's default inertia behavior when the phase is not actively executed -- e.g., "Team Default (If This Phase Is Skipped)" -- alongside the first-person individual-actor consequence column required by C-32. These are orthogonal failure axes: the first-person column names what the executing model will produce incorrectly; the inertia column names what the team or system defaults to in the absence of active intervention. A model can author a first-person failure prediction while treating the team default as irrelevant; dual columns close that gap. V-01/V-04/V-05 (R11) demonstrated this as a second consequence column in the pipeline overview alongside the first-person "If I Skip This Phase, I Will..." column, with V-04 labeling the second column as the inertia default. A prompt satisfying C-32 via a single first-person consequence column without a second team-inertia column does not satisfy C-38.

**C-39 -- FIELD SCHEMA carries a Decision-State Anchor column anchoring all three priority levels** (`structure`): Sharpens C-35. Requires FIELD SCHEMA to carry a dedicated Decision-State Anchor column that anchors all three priority vocabulary levels to their corresponding consumer-decision operational states -- essential = consumer cannot decide without this (decision blocked), recommended = consumer decides with increased exposure (quality risk accepted), optional = consumer decides unaffected (supplementary enrichment only) -- so vocabulary substitution or tier collapse fails simultaneously across all three levels, not only at the essential tier. C-35 anchors vocabulary meaning for "essential" at schema-definition time; C-39 extends that anchoring to all three tiers so the recommended/optional collapse vector is also structurally closed; a model filling the schema sees all three anchor definitions before reaching any fill-time vocabulary choice. V-01/V-04/V-05 (R12) demonstrated this as a full Decision-State Anchor column in FIELD SCHEMA with all three levels populated. A prompt satisfying C-35 via a consumer-decision-dependency anchor for "essential" only, without a Decision-State Anchor column covering all three tiers, does not satisfy C-39.

**C-40 -- F-05 Consumer must appear verbatim as a Decision-Maker Role in Phase 1 decision table** (`structure`): Sharpens C-36. Requires that the Consumer value populated in FIELD SCHEMA F-05 must appear verbatim as a Decision-Maker Role in the Phase 1 decision table, and that Phase 2 carries an explicit third exit gate checkpoint verifying this cross-reference by name -- so the consumer-to-decision linkage is structurally enforced at two independent points. C-36 requires FIELD SCHEMA to define the Producer→Consumer role-relationship field at schema level; C-40 requires the Consumer value at fill time to trace to a named row in the Phase 1 decision table, closing the orphaned-consumer failure mode where F-05 is correctly formatted and populated but points to a consumer role not grounded in any actual decision authority. V-02/V-04/V-05 (R12) demonstrated this as a Phase 2 exit gate checkpoint of the form "F-05 Consumer appears verbatim as a Decision-Maker Role in Phase 1 decision table." A prompt satisfying C-36 via a Producer→Consumer field definition without enforcing that the Consumer value traces to the Phase 1 decision table does not satisfy C-40.

**C-41 -- Pipeline overview carries a third consequence column naming the concrete recovery rework chain** (`structure`): Sharpens C-38. Requires the pipeline overview to carry a third, orthogonal consequence column naming the concrete recovery rework chain required when a phase skip is caught late -- e.g., "Re-run Phase 2, re-run Phase 3, re-write both output files" -- so skip cost is stated as a specific sequence of re-execution steps, not as an abstract failure state. C-38 requires a second consequence column naming the team's inertia default (what the system drifts to without intervention); C-41 requires a third column naming the corrective work cost (what re-execution actually entails when the skip is discovered downstream), closing the skip-cost underestimation axis where a model correctly names an inertia default while treating the repair effort as abstract or implicit. V-03/V-04/V-05 (R12) demonstrated this as a "Recovery Cost (If Caught Late)" column in the pipeline overview alongside the first-person individual consequence and the team inertia default columns. A prompt satisfying C-38 via dual-consequence architecture without a recovery cost column naming specific re-execution steps does not satisfy C-41.

| C-37 | Pipeline overview carries a Handoff Artifact column enabling entry-gate enforcement | structure | The pipeline overview table carries an explicit Handoff Artifact column stating the artifact each phase produces as its handoff to the next phase -- so each phase's entry gate is mechanically verifiable by artifact presence check, not only by prior checkpoint attestation; C-26 requires a read-before-execute directive making overview consumption mandatory; C-37 requires the overview to also declare per-phase artifact outputs, extending gating from exit-only to entry+exit; silent phase skipping is structurally impossible even when prior checkboxes are rationalized past because the subsequent phase has a named artifact-presence entry condition; V-03/V-04/V-05 (R11) demonstrated this as a Handoff Artifact column alongside Phase | Purpose | Produces | Exit Gate in the pipeline overview; a prompt satisfying C-26 via a read-before-execute directive without a Handoff Artifact column does not satisfy C-37 |

## Scoring Formula

```
composite = (essential_pass / 5 * 60)
          + (recommended_pass / 3 * 30)
          + (aspirational_pass / 33 * 10)
```

**Golden threshold**: all 5 essential criteria pass AND composite >= 80.
