# Scoring Rubric: org-roles (v8)

Evaluates output from the `org-roles` skill, which generates a typed role registry for a domain by analyzing product, codebase, or spec context and emitting structured role files to `.craft/roles/{area}/`.

**v8 changes**: Added C-34 through C-36 from Round 7 excellence signals. Aspirational denominator updated from 25 to 28. Notes for Evaluators extended.

---

## Criteria

### Essential (60 pts total — all must pass)

| ID | Text | Category | Weight | Pass Condition |
|----|------|----------|--------|----------------|
| C-01 | **Role schema completeness** — every emitted role file contains all six required fields: `name`, `orientation` (with `frame` and `serves` sub-fields), `lens` (with `verify_questions` and `simplify_rules` sub-fields), `expertise` (with `depth` and `relevance` sub-fields), `scope`, and `collaborates_with`. | correctness | essential | PASS if 100% of role files contain all six top-level fields and the required sub-fields under `orientation`, `lens`, and `expertise`. FAIL if any field or sub-field is absent or empty in any role file. |
| C-02 | **Devil-advocate role present** — an inertia-check role exists whose stated purpose is to challenge every proposal by arguing the status quo is sufficient. | correctness | essential | PASS if at least one role file has a name or orientation that explicitly identifies it as a devil-advocate / status-quo challenger, and its `lens.verify_questions` include at least one question of the form "why is the current approach insufficient?" FAIL if no such role exists or its job is generic skepticism without a status-quo framing. |
| C-03 | **Stock roles present** — PM, Architect, and Strategy roles are always emitted regardless of domain input. | coverage | essential | PASS if the output directory contains at least one file identifiable as each of: a product manager role, a software/systems architect role, and a strategy/business role. FAIL if any of the three stock roles is absent. |
| C-04 | **Output location correct** — all role files are written to `.craft/roles/{area}/` as individual Markdown files, where `{area}` reflects the domain derived from context. | format | essential | PASS if all files land in a `.craft/roles/` subdirectory named after the target area, and each file is a `.md` file. FAIL if files are written to a different path, bundled into a single file, or the area subdirectory is missing or named generically (e.g., `default`). |
| C-05 | **At least one context-derived domain expert** — the role set includes at least one domain expert role whose name, frame, and verify questions are specific to the analyzed product/codebase/spec, not a generic template. | correctness | essential | PASS if at least one role has a `frame` that references a domain-specific concern (e.g., "data pipeline correctness", "auth boundary enforcement", "signal fidelity") that could not apply verbatim to an unrelated domain. FAIL if every role beyond the three stock roles is a generic archetype (e.g., "Domain Expert") with no domain-specific content. |

### Recommended (30 pts total — output is better with these)

| ID | Text | Category | Weight | Pass Condition |
|----|------|----------|--------|----------------|
| C-06 | **Orientation coherence** — `frame` describes the role's epistemic viewpoint and `serves` names the stakeholder or system concern that benefits from that perspective; the two sub-fields are internally consistent and non-redundant. | depth | recommended | PASS if, for every role, `frame` is a perspective statement (how this role sees the world) and `serves` is a beneficiary statement (who or what depends on this role's signal), and the two are logically related. FAIL if `serves` is a restatement of `frame`, or if either field is blank or generic ("users", "the team"). |
| C-07 | **Lens quality** — `verify_questions` are actionable and testable (answerable by reading artifacts or running code), and `simplify_rules` are opinionated constraints that reduce scope or remove ambiguity. | depth | recommended | PASS if each role's `verify_questions` list contains at least two questions that reference specific artifacts, behaviors, or measurable outcomes; and `simplify_rules` contains at least one rule that excludes or constrains (not just describes). FAIL if questions are rhetorical or unmeasurable, or if simplify_rules are restatements of scope rather than constraints. |
| C-08 | **Collaborates_with accuracy** — collaboration relationships are architecturally sound and directionally meaningful; if role A lists role B, role B's scope plausibly requires input from or produces output for role A. The instruction explicitly prohibits both phantom role references and universalist listing. | correctness | recommended | PASS if every `collaborates_with` entry names another role that exists in the output set, and the pairing reflects a genuine handoff or dependency (not just "everyone works with everyone"). The skill instruction must prohibit (1) phantom roles — names absent from the role set, and (2) universalist listing — "works with everyone" or equivalent. FAIL if either prohibition is absent from the instruction, or if the output contains phantom names or universalist entries. PARTIAL if only one of the two prohibitions is present in the instruction. |

| ID | Text | Category | Weight | Pass Condition |

| C-09 | **Role differentiation** — each role's `lens.verify_questions` surface concerns that no other role would catch; the set of roles together covers the decision surface without redundant overlap. | depth | aspirational | PASS if a reviewer can identify at least one verify question per role that is unique to that role's frame and would be missed by all other roles in this registry. FAIL if two or more roles share the majority of their verify questions or if any role's questions are a strict subset of another's. |

| C-11 | **Context-first derivation ordering** — domain experts are derived from context analysis before stock roles are loaded; the skill's step sequence forces the model to name domain-specific concerns before encountering the PM/Architect/Strategy archetypes. | design | aspirational | PASS if the skill's step sequence places context analysis and domain expert derivation in earlier steps than stock role emission, and the domain expert `frame` fields reference concerns identified in the context analysis step. FAIL if stock roles are defined before domain experts are derived, or if domain expert frames could have been written without reading context (i.e., they are generic archetypes with domain labels applied after the fact). |
| C-12 | **Failure-mode-named field constraints** — schema field instructions name the exact failure mode to avoid, not just the positive requirement; for sibling fields that tend to collapse into restatements (e.g., `frame`/`serves`, `verify_questions`/`simplify_rules`), the instruction explicitly states what not to do. | design | aspirational | PASS if at least two schema field instructions include a negative constraint that names the specific failure mode (e.g., "Must be a different statement from frame, not a paraphrase" rather than "frame and serves must differ"), and the stock role examples demonstrate the constraint by construction rather than instruction alone. | C-13 | **Explicit step output format** — every skill step that names an output artifact defines the artifact's format fields, required content, and structure explicitly; no step is a heading-only stub. | completeness | aspirational | PASS if every step that produces an artifact (role file, registry summary, or other output) specifies the exact fields, format, and minimum required content for that artifact within the step definition. FAIL if any step names an output (e.g., "Write the registry summary") without defining what that output must contain — heading-only stubs are a structural omission, not a documentation gap. |
| C-14 | **Collaborates_with typed dual-failure prohibition** — the `collaborates_with` field instruction names both failure modes with explicit type labels: FAILURE MODE (type 1) for phantom/non-existent role references; FAILURE MODE (type 2) for universalist listing ("works with everyone"). Typed labeling goes beyond the dual-prohibition required by C-08; it gives the model a recovery path when it detects a violation. | design | aspirational | PASS if the instruction labels both failure modes with distinct type identifiers (numbered, named, or equivalent). PARTIAL if both prohibitions are present (satisfying C-08) but neither carries a type label or named classification. FAIL if either prohibition is absent (C-08 already fails in that case; C-14 inherits the failure). |
| C-15 | **Registry heading-stub failure mode** — the REGISTRY.md step explicitly names the heading-only stub as a FAIL condition: a section header present with no required content beneath it is not a complete registry entry. This forces the model to distinguish structural presence from content completion. | design | aspirational | PASS if the registry step (or its associated output format spec) includes a named FAIL example of the form: "'## Registry Summary' with no content below it fails C-10" or equivalent. PARTIAL if the registry step defines a completion condition (see C-16) that implicitly prevents a heading stub, but does not name the stub failure mode explicitly. FAIL if no heading-stub failure mode is named and no completion condition is defined. |
| C-16 | **Step completion condition** — every output-producing step defines a binary completion test the model can apply before proceeding: not just a format template, but a checkable condition that determines whether the step is done. Format definition (C-13) and completion condition (C-16) are distinct requirements. | completeness | aspirational | PASS if every step that emits an artifact includes a completion condition expressed as a verifiable test (e.g., "all five fields are present", "every derived expert names its source concern", "no field is an empty string"). PARTIAL if at least one output step has a completion condition but others do not. FAIL if no step defines a completion condition and the model must infer doneness from format alone. |
| C-17 | **Worked examples in field constraints** — at least one failure-mode-named constraint (see C-12) includes a worked example or before/after illustration, not just a label. A label names the failure; a worked example shows it. PARTIAL if at least one worked example is present but fewer than three fields are illustrated. FAIL if all failure-mode constraints are label-only with no illustrative content. |
| C-18 | **Contrastive worked example pairs** — every field instruction that includes a worked example shows both a failure specimen and a passing specimen in the same field comment; showing only one direction leaves the boundary calibration incomplete. A label names the failure (C-12). A bad example shows it (C-17). A bad-plus-good pair closes the loop by also showing what correct looks like from the same vantage. | depth | aspirational | PASS if at least two schema field instructions include a WORKED EXAMPLE (bad) and WORKED EXAMPLE (good) pair in the same field comment — both directions in the same location. PARTIAL if at least one field has a contrastive pair but fewer than two fields have both directions. FAIL if all worked examples (even those satisfying C-17) show only one direction. |

PARTIAL if at least one step has a multi-item gate (three or more items) but other output steps have only a single-condition test or no condition. FAIL if no step has a multi-item gate. |
| C-20 | **Domain-specific registry extension** — the registry summary template explicitly designates at least one field beyond the required five as a domain-specific extension slot, and that field is grounded in the context analysis step; the extension is named in the step output spec before any role is written, not added ad hoc to the final artifact. Registry extensibility is evidence that the skill design reached beyond the required baseline into the domain it was asked to model. | design | aspirational | PASS if the registry step's output template includes at least one named extension field that is (a) distinct from the five required fields and (b) references a concern identified in the context/domain analysis step. FAIL if the registry template contains only the five required fields with no extension mechanism, or if any additional field is not grounded in an earlier context analysis step. |
| C-21 | **Uniqueness mandate by construction** — the skill instruction or domain expert derivation step explicitly requires at least one verify question per role to be uniquely attributable to that role's frame, stated in language the model applies while authoring questions, not as a post-hoc evaluation criterion. The canonical formulation is: "a question no other role in this registry would prioritize first." C-09 tests whether the output is unique; C-21 tests whether the instruction prevents duplication by construction before any output is produced. | design | aspirational | PASS if the role template or derivation step includes an explicit statement requiring at least one verify question per role to be uniquely tied to that role's frame — phrased so the model must confirm uniqueness as part of writing each question, not after the full set is generated. FAIL if uniqueness is detectable only by post-hoc evaluation (C-09 standard) and the instruction does not mandate it by name. |
| C-22 | **Cross-step artifact count integrity** — the registry summary's `total_roles` count (or equivalent aggregate field) is explicitly verified against the number of role files emitted in the role-writing step, via a gate item or inline cross-reference instruction; the model cannot complete the registry without checking the prior step's actual output count. C-10 tests registry content; C-22 tests whether the count field is independently grounded in a prior artifact rather than self-reported by the registry step itself. | completeness | aspirational | PASS if a gate item or cross-reference instruction explicitly requires the registry aggregate count to match the file or entry count produced in the role-writing step (e.g., "total_roles must equal the number of files written in Phase N"). FAIL if the count is written without a cross-step verification requirement — the registry step fills in the number from its own state rather than checking the prior step's output. |
| C-23 | **Extension field commitment block with provenance** — the registry extension field (C-20) is introduced in the skill's earliest step in a named commitment block that specifies three elements: (a) the extension field name, (b) the population source — the step and artifact that will provide the field's value, and (c) the extension's purpose — what domain concern it captures. C-20 requires the extension to be named before role writing begins; C-23 requires a formal three-element commitment structure in the first step, giving the model a complete contract before any context is read. | design | aspirational | PASS if the skill's first step contains a named commitment block (EXTENSION-COMMITMENT or equivalent) that explicitly states all three elements: field name, population source step, and purpose. FAIL if the extension field is introduced informally (name only, without population source or purpose), or if the commitment block appears in any step other than the first, or if any of the three elements is absent. |
| C-24 | **Schema field name precision gate** — the role-writing step's completion gate explicitly names at least two required schema field identifiers as string literals (e.g., `verify_questions`, `simplify_rules`), distinguishing between correct naming and near-miss variants that pass visual review but fail downstream consumers. C-01 tests field presence in output; C-24 tests that the gate enforces naming precision, preventing integration-breaking shortcuts (e.g., `verify`, `simplify`) from passing the gate. | completeness | aspirational | PASS if the role-file completion gate contains at least one item that checks schema fields by their exact identifier names — the field names are quoted or marked as literals in the gate condition, not referenced by description alone. FAIL if the gate checks field presence or count without naming the expected identifiers explicitly. |
| C-25 | **Per-expert attribute gate in domain derivation** — the context analysis step's completion gate enumerates at least four named attributes per derived domain expert (e.g., name, concern link, domain-vocabulary frame, verify focus), verified per expert in the gate rather than as an aggregate count; this ensures each derivation record is complete before stock roles are loaded. C-05 tests output specificity in the final artifacts; C-25 tests that the gate holds each derived expert to a minimum attribute record before the context analysis step is considered done. | completeness | aspirational | PASS if the context analysis gate includes an item that checks at least four named attributes for each domain expert individually (not just "N experts are listed"). FAIL if the gate verifies expert count or existence without enumerating the per-expert attributes that must be present for each entry. |
| C-26 | **Named binding field contract** — the skill defines a dedicated, numbered contract block (e.g., FC-1 through FC-N) that enumerates schema field identifiers as binding specifications — not as template comments but as standalone contract entries that all gates and templates can reference by identifier. C-24 tests that a gate quotes field names as literals; C-26 tests whether a single authoritative contract structure exists, so field specifications are defined once and referenced everywhere rather than restated per step. | design | aspirational | PASS if the skill contains a named, multi-entry contract block (labeled FC-N, FIELD CONTRACT, or equivalent) in which each entry binds a schema field identifier to its specification, and at least one gate or template references that block by name or number rather than restating the spec inline. C-21 tests that the instruction mandates uniqueness by language during role authoring; C-35 tests that the mechanism is a named artifact — with explicit format fields and a required uniqueness argument — whose completion gate must pass before the WRITE phase begins. | design | aspirational | PASS if the skill contains a step that requires producing a named per-role artifact (Diagnosis Card, pre-YAML reasoning record, or equivalent) for every role before any YAML is written, and that artifact's format explicitly includes a uniqueness argument that names the closest overlap role and the distinguishing difference. ## Scoring Formula

```
composite = (essential_pass / 5 * 60)
          + (recommended_pass / 3 * 30)
          + (aspirational_pass / 28 * 10)
```

**Golden threshold**: all 5 essential criteria pass AND composite >= 80.

| Band | Score | Meaning |
|------|-------|---------|
| Gold | all essential + >= 80 | Production-quality role registry |
| Silver | all essential + 70-79 | Usable but lens/collaboration depth missing |
| Bronze | 4/5 essential + any score | Structural gaps; not ready for downstream skills |
| Fail | < 4 essential pass | Output not useful |

## Notes for Evaluators

- **C-01 takes priority**: a role file missing any schema field invalidates C-06, C-07, C-08 for that role. Score what is present; note what is absent.
- **C-01/C-07 naming risk**: the schema must use exact field names `verify_questions` and `simplify_rules` (not `verify`/`simplify`). Downstream skills (org-review, org-chart) read role files by exact field name. A schema that uses shortened names passes manual review but fails integration.
- **C-02 (devil-advocate)** is non-negotiable for Signal's design philosophy. A generic "critic" role that challenges quality rather than the status quo does not satisfy this criterion.
- **C-05 vs. C-03**: the stock roles (C-03) are always expected; domain expert roles (C-05) prove the skill read context. Evaluate them independently.
- **C-08 vs. C-14**: C-08 requires that both prohibitions exist in the instruction. C-14 requires that they be explicitly typed with labels. A variate can PASS C-08 (both present) and PARTIAL C-14 (not typed). Evaluate C-08 first; if C-08 fails, C-14 inherits the failure.
- **C-09 vs. C-21**: C-09 is evaluated by reading the output — does each role have a unique verify question? C-21 is evaluated by reading the instruction — does it mandate uniqueness with explicit language before output is produced? A variate where the output happens to be unique (C-09 PASS) but the instruction never required it (C-21 FAIL) is common. Conversely, a variate whose instruction mandates uniqueness (C-21 PASS) but whose output fails (C-09 FAIL) is a design success with an execution failure. Evaluate both independently.
- **C-09** requires a comparative read across all role files — do not evaluate per-file in isolation.
- **C-10 vs. C-13**: C-10 tests whether the registry summary *content* is complete (all five fields present). C-13 tests whether *every step* that names an output defines its format. A skill can fail C-13 (heading-only stub in Step 5) while also failing C-10 (no summary content). Fix C-13 first — it is the structural cause of C-10 failures.
- **C-10 vs. C-15**: C-10 tests content completeness (five fields). C-15 tests whether the stub failure mode is named as a FAIL condition. A variate can PASS C-10 (all five fields present) and still FAIL C-15 (no explicit stub-failure-mode instruction). Evaluate independently.
- **C-10 vs. C-20**: C-10 tests the required five fields. C-20 tests for domain-specific extension beyond those five. A variate can PASS C-10 (all five present) and FAIL C-20 (no extension field). A variate cannot PASS C-20 while failing C-10 — extension presupposes the baseline.
- **C-10 vs. C-22**: C-10 tests registry content (all five fields present including total_roles). C-22 tests how the count field is populated — is it derived from a prior step's actual output? A variate can PASS C-10 (count present) and FAIL C-22 (count self-reported). A variate cannot PASS C-22 while failing C-10 — count integrity presupposes count presence.
- **C-11 (context-first ordering)**: evaluate by reading the step sequence, not the output. If stock roles appear in Step 1 and domain experts in Step 3, C-11 fails regardless of how domain-specific the output looks.
- **C-12 (failure-mode-named constraints)**: look for negative constructions in schema field comments — "not a paraphrase", "not a description", "not a generic archetype". A schema that only states the positive ("frame is an epistemic viewpoint") is weaker than one that names the failure ("frame must not restate serves").
- **C-13 vs. C-16**: C-13 is about output *format* (field names, structural shape). C-16 is about *completion condition* (a binary test the model applies before proceeding). A step can define its output format (PASS C-13) without defining when that output is considered done (FAIL C-16). Both are required for the strongest skill designs.
- **C-14 vs. C-29**: C-14 requires typed labels (type 1, type 2) on both prohibitions. C-29 requires those labels to use "CONTRACT VIOLATION" (or "breach") framing inside a named contract block. A variate can PASS C-14 (typed labels present) and FAIL C-29 (labels appear as "FAILURE MODE" in template). PARTIAL C-29 corresponds exactly to PASS C-14 with advisory framing. Evaluate C-14 first; C-29 inherits the failure if C-14 fails.
- **C-16 vs. C-19**: C-16 requires any completion condition per step. C-19 requires that condition to be a multi-item checklist (three or more independent items). A step with a single condition ("all five fields are present") PASSES C-16 and FAILS C-19. Evaluate C-16 first; if C-16 fails, C-19 inherits the failure.
- **C-17 vs. C-18**: C-17 requires at least one worked example per field comment (bad value, before/after, or counter-example). C-18 requires that at least two of those field comments show both a bad AND a good example — both directions in the same location. A field comment with only "FAILURE: task list — e.g., 'write tests, review PRs'" is C-17 level. The same comment that also adds "PASS: epistemic lens — e.g., 'Sees every proposal through the lens of operational cost'" is C-18 level. Evaluate C-17 first; a variate that fails C-17 cannot PASS C-18.
- **C-18 (contrastive pairs)**: count field comments that contain both directions, not fields that mention failure AND separate fields that mention success. Both directions must appear in the same field comment to constitute a contrastive pair.
- **C-19 (multi-item gate)**: count independently verifiable items per gate — items that could each individually fail while others pass. A gate that lists "1) output written, 2) output complete" is not multi-item in the C-19 sense; both items test the same condition from different angles. A gate where each item references a distinct schema element or constraint is multi-item.
- **C-20 (domain-specific extension)**: verify that the extension field is named in the step output spec before the role files are written — not discovered by reading the final artifact. An "Inertia surface" field that appears only in the finished REGISTRY.md does not satisfy C-20; it must appear in the step's output template so the model knows to populate it.
- **C-20 vs. C-23**: C-20 requires the extension field to be named before role writing begins. C-23 requires it to be named in the skill's first step with a formal three-element commitment (name + population source + purpose). A variate can PASS C-20 (extension named in Phase 2 prose, before Phase 4 role writing) and FAIL C-23 (no commitment block, or commitment appears in Phase 2 rather than Phase 0/first step). A variate cannot PASS C-23 while failing C-20.
- **C-21 (uniqueness mandate)**: look for language that names uniqueness as a writing constraint during role authoring, not just an evaluation criterion. The canonical signal is a formulation like "this question must be one that no other role in this registry would prioritize first." A rubric note telling the evaluator to check for uniqueness is not a C-21 pass; a template instruction telling the model to write a unique question is.
- **C-21 vs. C-35**: C-21 tests that the instruction mandates uniqueness by language during authoring. C-35 tests the stronger mechanism: a named pre-YAML artifact structure that requires a comparative uniqueness argument (naming the closest overlap role) as a format field. A variate can PASS C-21 (uniqueness language in template) and FAIL C-35 (no named pre-YAML artifact). A variate cannot PASS C-35 while failing C-21 — the pre-YAML artifact presupposes the instruction mandates uniqueness.
- **C-22 (cross-step count integrity)**: look for an explicit bridge between the registry count field and the prior step's output — a gate item like "total_roles equals the count from Phase N output" or an instruction like "verify total_roles against Phase N before writing the registry." A registry that happens to have the right count but derives it without checking is a C-22 FAIL.
- **C-22 vs. C-27**: C-22 is satisfied by any gate-level instruction that requires cross-step count verification. C-27 requires the stronger mechanism: a named variable declared in a PREPARE sub-step before WRITE begins. A variate with a post-write gate item that says "confirm total_roles matches Phase N file count" PASSES C-22 and FAILS C-27. Both matter: C-22 ensures the check exists; C-27 ensures it happens before writing starts.
- **C-23 (commitment block)**: the commitment block must appear in the skill's first (earliest) step and must contain all three elements. A block with field name and purpose but no population source FAILS C-23. A block that appears in Step 2 (even before role writing) FAILS C-23 if Step 0 or Step 1 exists without the block.
- **C-23 vs. C-30**: C-23 requires three elements in the commitment block (name + population source + purpose). C-30 requires the purpose element to be explicitly framed as answering the skill's opening diagnostic question. A purpose like "preserves the status-quo argument" is concrete and descriptive (PASS C-23) but does not name a diagnostic question (FAIL C-30). A purpose like "answers Phase 0 diagnostic: what status-quo argument will the inertia-advocate deploy?" satisfies both. A variate cannot PASS C-30 while failing C-23.
- **C-24 (schema field name precision gate)**: look for the field names quoted as literals in a gate condition — e.g., "gate item: `verify_questions` and `simplify_rules` are spelled in full, not shortened." A gate that says "all required fields are present" passes C-16/C-19 but not C-24. The distinguishing feature is that the field name strings appear explicitly in the gate text.
- **C-24 vs. C-26**: C-24 tests that a gate quotes field identifiers as literals. C-26 tests that a dedicated contract block exists as the authoritative source for those identifiers — so gates can reference the contract by number (FC-4, FC-5) rather than restating the spec. A variate can PASS C-24 (gate quotes literals inline) and FAIL C-26 (no contract block, just inline quoting). A variate that PASSes C-26 typically also satisfies C-24 by construction, but evaluate them independently.
- **C-25 (per-expert attribute gate)**: the four attributes must be checked per expert in the gate, not as portfolio-level conditions. A gate that says "all derived experts have domain-vocabulary frames" is closer but still fails C-25 if it does not enumerate all four attributes by name. Look for a gate item that lists each of the required attributes (e.g., "each expert entry has: name, concern link, domain-vocabulary frame, verify focus").
- **C-25 vs. C-34**: C-25 tests that the derivation gate checks at least four named attributes per expert. C-34 tests that a fifth attribute — inertia gap — is added to the per-expert gate record. A variate can PASS C-25 (four attributes) and FAIL C-34 (no inertia gap). A variate cannot PASS C-34 while failing C-25 — inertia-gap attribution presupposes the per-expert attribute gate exists.
- **C-26 (named binding field contract)**: look for a dedicated block with entries labeled FC-N, [FC-1], FIELD CONTRACT entries, or equivalent — not schema field comments within a template. The distinguishing feature is that the block is standalone and referenceable by identifier from other steps. A template that annotates each YAML field with detailed constraints is C-12/C-17/C-18 territory; a separate contract block whose entries are cited by ID in gates is C-26 territory.
- **C-27 (pre-write count variable)**: look for a PREPARE sub-step (or equivalent pre-write phase) within the registry step that names a count variable explicitly. The variable name need not be "PHASE5_COUNT" specifically — any named intermediate (e.g., ROLE_COUNT, N_FILES) satisfies C-27 if it is (a) declared before WRITE begins and (b) the WRITE phase is explicitly told to use it. A gate that says "verify count matches Phase N output" after writing fails C-27 even if it satisfies C-22.
- **C-27 vs. C-32**: C-27 requires a named pre-write variable declared in a PREPARE sub-step. C-32 requires that PREPARE sub-step to also name the specific bypass anti-pattern it prevents. A variate can PASS C-27 (PREPARE sub-step with named variable) and FAIL C-32 (no bypass anti-pattern named — the mechanism exists but does not explain what failure it prevents). A variate cannot PASS C-32 while failing C-27.
- **C-28 (generic placeholder name prohibition)**: look for named anti-patterns in the instruction — e.g., "'domain-expert', 'expert-1', 'generic-expert' are prohibited." A positive requirement ("names must be domain-specific") is C-05 level. The distinguishing feature of C-28 is that the instruction names at least one specific placeholder pattern rather than relying on the positive requirement alone to prevent it.
- **C-28 vs. C-31**: C-28 requires the prohibition to appear at least once and name at least one anti-pattern. C-31 requires the same prohibition to appear at a minimum of three independent checkpoints: the contract block, the derivation step prose, and a gate condition. A variate can PASS C-28 (prohibition present in one location) and FAIL C-31 (only one location). A variate cannot PASS C-31 while failing C-28 — multi-site enforcement presupposes the prohibition exists.
- **C-29 (contract violation framing)**: the key distinguishing word is "violation" or "breach" in the label — not just "type 1" or "type 2." Look for "CONTRACT VIOLATION (type 1)" or "VIOLATION TYPE 1" or "breach (type 1)" inside a named contract structure. "FAILURE MODE (type 1)" in a template comment is C-14 PASS and C-29 PARTIAL. The distinction matters because contract violation framing changes the model's categorical understanding of the constraint from "a thing that sometimes goes wrong" to "a thing that breaks the contract."
- **C-29 vs. C-33**: C-29 tests that the contract block carries CONTRACT VIOLATION framing. C-33 tests that the template at the collaborates_with write site also carries those labels, making the prohibition visible where the model is most likely to violate it. A variate can PASS C-29 (labels in contract block) and FAIL C-33 (labels not mirrored in template). PARTIAL C-33 means only one of the two labels is mirrored. A variate cannot PASS C-33 while failing C-29.
- **C-30 (extension purpose tied to diagnostic question)**: look for an explicit reference to a named diagnostic question in the purpose sub-field of the commitment block — the purpose must name what question it answers, not just what it stores. A purpose that says "records the inertia surface for downstream use" is content-descriptive (FAIL C-30). A purpose that says "answers Phase 0 diagnostic question: 'what status-quo claim will the inertia-advocate deploy?'" is diagnostic-linked (PASS C-30). The diagnostic question may be paraphrased but must be explicitly identified.
- **C-31 (multi-site enforcement)**: count the number of independent locations where the placeholder name prohibition and its anti-pattern examples appear — contract block counts as one, derivation step prose counts as one, derivation step gate counts as one, write step gate counts as one. Four locations (as in R6 V-01) is the strongest; three (as in R6 V-02) satisfies the criterion. Two locations or fewer fails C-31. Locations that are redundant within the same artifact section do not count separately.
- **C-32 (count bypass failure class)**: look for a sentence in or adjacent to the PREPARE sub-step that names the failure class it prevents — something equivalent to "deriving the count from a prior-phase plan rather than enumerating the actual output files is a failure." A PREPARE sub-step that says only "list all .md files written in Phase N, record the count as ROLE_COUNT" satisfies C-27 but fails C-32. The distinguishing feature of C-32 is that the instruction names the anti-pattern it was designed to prevent, not just the correct procedure.
- **C-33 (contract-to-template mirroring)**: look for the CONTRACT VIOLATION labels in two locations: the FC contract block (required by C-29) and the role template's collaborates_with field annotation. The template annotation need not be verbatim — "CONTRACT VIOLATION (type 1): phantom role reference" in the template alongside "[FC-10]: CONTRACT VIOLATION (type 1) — PHANTOM" in the contract block satisfies C-33. PARTIAL if only one label is mirrored. A contract block with both labels and a template with neither fails C-33 even when C-29 passes.
- **C-34 (inertia gap per expert)**: look for a fifth attribute in the derivation gate that names the expert's inertia gap, status-quo argument, or the resistance their domain would generate. The fifth attribute need not use the exact term "inertia gap" — "status-quo defense", "resistance claim", "conservative argument" all satisfy C-34 if the gate checks it per expert. A gate item that says "each expert has a named inertia gap" (checked per expert) satisfies C-34; a gate item that says "inertia-advocate has a status-quo question" (checked once for the role, not per expert) does not. The distinguishing feature is per-expert attribution of the inertia claim.
- **C-35 (pre-YAML per-role reasoning artifact)**: look for a named step that requires producing a structured per-role artifact before the WRITE phase begins. The key signals are: (a) the artifact has a defined format (not just a prose reflection), (b) the format includes a uniqueness argument field that requires naming the closest overlap role, (c) all artifacts must be completed before YAML writing begins. A step that says "think through each role's uniqueness before writing" without a structured format fails C-35. PARTIAL if the pre-YAML artifact exists but the uniqueness argument field is open-ended rather than requiring a named overlap role.
- **C-35 vs. C-36**: C-35 requires per-role pre-YAML artifacts. C-36 requires a cross-set scan gate operating on the completed artifact set. A variate can PASS C-35 (Diagnosis Cards required) and FAIL C-36 (no named cross-set scan step). A variate cannot PASS C-36 while failing C-35 — the cross-set scan presupposes the per-role artifacts exist.
- **C-36 (cross-set pre-YAML uniqueness scan)**: look for a named gate or step between the "all Diagnosis Cards complete" milestone and the "begin YAML writing" milestone. The scan must be a discrete named artifact or gate item — not just a template annotation in the YAML step. The key signal is sequential positioning: after per-role artifact completion, before YAML writing begins. PARTIAL if a cross-set check exists but is positioned inside the YAML-writing step rather than as a preceding gate.
