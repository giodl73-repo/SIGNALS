# Rubric: quest-rubric (v16)

Evaluates output from the `quest-rubric` skill, which produces a scoring rubric for a named
Signal skill. The rubric must function as a working objective function: each criterion must be
independently checkable, the tiers must follow the structural contract, and the essential
criteria must be grounded in the target skill's actual non-negotiable behaviors — not generic
quality observations that would apply to any skill.

**v2 additions**: Three aspirational criteria extracted from R1 excellence signals — the inertia
test (V-05), the concrete discrimination example (V-04), and phase-gate enforcement (V-01).

**v3 additions**: Five aspirational criteria extracted from R2 excellence signals — calibration
pair lifecycle position (ES-2/V-03), sub-step output sequencing (ES-1/V-03), multi-level gate
architecture (ES-3/V-03), audit role separation (ES-4/V-05), and audit trail captures reasoning
(ES-5/V-05).

## Criteria

### Essential (must all pass — without these the output is not useful)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-01 | **Field completeness** — Every criterion entry contains all five required fields: ID (C-NN), text with a bold label, weight (essential/recommended/aspirational), category (correctness/depth/format/coverage/behavior), and a pass condition. | format | essential | Count the fields present for each criterion. Pass if no criterion is missing any of the five fields. A criterion with "see above" or an empty pass condition is a fail. |
| C-02 | **Tier count compliance** — The rubric contains 3-5 essential criteria, 2-3 recommended criteria, and 1-2 aspirational criteria — no more, no fewer. | correctness | essential | Count criteria in each tier. Pass only if all three counts fall within the specified ranges. A rubric with 2 essential criteria or 4 recommended criteria fails immediately. |
| C-03 | **Essential criteria grounded in target skill** — Essential criteria cover behaviors that are specific and non-negotiable for the named target skill, not generic output-quality criteria that would apply to any skill. | correctness | essential | Read the essential criteria and ask: could these criteria be copy-pasted unchanged onto a rubric for a completely different skill? Pass if fewer than 2 of the essential criteria are purely generic (e.g., "output is well-formatted" with no connection to the skill contract). |
| C-04 | **Testable pass conditions** — Every pass condition names a specific, observable outcome that an evaluator can verify without subjective judgment. | correctness | essential | For each pass condition, ask: can two independent evaluators reach the same pass/fail verdict without discussion? Pass conditions containing only vague language ("good", "sufficient", "appropriate") with no concrete anchor are a fail. |
| C-05 | **Composite score formula and golden threshold present** — The rubric includes the composite scoring formula using the standard weights (essential 60, recommended 30, aspirational 10) and states the golden threshold condition. | format | essential | Check that the score section contains a formula matching the standard composite structure and a golden threshold statement. Missing formula or threshold is a fail. |

### Recommended (output is better with these)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-06 | **Category diversity** — Criteria span at least three distinct categories across the full rubric, avoiding clustering in a single dimension. | coverage | recommended | Count distinct categories used. Pass if >= 3 of the 5 canonical categories (correctness/depth/format/coverage/behavior) appear across all criteria. |
| C-07 | **Criteria ordered by severity within tiers** — Within the essential tier, criteria are listed from most structurally critical (output unusable if this fails) to least. Same ordering principle applied to recommended and aspirational tiers. | depth | recommended | Read the essential criteria in order. If swapping C-01 and C-04 would produce a more logical triage sequence, the ordering fails. Pass if the first essential criterion represents the single highest-stakes failure mode for the target skill. |
| C-08 | **Quick checklist present** — Output includes a checklist block (markdown checkboxes) summarizing all criteria IDs and their key assertion for rapid manual scoring. | format | recommended | A checklist section exists with one checkbox line per criterion referencing its ID and a short description. Pass if all criteria IDs appear in the checklist. |

### Aspirational (raise the bar once essential/recommended are stable)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-09 | **Criteria enable iteration** — When a criterion fails, its pass condition gives enough specificity that the output author knows exactly what to fix without running the skill again to ask. Each criterion's failure mode is self-describing. | behavior | aspirational | For each criterion, ask: if this criterion failed, does the pass condition text tell the author the minimum change needed to achieve a pass? Pass if >= 80% of criteria have self-describing failure modes (not just "does not pass"). |
| C-10 | **No redundancy or contradiction** — No two criteria overlap such that passing one guarantees passing the other, and no two criteria impose logically contradictory requirements on the output. | depth | aspirational | For each pair of criteria, ask: is there a rubric output that passes one but fails the other for a non-trivial reason? If two criteria always agree, they are redundant and one should be folded in. Pass if all criteria add unique signal with no fully-overlapping pair. |
| C-11 | **Discriminating pass conditions** — Each pass condition tests a behavior that a generic rubric ("the output is clear and comprehensive") would fail to catch. Distinct from C-04 (observability): a condition can be observable but still redundant with a vague rubric if it checks only structural presence rather than skill-specific contract fulfillment. | correctness | aspirational | For each pass condition, ask the inertia test: "Could a rubric that says 'the output is clear and comprehensive' still pass this condition?" If yes for any pass condition, that condition has not been sharpened enough. Pass if every pass condition names at least one count, named field, structural pattern, or enumeration that only the target skill's contract could supply — not shared structural minimums. |
| C-12 | **Calibration example present** — The rubric includes at least one concrete positive/negative pair showing the boundary between acceptable and failing output for the most critical essential criterion. The pair must name the skill — e.g., "generic: 'output is low quality' / grounded: 'the output has no formula so quest-golden cannot compute a composite score'" — so a new evaluator can self-calibrate without a training session. | behavior | aspirational | A calibration example block exists containing one labeled generic example (shows what NOT to write) and one labeled grounded example (shows what to write) for at least one essential criterion. Both examples must reference the target skill by name or reference its output contract. Pass if a reader unfamiliar with the rubric could use the pair to correctly classify two additional borderline cases. |
| C-13 | **Phase-gate enforcement present** — The rubric's structure includes at least one hard gate that blocks forward progress until a structural requirement is verified. A gate must be explicit (e.g., "DO NOT proceed to the next step until...") and must reference a checkable condition, not an aspiration. | behavior | aspirational | A gate statement is present using explicit blocking language ("DO NOT proceed", "stop here", "must complete before advancing"). The gate condition is checkable: it names a count, a field, or a structural property, not a quality judgment. Pass if violating the gate would result in a detectably incomplete output (the gated step is one that earlier steps feed). |
| C-14 | **Calibration pair produced at point of drafting** — The calibration pair required by C-12 is written while the pass condition is being drafted, not retrospectively after all criteria are sealed. Retrospective calibration pairs lose context and risk being rationalized from the final condition rather than derived alongside it. | behavior | aspirational | The rubric's workflow positions the calibration pair as a step immediately following pass-condition drafting for each criterion, before the criterion row is sealed. A workflow that permits all calibration pairs to be written at the end, after all criteria rows are finalized, does not pass. The ordering must be: draft pass condition → write calibration pair → seal criterion — not: draft all pass conditions → seal all criteria → write all calibration pairs. |
| C-15 | **Sub-step output sequencing makes omission detectable** — When the rubric requires checking mechanisms (inertia test, calibration pair), each mechanism's output appears as an independent artifact in the output stream before the criterion is recorded in the summary table. A builder who skips a check produces a table row with no preceding artifact — a structural gap detectable by inspection, not a quality judgment. | behavior | aspirational | For each mandatory check step (e.g., inertia test, calibration pair), the rubric's instructions require that step's output to appear in the output before the criterion is entered into the table (not embedded in the table cell or omitted silently). A rubric where all checking can be done mentally with only the final table as output does not satisfy this criterion. Pass if a reviewer scanning the output would see a detectable structural gap — not just a short table cell — wherever a required check was skipped. |
| C-16 | **Multi-level gate architecture** — The rubric enforces gating at two distinct granularities: a phase gate ensuring criterion-building begins from a sufficient structural foundation, and at least one per-criterion gate ensuring each criterion is complete before the next begins. A single gate at either level alone leaves the other level unprotected. | behavior | aspirational | Identify the phase gate (a condition that must pass before any criterion-building begins) and at least one per-criterion gate (a condition that must pass before moving from one criterion to the next). Both must use explicit blocking language. Pass if the rubric has both levels of gating and neither relies solely on the author's self-discipline to enforce. |
| C-17 | **Audit role or phase separation** — The rubric's workflow separates the criterion-authoring concern (achieving coverage: do the criteria address the right behaviors?) from the criterion-auditing concern (achieving specificity: are the pass conditions discriminating?). When both concerns are addressed in a single pass, the author optimizes for completion rather than for quality of each concern. | depth | aspirational | The rubric's instructions include two structurally distinct phases or roles: one focused on criterion construction (ensuring all required behaviors are covered) and one focused on criterion quality verification (ensuring each pass condition is discriminating, non-redundant, and calibrated). A rubric where both concerns are addressed in a single pass through the criteria table does not satisfy this criterion. Pass if the workflow explicitly sequences authoring before auditing, with a gate or role boundary between them. |
| C-18 | **Audit trail captures reasoning, not only verdicts** — The audit phase produces a structured artifact (table or equivalent) recording not just whether each pass condition passed the inertia test, but which specific skill-contract element made it discriminating. Future rubric maintainers can trace which part of the skill's contract grounds each criterion, making the grounding visible without re-running the evaluation. | behavior | aspirational | The audit artifact (inertia-test table or equivalent) includes a column or field for each criterion recording the discriminating element — the specific skill-contract behavior, count, named field, or structural pattern that grounds the pass condition. An audit table with only "PASS/FAIL" columns does not satisfy this criterion. Pass if a maintainer reading the audit artifact could identify which part of the target skill's contract would need to change before each criterion became invalid. |
| C-19 | **Named-block artifact architecture** — Mandatory check outputs (inertia test records, calibration pairs) are required as named markdown section blocks (e.g., `### INERTIA-RECORD [C-NN]`, `### CALIBRATION-PAIR [C-NN]`) rather than inline prose within the drafting flow. Named blocks create heading-scan detectability: a missing block corresponds to a missing heading, verifiable without reading content. | format | aspirational | The rubric's instructions require at least two categories of check outputs (e.g., inertia-test records and calibration pairs) to appear as named markdown section headings — not as inline paragraphs or table cells. Pass if a reviewer could detect a missing check artifact by scanning section headings alone, without reading the content of any section. A rubric where all check outputs are embedded as text within a single drafting flow does not satisfy this criterion. |
| C-20 | **Gate must block forward progress, not repair backward omissions** — A retroactive repair step ("if the block is missing, write it now") is not a gate. Gates prevent a builder from proceeding until a condition is satisfied; repair steps detect a past omission after criterion rows are already sealed. | behavior | aspirational | For each gate in the rubric, verify that it is positioned before the gated work (not after it). A gate that says "if you have not yet written the calibration pair, write it now before continuing" is a retroactive repair masquerading as a gate: the criterion row may already have been sealed. Pass if every gate is positioned at the entry of the step it protects, with explicit blocking language that prevents the protected step from beginning rather than detecting its omission. |
| C-21 | **Audit table consolidates all criteria into a single cross-criterion view** — The audit phase produces a single table where each row corresponds to one criterion, enabling a maintainer to scan all criteria simultaneously and detect systemic specificity failures (e.g., every criterion anchors to the same skill-contract element, leaving other elements ungrounded). Cross-criterion visibility is not achievable by reading per-criterion records sequentially. | depth | aspirational | The audit phase output is structured as a single table where rows correspond to criteria (not to individual inertia-test records scattered through the output). The table must include at least: criterion ID, pass/fail verdict, and the discriminating element (per C-18). Pass if a maintainer reviewing the table could identify a systemic pattern (e.g., all criteria ground in output format, none in behavioral contract) without reading any other output section. |
| C-22 | **Structural sequencing enforced by both ordering constraint and explicit anti-deferral instruction** — Structural ordering constraints (e.g., requiring named blocks in sequence: CALIBRATION-PAIR before CRITERION) are necessary but not sufficient to guarantee per-criterion production. A builder who understands the sequence can still defer all calibration pairs to the end by producing empty placeholder headings. An explicit behavioral instruction paired with the structural constraint closes this gap. | behavior | aspirational | The rubric includes both: (a) a structural ordering constraint (block X must appear before block Y in the output) and (b) an explicit behavioral instruction prohibiting deferral (e.g., "produce this block now, before sealing the criterion row — do not defer to end of output"). A rubric with structural ordering alone, or with a behavioral instruction absent from the operative step, does not satisfy this criterion. Pass if the two mechanisms are co-located at the operative sub-step (not split between a phase header and a sub-step body). |
| C-23 | **Structural verification phase gates Auditor entry** — C-20 requires per-criterion gates to block forward progress rather than repair backward omissions. C-23 extends this principle one level up: the structural heading-scan (or equivalent structural completeness check) must itself be a hard gate that blocks Auditor Phase entry. A structural scan that allows the Auditor phase to begin despite detected heading-pattern violations is a detection step, not a gate. | behavior | aspirational | The Structural Verification step includes explicit blocking language that prevents the Auditor Phase from beginning until all structural checks pass. Language such as "if any scan fails, do not proceed to the Auditor Phase" is required. A structural scan that reports findings as observations — allowing the Auditor Phase to proceed regardless of result — does not satisfy this criterion. Pass if violating the structural check produces a hard stop that the workflow cannot route around without ignoring the gate language. |
| C-24 | **Three-level verification chain covers non-overlapping defect classes** — C-16 requires two gate levels (phase and per-criterion). C-24 requires the full verification architecture to cover three distinct defect classes in sequence, each addressed by a mechanism the prior level cannot handle: (1) per-criterion omission prevention at write time, (2) structural pattern bypass detection post-phase, and (3) cross-criterion specificity analysis at audit time. | depth | aspirational | Identify the three verification mechanisms and the defect class each addresses. The defect classes must be non-overlapping: a defect caught by mechanism 2 must be structurally undetectable by mechanism 1 (because it requires a completed phase to scan), and a defect caught by mechanism 3 must require cross-criterion comparison (not per-criterion checking). Pass if the three mechanisms address: (1) builder skipping a required check mid-criterion, (2) builder completing all per-criterion checks but producing the wrong structural pattern, and (3) builder producing correct patterns but with redundant or non-discriminating pass conditions across criteria. |
| C-25 | **Anti-deferral prohibition co-located at the operative sub-step, not only in phase header** — C-22 requires an explicit anti-deferral behavioral instruction paired with structural sequencing. C-25 specifies where that instruction must appear: inside the sub-step body it governs (e.g., directly inside the calibration-pair sub-step instruction), not only in a phase preamble or section header that a builder could skip while following the sub-steps. | behavior | aspirational | Locate the anti-deferral instruction required by C-22. Verify that it appears inside the sub-step body where the gated action is performed (not only in a phase header or preamble above the sub-step). A rubric where the anti-deferral instruction appears only in a phase introduction paragraph — with no restatement inside the operative sub-step instruction — does not satisfy this criterion. Pass if a builder following only the sub-step instructions (skipping all preamble text) would still encounter the anti-deferral prohibition at the point where deferral is possible. |
| C-26 | **Phase-level workflow transitions marked by named section headings** — C-19 requires per-criterion artifact outputs to appear as named markdown section blocks, making their absence detectable by heading scan without reading content. C-26 extends this principle one level up: each workflow phase transition is marked by a named section heading (e.g., `### AUTHOR PHASE`, `### STRUCTURAL VERIFICATION`, `### AUDITOR PHASE`), making the position of structural verification between Author and Auditor phases verifiable by heading scan. | format | aspirational | The workflow's phase transitions are marked by named section headings that appear in the output. A reviewer scanning only section headings (not reading prose) can confirm: (a) the Author Phase precedes Structural Verification, (b) Structural Verification precedes the Auditor Phase, and (c) the Auditor Phase is the final phase. A workflow where phase transitions are conveyed only through numbered steps or prose instructions — with no heading-level markers in the output — does not satisfy this criterion. Pass if the phase sequence is verifiable by heading scan without reading any instruction or prose content. |
| C-27 | **Cross-criterion verification produces a named artifact with a forward-blocking gate** — C-15 requires each mandatory per-criterion check to emit an independent output artifact before the criterion is sealed, making omission detectable by structural gap. C-19 requires those artifacts to be named markdown section blocks. C-27 extends both to cross-criterion verification: any mechanism that checks properties across multiple criteria simultaneously (e.g., redundancy check, category coverage check) must produce a named section block and carry a forward-blocking gate that prevents subsequent steps from beginning until the cross-criterion check is complete. | behavior | aspirational | The rubric includes at least one cross-criterion verification mechanism (operating on the set of criteria together, not on individual criteria). This mechanism produces a named section block (e.g., `### REDUNDANCY-CHECK-TABLE`) and carries explicit blocking language preventing forward progress (e.g., "DO NOT proceed to Structural Verification until this table is complete and all pairs are confirmed non-redundant"). A cross-criterion check that produces only a prose summary embedded in a phase header, without a named block and a gate, does not satisfy this criterion. Pass if the cross-criterion check artifact is detectable by heading scan and the gate is explicit. |
| C-28 | **Scoring formula denominators explicitly annotated with the criterion ID range they cover** — The scoring formula uses a fixed denominator for each tier (5 for essential, 3 for recommended, N for aspirational). As aspirational criteria are added across rubric versions, the aspirational denominator changes. Without an annotation, a scorer must count the aspirational criteria in the table to verify the denominator is current — a manual step that is skipped in practice, allowing stale denominators to persist undetected. | format | aspirational | The scoring formula in the rubric includes an inline comment or annotation (e.g., `/* C-09 through C-34 */`) stating the criterion ID range covered by the aspirational denominator. The annotation must be co-located with the formula (not in a separate note), so a scorer verifying the denominator can do so by reading the formula line alone. A formula with a numeric denominator and no ID-range annotation does not satisfy this criterion. Pass if a scorer could verify the aspirational denominator's correctness without counting rows in the criteria table. |
| C-29 | **Mechanism overview declares structural output format requirement alongside behavioral requirement** — C-27 requires cross-criterion verification mechanisms to produce named section blocks; C-19 requires per-criterion check mechanisms to produce named section blocks. Both criteria govern what must be produced. C-29 requires the mechanism overview (or equivalent orientation section) to declare the structural format (named `###` heading) each mechanism will produce, so that a builder reading the orientation section can derive the complete expected heading structure before any phase begins. Without this declaration, the structural contract is discoverable only by reading the operative phase instructions — the builder cannot build an output skeleton from the orientation section alone. | behavior | aspirational | The rubric's MECHANISMS section (or equivalent named orientation block) lists each mechanism governed by C-19 and C-27 and names: (a) the mechanism's function (what it checks), and (b) the structural output format it produces (the named `###` heading it will emit, e.g., `### INERTIA-RECORD [C-NN]`). A MECHANISMS section that lists mechanism names and functions without naming the output heading pattern does not satisfy this criterion. Pass if a builder reading only the MECHANISMS section could construct the complete expected section heading sequence for a full rubric output, without reading any phase-level instructions. |
| C-30 | **Structural Verification explicitly declares its phase-bounded coverage scope and names the deferral destination for out-of-scope checks** — C-24's three-level verification chain assigns each level a distinct defect class. C-23 requires Structural Verification to gate Auditor entry. Neither criterion requires SV to declare what it does not check. Without an explicit scope declaration, a builder using SV may believe it verifies heading patterns from all phases — including Auditor Phase headings — when it covers only Author Phase output. This silence creates a false sense of completeness: heading-pattern errors in the Auditor Phase output are not caught until a re-run, if at all. | behavior | aspirational | The Structural Verification step's instructions include an explicit scope statement identifying: (a) the phase whose output it covers (Author Phase only, or equivalent), (b) at least one named category of heading check that is out of scope (e.g., Auditor Phase heading patterns), and (c) the mechanism or phase where the deferred check will be performed. ## Scoring

```
composite = (essential_pass / 5 * 60)
          + (recommended_pass / 3 * 30)
          + (aspirational_pass / 40 * 10)  /* C-09 through C-48 */
```

**Note**: aspirational denominator is 40 (C-09 through C-48).

**Golden threshold**: all 5 essential criteria pass AND composite >= 80.

| Band | Score | Interpretation |
|------|-------|----------------|
| Golden | all essential + >= 80 | Ship-ready rubric — use as objective function |
| Acceptable | all essential + 70-79 | Minor gaps, usable with reviewer awareness |
| Marginal | all essential + < 70 | Coverage or ordering gaps, iterate before use |
| Fail | any essential fails | Not a valid objective function regardless of total score |
