# Rubric: quest-variate (v9)

Evaluates output from the `quest-variate` skill, which generates N distinct prompt variations
of a skill body. Each variation must be complete and runnable, vary along exactly one axis,
and carry a labeled hypothesis.

**v9 changes**: Added C-38 through C-40 from Round 8 excellence signals. No essential or
recommended criteria changed. Aspirational denominator updated from /29 to /32.

---

## What changed v8 -> v9

3 new aspirational criteria (C-38 through C-40), each traceable to a specific R8 excellence signal:

| ID | Criterion | Source signal | Distinction from prior criteria |
|----|-----------|---------------|---------------------------------|
| C-38 | **Point-of-production anti-pattern prohibition with corrective instruction** | R8 V-02 C-36 evidence: PROHIBITION block at each phase -- "A `Declared responsibility:` label whose content echoes the phase title... is a required rewrite" -- the failure-mode text is placed physically adjacent to the label production site, blocking advancement at the moment of authorship | C-36 requires the `Declared responsibility:` label to use an action-verb phrase; C-36 can be satisfied by a post-hoc check that catches title-echoing content after it is written. C-38 requires the failure mode to be explicitly named and declared a required rewrite at the production site -- physically adjacent to where the label is authored, not in a downstream audit step. Converts the criterion from a convention checked after authorship to a constraint enforced during authorship. A variation that satisfies C-36 (correct action-verb content is present) via correct convention alone, without a prohibition at the production site, does not satisfy C-38. |
| C-39 | **Named structural key uniformity auditor role with per-key result emission and phase gate** | R8 V-03 C-37 evidence: "Structural Key Uniformity Auditor: post-generation set-level scan against all 4 required keys, emits per-key results, blocks Step 3 on FAIL: 'Required rewrite before Step 3 may begin'" -- a dedicated named role fires after all bodies are written, scans the complete set, and hard-gates the findings phase | C-37 requires structural key names to propagate uniformly across all variation bodies in the set; C-37 can be satisfied by a Variation Author per-body gate (C-15) that prevents dropout before advancing to the next body. C-39 requires a separately named enforcement role that (a) fires post-generation as a set-level scan, (b) operates against an explicitly declared list of required structural keys, (c) emits structured per-key results (pass/fail per key per body, not a single aggregate verdict), and (d) blocks advancement to the findings phase on any per-key fail with an explicit "required rewrite before [next phase] may begin" instruction. Detection is promoted from per-body-authorship prevention to post-generation set-level audit with a named role and hard phase gate. A C-37-compliant set where key uniformity is enforced solely by the Variation Author per-body gate does not satisfy C-39. |
| C-40 | **Check-step inline expected-form declaration for structural keys** | R8 V-04 C-35 evidence: Step 2A Key 1 check -- "Key 1 (Consistency verdict: [CONSISTENT \| AXIS DIVERGENCE] in Step 2B)" -- the check instruction carries the complete expected form of the key, including bracket notation, inline at the check site rather than only at the field definition site | C-35 requires bracket notation declaring both option values at the token field definition site. C-40 requires the notation to also appear at the check site: when a key-inspection check step verifies structural key compliance, the check instruction includes the complete expected form inline at the check site -- including bracket notation where applicable -- not just the key name. Makes the check self-documenting: a generator running the check can confirm compliance by comparing the output against the embedded expected form without consulting a separate criterion definition or field declaration. A C-35-compliant prompt where check steps name the key without embedding the expected form does not satisfy C-40. Only activated when a key-inspection check step is present (e.g., Step 2A key list, auditor scan list). |

**Scoring formula**: aspirational denominator bumped from `/29` to `/32`. All bands and weights unchanged.

## Criteria

### Essential (must all pass -- without these the output is not useful)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-01 | **Runnable completeness** | correctness | essential | Every variation is a full, standalone skill body. Each V-NN block can be copied into a skill file and executed without referencing any other variation. No "same as V-01 except..." constructs. All declared steps and sections are written out in full -- not placeholders, not ellipses. |
| C-02 | **Count and labeling** | format | essential | Output contains exactly the requested N variations (default 5 when N is unspecified), labeled V-01 through V-0N in sequence with no gaps. Fewer than N is a fail; extra unlabeled content does not substitute for missing labeled variations. |
| C-03 | **Axis declaration** | format | essential | Every V-NN header or preamble includes an `axis:` field naming one of the defined axes, and a `hypothesis:` field stating the expected behavioral effect. Both fields must be present and non-empty. |
| C-04 | **Single-axis isolation** | correctness | essential | Each variation changes exactly one axis relative to a shared baseline. Reading two variations side-by-side, the structural difference maps to exactly one named axis. Changes that bleed across axes are a fail. Exception: variations explicitly labeled as combination passes. |
| C-05 | **Genuine distinctness** | correctness | essential | No two variations are superficially equivalent (e.g., minor synonym substitution that does not actually change the axis). Each variation must be distinguishable by its axis choice in a way that would produce observably different model behavior. |

---

### Recommended (output is better with these)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-06 | **Axis spread across the full axis vocabulary** | coverage | recommended | The N variations cover at least 3 distinct axes from the axis vocabulary (role sequence, output format, lifecycle emphasis, stock role selection, phrasing register, scoring granularity). Not all variations on the same axis. |
| C-07 | **Hypotheses are falsifiable** | depth | recommended | Every hypothesis is specific enough to be testable: it predicts a directional outcome (e.g., "V-03 will produce more granular scoring because..."), not a vague preference ("this might be better"). |
| C-08 | **Baseline is explicit or inferable** | format | recommended | The output either states the baseline skill body once up front, or makes the stable baseline clearly inferable from the variation set so a reviewer can reason about what changed. |

### Aspirational (raise the bar once essential/recommended are stable)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-09 | **Combination roadmap appended** | depth | aspirational | After the single-axis variations, the output includes a brief combination matrix or table identifying the most promising axis pairings for a follow-up pass, with rationale. |
| C-10 | **Hypothesis tension surfaced** | depth | aspirational | At least one variation explicitly notes a trade-off or tension between axes (e.g., "increasing scoring granularity in V-04 conflicts with the concise output format in V-02"), demonstrating awareness of interaction effects. |
| C-11 | **Explicit failure condition on every hypothesis** | depth | aspirational | Every hypothesis includes a stated failure condition: a specific, observable outcome that would refute the mechanism if it occurred (e.g., "if V-03 does not improve X, the mechanism is refuted"). Enables a negative experimental result to be recovered cleanly rather than explained away. |
| C-12 | **Evaluation order guidance** | depth | aspirational | After the variation bodies, the output includes a ranking or table ordering the variations by evaluation priority -- incorporating cost, independence from other variations, and dependency relationships -- to support a multi-run experimental plan. |
| C-13 | **Hypothesis tension note pre-combination** | depth | aspirational | At least one axis-pair conflict is explicitly named *before* the combination roadmap, with a prediction of which variable will dominate the outcome if the combination is run. Prevents a confounded round-2 result by forcing a phase-priority choice before the combination is designed. |
| C-14 | **Criterion-targeted axis selection** | depth | aspirational | At least one variation names the specific rubric criterion its declared axis is designed to improve. The mechanism connecting the axis to the criterion must be explicit (e.g., "annotated-rationale constraints cause mechanism-citing hypotheses, improving C-07 pass rates"). Naming an outcome-level effect without connecting it to a criterion ID does not pass. |
| C-15 | **Inline generation loop gate** | correctness | aspirational | At least one variation embeds a self-check gate *inside* the variation-generation loop -- fired after each variation body, before advancing to the next -- with an explicit "fix before advancing" or equivalent instruction. A gate that appears only as a post-generation review step does not pass. The gate must name which criteria it is checking. |
| C-16 | **Per-axis pole declaration before generation** | format | aspirational | Before any variation body is generated, the output includes a structured declaration of the current (baseline) pole for every axis in the axis vocabulary. A champion table or equivalent qualifies; a single "the baseline skill body is [X]" reference does not -- each axis must have its current pole named individually. Directly enables C-04 isolation checks by making the stable element visible per axis. |
| C-17 | **Pre-generation axis lock** | correctness | aspirational | Combination passes (or any variation where axis isolation is a risk) include an explicit instruction declaring axis assignments immutable once the generation phase begins ("do not revise axis assignments after Phase N begins" or equivalent). Converts C-04 single-axis isolation from an intention to an enforced protocol constraint. Only evaluated for combination passes; single-axis-only outputs pass by default. |
| C-18 | **Single-axis comparison denominator in combination failure conditions** | depth | aspirational | Combination pass failure conditions name the most relevant single-axis variation as the comparison baseline rather than the bare unvaried skill (e.g., "if the combination does not exceed V-01 alone..." rather than "if the outcome does not improve..."). The named variation must be the one that most directly tests the dominant axis of the combination. Only evaluated for combination passes; single-axis-only outputs pass by default. |
| C-19 | **Structured four-part hypothesis schema** | format | aspirational | At least one variation's hypothesis field is composed of all four named parts: (1) criterion-target (C-NN by ID), (2) directional prediction (increases / decreases), (3) causal mechanism (because...), and (4) explicit failure condition (if...). Passing C-11 + C-14 individually does not imply passing C-19 -- C-19 requires all four parts to be present and structurally labeled within a single hypothesis entry. |
| C-20 | **Dual mechanistically-distinct failure conditions** | depth | aspirational | At least one hypothesis states two failure conditions that are mechanistically distinct: one for *outcome failure* (the axis produced no measurable effect on the target criterion) and one for *implementation failure* (the mechanism was instantiated incorrectly even if the outcome metric is met, producing a result that independently violates a separate criterion). The second failure condition must name the separate criterion it would fail. Single-failure-condition hypotheses (C-11 passes) do not satisfy C-20. |
| C-21 | **Rubric-ID exception citation for combination passes** | format | aspirational | When a combination pass invokes the C-04 single-axis isolation exception, the variation body cites the criterion ID being excepted by number (e.g., "C-04 exception explicitly invoked") rather than using an informal label ("combination pass" or equivalent). Converts the exception from an implied permission to a traceable protocol citation with a clear audit trail. Only evaluated for outputs that include at least one combination pass. |
| C-22 | **Criterion-keyed combination roadmap rows** | depth | aspirational | Each entry in the combination roadmap names the specific rubric criterion IDs the combination is designed to improve, not just a narrative rationale. A row qualifies only if it maps the axis pairing to at least two criterion IDs and includes a mechanism sentence explaining why the pairing addresses those criteria jointly. Exceeds C-09 (which requires rationale) -- this maps each planned combination to an experimentally verifiable criterion goal. |
| C-23 | **Phased prompt architecture** | correctness | aspirational | The variation output uses an explicitly numbered, labeled phase structure -- at minimum: a commitment/planning phase (declare axes, lock assignments), a generation phase (produce bodies with embedded gate), and a findings phase (compare results) -- each with its declared responsibility stated in the prompt. Exceeds C-15 (which requires only an inline gate) by requiring a complete phased container with named transition points between phases. Evaluated when any loop gate is present; single-axis-only outputs with no gate pass by default. |
| C-24 | **Pre-generation variation score predictions** | depth | aspirational | Before the generation phase begins, the output includes a variation map that predicts, for each planned variation, which rubric criteria are expected to improve and in which direction. Constitutes a quantitative pre-commitment that can be evaluated against the actual findings. A champion inventory without per-variation criterion predictions does not pass. Evaluated for any output that includes a planning or commitment phase. |
| C-25 | **Dual failure condition sub-fields as separately labeled keys** | format | aspirational | When a hypothesis contains two failure conditions (satisfying C-20), each condition appears as a distinct named key -- `failure-condition-outcome:` and `failure-condition-implementation:` -- rather than two sentences embedded within a single `failure-condition:` field. The separation makes C-20 compliance structurally detectable by field presence: a scorecard can check for the `failure-condition-implementation:` key without parsing prose. A variation that satisfies C-20 by content (two distinct mechanisms stated in one field) does not satisfy C-25. Only evaluated for hypotheses that claim or target C-20. |
| C-26 | **Per-body axis-freeze protocol inside the generation loop** | correctness | aspirational | Inside the generation phase, before writing each variation body, the prompt instructs the generator to: (1) read the committed axis from the pre-generation declaration; (2) explicitly name every other axis that could tempt drift and freeze each by name; (3) proceed to write only the committed axis change. Converts C-04 single-axis isolation from a post-body audit (did anything else change?) into a per-body pre-write checklist (what else might change -- freeze it now), surfacing contamination risk before it occurs rather than catching it after. Exceeds C-17 (a document-level axis lock at the start of generation) by requiring the freeze to execute per-body inside the generation loop. Only evaluated when a generation phase with multiple variation bodies is present; single-variation outputs pass by default. |
| C-27 | **Pre-generation variation map as a standalone top-level artifact** | format | aspirational | The pre-generation variation map (satisfying C-24) is placed as a distinct, independently labeled section at document scope -- not embedded inside a phase body, planning sub-stage, or preamble block -- and carries an explicit immutability instruction at the section level ("Do not revise after Phase N begins" or equivalent). A C-24-compliant map that lives inside a planning phase block without its own top-level section header and a freeze instruction does not satisfy C-27. C-24 requires the *content* (per-variation per-criterion directional predictions); C-27 requires the *artifact structure* (separate header, document-scope placement, named immutability). |
| C-28 | **Count-verifiable axis-freeze completeness gate** | correctness | aspirational | The per-body axis-freeze protocol (satisfying C-26) includes an explicit count-completeness gate: the protocol states the exact number of axis entries required and instructs the generator to refuse to proceed if the count is not met (e.g., "A freeze list with fewer than N entries is incomplete -- do not proceed"). Makes freeze completeness mechanically verifiable by count without reading the content of each entry -- a count audit supplementing the name audit required by C-26. A C-26-compliant freeze that names each axis but does not declare a required-entry count does not satisfy C-28. Only evaluated when a per-body axis-freeze protocol is present (i.e., C-26 must pass or be targeted). |
| C-29 | **VARIATION MAP declared as authoritative read-source for the in-loop freeze protocol** | correctness | aspirational | The standalone VARIATION MAP (satisfying C-27) includes an explicit declaration that the per-body AXIS-FREEZE PROTOCOL reads its axis assignments from this map (e.g., "The AXIS-FREEZE PROTOCOL in Phase 2 reads axis assignments from this table"), and each axis entry in the AXIS-FREEZE PROTOCOL carries a label confirming it was read from the VARIATION MAP. Converts two independent mechanisms -- pre-generation commitment (C-27) and per-body enforcement (C-26) -- into a traceable protocol chain with a single named source of truth, making it verifiable that the in-loop freeze is executing the same assignments that were immutably declared before generation began. A prompt that passes C-26 and C-27 independently, without linking them via an explicit read-source declaration in both artifacts, does not satisfy C-29. Only evaluated when both C-26 and C-27 are present. |
| C-30 | **Failure-condition predictions embedded in the pre-generation variation map** | depth | aspirational | The pre-generation variation map (satisfying C-24) includes, for each variation, a separately labeled prediction of which failure mode -- outcome failure or implementation failure -- is expected to manifest if the hypothesis is false. Two distinct prediction fields or columns per variation row qualify: e.g., `failure-condition-outcome prediction` (the axis produced no measurable effect on the target criterion) and `failure-condition-implementation prediction` (the mechanism was instantiated incorrectly, independently violating a named criterion). A map with criterion improvement direction predictions (C-24) but no failure-mode predictions does not satisfy C-30. C-30 extends C-24 from a "what improves" record to a "what breaks and how" record, creating a verifiable implementation risk pre-commitment at the planning stage. |
| C-31 | **Cross-artifact axis consistency gate in the generation loop** | correctness | aspirational | The inline generation loop gate (satisfying C-15) includes an explicit cross-artifact consistency check that verifies the axis declared in the standalone VARIATION MAP matches the axis declared in the current variation's per-variation planning header before the body is written (e.g., "Verify VARIATION MAP axis matches Phase 1 header axis before advancing"). Detects silent divergence between the immutable pre-generation commitment and the per-variation declaration -- a case not caught by C-26 (which freezes drift during writing) or C-29 (which links the freeze protocol to the map) because the inconsistency would exist in the planning headers before writing begins. Only evaluated when a standalone VARIATION MAP (C-27) and per-variation planning headers with declared axes (C-03 format) are both present. |
| C-32 | **Named verdict token in inline consistency gates** | format | aspirational | When an inline generation loop gate performs a cross-artifact consistency check (satisfying C-31), the check emits a named binary verdict as a labeled output line -- e.g., `verdict: CONSISTENT` or `verdict: AXIS DIVERGENCE -- halt` -- rather than only a conditional instruction ("if mismatch, stop"). The verdict token makes gate outcome mechanically readable: a scorecard can detect pass/fail by presence of the token without parsing conditional prose reasoning. A C-31-compliant gate that uses only conditional prose without a named verdict token does not satisfy C-32. Only activated when a C-31-compliant cross-artifact consistency check is present. |
| C-33 | **Per-phase declared responsibility as a structurally labeled key** | format | aspirational | In the phased prompt architecture (satisfying C-23), each phase carries its declared responsibility as an explicitly labeled key -- e.g., `Declared responsibility: [text]` -- not as a prose sentence embedded in the phase description or implied from the phase title. The labeled key makes C-23 compliance structurally detectable by field presence rather than prose parsing: a scorecard can confirm each phase has a responsibility declaration by checking for the key, without reading the phase narrative. A C-23-compliant output where responsibility is stated in prose without a key label does not satisfy C-33. A partial-phase output (e.g., only some phases carry the label) does not satisfy C-33 -- all phases in the structure must carry the key. Only activated when a phased prompt architecture is present. |
| C-34 | **Round-qualified single-axis comparison denominator** | depth | aspirational | When a combination failure condition names a single-axis variation as its comparison baseline (satisfying C-18), the citation includes an explicit round qualifier -- e.g., "V-01 R6 alone (C-25 single-axis baseline)" rather than "V-01 alone." The round tag makes the reference unambiguous across experimental iterations where the same variation label (V-01, V-02, etc.) recurs in different rounds with different bodies: a cross-round analysis can resolve "V-01 R6" unambiguously, while "V-01" is ambiguous. A C-18-compliant failure condition that names the variation ID without a round qualifier does not satisfy C-34. Only evaluated for combination passes with a named single-axis comparison denominator (i.e., C-18 must pass or be targeted). |
| C-35 | **Bracket-notation binary option declaration in verdict token fields** | format | aspirational | When a named verdict token is declared in an inline gate (satisfying C-32), the token field definition includes explicit bracket notation showing both possible outcome values at the declaration site -- e.g., `Consistency verdict: [CONSISTENT \| AXIS DIVERGENCE]` -- rather than only the emitted value or a prose description of the options. Makes the full binary decision space readable at field definition without requiring inference from conditional branches elsewhere in the gate. A C-32-compliant gate that emits a verdict token without declaring both option values using bracket notation at the token definition site does not satisfy C-35. Only activated when a C-32-compliant verdict token is present. |
| C-36 | **Responsibility label content grounded in protocol action verb** | format | aspirational | When a phase carries a `Declared responsibility:` labeled key (satisfying C-33), the label content uses an action-verb phrase that directly names the mechanical operation of that phase (e.g., "lock axis assignments," "write committed variation bodies," "confirm in-loop axis consistency") rather than a generic phase-title synonym (e.g., "planning," "generation," "review") or a restatement of the phase name. Makes C-33 compliance structurally meaningful: label content can be mechanically checked against the phase's protocol action, not just key presence. A C-33-compliant phase that carries the key with generic or title-echoing content does not satisfy C-36. All phases in the architecture must carry action-verb-grounded labels; a partial set fails C-36. Only activated when C-33 is present. |
| C-37 | **Structural key names propagate uniformly across all variation bodies in the set** | correctness | aspirational | Within a variation set, any structural key name that appears in one variation body (e.g., `failure-condition-outcome:`, `Declared responsibility:`) must appear consistently in all other variation bodies where the corresponding field is required, regardless of which axis those bodies vary. An axis change that does not target phase structure or hypothesis key naming cannot introduce key-name regression (reverting `failure-condition-outcome:` to `failure-condition:`) or key dropout (losing `Declared responsibility:` from phases not touched by the axis change) in other bodies. Evaluated as a set-level scan: all bodies are checked for the named key; non-uniform results fail. Only activated when at least one body in the set passes a structural key-naming criterion (C-25, C-33) and at least one other body contains the same field category. |
| C-38 | **Point-of-production anti-pattern prohibition with corrective instruction** | correctness | aspirational | When a quality criterion for label content is targeted (e.g., C-36 action-verb grounding), the variation body includes, physically adjacent to each label production site, an explicit prohibition that: (a) names the failure mode by description (e.g., "a label whose content echoes the phase title"), and (b) declares violation a required rewrite -- not a post-hoc audit signal or a downstream review step. Converts the criterion from a convention that is checked after authorship to a constraint enforced at the moment of authorship: the generator encounters the prohibition while writing the label, not after. A variation that satisfies C-36 (correct action-verb content present) via a pre-generation convention or post-generation scorecard check, without placing the prohibition at the production site, does not satisfy C-38. Only activated when C-36 (or an equivalent content-grounding criterion) is present. |
| C-39 | **Named structural key uniformity auditor role with per-key result emission and phase gate** | correctness | aspirational | When C-37 (structural key uniformity) is targeted, the prompt architecture includes a separately named enforcement role -- e.g., "Structural Key Uniformity Auditor" -- that: (a) fires post-generation as a set-level scan over all variation bodies, (b) operates against an explicitly declared list of required structural keys, (c) emits structured per-key results (pass/fail per key per body, not a single aggregate verdict), and (d) blocks advancement to the findings phase on any per-key fail with an explicit "required rewrite before [next phase] may begin" instruction. Distinct from C-37 (which requires uniform propagation as an outcome) by requiring a named enforcement role and structured output format; distinct from C-15 (which requires a per-body gate inside the loop) by requiring the role to fire post-generation over the complete set. A C-37-compliant set where key uniformity is enforced solely by the Variation Author per-body gate (C-15) does not satisfy C-39. Only activated when C-37 is present. |
| C-40 | **Check-step inline expected-form declaration for structural keys** | format | aspirational | When a key-inspection check step verifies structural key compliance -- e.g., a Step 2A key list, auditor scan list entry, or equivalent check instruction -- the check includes the complete expected form of the key inline at the check site, not just the key name. Makes the check self-documenting: a generator running the check can confirm compliance by comparing the output against the embedded expected form without consulting a separate criterion definition or field declaration. A prompt where check steps name the key without embedding the expected form (e.g., "verify Consistency verdict is present in Step 2B") does not satisfy C-40 even if C-35 is satisfied at the declaration site. Only activated when a key-inspection check step is present (e.g., Step 2A key list, auditor scan list). |

## Scoring

```
composite = (essential_pass / 5 * 60)
          + (recommended_pass / 3 * 30)
          + (aspirational_pass / 32 * 10)
```

**Golden threshold**: all 5 essential criteria pass AND composite >= 80.

| Band | Score | Meaning |
|------|-------|---------|
| Gold | >= 80, all essential pass | Ship-ready variation set |
| Silver | >= 65, all essential pass | Usable with minor gaps |
| Bronze | >= 50, >= 4 essential pass | Needs revision before use |
| Fail | any essential fails | Not usable |

## Evaluator Notes

- C-01 is the hardest essential to fake: spot-check by asking "could I paste V-02 directly into a skill file and run it?" If no, C-01 fails.
- C-04 failures are common when a skill body is complex -- authors tend to fix two things at once. One axis change per variation is a strict constraint.
- **C-14 vs C-07**: C-07 requires a falsifiable hypothesis. C-14 requires that the axis was *chosen* to target a specific rubric criterion, with the mechanism stated. A variation can pass C-07 and fail C-14 if the axis-to-criterion connection is never made explicit.
- **C-15 vs C-08**: C-08 checks whether a baseline is present. C-15 checks whether a correctness gate fires inside the loop. They are independent -- a variation can pass C-08 and fail C-15.
- **C-16 vs C-08**: C-08 passes if the baseline is "inferable." C-16 requires a structured per-axis declaration of current poles before generation begins. A set can pass C-08 and fail C-16.
- **C-17 and C-18**: Only evaluated for outputs that include at least one combination pass. If all variations are single-axis, both criteria are vacuously true and score as full passes.
- **C-19 vs C-11 + C-14**: A variation can pass both C-11 (has a failure condition) and C-14 (names a criterion ID) while still failing C-19 if those elements live in separate variations or all four parts are not present in a single, compositionally structured hypothesis entry.
- **C-20 vs C-11**: C-11 requires one explicit failure condition. C-20 requires two failure conditions that are mechanistically *different*: one tests whether the axis produced its claimed outcome, the second tests whether the mechanism was implemented in a way that would independently violate another named criterion. If both failure conditions describe the same mechanism failure, C-20 does not pass.
- **C-21**: Only evaluated for combination passes. The criterion ID must appear verbatim or in an unambiguous form (e.g., "C-04 exception explicitly invoked"). A phrase like "per the combination rule" does not pass.
- **C-22 vs C-09**: C-09 passes with a narrative rationale per roadmap row. C-22 requires criterion IDs in addition. A roadmap that says "combining role-sequence with inertia-framing should improve completion quality" fails C-22; one that says "targets C-01, C-04, C-07" passes.
- **C-23 vs C-15**: C-15 requires a gate inside the loop. C-23 requires an explicitly labeled, numbered phase structure with declared responsibilities per phase. A prompt with a loop gate but no phase labeling passes C-15 and fails C-23. A prompt with phases but no inline gate fails C-15 and may pass C-23.
- **C-24**: The variation map must be directional per criterion per variation -- not a single summary claim. "V-03 is expected to improve C-07 pass rates" per row qualifies. "These variations will improve output quality overall" does not.
- **C-25 vs C-20**: C-20 requires two mechanistically-distinct failure conditions by *content*. C-25 requires they appear as *two separately named keys* in the hypothesis structure. A hypothesis can pass C-20 and fail C-25 if both conditions are written as prose sentences within a single `failure-condition:` field. C-25 only activates for hypotheses that contain or target C-20.
- **C-26 vs C-17**: C-17 is a one-time document-level axis lock at the start of generation ("do not revise after Phase N begins"). C-26 requires the freeze to execute inside the loop, once per variation body -- naming the specific axes at risk for *that body* before writing it. A prompt with a C-17-compliant document lock but no per-body freeze step passes C-17 and fails C-26.
- **C-27 vs C-24**: C-24 passes if per-variation per-criterion directional predictions exist anywhere before generation begins, including inside a planning sub-stage. C-27 requires those predictions to live in a section with its own top-level header and an explicit immutability declaration at the section level. A table embedded inside "PHASE 1B: Score Predictions" without a separate named artifact heading fails C-27.
- **C-28 vs C-26**: C-26 passes if the per-body freeze protocol names each axis to be frozen. C-28 requires the protocol to additionally declare the required number of entries and refuse to proceed if the count is not met. A freeze protocol can name all six axes (passing C-26) without stating "fewer than six entries is incomplete" (failing C-28). Count-verifiability is the distinguishing requirement.
- **C-29 vs C-26 + C-27**: C-26 and C-27 can both pass independently -- a prompt can have a per-body freeze (C-26) that names axes generically, and a standalone map (C-27) with an immutability instruction, without either referencing the other. C-29 requires both artifacts to be explicitly linked: the VARIATION MAP declares it is the read-source, and the per-axis freeze entries confirm they read from it. An output where C-26 and C-27 are structurally independent passes both and fails C-29.
- **C-30 vs C-24**: C-24 requires a map predicting which criteria improve and in which direction. C-30 requires the same map to also contain per-variation predictions of which failure mode will manifest (outcome failure vs implementation failure). Both can be satisfied by the same map with additional columns; C-24 is a prerequisite for C-30.
- **C-31 vs C-15**: C-15 requires an inline gate that names criteria. C-31 requires the gate to include a specific cross-artifact check: does the axis in the VARIATION MAP match the axis in this variation's planning header? A gate that names criteria (C-15 pass) without performing this comparison (C-31 fail) is common when the planning header and variation map are authored at different points. Only activated when both artifacts are present.
- **C-29 and C-31 relationship**: C-29 ensures the freeze protocol reads axis assignments from the VARIATION MAP. C-31 ensures the gate verifies those assignments are consistent between the map and the per-variation header. They address different divergence risks: C-29 prevents the freeze from using stale values; C-31 detects when the per-variation header was updated inconsistently with the map.
- **C-32 vs C-31**: C-31 requires the cross-artifact consistency check to fire and halt on divergence. C-32 requires the check to also emit a named binary verdict token as a labeled output line. A gate can satisfy C-31 (check fires, halt instruction present) while failing C-32 (outcome is expressed only as a conditional prose instruction with no named token). The verdict token is the distinguishing requirement.
- **C-33 vs C-23**: C-23 passes if the declared responsibility for each phase is "stated in the prompt" -- prose, title, or key all qualify. C-33 requires specifically a labeled key (e.g., `Declared responsibility: [text]`). An output where responsibilities are communicated through phase titles or embedded prose sentences passes C-23 and fails C-33. All phases must carry the key; a partial set (only some phases labeled) fails C-33.
- **C-34 vs C-18**: C-18 passes if the combination failure condition names a specific single-axis variation as the comparison denominator. C-34 requires that name to include a round qualifier. "V-01 alone" passes C-18 and fails C-34. "V-01 R6 alone" passes both. Round-qualification is especially load-bearing in multi-round experimental programs where variation labels repeat. Only activated for combination passes where C-18 is applicable.
- **C-35 vs C-32**: C-32 passes if a named binary verdict token is emitted as a labeled output line. C-35 requires the token field definition to also declare both option values using bracket notation at the declaration site. A gate that emits `verdict: CONSISTENT` without also showing `[CONSISTENT | AXIS DIVERGENCE]` as the declared option space passes C-32 and fails C-35. The bracket declaration is the distinguishing requirement.
- **C-36 vs C-33**: C-33 requires a `Declared responsibility:` key to be present in every phase. C-36 requires the key's content to be an action-verb phrase naming the phase's protocol operation. A phase with `Declared responsibility: Generation phase` (title echo) passes C-33 and fails C-36. A phase with `Declared responsibility: write committed variation bodies` passes both. All phases must carry action-verb-grounded content; a partial set (some phases with action verbs, others with generic labels) fails C-36.
- **C-37 vs C-25 + C-33**: C-25 evaluates whether a specific hypothesis uses `failure-condition-outcome:` key naming. C-33 evaluates whether a specific body's phases carry `Declared responsibility:` keys. C-37 is a set-level scan: it fails if any body in the set uses a regressed key name (`failure-condition:` when another body uses `failure-condition-outcome:`) or drops a structural key from phases that the axis change did not target. A set where all individual bodies pass C-25 and C-33 automatically satisfies C-37; C-37 only activates as a differentiator when at least one body fails C-25 or C-33 and the failure is traceable to axis-change-induced regression rather than a design choice.
- **C-38 vs C-36**: C-36 requires action-verb-grounded content in the `Declared responsibility:` key -- it can be satisfied by a correct author writing the right content without any prohibition in the prompt. C-38 requires the prompt to explicitly name the failure mode (title-echoing content) and declare it a required rewrite, at the production site where the label is written. An output where a Variation Author simply uses correct action-verb content without encountering a prohibition instruction passes C-36 and fails C-38. Placement is the distinguishing requirement: the prohibition must be adjacent to the label, not in a checklist or separate audit section.
- **C-39 vs C-37 + C-15**: C-15 requires a gate inside the generation loop, per body, before advancing. C-37 requires uniform key propagation as an outcome. C-39 requires a separately named audit role that fires after all bodies are generated, scans the complete set, and produces structured per-key results. The three can all pass independently: C-15 prevents per-body regression during authorship; C-37 verifies the outcome is uniform; C-39 requires the scan to be named, structured, and gated. A prompt that prevents key dropout via C-15 alone (no named auditor) satisfies C-37 if the outcome is uniform but does not satisfy C-39.
- **C-40 vs C-35**: C-35 requires bracket notation at the token field *definition* site. C-40 requires bracket notation also at the *check* site. A prompt where `[CONSISTENT | AXIS DIVERGENCE]` appears at the declaration site (Step 2B) but the check step (Step 2A or auditor scan list) names the key without the bracket form passes C-35 and fails C-40. The self-documenting check site is the distinguishing requirement.
