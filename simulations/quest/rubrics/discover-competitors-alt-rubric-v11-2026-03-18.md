**Two new criteria added — C-30 and C-31:**

**C-30 — Bracket-notation substitution rule declared for REQUIRED OUTPUTS column headers**
C-29 tests the *outcome* (committed value appears in the column header). C-30 tests the *mechanism*: the template must use `[TOKEN]` bracket syntax with an explicit substitution instruction — not the static abstract word `TOKEN`. This is the structural distinction between V-01/V-03 (fail) and V-02/V-04/V-05 (pass). Requires C-29.

**C-31 — GATE-0 REQUIRED OUTPUTS table retains template form with explicit propagation declaration**
GATE-0's own table must keep `[TOKEN] required?` as the column header (not substitute the committed value) and carry a propagation declaration row stating that the committed value substitutes `[TOKEN]` in all GATE-1+ tables. This makes GATE-0 simultaneously a declaration artifact and a substitution contract, both readable from the table alone. A GATE-0 that writes the committed value into its own header is correct but undocumented — the propagation rule is consumed without being stated. Requires C-30.

## Criterion Rationale (v11 additions)

**Why C-30 (Bracket-notation substitution rule declared for column headers):**
R10 V-01 and V-03 used `TOKEN required?` as a static column header — the abstract word, not the committed identifier — and both failed C-29 because the column header did not carry the committed value. V-02, V-04, and V-05 used `[TOKEN] required?` with an explicit substitution instruction, so after GATE-0 declared `TOKEN: SIGNAL-LOCK`, subsequent tables produced `SIGNAL-LOCK required?` as the column header. C-29 captures the outcome: committed value appears in the header. C-30 captures the mechanism: the substitution rule is explicitly declared, not implicitly assumed. A variation that produces committed-value headers through ad hoc rewording rather than a declared substitution rule is brittle — the rule can be dropped when the rubric is adapted or the skill is iterated. A declared substitution instruction makes the mechanism durable: the header is committed-value by contract, not by coincidence.

**Why C-31 (GATE-0 retains template form with propagation declaration):**
R10 V-02, V-04, and V-05 showed that GATE-0's own REQUIRED OUTPUTS table retaining the `[TOKEN]` template form — rather than substituting the committed value — serves a structural function: it makes GATE-0 both the declaration artifact and the propagation specification. When GATE-0's table carries a row stating "declared value becomes the substituted column header in all subsequent gates," a reader consulting only GATE-0's table learns two things: what TOKEN is being committed, and how that commitment propagates to every subsequent gate. A GATE-0 that substitutes the committed value into its own header consumes the template without documenting the propagation rule, leaving GATE-1+ headers correct but unexplained. The template-plus-declaration pattern makes the substitution contract self-contained at the declaration gate: GATE-0's table is a complete specification of both the token value and the propagation mechanism, independently verifiable without reading GATE-1+ tables or body prose.
