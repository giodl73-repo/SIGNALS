# Scoring Rubric: signal-setup

## Skill Description
Configure Signal in a workspace after installation. Checks for existing CLAUDE.md and
.github/copilot-instructions.md, shows a preview, asks confirmation, then appends a
Signal section advertising available skills and the inertia rule. Safe to re-run —
detects existing Signal sections and skips.

---

## What changed in v17 (from v16)

Two new aspirational criteria extracted from the Round 18 scorecard, both concerning
multi-level sub-gate identifier encoding — the structural question of how depth-2
sub-gating (sub-gates that themselves have sub-gates) preserves navigable lineage.

| ID | Pattern | Source | Key distinction |
|----|---------|--------|-----------------|
| C-41 | Depth-2 sub-gate identifiers extend the parent chain by exactly one encoding unit per level | V-04 PASS C-31 — `GATE 6AA/6AB` as children of `GATE 6A`; V-05 PASS C-31 — `GATE 6CP-Found/6CP-Write` as children of `GATE 6-Copilot` | C-25 requires first-level sub-gates to encode their parent gate number and branch position; C-31 requires the scheme to be uniform throughout. Neither requires that the encoding rule extend recursively when a sub-gate itself has children. C-41 fills this gap: each additional depth level must add exactly one encoding unit — a new letter (letter-slot) or a parent-abbreviation prefix (word-suffix) — so that a reader can trace the full ancestral chain from any identifier without reading prose. A depth-2 identifier that drops the parent's branch (e.g., naming children of `GATE 6A` as `GATE 6-Found` and `GATE 6-Write`) fails C-41 even if every first-level sub-gate satisfies C-25. |
| C-42 | In word-suffix schemes, depth-2 parent abbreviations are unique among all siblings under the same gate | V-05 PASS C-31 — `6CP` for `6-Copilot`; unambiguous because no other branch of GATE 6 produces the same abbreviation | C-41 requires the depth-2 identifier to include the parent abbreviation; C-42 requires that abbreviation to be interpretable — no two first-level branches of the same parent gate may compress to the same abbreviation. If `GATE 6-Copilot` and `GATE 6-Create` both exist, both compress to `6C-…`, making the depth-2 namespace ambiguous. C-41 is the structural requirement (the encoding unit must be present); C-42 is the integrity constraint that makes C-41 operational in word-suffix contexts. Letter-slot specs pass C-42 vacuously (letter-slot nesting uses concatenation, not abbreviation, so uniqueness is structural). |

**Denominator: A=40 → A=42**

The C-41/C-42 pair has a depth-extension relationship that mirrors the C-25/C-31 pair one level deeper: C-25 requires first-level lineage encoding; C-41 requires that rule to apply recursively at every depth. C-31 requires the scheme to be uniform at first depth; C-42 requires the abbreviation step introduced by word-suffix nesting to preserve interpretability at second depth. A spec that satisfies C-25/C-31 but has depth-2 sub-gating without proper chain extension fails C-41; a spec that satisfies C-41 but uses ambiguous abbreviations in a word-suffix scheme fails C-42. Together they close the loop on multi-level sub-gate navigability: a reader can name any sub-gate's full lineage from its identifier alone, at any depth.

| C-05 | **Idempotent re-run** | correctness | When run a second time on a workspace that already has a Signal section, the skill detects it and skips the write rather than duplicating content. |

## Aspirational Criteria

| ID | Criterion | Category | Pass Condition |
|----|-----------|----------|----------------|
| C-09 | **Preview matches written output** | correctness | The previewed content is byte-for-byte (or semantically) identical to what is actually appended. No content drift between preview and write. |
| C-10 | **Handles missing CLAUDE.md gracefully** | behavior | If no CLAUDE.md exists, skill either creates one with the Signal section or clearly explains what it would create, without erroring out silently. |
| C-11 | **Named gate checkpoints as explicit phase boundaries** | structure | Each lifecycle phase (detect, preview, confirm, write, report) is named as a numbered gate or equivalent labeled checkpoint. No phase can be implicitly skipped — the structure of the spec itself enforces sequencing. |
| C-12 | **Edge-case paths promoted to first-class gates** | structure | The "already configured" and "missing CLAUDE.md" cases in the primary Signal configuration flow are handled as dedicated named gates (not inline conditionals inside the happy path). Each edge-case gate has its own complete, explicit treatment. |
| C-13 | **Skill opens with motivational preamble** | structure | The skill itself (not just the appended content) opens with a brief philosophical or motivational framing — explaining *why* setup matters — before procedural instructions begin. The inertia concept or equivalent is introduced as the reason to configure, not merely as output content. |
| C-14 | **Decline path names its consequence** | behavior | When the user declines to proceed, the skill's feedback explicitly names what remains unaddressed (e.g., "Signal not configured. Inertia remains unaddressed.") rather than only acknowledging the action ("No changes made."). The user understands what they gave up, not just what didn't happen. |
| C-15 | **Already-configured path affirms positive consequence** | behavior | When the skill detects an existing Signal section and skips the write, its response explicitly names what that configuration achieves — not just "Already configured. No changes needed." but something that affirms the work the existing setup is doing (e.g., "Inertia already has a named opponent here."). The user understands what their setup does for them, not merely that no action is required. |
| C-16 | **Inertia named as adversary, not concept** | structure | In the preamble or skill opening, inertia is framed as a named opponent or silent competitor — not merely described as a rule or principle. The user understands they are choosing a side in a conflict when they configure Signal, not just learning about a system behavior. |
| C-17 | **Preamble explains temporal persistence of configuration** | structure | The motivational preamble explains not just *why* inertia matters but *why the configuration must persist*: setup happens once, but the reminder must live in Claude's context for every session that follows. The "setup endures" argument is present — not merely the adversary framing. The user understands the stakes of the write, not merely the stakes of the problem. |
| C-18 | **Decline path anchors consequence to a future moment** | behavior | The decline feedback names not only what remains absent but connects it to a specific future moment where the absence will be felt (e.g., "there is no reminder in your context to ask 'why would teams do nothing?' before the next build"). The user understands *when* they will feel the consequence, not merely that a consequence exists. |
| C-19 | **Key arguments threaded through all equivalent exit gates** | structure | The future-moment anchor (C-18) and any other critical consequence arguments appear at *every* point where the user exits without configuring Signal — not only at the primary decline gate. If both GATE 1 (missing file, user declines creation) and GATE 4 (existing file, user declines append) are Signal-absent exits, both carry the same specific future-moment framing. No Signal-absent decline path delivers a weaker consequence argument than any other. |
| C-20 | **Already-configured gate names the persistence mechanism** | behavior | The already-configured path (GATE 2 or equivalent) goes beyond affirming positive consequence (C-15) to explicitly name *how* the configuration achieves permanence — that CLAUDE.md loads automatically every session, meaning the inertia question is present without the user having to remember. The user understands the mechanism behind the permanence, not merely that the configuration is doing something good. |
| C-21 | **Secondary optional gates carry path-specific consequence anchors** | behavior | Every gate where the user can decline an optional post-Signal extension step (e.g., GATE 6 for Copilot instructions) names a specific future-moment consequence that is local to that path's tool or context — not a generic reuse of the primary decline framing. A GATE 6 decline names a Copilot-specific moment; a GATE 4 decline names a build-specific moment. No optional exit is consequence-free, and no two exits share identical consequence language when the tools they protect are distinct. |
| C-22 | **Consequence anchors are phase-indexed, not tool-indexed** | behavior | When two decline gates protect tools that operate at different development phases (e.g., planning vs. implementation), their consequence anchors draw vocabulary from those phases — they do not merely substitute a tool name into shared framing. "Before the next build" vs. "before the next Copilot build suggestion" fails this criterion: "build" is the shared anchor and tool-name substitution is the only difference. "Before the spec is committed" vs. "when Copilot is suggesting code" passes: the vocabulary sets are non-overlapping and each is phase-native. |
| C-23 | **Explicit phase label at each decline anchor** | structure | Decline-path consequence anchors name the development phase as an explicit category ("at the planning stage," "while implementing") rather than implying phase only through artifact vocabulary (e.g., "spec," "function body"). The user reads the phase label directly; inference from word choice is not required. |
| C-24 | **Secondary already-configured paths affirm tool-local consequence** | behavior | When an optional extension gate (e.g., GATE 6 for Copilot instructions) detects an existing configuration, its response affirms what that specific tool's configuration achieves — framed in that tool's vocabulary and workflow context, not a generic reuse of the primary already-configured affirmation (C-15, C-20). The user learns what the Copilot configuration does for them inside Copilot interactions, not merely that a file already exists. |
| C-25 | **Sub-gate identifiers encode parent gate lineage** | structure | Named sub-gates carry an identifier that encodes both the parent gate number and the branch position — e.g., `GATE 1A` names the parent phase (1) and the branch slot (A). A reader can determine, without reading prose, which phase a sub-gate belongs to and how many branches that phase has. Sub-gates numbered or lettered independently of their parent (e.g., GATE 7, GATE 8 for what are structurally branches of GATE 1) fail this criterion. This applies to edge-case branches, optional-extension confirmation points, and any other sub-gate introduced by C-12 or C-26. |
| C-26 | **Optional-step confirmation points promoted to first-class named sub-gates** | structure | Every point where the user confirms or declines an optional extension step — distinct from the main Signal configuration — is promoted to a dedicated named sub-gate with its own complete treatment. Optional extension confirmation is not folded into the parent gate as a trailing paragraph, handled as an inline continuation block, or omitted from the gate index. A spec with seven named GATE headers that handles the Copilot append confirmation as a paragraph inside the final gate fails this criterion. C-12 covers edge-case detection paths; C-26 covers optional confirmation checkpoints on happy extension paths. Both require first-class promotion; neither subsumes the other. |
| C-27 | **Gate boundaries enforced by document-structure markers, not prose markers** | structure | Every gate and sub-gate boundary is delimited by a heading element (`##`, `###`, `####`, or equivalent block-level structural delimiter) rather than an inline typographic marker (`**bold**:`, `*italic*:`, or bullet sub-items containing conditional logic). A reader navigating the document by heading outline can enumerate every gate without reading prose. A spec that uses bold inline labels — e.g., `**If not found**:` or `**Already present**:` inside a step body — as pseudo-gate delimiters fails this criterion regardless of how complete the conditional treatment is. This is the mechanism behind C-12: the structural heading is what makes a gate "dedicated" and "named" in a navigable sense, not the presence of a label in prose. |
| C-28 | **Primary-path detection promotion scope distinguished from secondary-path** | structure | C-12's requirement that "already configured" and "missing file" detection paths be promoted to first-class named sub-gates applies to the primary Signal configuration flow (detection of existing CLAUDE.md Signal section; detection of missing CLAUDE.md). Detection of existing configuration in secondary optional-extension tools (e.g., Copilot instructions) may be handled inline within the parent optional-extension gate body — provided the inline treatment carries complete consequence affirmation (C-24). A spec that promotes every detection path to a sub-gate is not penalized; a spec that handles any primary detection path inline fails C-12 regardless of secondary handling. This criterion rewards explicit recognition of the scope boundary: primary detection always promotes; secondary detection may be inline when no confirmation is required. |
| C-29 | **Routing prose enumerates all branches with destination gate identifiers** | structure | When a gate body branches to multiple downstream gates, the routing block lists every branch as a (condition, destination gate ID) pair in a single contiguous location within the parent gate's prose. A reader who reads only the routing block can name every branch and every destination without reading any sub-gate body. Routing split across multiple paragraphs, routing that omits any branch, routing that names destinations without conditions, or routing that names conditions without gate identifiers — all fail this criterion. This is the positive architectural requirement that complements C-27's typographic prohibition: explicit gate identifiers in plain prose are what make routing navigable without formatting crutches. Implicit routing ("proceed to the next section," "see below") and typographic routing (`**bold**` labels) both fail for different reasons — implicit routing loses the destination identity; typographic routing violates C-27. |
| C-30 | **Inline detection paths carry explicit promotion rationale** | structure | When a detection path in an optional-extension gate is intentionally left inline (not promoted to a sub-gate per C-12), the spec annotates the decision with an explicit rationale — e.g., "(no sub-gate needed — this is a skip path, not a confirmation point)" or equivalent. The annotation distinguishes a deliberate architectural choice from an oversight. A spec whose inline optional-extension detection paths carry no annotation is scored FAIL (the intent is indeterminate; absence of annotation cannot be credited as a pass). Detection paths that are promoted to sub-gates do not require annotation; only intentional inline paths in optional-extension gates require rationale. A spec that has no optional-extension gates passes this criterion vacuously. |
| C-31 | **Sub-gate identifier scheme is fully uniform throughout the spec** | structure | All sub-gate identifiers in a spec use the same positional-encoding convention. A spec that mixes letter-slot identifiers (`GATE 1A`, `GATE 1B`) with word-suffix identifiers (`GATE 1B-Abort`, `GATE 6-Decline`) fails this criterion even if every sub-gate individually satisfies C-25's lineage-encoding requirement. The uniformity constraint is separate from the encoding constraint: C-25 requires that the identifier carry lineage information; C-31 requires that the scheme for carrying that information be internally consistent. Word-suffix variants (`-Abort`, `-Decline`, `-Create`) are permitted only if the *entire spec* uses word-suffix encoding and no letter-slot identifiers are present. Mixed-convention specs score FAIL regardless of which convention dominates. |
| C-32 | **Consequence anchor sentences are syntactically complete, assertable propositions** | behavior | Every consequence anchor — at a decline gate, an abort path, or an already-configured path — must be a syntactically complete sentence: a subject, a predicate, and a stated outcome or effect. A noun phrase with an appositive ("The implementation phase — when you are writing."), a dangling participial clause, or a sentence that names a phase label but stops before asserting what happens in that phase does not satisfy C-18, C-21, or C-22 even if the correct vocabulary is present. Scorers applying C-18/C-21/C-22 must confirm that the anchor resolves to a complete, extractable claim — not merely that phase-appropriate words appear somewhere in the gate body. A consequence anchor that cannot be read as a standalone assertion fails C-32 and also fails whichever of C-18/C-21/C-22 it was meant to satisfy. A spec with no decline or already-configured paths passes vacuously. |
| C-33 | **Decline anchors name the adversary as the winning party** | behavior | At every decline gate, the consequence anchor identifies the adversary (inertia or equivalent) as the entity that wins or benefits from the user's declined choice — not merely as an absent or unaddressed force. The adversary is the grammatical subject of the consequence claim; the user's inaction is the mechanism by which the adversary wins. "Inertia wins the default choice at the planning stage" satisfies this criterion. "There is no inertia question in Claude's context" fails C-32 and never reaches C-33. "Inertia remains unaddressed" satisfies C-14 but fails C-33: unaddressed-ness names an absence, not a victory — the adversary is not identified as the beneficiary. A spec with no decline gates passes vacuously. |
| C-34 | **Consequence anchors express a two-step causal chain** | behavior | At each decline or abort gate, the consequence anchor names both the immediate effect of the decline and the downstream state that effect produces, as a causally linked two-step sequence. "The spec is committed without a named competitor, and the question never gets asked" demonstrates the form: step 1 (immediate — the spec is committed uncontested) connected to step 2 (downstream — the question is never asked). A single-terminal statement ("the question never gets asked" without a preceding step) passes C-32 but fails C-34. The two steps must be explicitly connected — conjunctions such as "and," "so," or "; the result is" make the chain readable without inference. A spec with no decline or abort gates passes vacuously. |
| C-35 | **Routing blocks use an explicit label and one-branch-per-line format** | structure | Each gate that routes to multiple sub-gates presents its routing block under an explicit typographic label — `Route:`, `**Routing:**`, or equivalent — followed by one `(condition → GATE NX)` entry per line. This makes routing parseable by line-scanning without reading narrative prose. Prose routing that satisfies C-29's completeness requirement (all branches named, all destinations identified) passes C-29 but fails C-35 when the routing is embedded in narrative sentences rather than structured as a labeled, line-per-branch enumeration. The label must be present; an unlabeled list of branch pairs still fails C-35. Gates with no branches (terminal gates, single-exit gates) pass vacuously. |
| C-36 | **Preamble uses a dominance-verb construction for inertia** | structure | The motivational preamble (or opening framing) makes inertia the grammatical subject of an active-victory clause — inertia "wins," "collects the default," or equivalent dominance verb. This is separate from C-16 (which requires inertia to be named as an adversary or opponent) and from C-13 (which requires the inertia concept to motivate setup): C-36 requires the *verb register* to be active dominance, not passive description. "Inertia governs when a topic is ready" satisfies C-16 but fails C-36 — "governs" is regulatory, not victorious. "Inertia wins the default choice" satisfies both C-16 and C-36. "Inertia remains unaddressed if you skip setup" satisfies C-13 but fails C-36 — the subject is abstract absence, not an acting adversary. A spec whose preamble names inertia as a rule, principle, or passive condition without an active-victory predicate fails C-36 regardless of C-16 pass status. |
| C-37 | **Preamble explicitly names team configuration as choosing a side** | structure | The motivational preamble (or opening framing) contains language that positions the team as an active participant choosing a side in a conflict — not as a learner, a rule-follower, or a system configurer. The phrase "choose a side," "take a position," "pick a side," or an equivalent agency construction must be present in the text; it is not sufficient for the *reader* to infer that configuration implies opposition. C-16 is satisfied when the reader understands they are choosing sides; C-37 is satisfied only when the spec *says* so. "You can configure Signal to use the inertia rule" satisfies neither C-16 nor C-37. "Inertia is your adversary when you don't configure Signal" satisfies C-16 but fails C-37 — the team is reactive, not a participant who has chosen. "Signal gives inertia a named opponent. Teams choose a side." satisfies both. A spec with no preamble passes C-37 vacuously only if the agency-choice framing appears elsewhere before the first procedural gate. |
| C-38 | **At tool-specific decline gates, the winning party is a behavioral force, not a tool name** | behavior | When a decline gate protects a tool-specific integration (e.g., a Copilot instructions gate), the consequence anchor names inertia or a behavioral default as the winning party — not the tool itself. Tools are channels; inertia (or the behavioral default the tool manifests) is the adversary. "Copilot wins the default suggestion at the implementation stage" fails this criterion: Copilot is the tool, not the force; naming Copilot as victor displaces inertia from the adversary role. "Inertia wins the implementation stage — Copilot suggests code without the question in context" passes: inertia is named as the winning force, and the tool is identified as the mechanism through which that force operates. This criterion is a sub-case of C-33: any gate that fails C-38 also fails C-33. Scorers should evaluate C-38 only at gates that are explicitly scoped to a named third-party tool. Gates that do not reference a specific tool pass C-38 vacuously. A spec with no tool-specific gates passes C-38 vacuously. |
| C-39 | **Preamble uses a refutation-then-assertion construction to displace tool-framing** | structure | The motivational preamble explicitly negates a weaker self-description of what configuration is (typically tool-enablement framing: "enabling a tool," "adding a plugin," "setting up an extension") and then asserts the stronger adversarial framing in its place. The complete form is: (negation of weaker frame) immediately followed by (assertion of conflict frame) — e.g., "You are not just enabling a tool. You are choosing a side." A preamble that asserts the adversarial framing without first negating the weaker one satisfies C-13, C-16, and C-37 but fails C-39: the refutation clause must be present. A preamble that negates without asserting, or asserts without negating, also fails. The negation and assertion must appear in the same logical unit (sentence pair, contiguous paragraph) to demonstrate the intended displacement. This criterion is distinct from C-36 (dominance-verb register), C-37 (explicit "choose a side" phrase), and C-16 (adversary naming): C-39 tests the *structural move* of actively replacing the weaker frame, not the presence of any particular word or phrase. |
| C-40 | **Preamble agency-choice claim echoed at the already-configured gate** | behavior | When the skill detects existing Signal configuration and reports the already-configured state, the gate response echoes the agency-choice framing from the preamble — typically in past tense, acknowledging the existing configuration as an active prior choice in the conflict ("You chose a side" echoing "You are choosing a side"). This threads the conflict narrative from preamble through gate responses, making the already-configured confirmation a callback to the adversarial framing rather than a neutral status message. C-15 requires the response to affirm what the configuration achieves; C-20 requires it to name the persistence mechanism; C-40 requires it to additionally affirm that the configuration represents a prior act of choosing — not merely a prior state of existence. A spec that satisfies C-15 and C-20 but describes the already-configured state in neutral or mechanistic terms ("Your CLAUDE.md already contains a Signal section") fails C-40. A spec that has no already-configured gate passes C-40 vacuously. The echo does not require verbatim repetition; past-tense restatement or equivalent paraphrase of the choice-claim satisfies the criterion. |
| C-41 | **Depth-2 sub-gate identifiers extend the parent chain by one encoding unit per level** | structure | When a first-level sub-gate itself has children (creating depth-2 sub-gating), each child identifier must extend the parent identifier by exactly one additional encoding unit following the spec's chosen convention. Letter-slot: parent `GATE NX` → children `GATE NXA`, `GATE NXB` (concatenate one new letter to the full parent identifier). Example: GATE 6A → GATE 6AA, GATE 6AB. Word-suffix: parent `GATE N-Word` → children `GATE NABB-Word2` where ABB is a concise, unambiguous abbreviation of the parent's word-suffix and Word2 is the child's own branch label. Example: GATE 6-Copilot → GATE 6CP-Found, GATE 6CP-Write. A depth-2 sub-gate that omits the parent's branch encoding (e.g., a child of GATE 6A named GATE 6-Found, or a child of GATE 6-Copilot named GATE 7) fails this criterion — the reader cannot reconstruct the grandparent relationship from the identifier alone. C-25 covers first-level parent lineage encoding; C-41 requires the same recursive principle to apply at every depth level so that any identifier in the spec is independently navigable. Specs with no depth-2 sub-gating pass vacuously. |
| C-42 | **In word-suffix schemes, depth-2 parent abbreviations are unambiguous among siblings** | structure | When a word-suffix scheme produces depth-2 identifiers by compressing the parent's word-suffix into an abbreviation (C-41), each abbreviation must be unique among all first-level branches sharing the same parent gate. Two first-level branches of the same gate must not produce the same abbreviation when compressed for use as a depth-2 prefix. Example: a gate containing both `GATE 6-Copilot` and `GATE 6-Create` as first-level sub-gates would generate the same prefix `6C-…` for both, creating an unresolvable depth-2 namespace — this fails the criterion. A spec must either choose non-colliding branch names or use longer abbreviations to restore uniqueness. Scorers applying C-41 to word-suffix specs should simultaneously check C-42: C-41 requires the abbreviation to be present; C-42 requires it to be interpretable without ambiguity. Letter-slot specs pass C-42 vacuously — letter-slot depth-2 identifiers use letter concatenation, which has no abbreviation step and no collision risk. Specs with no depth-2 sub-gates in a word-suffix scheme pass C-42 vacuously. |
