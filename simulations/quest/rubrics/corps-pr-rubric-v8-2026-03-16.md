## Three new criteria from Round 7

**C-25 — Three-clause labeled Budget verdict formula** *(C-21 output-audibility gap)*

R7 V-01 moved C-21 from PARTIAL to PASS by requiring the Budget verdict to appear as exactly three labeled output lines: `WORSE rows: [list]`, `BETTER rows: [list]`, `Conclusion: [text]`. C-21 requires the verdict to be *derived* from row Net directions -- but derivation is content, not structure. The three-clause formula makes it *string-matchable*: a reviewer can verify row citation without reading prose, and a missing category must appear explicitly (e.g., `BETTER rows: none`). Without the formula, an inferrable prose verdict satisfies C-21 but fails C-25.

**C-26 — Named output result lines for C1 and C2 sub-conditions** *(C-22 silent-failure gap)*

R7 V-02 moved C-22 from PARTIAL to PASS by requiring each sub-condition to produce a named output result line: `C1 RESULT: ALL CLEAR / BLOCKED -- [item]` once before Phase C begins, and `C2 RESULT: SATISFIED / UNSATISFIED -- [citation missing]` per role section. C-22 requires dual-clause *structure in the declaration*; C-26 requires dual-clause *visibility in the output*. Without result lines, a C1 or C2 failure is detectable only by reading the pipeline declaration. With result lines, any failure is a visible output gap -- the BLOCKED or UNSATISFIED value appears where the reviewer can see it.

**C-27 — IA READ RECEIPT initial position pre-committed in a named block before findings** *(C-23 timing gap)*

R7 V-03's C-23 axis: C-23's structural requirement (receipt block before findings) is satisfied whether the initial position is written before or after findings generation. A role can write findings, observe the verdict, and then write a consistent initial position -- post-hoc rationalization that passes structural inspection. C-27 closes this by requiring the initial position to appear in a distinct labeled `PRE-COMMITMENT:` block that precedes the findings table in the output. The ordering is now independently verifiable from the output text alone.

## What changed from v7

**C-25 closes the C-21 output-audibility gap.** C-21 requires the Budget verdict to explicitly name which rows are WORSE and which are BETTER as the basis for its tradeoff conclusion. R7 V-01 showed a tighter pattern: the verdict is expressed as three labeled lines -- "WORSE rows:", "BETTER rows:", and "Conclusion:" -- making it string-matchable. A reviewer can verify the cited rows without reading prose; a Conclusion that does not follow named rows is visibly inconsistent. Without the three-clause formula, the verdict can cite rows in prose in a way that is inferrable but not string-matchable -- C-21's derived-verdict requirement is satisfied by content but not by structure. C-25 requires the three-clause labeled format as an output artifact, not merely as a content expectation. An empty-category clause (e.g., "BETTER rows: none") must still appear -- omitting the clause when a category is empty is indistinguishable from an output that forgot to audit that direction.

**C-26 closes the C-22 silent-failure gap.** C-22 requires the Phase C entry condition to be structured as two independently auditable sub-conditions: C1 (pre-flight Phase B exit checklist) and C2 (per-role IA read prerequisite). R7 V-02 showed a tighter pattern: each sub-condition produces a named output result line -- "C1 RESULT: ALL CLEAR / BLOCKED" once before Phase C begins, and "C2 RESULT: SATISFIED / UNSATISFIED" per role section. Without the result lines, C1 and C2 are declared in the pipeline but produce no output evidence of execution -- a failure is detectable only by reading the declaration and auditing whether the pipeline was followed. With the result lines, any failure is a visible output gap: a missing or BLOCKED C1 RESULT halts Phase C visibly; a missing or UNSATISFIED C2 RESULT per role is visible at that role's section. C-26 requires the named result lines as output artifacts, not only as pipeline-level declarations.

**C-27 closes the C-23 timing gap.** C-23 requires each technical role section to begin with a named IA READ RECEIPT output block containing the role's initial position on the most relevant cost row. R7 V-03 showed the remaining gap: the initial position field passes C-23's structural requirement whether it is written before or after findings are generated -- a role can write its findings first, observe the verdict, and then write an initial position consistent with those findings. This is post-hoc rationalization disguised as pre-commitment. C-27 closes this by requiring the initial position to appear in a named pre-commitment block ("PRE-COMMITMENT:") that is structurally distinct from and prior to the findings table. The structural separation makes the timing claim independently auditable: if the pre-commitment block precedes findings in the output, the ordering is verifiable. A receipt that interleaves the initial position with findings, or that appends it after the findings table, fails even if the content accurately reflects what the role would have said in advance.
