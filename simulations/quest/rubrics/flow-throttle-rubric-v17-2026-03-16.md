# flow-throttle Rubric — v17

**Skill**: `flow:throttle` | **Version**: v17 | **Date**: 2026-03-16
**Max composite**: 270 | **Golden threshold**: all 5 essential PASS + composite >= 80

---

## What Changed and Why

**C-13 — Pre-analysis source commitment** (aspirational, carried from v3)
V-03's Phase 0 source register revealed a failure mode that C-11 alone cannot prevent: retroactive citation, where a model states a number and then vaguely attributes it after the fact. C-11 checks that every number has a named source; C-13 checks that the source was *committed before the analysis began*. An output can pass C-11 (attribution present) while failing C-13 (attribution assembled post-hoc). Together they form a two-gate evidence integrity check: C-11 asks "is there a source?", C-13 asks "was the evidence base locked before the claims were built?" The required form is a named source register section that appears before any tier or threshold in the body; tiers without a register entry must be flagged `undocumented` rather than assigned inferred values.

**C-14 — Binding constraint exclusivity statement** (aspirational, carried from v3)
V-01 and V-04 both enforced an at-most-one binding tier constraint (a single Binding? = Y per inventory). The signal: designating a primary bottleneck without explaining why competing tiers are *not* the binding constraint is an assertion, not an argument. C-01 requires naming the primary bottleneck and its causal reason; C-14 requires contrastive reasoning for at least one non-binding tier — naming the mechanism that prevents it from failing before the primary bottleneck under the described load pattern (e.g., "Tier B's limit of 500 RPS is not reached before Tier A's 100 RPS connection limit because parallel branch fan-out hits the connection tier first"). Without C-14, a multi-tier analysis cannot distinguish genuine causal priority from arbitrary selection.

**C-15 — Role-locked sequencing pipeline** (aspirational, carried from v4)
V-01 and V-05 both scored 120/120 under v3. The distinguishing structural mechanism: a three-role pipeline with locked inter-phase dependencies. Domain Expert commits the evidence base first; Architect traces only from that base; UX Advocate accounts for every tier the Expert named and no others. The critical property is that no role may introduce data that a prior role did not produce — the handoff between phases is explicit and one-directional. This locked sequencing eliminates retroactive citation more reliably than any single instruction (including C-13), because it makes out-of-order claims architecturally impossible rather than merely discouraged. C-13 requires that the source register appear before the first number; C-15 requires that the entire analysis honor a phase boundary: evidence committed → claims built from evidence → UX derived from claims. An output that organizes all analysis in a single undifferentiated block fails C-15 even if every number is individually attributed, because the structural dependency that prevents late evidence injection is absent.

**C-16 — Phase-boundary data-lock prohibition** (aspirational, carried from v5)
The Round 4 comparison of V-01 and V-02 exposed a gap above C-15. V-02 has named phases and positive handoff banners ("Source Register closed. Phase B draws only from this table.") — it satisfies C-15's requirement for named phases with explicit handoff statements. But V-02 scores C-15 PARTIAL because it lacks the *negative constraint form*: V-01's phase boundaries close with explicit prohibitions ("You may not introduce any new tier, limit value, or source reference after this line"), while V-02 relies on table conventions and positive references to prior artifacts. The distinction matters because positive references tell the analyst what to use; prohibitions name the failure mode and make violations auditable. A reader can verify compliance with "no new source references may appear" by scanning downstream content; they cannot verify compliance with "draw only from the source register" without reconstructing the analysis. C-15 requires that the structural dependency be visible in the output shape; C-16 requires that each phase boundary includes a prohibition statement naming what data class may not enter after it. An output can satisfy C-15 (named phases, handoff present) while failing C-16 (handoffs are positive-only, no prohibition).

**C-17 — Consequence-phase exhaustive enumeration anchor** (aspirational, carried from v5)
The Round 4 V-02 analysis scored C-04 PARTIAL because the prompt lacks an explicit "for each tier in [Phase B Tier Inventory]" anchor in the consequence phase. V-01 earns C-04 PASS because Phase 3 opens with "For each tier in the Phase 1 inventory" — the prior enumeration artifact is named as the controlling input, making coverage exhaustive by construction and verifiable by inspection. C-04 checks whether every named tier has a UX consequence (a content completeness check); C-17 checks whether the consequence phase includes an explicit enumeration anchor that makes completeness verifiable by reference to a prior artifact rather than by reading the entire output. The distinction is auditing posture: an output can satisfy C-04 by coincidence (all tiers happen to appear without an anchor) while failing C-17 (no mechanism makes omission detectable without full reconstruction). The anchor must name the specific prior artifact ("For each tier in the Phase 1 Tier Inventory"), not a vague class ("for each tier identified").

**C-18 — Constraint-purpose annotation (inertia framing)** (aspirational, carried from v6)
The Round 5 V-05 axis surfaces an enhancement above C-16 and C-17: prohibition statements and enumeration anchors gain auditing force when annotated with the failure mode they prevent, not merely stated as rules. V-05's "inertia framing" names the three failure modes that C-16/C-17 existence corrects — retroactive source injection (a number enters after the evidence phase closes), scope creep (a tier name enters the consequence phase that was absent from the evidence phase), and coverage elision (a tier is omitted from consequence coverage without a detectable gap). An output can satisfy C-16 (prohibition present) and C-17 (enumeration anchor present) while failing C-18 (neither prohibition nor anchor names the failure mode it prevents). The annotation transforms each structural constraint from a form requirement into a diagnostic: a reader can verify not only whether the constraint is present but whether the output would have violated it without the constraint. The required form is an inline annotation at each prohibition and anchor, e.g., "— prevents retroactive source injection" or "— prevents scope creep into consequence phase." Constraints without annotations are present but not self-explaining.

**C-19 — Dedicated global criterion-coverage step** (aspirational, carried from v6)
C-12's PARTIAL pattern across V-01–V-04 reveals a structural gap: count-match checks (e.g., "N tiers named = N UX rows") verify section-level content completeness but do not map output content against essential criterion categories by name. V-05 adds a dedicated global self-verification step that names each essential criterion (C-01 through C-05) by category, confirms its presence in the output, and reports a per-criterion verdict. This is structurally distinct from C-12's aspirational self-check in two ways: it must be a named, dedicated step (not prose embedded in the body), and it must name each criterion category explicitly (not merely confirm that counts match). An output that includes a "coverage check" section confirming "5 tiers named, 5 UX rows present" satisfies C-12 partially but fails C-19, because C-19 requires the step to map output content to criterion categories (bottleneck identified → C-01, propagation traced → C-02, tiers enumerated → C-03, UX described → C-04, burst path found → C-05). C-12 and C-19 are independent: an output can satisfy C-12 (self-check present) without satisfying C-19 (step is implicit or section-scoped), and an output that satisfies C-19 necessarily satisfies C-12.

**C-20 — Prose-restriction declaration** (aspirational, carried from v7)
The Round 6 V-02 axis surfaces a format-level discipline: explicitly declaring the complete set of contexts where prose is permitted, rather than allowing mixed narrative/table output by default. V-01's C-16 prohibitions restrict which *data classes* may enter each phase; C-20 restricts which *output formats* are permitted outside named exceptions. The distinction: a prohibition statement prevents late data injection; a prose-restriction declaration makes format deviations auditable. An output with a declared prose-restriction list ("prose permitted only for: binding-tier causal reason, burst narrative, self-verification verdict notes") makes any prose appearing outside a declared context a violation identifiable without full content review. An output that mixes narrative and tables without declaring restrictions may be structurally correct (all data present) while failing C-20 (format deviations are undetectable without full review). The required form is an explicit declaration appearing before the first table, enumerating the complete and closed list of prose-permitted contexts so that the constraint is self-enforcing rather than implicit.

**C-21 — Cross-artifact referential integrity** (aspirational, carried from v7)
V-02's table cascade (TABLE C "one row per tier from TABLE A; no tier may be omitted, no tier may be added"; TABLE F "a Mitigation + Tradeoff row for every TABLE E risk") surfaces a pattern above C-17's single enumeration anchor at the consequence-phase entry. C-17 requires that the consequence phase name its controlling prior artifact; C-21 requires that *every derived artifact* — UX table, mitigations table, severity ranking — includes an explicit upstream reference by identifier (table number or phase name) and states its row-count or coverage constraint. Without C-21, only the consequence-phase entry is anchored; downstream tables (mitigations, rankings) can drift from their source artifacts without producing a detectable gap. An output can satisfy C-17 (consequence phase opens with an anchor naming the tier inventory) while failing C-21 (mitigations table has no explicit reference to the risk table it derives from, making mitigation-count drift undetectable). The required form is a header line at each derived table stating its source artifact and coverage obligation, e.g., "One row per risk from TABLE E. No risk may be omitted." The header must be co-located at the table itself — an adjacent prose statement before the table satisfies the semantic requirement but fails the form requirement, because a prose statement does not travel with the artifact when the artifact is excerpted or reordered.

**C-22 — Typed-row risk taxonomy with minimum-coverage guarantee** (aspirational, carried from v7)
V-02's TABLE E uses a Type column (Burst / Cascade / RetryAfter) with at least one required row per type. This is structurally distinct from C-09's count requirement ("≥ 2 mitigations with tradeoffs"): a count requirement can be satisfied by two rows of the same failure category, leaving an entire category uncovered without detection. A typed-row taxonomy enforces minimum coverage by failure mode category, making category-level gaps detectable by type scan rather than by full content review. An output can satisfy C-09 (two mitigations present) while failing C-22 (both mitigations address the same failure type; burst or cascade category is absent but count requirement is met). The required form is a risk/gap table with a Type column containing a named taxonomy (at minimum: Burst, Cascade, RetryAfter) and a per-type minimum-row constraint stated in the prompt. An output that lists risks without type classification fails C-22 even if the required risk categories happen to appear, because the mechanism that makes category absence detectable is absent.

**C-23 — Inline prose-exception citation** (aspirational, carried from v8)
The Round 7 V-04 variation surfaces a format-discipline signal above C-20. V-04's output carries an inline reference at each prose passage: "prose permitted here per the Prose-Restriction Declaration" identifying the specific exception that authorizes the passage. C-20 requires that the complete set of prose exceptions be declared before the first table; C-23 requires that each prose passage in the body carry an inline citation naming the specific declared exception that authorizes it (e.g., "— burst narrative exception" or "— per Prose-Restriction Declaration item 3: self-verification verdict"). The distinction: C-20 declares the legal set of prose contexts once; C-23 makes each prose use self-authorizing at the point of use. Without inline citations, detecting an unauthorized prose passage requires cross-referencing the declaration against every prose occurrence in the output — a reconstruction task. With inline citations, an unauthorized prose passage is immediately visible as one that carries no citation or cites a non-existent exception — a scan task. An output can satisfy C-20 (declaration present, exceptions enumerated) while failing C-23 (individual prose passages carry no inline authorization tag, making violations detectable only by reconstruction). The required form is an inline tag at each prose passage naming the specific exception from the C-20 declaration that permits it.

**C-24 — Type-completeness gate before mitigation phase** (aspirational, carried from v8)
The Round 7 V-03 variation surfaces a gate pattern above C-22. V-03's TYPE SCAN GATE requires explicit confirmation that all three type categories are present in the risk table before the mitigation phase may open: "(a) at least one Burst row present; (b) at least one Cascade row present; (c) at least one RetryAfter row present." C-22 requires that the risk table include a Type column with a named taxonomy and per-type minimum-row constraints; C-24 requires that an explicit gate step enumerate each expected type category, confirm its presence, and structurally block progression to the mitigation phase if any category is absent. The distinction: C-22 ensures the risk table is correctly typed and all categories are present in the finished output; C-24 ensures the type completeness check is a named, explicit structural step rather than an implicit property of the table verifiable only by inspection. An output can satisfy C-22 (Type column present, all three categories represented) while failing C-24 (type completeness is verifiable by reading the table but no gate step makes a missing category a structural block — the mitigation phase can open regardless of risk table completeness). The required form is a named TYPE SCAN GATE step (or equivalent label) that appears between the risk inventory and the mitigation table, enumerates each expected type category by name, reports a presence/absence verdict per category, and explicitly states the overall proceed/block decision before the mitigation table begins.

**C-25 — Structured-output exemption class in prose-restriction declaration** (aspirational, carried from v9)
The Round 8 V-05 variation surfaces a boundary-resolution signal above C-20 and C-23. V-05's PROSE-RESTRICTION DECLARATION adds an explicit enumeration of non-prose structured-output types exempt from C-23 inline tagging: "Gate decision lines (PROCEED / BLOCKED) and cross-artifact header lines are structured output, not prose, and do not require inline prose citation tags." C-20 declares where prose is *permitted*; C-25 requires the declaration to also name what is *not prose at all*, resolving the prose/structured-output boundary at the point of definition. Without this element, an executor applying C-23 to a prompt that combines C-20, C-23, and C-24 must judge whether gate verdicts and artifact headers require `[prose: item N]` tags — a content-interpretation task that can produce false compliance (tagging gate output as prose) or ambiguous violations (omitting tags from gate output without a stated basis for the omission). With the exemption class declared, gate decision lines and cross-artifact headers are self-classifying: a reader can determine by element type whether a C-23 tag is required, without interpreting content. An output can satisfy C-20 (prose-restriction declaration present, exceptions enumerated) while failing C-25 (the declaration enumerates only prose-permitted contexts; structured-output types that are exempt from tagging are not named, leaving the prose/structured boundary implicit). The required form is a named exemption clause within or immediately following the C-20 prose-restriction declaration, enumerating each structured-output type (gate verdicts, artifact header lines, or equivalent) that is exempt from C-23 inline citation requirements.

**C-26 — Gate inertia annotation** (aspirational, carried from v9)
The Round 8 V-05 variation surfaces a gate-level instance of the C-18 inertia-framing pattern. V-05's TYPE SCAN GATE carries a "Purpose" statement: "C-22 ensures the Type column is present and all categories appear in the finished output. This gate ensures type completeness is an explicit named structural check, not an implicit property verifiable only by reading TABLE E in full. A missing category is a structural block here, not a detectable gap in hindsight." C-24 requires the gate to exist, enumerate categories, and produce a PROCEED/BLOCK verdict; C-26 requires the gate to carry an annotation that names the specific failure mode it prevents and explains the precise gap above C-22 that makes the gate necessary. The distinction: C-18 applies inertia framing to phase-boundary prohibitions and enumeration anchors; C-26 applies the same pattern to a gate step, making the gate self-explaining rather than merely present. An auditor reading only the gate section can reconstruct why it exists and what the output would look like if it were absent — without cross-referencing the rubric. An output can satisfy C-24 (gate present, category enumeration, PROCEED/BLOCK decision) while failing C-26 (gate states the check requirements and delivers a verdict but carries no annotation naming the failure mode it prevents or explaining why it is necessary given C-22's presence). The required form is a named annotation within the gate step — "Purpose," "Why this gate," or equivalent label — that references C-22 by name or equivalent, identifies the failure mode prevented (category elision — missing type category detectable only by full table inspection), and states the gap the gate closes above C-22 (C-22 ensures the table is correctly structured; this gate ensures a missing category is a structural block, not a post-hoc detectable gap).

**C-27 — Closed-register declaration for structured-output exemption class** (aspirational, carried from v10)
The Round 9 V-03 and V-05 variations add a closure statement to the C-25 structured-output exemption register: "No other element types are added to this register without revising this contract." C-25 requires the declaration to name at least one structured-output type as exempt from C-23 tagging; C-27 requires the exemption register to declare itself a formally closed set, making any unregistered element type's classification status explicitly unresolved rather than implicitly exempt. The closure declaration is necessary because an open enumeration reintroduces the content-inspection problem C-25 was designed to eliminate: an executor encountering an element type not in the list must infer whether it is prose or structured output — the same classification task C-25 was intended to resolve at the declaration level. With the closure declaration, an unlisted type is unambiguously unclassified, its absence is a detectable gap (not an implicit permission), and adding a new type requires an explicit contract revision that is auditable by history. V-03's form: tabular register with closing declaration "No other element types are added to this register without revising this contract." V-05's form: four-column table (Element type / Canonical form / Prose? / C-23 tag required?) with explicit No/No entries and equivalent closure language. An output can satisfy C-25 (exemption types named, boundary defined for listed types) while failing C-27 (register is not declared closed; an executor encountering an unregistered element type — such as a new gate variant or a phase-boundary label — must revert to content inspection to determine its C-23 obligation). The required form is a closure declaration within or immediately following the C-25 exemption clause, stating that the register is complete and exhaustive and that additions require explicit revision to the declaration.

**C-28 — Gate-annotation audit test** (aspirational, carried from v10)
The Round 9 V-03 and V-05 variations extend the C-26 gate annotation with a dedicated audit test field that provides verification instructions for both the compliance path (gate present and functioning) and the failure path (detecting the failure mode when the gate is absent). C-26 requires the annotation to reference C-22, name the failure mode prevented, and explain the gap above C-22; C-28 requires an additional audit test element within that annotation giving a reader actionable verification instructions — what to look for to confirm gate compliance and what the output would look like if the gate were absent. V-03's form: "Field 3 — Audit test" with specific scan instructions. V-05's form: dual-path verification with separate gate-present and gate-absent instructions ("Gate present audit method: scan for TYPE SCAN GATE between TABLE E and TABLE F, confirm per-category verdicts and PROCEED/BLOCK decision present; Gate absent audit method: check whether mitigation table opens without prior type-completeness confirmation"). The distinction: C-26 ensures the annotation explains *why* the gate exists; C-28 ensures the annotation explains *how to verify* that it is present and functioning, transforming the annotation from a rationale statement into a self-contained compliance test. An output can satisfy C-26 (annotation present, failure mode named, gap above C-22 stated) while failing C-28 (annotation describes purpose but provides no audit instructions — a reader cannot verify gate compliance without independently reconstructing the expected gate form and failure signature). The required form is a named audit test element ("Audit test," "Verification method," or equivalent label) within the C-26 annotation, stating at minimum: (a) the observable evidence that confirms gate compliance and (b) the expected output signature when the gate is absent.

**C-29 — Closed-register inertia annotation** (aspirational, carried from v11)
The Round 10 V-04 axis applies the C-18/C-26 inertia-framing pattern to the C-27 closure declaration itself. V-04's closure declaration carries an annotation that demolishes the implicit-complement reasoning the closure declaration forecloses: "anything not named in the list is implicitly classified as structured output" → this requires inspecting element content to determine its classification → this reintroduces the content-inspection problem C-25 was designed to eliminate → the closure declaration forecloses this by making any unlisted type explicitly unclassified rather than implicitly exempt. C-27 requires the closure declaration to be present and to state that additions require revision; C-29 requires the declaration to also carry a named annotation that (a) identifies the implicit-complement inference it forecloses, (b) traces the reasoning chain from that inference to the exact C-25 failure mode it reintroduces, and (c) explains how the closure declaration prevents it. The structural effect: the closure declaration becomes self-explaining — an auditor reading only the exemption clause can reconstruct why it must be closed and what would happen without it, without cross-referencing the rubric. An output can satisfy C-27 (closure declaration present, additions-require-revision stated) while failing C-29 (the declaration is a bare rule statement carrying no annotation naming the implicit-complement reasoning it destroys or the content-inspection problem that reasoning reintroduces). The required form is a named annotation within or immediately following the C-27 closure declaration — "Why closed," "Inertia," or equivalent label — naming the implicit-complement inference, tracing it to the content-inspection problem, and stating how closure forecloses it.

**C-30 — Concrete unregistered-element examples in closure annotation** (aspirational, carried from v11)
The Round 10 V-05 axis extends C-29's inertia annotation with specific named examples of element types that are *not* in the exemption register and would trigger the content-inspection problem if the register were open. C-29 requires the annotation to name the abstract failure mode (implicit-complement inference leads to content inspection); C-30 requires the annotation to also name at least one concrete unregistered element type — such as a new gate variant, a phase-boundary label, or an inline decision token — and state the classification ambiguity it would create under an open register ("an executor encountering [example element] would need to inspect its content to determine whether it requires a C-23 tag"). The structural distinction: C-29 states *that* an unlisted type forces content inspection; C-30 makes the claim independently testable by naming a specific element type a reader can evaluate against the register. An output can satisfy C-29 (annotation present, implicit-complement reasoning named, content-inspection problem traced) while failing C-30 (annotation uses abstract language only — "any unlisted type would require content inspection" — without naming a specific unregistered element type that instantiates the problem). The required form is at least one named example in the C-29 annotation, identified by element type, with an explicit statement of the classification ambiguity that type would create if the register were open.

**C-31 — Named-observable gate audit test** (aspirational, carried from v11)
The Round 10 V-05 axis extends C-28's audit test by requiring named, independently verifiable observables in each path rather than general scan instructions. C-28 requires the audit test to state (a) what to look for to confirm gate compliance and (b) what the output looks like when the gate is absent; C-31 requires each path to name its observables by artifact identifier and structural position so that each observable is verifiable in isolation without parsing surrounding context. V-05's form: compliance path names three specific observables — TYPE SCAN GATE step position (between TABLE E and TABLE F), per-category verdict presence (three Y/N verdicts), PROCEED/BLOCKED decision presence (before TABLE F opens); absence path names the specific observable — TABLE F opens immediately after TABLE E with no intervening gate step. The distinction: a C-28-compliant audit test can say "scan for the gate between the risk table and mitigation table" (locating the gate by general region); a C-31-compliant audit test names each observable with the artifact references and structural position required to verify it independently (step position, verdict count, decision presence). An output can satisfy C-28 (audit test present, compliance-path instruction present, gate-absent signature present) while failing C-31 (instructions are general — "verify the gate exists and produces verdicts" — rather than naming each observable as a discrete, independently falsifiable check). The required form is a named observables list or equivalent structure within the C-28 audit test, where each observable names at minimum its artifact reference (by table or step identifier) and structural position (before/after which artifact).

**C-32 — Gate-annotation escape-route demolisher** (aspirational, carried from v11)
The Round 10 V-05 axis extends the C-26 gate annotation with an escape-route demolisher: an explicit rebuttal of the argument that C-22's structural requirements make the gate redundant. C-26 requires the annotation to reference C-22, name the failure mode prevented (category elision), and explain the gap the gate closes above C-22; C-32 requires the annotation to additionally name and rebut the specific counter-argument a reader or executor might raise to dismiss the gate as unnecessary given C-22's presence. The counter-argument to demolish: "C-22 already requires the Type column with all three categories and a per-type minimum-row constraint; therefore, any output satisfying C-22 already has type completeness — the gate is redundant." The rebuttal: C-22 ensures the risk table is correctly structured *when the output is complete*; a missing type category is detectable only by reading the finished TABLE E in full — the gate makes category absence a structural block *before* the mitigation phase opens, not a detectable gap *after* the output is assembled. Without the demolisher, a reader who notes C-22's structural requirements can reason their way to skipping the gate without a visible logical error; the demolisher closes this escape route by naming the reasoning and explaining its flaw. An output can satisfy C-26 (purpose annotation present, C-22 referenced, failure mode named, gap above C-22 stated) while failing C-32 (annotation explains what the gate does and why it exists but does not address the specific argument that C-22's requirements already provide the same guarantee, leaving the gate open to challenge by any reader who notes the structural overlap). The required form is a named demolisher element within the C-26 annotation — "Escape route," "Why C-22 alone is insufficient," or equivalent label — that states the specific counter-argument and explains why it fails.

**C-33 — Annotation field-count declaration** (aspirational, carried from v12)
The Round 11 V-04 combined axis reveals that when C-29 and C-30 are satisfied simultaneously — a two-field inertia annotation (C-29) extended with a concrete-example field (C-30) — the annotation produces a three-field structure whose completeness should be self-declared rather than requiring a reader to count fields and match them against criteria. V-04's three-field form demonstrates that C-29 and C-30's independent labeled-field requirements, when present together, create a named canonical structure: (1) implicit-complement inference field, (2) content-inspection-chain field, (3) concrete-example field. C-33 requires this multi-field annotation to carry a header line declaring its total field count and naming each field, so that annotation completeness is verifiable by count-scan before content inspection. Without the field-count declaration, verifying that all required fields are present requires knowing which criteria apply and reconstructing field-to-criterion mapping; with the declaration, completeness is a count check ("header declares 3 fields; 3 labeled fields present — complete"). An output can satisfy C-29 (two named fields present) and C-30 (third field — concrete example — present) while failing C-33 (all three fields exist and are labeled but no header declares the field count or field names, making field-completeness verification a criterion-matching task rather than a count-scan task). The required form is a header line at the annotation entry point — co-located with the annotation label ("Why closed," "Inertia," or equivalent) — declaring the total number of fields and naming each field, e.g., "Three fields: (1) Implicit-complement inference, (2) Content-inspection chain, (3) Concrete unregistered example."

**C-34 — Closed observable set in audit test** (aspirational, carried from v12)
The Round 11 V-04 combined axis reveals that the C-31 named-observable audit test is structurally parallel to the C-25/C-27 exemption register: both enumerate discrete elements whose completeness a reader must verify. C-27 closes the exemption register so that any unlisted element type is explicitly unclassified rather than implicitly permitted; C-34 applies the same closure pattern to the C-31 observable list by requiring the audit test to declare the total count of observables per path at each path's entry point, converting the observable list from an open enumeration into a closed set. Without a count declaration, an auditor verifying the audit test cannot confirm when their check is complete — the list is structurally open-ended. With a count declaration ("Gate present — 5 observables:"; "Gate absent — 1 observable:"), verification completion is a count confirmation: an auditor who has located N observables and the header declares N knows they are done. An output can satisfy C-31 (named observables present with artifact IDs and structural positions) while failing C-34 (observable list carries no count declaration — an auditor cannot confirm completeness without reading the entire list and independently judging whether additional observables were intended). The required form is a count declaration at the entry point of each audit path — co-located with the path label ("Gate present," "Gate absent," or equivalent) — stating the total number of named observables in that path, e.g., "Gate present — 5 observables:" followed by the numbered list, and "Gate absent — 1 observable:" followed by the single entry.

**C-35 — Format-contract phase lock** (aspirational, carried from v13)
The Round 12 V-05 combined axis reveals a structural distinction above C-33. In V-04, the field-count header satisfying C-33 is present as an inline instruction — the annotation header is well-formed because the prompt instructs it to be. In V-05, the Format Analyst (Role 1) produces the field-count header as a formally-declared **FORMAT CONTRACT** deliverable, and an explicit "FORMAT CONTRACT COMPLETE" statement is issued before Role 2 activates, locking the header at the role-handoff boundary. Roles 2 and 3 cannot introduce or modify the field-count declaration because the contract was sealed at the phase boundary. This is the C-16 prohibition pattern applied to format artifacts rather than data classes: C-16 prohibits late *data* injection at phase boundaries; C-35 prohibits late *format* modification at the role-handoff boundary. C-33 requires the field-count header to be present; C-35 requires that the header be produced as a named contract artifact and locked by an explicit contract-closure statement before downstream roles activate. The structural effect: C-33 compliance is enforced by role architecture rather than by instruction proximity — a downstream role cannot override the field-count declaration even if its instructions do not forbid it, because the contract was sealed before it activated. An output can satisfy C-33 (field-count header present and correct) while failing C-35 (the header appears as an inline instruction rather than as a locked contract artifact — a downstream role operating under different instructions could modify it without producing a detectable structural violation). The required form is: (a) a named FORMAT CONTRACT section produced by the first-phase role listing the field-count header as a deliverable, and (b) a FORMAT CONTRACT COMPLETE statement appearing at the role-handoff boundary before the next role activates.

**C-36 — Role-boundary observable-count declaration** (aspirational, carried from v13)
The Round 12 V-05 combined axis reveals a structural distinction above C-34 parallel to C-35's distinction above C-33. In V-04, the per-path observable-count declarations satisfying C-34 appear as inline instructions within the audit test body — they are well-formed because the prompt places them at the path-label level. In V-05, the observable-count declarations for the TYPE SCAN GATE's audit paths appear as **role-activation conditions** for the Consequence Analyst (Role 3): the gate's path-identifier count labels ("Gate-present audit method — 5 observables:" and "Gate-absent signature — 1 observable:") are the structural entry conditions under which Role 3 activates, not instructional content inside Role 3's body. The count declarations are the gate's structural form at the role-boundary level rather than at the instruction level — they appear at the architectural seam that determines whether Role 3 activates at all. This makes count-declaration omission a role-activation failure (Role 3 cannot start without the count-labeled gate output) rather than merely a formatting gap in Role 3's audit section. C-34 requires count declarations at path entry points co-located with path labels; C-36 requires those declarations to appear as role-activation conditions, so that their absence structurally prevents the consequence phase from opening rather than merely producing a malformed audit test. **C-42 — Audit block field for inventory gate compliance** (new aspirational, from v17)

**C-43 — Two-signal handoff chain for inventory verification** (new aspirational, from v17)
V-03 and V-05 both use a dedicated intermediate role (Role 1.5 — INVENTORY VERIFIER) that creates a two-signal handoff chain: FORMAT CONTRACT COMPLETE (Role 1 → Role 1.5) and INVENTORY VERIFIED (Role 1.5 → Role 2). The insight from V-05's inertia frame makes the structural necessity explicit: FORMAT CONTRACT COMPLETE confirms that Role 1's sections are present — it does not verify that the annotation inventory is complete to its declared count. A partial Section C (e.g., 6 of 7 annotations) satisfies section presence while failing count verification. When a single gate status field embedded in Role 2's entry condition (C-41's minimum pass form) serves as the only transition, the signal that closes Role 1 and the signal that opens Role 2 are conflated — "sections present" and "count verified" are indistinguishable. The two-signal chain resolves this conflation: Signal 1 (FORMAT CONTRACT COMPLETE) confirms Role 1 completion; Signal 2 (INVENTORY VERIFIED) confirms count correctness. Role 1.5 has single responsibility — its only job is to confirm count = declared; it "has no other responsibility and produces no analysis output." Role 2's entry condition checks Signal 2 by name; a Role 2 that activates on Signal 1 alone bypasses count verification. This parallels the C-34→C-36 structural argument applied to the verification role's signal design: C-41 requires a Phase 1 gate; C-43 requires that gate to use a distinct verification role whose activation signal is separate from Role 1's completion signal.

**C-44 — Inertia-frame rejection at the inventory verification boundary** (new aspirational, from v17)
V-05 names and rejects the bypass failure mode at the inventory verification gate — the "looks complete" framing — with a structural proof that the bypass is self-defeating. The proof has four steps: (1) FORMAT CONTRACT COMPLETE confirms that Role 1's sections are present; (2) it does not verify that Section C is complete to its declared count; (3) a partial Section C satisfies Role 1's section presence check while failing the count declared in Section C's header; (4) bypassing the gate converts the Section C count from a verifiable gate into an unverified assertion — a discrepancy is detectable only after Role 2 has produced output, not before. This is structurally parallel to C-26 (escape-route frame at Step 1B proving that load-shape classification cannot be deferred without creating a circular dependency). C-26 names the "STATUS tracks volume thresholds" deferral temptation and proves it is self-defeating because Step 2G depends on per-tier verdicts that Step 1B is supposed to supply; C-44 names the "looks complete / FORMAT CONTRACT COMPLETE implies completeness" bypass temptation and proves it is self-defeating because FORMAT CONTRACT COMPLETE confirms section presence, not count completeness. Both proofs convert a policy prohibition into a logical necessity: you cannot satisfy the downstream requirement by skipping the intermediate step. An output that passes C-41 (gate mechanism present) but omits the inertia-frame proof fails C-44.
