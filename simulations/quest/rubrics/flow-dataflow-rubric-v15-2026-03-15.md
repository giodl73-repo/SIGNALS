**Three new criteria added — source and logic:**

**C-35 — Phase gate PG-NN pre-registration** (from V-01)
V-01's design axis was "Gate tables pre-declared in scaffold as PG-NN artifacts." C-34 already requires the gate block to exist; the V-01 signal is that *naming* each gate PG-01, PG-02, etc. in the scaffold makes it independently cross-referenceable from C-23 and C-16. A gate that satisfies C-34 structurally but has no PG-NN entry in the scaffold cannot be cited by downstream annotations — the naming is what converts a gate block into a referenceable artifact.

**C-36 — Count parity verification gate** (from V-02)
V-02 scored C-32 PASS via "`[N] this stage, [M] total` running total + Phase 3 parity gate verifies count parity." C-32 requires per-stage counts; the V-02 signal is the separate arithmetic gate that asserts `sum(N1+N2+...+Nk) == M cumulative`. Counting per stage ≠ verifying the sum. The parity gate is a separate pre-declared artifact that closes the accumulation audit loop.

**C-37 — Structural compliance command register (CMD-NN)** (from V-03)
V-03 scored all three new criteria PASS through a CMD-01–CMD-18 register with explicit DO NOT prohibitions. The signal is that a named register makes every structural requirement individually addressable and every shortfall citable as a named CMD breach — the register is a meta-contract over all other structural criteria, pre-registered in the scaffold under C-24's navigational contract.

| C-24 | Pre-declared complete output scaffold   | structure   | Before the first stage trace entry, a dedicated scaffold section declares every output table that will appear in the response -- by table number, table name, stated purpose, and which upstream tables it references. | C-25 | Explicit stage-to-role assignment map   | structure   | In multi-role prompt designs (two or more explicitly named role sections), the STEP 0 scaffold (C-24) must include a dedicated stage assignment table mapping every pipeline stage from C-01 to exactly one named role. | C-26 | Pre-trace failure mode prediction register with lifecycle resolution | depth | Before the first stage trace entry, a dedicated prediction register declares every anticipated failure mode in the pipeline. After all stage traces are complete, a resolution section resolves every FM-NN entry with one of three statuses: CONFIRMED (prediction materialized -- cites the specific NH-NN or LP-NN it produced), EXONERATED (prediction did not materialize -- states the specific reason why the failure mode was absent), or NEW (emerged during trace without prior prediction -- assigned an NH-NN or LP-NN identifier retroactively). A prediction register with entries that lack resolution status fails. A CONFIRMED entry that does not cite its corresponding NH-NN or LP-NN fails. An EXONERATED entry with no stated reason fails. An FM-NN entry that passes through a relevant stage with no inline acknowledgment fails. An inline entry that commits to a resolution status before the post-trace section fails. This pattern enables cross-pipeline evidence to accumulate across all stages before the resolution audit commits -- deferred acknowledgment is the structural mechanism that permits evidence revision before closure. |
