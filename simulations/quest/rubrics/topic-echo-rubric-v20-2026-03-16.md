**Pattern relationships:**

The three R19 patterns form a coherent set that closes the currency, comprehensiveness, and contract-parity gaps that emerge when C-60 is added to a C-58+C-59 base:

- **C-61** applies to the GATE-TOKEN-REGISTRY required by C-58. C-58 converts token completeness from a scan-dependent to a diff-detectable property by declaring an inventory of expected gate tokens before Step 0. That inventory covers GT-01..GT-03 -- the three tokens established by C-55. When C-60 is simultaneously satisfied, a fourth gate token is produced (GT-04, CROSS-CITATION-CONSISTENT) at Step 9. A C-58-compliant registry does not automatically include GT-04: C-58's registry was designed for the three-token world of C-55, and C-60 adds a gate criterion after the registry was first established. Without C-61, the registry is accurate but not current -- a reviewer diffing against a GT-01..GT-03 inventory cannot detect a missing GT-04 because GT-04 is not declared as expected. C-61 closes this registry-currency gap: the registry self-extends to include GT-04 whenever C-60 is also satisfied. This applies to gate tokens the same currency requirement that C-52 applies to citations (a citation manifest that accounts for TYPE-A and TYPE-B but not TYPE-C fails even if TYPE-C citations exist): a registry that accounts for the tokens it knew about at first establishment but not for tokens produced by subsequently satisfied criteria is not a complete inventory.

- **C-62** applies to the TOKEN CONTENT REQUIREMENT section, which declares format contracts for gate tokens. C-55 required each gate token to carry inline specifics; C-59 extended GT-01's format; C-60 introduced GT-04. The TOKEN CONTENT REQUIREMENT section that satisfies C-59's requirements covers GT-01 in extended form, GT-02, and GT-03 -- but lacks a format contract for GT-04. A reader consulting TOKEN CONTENT REQUIREMENT cannot verify the complete format contract for CROSS-CITATION-CONSISTENT: what columns the per-pair table must include, how the MATCH assertion must be stated, what count the token must carry. C-62 closes this format-specification gap: TOKEN CONTENT REQUIREMENT must remain comprehensive as new gate tokens are added, covering all four tokens with full format contracts. This parallels C-49's requirement that the BC-COVERAGE-RECORD output prefix each field header with its schema ID: just as C-49 requires the format contract link to be active in the consuming output, C-62 requires the format contract declaration to be current for every token the artifact produces.

- **C-63** applies to the verification contract of the CROSS-CITATION-CONSISTENT token. C-55 established the self-certifying standard for the three original gate tokens: inline specifics must be sufficient to verify the pass condition without consulting underlying artifact sections. The standard is that the token is the audit record, not a pointer to the audit record. C-60 introduced GT-04 and required the per-citation table and the token -- but did not require the verification contract to be stated explicitly as traversal-free. A variation satisfying C-60 produces the table and the token; the traversal-free property is real but undeclared. Without an explicit note, a reviewer new to the artifact cannot confirm from text alone whether CROSS-CITATION-CONSISTENT can be verified from the per-citation table without also consulting the sections whose inline references it cross-checks. C-63 closes this contract-parity gap: the verification note for GT-04 must be as explicit as the self-certifying contracts established by C-55 for GT-01..GT-03. The note must name what sections are not required (STILL-LIVE FILTER, CITATION-COMPLETENESS-MANIFEST), paralleling how C-55's token format requirements specify the exact inline fields that make each token self-certifying.

## Essential Criteria (weight: 60 points total, 12 pts each)

Five criteria. All must pass to qualify as golden or passing.

| ID | Criterion | Category | Pass Condition |
|----|-----------|----------|----------------|
| C-01 | **Named surprise with departure framing** | correctness | The artifact names at least one finding explicitly framed as a departure from expectation. The framing must be structural -- the surprise is labeled or marked as such, not merely described. A finding presented as a neutral observation that happens to be unexpected does not pass. Minimum acceptable framing: "We expected X to be negligible; it turned out to dominate" (surprise). |
| C-02 | **Signal tracing per surprise** | correctness | Every named surprise is traced to a specific source signal or artifact by name. Generic attribution is insufficient. Each surprise must name the artifact that produced it. A surprise without a traceable source cannot be audited by the next team and fails. |
| C-03 | **Design impact assessed per surprise** | depth | Every named surprise carries an explicit assessment of its impact on design direction. Impact may be confirming, redirecting, or destabilizing. An impact statement that does not identify which design decision is affected fails. "This changes things" without naming what changes is insufficient. |
| C-04 | **Synthesis not summary** | behavior | The artifact contains only surprises and their analysis. Expected findings are either absent or explicitly marked as context-only. An artifact that enumerates findings and labels some "interesting" fails -- the surprise/expected partition must be explicit and structural. |
| C-05 | **Surprise specificity** | correctness | Each surprise is specific to this investigation's signal set, not a generic observation that could appear in any investigation. "APIs are harder to design than expected" fails -- it applies universally. "The scout:competitors signal revealed that all three competitors treat X as a solved problem, contrary to our assumption that the space was unsolved" passes -- it is falsifiable against this investigation's artifacts. |

## Recommended Criteria (weight: 30 points total, 10 pts each)

Output is better with these. Golden typically requires at least 2 of 3.

| ID | Criterion | Category | Pass Condition |
|----|-----------|----------|----------------|
| C-06 | **Expectation counterfactual** | depth | Each surprise explicitly names what was expected versus what was found. The counterfactual structure is stated, not implied. A reader new to the topic can reconstruct the original assumption from the artifact alone without consulting any prior document. |
| C-07 | **Institutional framing** | behavior | The artifact is explicitly addressed to a future team. At least one element signals forward-facing intent: surprises framed as things the next team would need to know, a forward statement naming the specific avoided failure, or an explicit handoff to the consuming phase. |
| C-08 | **Cross-signal pattern identification** | coverage | If two or more surprises share an underlying cause or implication, that relationship is named with a statement of what they have in common. A list of independent surprises that happen to share a root cause fails -- the relationship must be explicitly identified. |
