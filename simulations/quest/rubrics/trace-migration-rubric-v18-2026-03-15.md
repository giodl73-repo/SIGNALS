**Key evaluator distinctions (v18 additions):**

- **C-51 vs C-36**: C-36 tests whether the STATUS QUO COST section — the three-part inertia-contrast baseline — is established in a named section that precedes Q1, making the cost-of-inaction frame an entry condition for Phase A's per-entity analysis. C-51 tests whether that section precedes the parse phase itself — before constraint type enumeration begins, before Phase A's opening, before any analytical structure is established. An output where the STATUS QUO COST section precedes Q1 (PASSING C-36) but follows the parse-phase CONSTRAINT TYPE REGISTRY and phase boundary setup PASSES C-36 and FAILS C-51: the baseline is established before the per-entity analysis, but the analyst enumerated constraint types before committing to the cost-of-inaction frame. When C-51 passes, the inertia cost is not a Phase A framing section but an architectural premise: the analyst sees the cost-of-inaction before parsing any constraint, before establishing any registry, before constructing any phase boundary. The placement makes the inertia frame the first analytical commitment in the entire prompt rather than the first analytical commitment in Phase A.

- **C-52 vs C-39**: C-39 tests whether the Phase B B2 section contains a cross-role causal chain: a shared infrastructure condition whose disruption produces direct, named consequences in at least two distinct domain roles — the substrate failure demonstrating both downstream failures simultaneously. C-52 tests whether B2 is explicitly named as the close of the STATUS QUO COST FRAME opened in the pre-parse or pre-Q1 section. An output where B2 provides a fully specified cross-role chain (PASSING C-39) but whose prompt contains no explicit statement linking B2 to the STATUS QUO COST FRAME narrative PASSES C-39 and FAILS C-52: the cross-role chain provides the cost-of-inaction evidence that would close the frame, but the closure is achieved by structural coincidence and recognized by the evaluator rather than named by the prompt. When C-52 passes, the STATUS QUO COST FRAME is a named structural pair: the opening (pre-parse or pre-Q1) commits the analyst to framing all analysis within the inertia-cost premise, and B2's explicit closure statement names the moment when the accumulated cross-role cost is demonstrated. A prompt that PASSES C-51 and C-52 creates a fully scoped frame: opened before parsing, closed at B2, with every analytical section between those anchors operating within the named premise. C-39 must also pass — C-52 cannot pass if C-39 fails.

- **C-53 vs C-48**: C-48 tests whether each ENTRY REFERENCE position includes an explicit inline note naming the self-documenting property — that the compound "(BINARY FIELD)" annotation at this position makes the gate condition interpretable without consulting the EXIT BLOCK. When C-48 passes, the note may take various forms: an instruction to include the note ("Include an inline note at this position stating the self-documenting property explicitly"), a pre-written note the model fills, a table row with "Do not leave this row blank," or an exact phrasings list. C-53 tests whether the prompt goes further and names the absence of the note as a structural incompleteness: "An ENTRY REFERENCE block that carries the annotation but contains no inline rationale is structurally incomplete" (V-03's formulation) or equivalent. The distinction is verdict formulation: when C-48 passes without C-53, omission of the inline rationale is a missing-element observation — a content gap discovered by reading the ENTRY REFERENCE block; when C-53 also passes, omission is a named structural failure — the block is structurally incomplete by the prompt's own definition, detectable at the same level as a missing BOUNDARY PROTOCOL header or an absent EXIT BLOCK. The verdict formulation elevates the inline rationale from an annotation convention to a block-level completeness requirement: a structurally complete ENTRY REFERENCE block must carry both the compound annotation (C-46) and the inline rationale note (C-48), and the prompt's own definition names the incompleteness condition explicitly (C-53). C-48 must also pass — C-53 cannot pass if C-48 fails.

**Key evaluator distinctions (v17 additions):**

- **C-48 vs C-46**: C-46 tests whether the "(BINARY FIELD)" compound annotation is present at ENTRY REFERENCE positions — the same annotation that appears at EXIT BLOCK positions, making the binary constraint readable at both structural anchors without cross-referencing. C-48 tests whether the ENTRY REFERENCE block includes an explicit note stating why the annotation is present there and what it enables: that the compound type is self-documenting at this position without requiring the EXIT BLOCK to be consulted. An output where ENTRY REFERENCE positions carry "PARSE GATE (BINARY FIELD) = OPEN required" (PASSING C-46) but contain no inline note about the annotation's self-documenting property PASSES C-46 and FAILS C-48: the annotation is symmetric, but a reader who encounters only the ENTRY REFERENCE block must infer its purpose from the EXIT BLOCK's design intent rather than finding the rationale at the point of use. C-48 applies the design-intent legibility principle locally: the annotation is not only present but explained where it appears.

- **C-49 vs C-47**: C-47 tests whether the PROTOCOL COUNT MANIFEST contains a self-referential completeness assertion — a rule that names the condition under which the manifest would be incomplete ("A gate named in any block but absent from this manifest = incomplete registry" or equivalent). C-49 tests whether the assertion is accompanied by an action directive that operationalizes it: a positive instruction ("Apply this rule to detect omissions," "Check this rule against all gate-bearing blocks before proceeding") or a negative prohibition ("do not rely on cross-document search"), or both. An output where the manifest states the completeness rule without an action directive PASSES C-47 and FAILS C-49: the rule exists, the incompleteness condition is named, but the evaluator's use of the rule is unspecified. The directive is the difference between a rule that is written and a rule that is used: when C-49 passes, the evaluator is not only informed of the incompleteness condition but directed to act on it, and in the strongest form the rule also names the alternative it displaces.

- **C-50 vs C-47 / C-49**: C-47 tests whether the manifest states a self-referential completeness rule. C-49 tests whether the rule includes an action directive. C-50 tests whether the rule's coverage scope is self-defining by explicitly naming the block types that constitute the gate inventory. An output that states the completeness rule with "any block" as the scope term (PASSING C-47) leaves the coverage domain as an inference: a reader applying the rule must reconstruct the set of block types that can contain gates by reading the prompt's gate architecture. When C-50 passes, the rule names the three types: "BOUNDARY PROTOCOL block, EXIT BLOCK, or ENTRY REFERENCE position." The scope is auditable from the rule alone: no other prompt sections need to be consulted to know what "any block" means. A gate in a block type not enumerated by the rule would constitute a gap — making the rule's scope a design constraint on the prompt's gate architecture, not only a verification tool against the current inventory.

**Key evaluator distinctions (v16 additions):**

- **C-46 vs C-17 / C-34**: C-17 tests whether gate state fields carry compound "(BINARY FIELD)" type annotation at their definition sites — EXIT BLOCK positions where the gate is first declared. C-34 tests whether every phase-transition boundary has bilateral structural coverage: a named EXIT BLOCK at the preceding phase's close and a named ENTRY REFERENCE at the succeeding phase's open. C-46 tests whether the "(BINARY FIELD)" compound annotation is present at BOTH structural positions, not only at the EXIT BLOCK. An output that PASSES C-17 (EXIT BLOCK annotations carry the compound type annotation) and PASSES C-34 (bilateral structure present at every boundary) but whose ENTRY REFERENCE positions carry "PARSE GATE = OPEN required" rather than "PARSE GATE (BINARY FIELD) = OPEN required" PASSES C-17 and C-34 and FAILS C-46. The annotation at the ENTRY REFERENCE position is independently meaningful: a reader consulting only Phase B's opening to verify the Phase A-to-B gate requirement sees "PARSE GATE = OPEN" — a pass/fail state — but does not see the binary constraint that makes the gate mechanism interpretable without cross-referencing Phase A's EXIT BLOCK. C-46's value is that it extends the self-documenting principle of C-17 to both anchors of every bilateral gate, so neither the EXIT BLOCK nor the ENTRY REFERENCE requires its partner to be interpretable.

- **C-47 vs C-37 / C-45**: C-37 tests whether a named PROTOCOL COUNT MANIFEST exists at Phase B entry listing all boundary names, gate names, and gate states. C-45 tests whether the manifest enumerates all named gates in the prompt — boundary-transition gates and content-category gates — in a single table. C-47 tests whether the manifest contains an explicit rule that defines the condition under which it would be incomplete — a self-referential completeness assertion. An output that PASSES C-45 (all named gates appear as manifest rows) but contains no explicit incompleteness-detection rule PASSES C-45 and FAILS C-47: the manifest is complete at this snapshot, but its completeness is asserted by the evaluator rather than by a rule within the manifest itself. The practical consequence: if a new gate is added to any block after the manifest is written, the manifest has no mechanism to detect the omission — the evaluator must re-enumerate all gate-bearing blocks. When C-47 passes, the stated rule creates a persistent completeness contract: the manifest's authority and its incompleteness condition are both named, and any deviation from the rule is a detectable structural failure rather than a silent omission.

**Key evaluator distinctions (v15 additions):**

- **C-44 vs C-42 / C-21 / C-34**: C-42 tests whether a named binding block exists before Q1, creates ALIGNMENT STATE as a structural record, and mandates that the three artifacts — COST LEDGER row (a), Phase A Q1 analysis domain, B2 chain substrate — name the same infrastructure condition class. C-21 and C-34 test whether every phase-transition boundary has a named binary gate present at both positions. C-44 connects them: the same EXIT BLOCK that carries the phase-transition gate also carries the alignment gate, and both must be in the required state before Phase B may proceed. - **C-45 vs C-37 / C-43 / C-44**: C-37 tests whether a named PROTOCOL COUNT MANIFEST appears at Phase B entry listing all phase-transition BOUNDARY PROTOCOL blocks with gate states. C-43 tests whether a COST LEDGER SUBSTRATE GATE block exists with a conditional return-to-parse instruction for row (a) category violations. C-44 tests whether ALIGNMENT STATE is wired into the Phase A-to-B EXIT BLOCK as a gate condition. C-45 tests whether the PROTOCOL COUNT MANIFEST is extended to enumerate all named gates in the prompt — boundary-transition gates and content-category gates — in a single table. An output where the PROTOCOL COUNT MANIFEST lists three BOUNDARY PROTOCOL entries (PASSING C-37) and the COST LEDGER SUBSTRATE GATE is a named block (PASSING C-43) but the manifest does not include a row for the COST LEDGER SUBSTRATE GATE or ALIGNMENT STATE gate PASSES C-37, C-43, and C-44 and FAILS C-45: each gate exists, each gate is named, but the manifest omits the content-category layer. The practical consequence: a reviewer using only the manifest can verify boundary-gate completeness but cannot detect a closed content-category gate without reading phase body interiors or BOUNDARY PROTOCOL blocks. When C-45 passes, the two-surface verification principle introduced by C-35 and C-37 extends to the full gate architecture: section headers enumerate boundary blocks, the manifest enumerates all gates, and a discrepancy between the two surfaces — a gate named in a BOUNDARY PROTOCOL block but absent from the manifest, or a manifest row with no corresponding named block — is a cross-document structural failure detectable without reading any analytical content.

**Key evaluator distinctions (v14 additions):**

- **C-42 vs C-40 / C-41**: C-40 tests whether Phase A Q1 is the Operations/infrastructure role. C-41 tests whether COST LEDGER row (a) names a schema or infrastructure condition. C-42 tests whether the prompt contains an explicit named binding statement that requires the three artifacts — COST LEDGER row (a), Phase A Q1 analysis domain, B2 chain substrate — to name the same infrastructure condition or condition class. An output where Phase A is Operations-first (PASSING C-40) and COST LEDGER row (a) names a schema condition (PASSING C-41) achieves three-artifact alignment as an evaluator inference; the alignment is real but not structurally mandated. C-42 passes only when the prompt makes the alignment a stated requirement: a model following only C-40 and C-41 constraints could produce an Operations-first Q1 and an infrastructure-grounded COST LEDGER that analyze different infrastructure domains — satisfying both criteria independently while producing a B2 chain substrate that cites a third domain. The binding statement forecloses that divergence by naming all three artifacts in a single constraint: the same condition that anchors COST LEDGER row (a) must be the condition Q1 analyzes and the substrate B2's chain cites. An output that PASSES C-40, PASSES C-41, and PASSES C-42 achieves a three-artifact alignment that is prompt-enforced; an output that PASSES C-40 and C-41 but FAILS C-42 achieves it coincidentally.

- **C-43 vs C-41 / C-21 / C-34**: C-41 tests whether COST LEDGER row (a) names the correct category (schema/infrastructure condition rather than business process or financial metric). C-43 tests whether a violation of C-41 triggers a hard structural response — a conditional return-to-parse instruction — rather than proceeding with a misaligned substrate. C-21 and C-34 establish the binary gate mechanism for phase transitions: a gate in the wrong state blocks phase entry. C-43 extends that mechanism to the COST LEDGER row (a) category check: a wrong-category row (a) is treated as a parse-phase failure requiring restart rather than a style gap to note and continue past. An output where the COST LEDGER row (a) constraint is enforced by negative prohibition language alone PASSES C-41 and FAILS C-43: the constraint is active, but a model that produces a wrong-category row (a) can proceed to Q1 with a misaligned substrate and produce a B2 chain substrate that diverges silently from the COST LEDGER framing. When C-43 passes, the row (a) category constraint and the binary gate mechanism (C-21, C-34) operate at the same enforcement level: both block downstream phase entry on violation, and both create a return path to the phase where the violation occurred. The conjunction of C-41 and C-43 makes the COST LEDGER a hard gate on Phase A entry rather than a structural preference that Phase A proceeds past regardless of row (a) content.

**Key evaluator distinctions (v13 additions):**

- **C-40 vs C-39**: C-39 is a B2 content criterion — it passes if B2 names a cross-role causal chain linking a shared infrastructure substrate failure to named consequences in at least two distinct domain roles. C-40 is a Phase A ordering criterion — it passes only if Phase A's Q1 section analyzes the Operations or infrastructure role, establishing the shared infrastructure substrate before domain-role sections (Q2, Q3, etc.) analyze Commerce, Finance, or other domain roles. ## Essential Criteria (60 points total)

| ID | Criterion | Category | Weight | Pass Condition |
|----|-----------|----------|--------|----------------|
| C-01 | **Before/After State** | correctness | essential | For each changed entity (table, column, constraint, index), the output explicitly shows the state before the migration and the state after. "Changed" includes additions, removals, type changes, nullability changes, and constraint changes. An entity described in generic terms without before/after values does not satisfy this criterion. |
| C-02 | **Data Loss Path Identification** | correctness | essential | The output identifies at least one concrete data loss path if one exists, OR explicitly states "no data loss paths found" with reasoning. A data loss path is any migration step that can silently drop rows, truncate values, or discard columns without surfacing an error. |
