## V-01 — Single Axis: Lifecycle Emphasis (Phase-Embedded Champion Formation Event)

**Variation axis:** Lifecycle emphasis — The simulation phases contain a mandatory CHAMPION
FORMATION EVENT block in any phase where a champion first emerges. This block is a fill
contract with three required fields: (a) the role becoming a champion, (b) the bridging
mechanism (how they will accelerate spread to the next cohort), and (c) the specific incumbent
argument this champion is positioned to counter. The event is pinned to a named month. The
satellite SECTION 3 champion table is derived from these formation events — each champion row
in SECTION 3 must cite its formation phase and month. A champion appearing in SECTION 3 without
a corresponding CHAMPION FORMATION EVENT in the simulation body fails C-14.

**Hypothesis:** R2 V-05 has "Champion forming?: [role; mechanism; what THE INCUMBENT argument
they're positioned to answer]" as a question prompt inside PHASE 2 only. This surfaces the
right information but as a discretionary question answer, not a structural requirement. C-14's
pass condition is "at least one simulation phase block contains an embedded champion formation
entry with role, mechanism, and incumbent argument targeted, pinned to a specific month." The
difference between a question that CAN surface this and a fill-contract block that MUST surface
it is structural guarantee. Making the CHAMPION FORMATION EVENT a labeled block with named
fields — like CHASM-A/B/C for the chasm — converts the champion emergence moment from optional
annotation to first-class phase event. The satellite SECTION 3 derivation rule also closes the
remaining gap: if SECTION 3 is populated from formation events, and formation events carry
incumbent argument fields, C-09 (cross-archetype influence dynamics with mechanism and timing)
and C-12 (causal champion column) pass as byproducts.

## V-02 — Single Axis: Output Format (Mechanically Cited Traceability Audit)

**Variation axis:** Output format — After PHASE 5 CLOSE, before the THREE-AXIS GAP AUDIT, a
dedicated TRACEABILITY AUDIT block is required. It lists all 16 roles and for each provides:
(a) adoption month, (b) archetype cohort, and (c) a Row Citation in the parseable format
`[PHASE N / Month M / Row position]` identifying the exact structural location in the simulation
body where this role's adoption is evidenced. Generic citations ("PHASE 2", "Month 3") fail the
Row Citation column — the citation must identify the specific row within the specific month
block. Roles not yet adopted are listed as non-adopters with a Row Citation pointing to the
PHASE 5 CLOSE non-adopter list. No role passes the TRACEABILITY AUDIT on assertion alone.

**Hypothesis:** R2 V-05's THREE-AXIS GAP AUDIT Axis A has a "Source (phase + month where named)"
column, which is where C-15 was first observed as an excellence signal. But Axis A's source
column accepts any phase-and-month label — "PHASE 2 M3" passes without specifying which row
in that month block. C-15's pass condition is that "each traceability claim carries a citation
back to the simulation row." A row-level citation format `[PHASE N / Month M / Row position]`
makes the citation structurally parseable: "Row 2 of Month 3 of PHASE 2" is a specific address
that can be verified against the simulation body without interpretation. Separating the
TRACEABILITY AUDIT from Axis A also makes C-15 a standalone structural check rather than one
column buried inside a three-axis block — its compliance verdict is independent and visible.

## V-03 — Single Axis: Role Sequence (Pre-committed Adoption Schedule)

**Variation axis:** Role sequence — SECTION 1's archetype mapping table includes a "Committed
adoption month" column that is filled before the simulation begins. Each of the 16 roles
receives a declared adoption month (M1 through M9) or "non-adopter." The simulation is then
executed against this pre-declared schedule: each role must appear in its committed month. Any
deviation from the committed schedule requires an explicit explanation in the simulation row
("committed M3, delayed to M5 because: [reason]"). PHASE 5 CLOSE includes a SCHEDULE
RECONCILIATION block that compares committed months against actual months for all 16 roles.
The TRACEABILITY AUDIT (Axis A) is simplified: each role's adoption month is already declared
in SECTION 1, and the audit's job is to confirm the simulation executed against the declared
schedule or explain why not.
