## Prompt Body

You are running `/org-chart {topic}`.

DEFAULT POSITION: THE TEAM CAN OPERATE FLAT.

---

PHASE 1: LOAD AND CLASSIFY ROLES

Glob `.craft/roles/`. Produce:

  ROLES-LOADED:
  - [role name] -- [one-line description from role file]

If no files found:
  ROLES-NOTE: No .craft/roles/ files found. Using inferred roles:
  - [role name] -- [description]

Every role name appearing anywhere in this document must be declared here.
Then produce ROLE-TYPE-CLASSIFICATION (Engineering roles first, then Operations,
then PM / Design / Research / Other):

  ROLE-TYPE-CLASSIFICATION:
  - [role name] -- [Engineering / PM / Design / Operations / Research / Other]

DO NOT proceed until every role is classified.

Emit: === [PHASE GATE: ROLES COMPLETE -- INERTIA ASSESSMENT BEGINS] ===

---

PHASE 2: INERTIA ASSESSMENT -- FOUR SUB-SECTIONS

Write before any org diagram. You are the advocate for staying flat.

Sub-section 1 -- Case for Staying Flat

  | Mechanism Name           | Type         | Frequency / Participants    | Estimated Coordination Cost |
  |--------------------------|--------------|-----------------------------|-----------------------------|
  | (observable mechanism)   | (type value) | (frequency, count)          | (eng-weeks/qtr)             |

Type closed set: Channel / Informal Role / Recurring Cadence / Shared Artifact.
Minimum 2 rows, >= 2 distinct type classes.
When both thresholds met, emit:
  --- [FLAT-CASE-CLOSED: {N} mechanism rows. {K} of 4 type classes. Total cost: [sum]. How We Coordinate Today begins.] ---

Sub-section 2 -- How We Coordinate Today
Describe coordination patterns in practice. Do not repeat Sub-section 1 table entries.

Sub-section 3 -- Rebuttal
Name the ONE failure mode Sub-section 1 mechanisms cannot resolve. Use a specific
observable with a quantifier (count, duration, date, or dollar magnitude).

  UNCOVERED: [function or decision class with no current owner under flat structure]

Sub-section 4 -- VERDICT

  NET-COST-COMPARISON:
    Flat coordination cost:  [total from FLAT-CASE-CLOSED]
    Structure overhead:      [meeting load + charter maintenance + escalation delay]
    Net delta:               [signed delta]

  FLAT-CASE-PRESSURE: [rating] -- [one sentence: mechanism count + cost anchor + named failure mode]
  Rating closed set: Strong (5) / Moderate (4) / Weak (3) / Negligible (2) / Insufficient (1)

  Declare: CAN-OPERATE-FLAT or STRUCTURE-WARRANTED (must follow from sign of net delta).

  RE-ASSESSMENT TRIGGER: concrete measurable threshold in cost units.

---

FLAT-VERDICT BRANCH

IF VERDICT = CAN-OPERATE-FLAT:
  All structural sections ABSENT (ASCII diagram, Headcount, Rhythm Table, Charters,
  Register, Roadmap, Anti-Pattern Watch).
  Produce ONE affirmative investment action for the flat state.
  End artifact: Generated by: /org-chart {topic} -- {date}

IF VERDICT = STRUCTURE-WARRANTED: continue.

Emit: === [PHASE GATE: INERTIA COMPLETE -- STRUCTURE SECTION BEGINS] ===

Draw using box-drawing characters (+, -, |). Minimum three hierarchy levels.
Committees as distinct nodes -- not inside area boxes.

All role names from ROLES-LOADED or ROLES-NOTE only.

Emit: === [PHASE GATE: STRUCTURE COMPLETE -- HEADCOUNT AND RHYTHM BEGIN] ===

  |----------|-----------|-----------------|----------------|---------------------------------------|
  | (area)   | N         | [role] (type)   | (CF-coded ref) | (types) -> [named destination]        |

## What Made It Golden

Five patterns distinguish R20-V-05 from the R20-V-01 baseline (which scored 99.58):

### 1. Three-field GATE-CONDITION-MET signature (A-46)
V-01's GATE-CONDITION-MET block acknowledged the gate in prose. V-05 decomposes the
acknowledgment into three separately labeled fields -- SOURCE-GATE (exact gate text),
GATE-VERDICT (one-phrase authorization), and ARRIVAL-COUNTERPART (section name). Each
field is independently scannable without parsing prose context. This closes the
A-23/A-43 dual-site pattern: the departure site (HEADCOUNT-SUM-GATE, A-40) declares
the condition; the arrival site (GATE-CONDITION-MET, A-43/A-46) acknowledges it with
a complete three-field signature.

### 2. Ordinal label with inline cardinality-source back-reference (A-47)
V-01's AUDIT-BLOCK ordinal labels (`1 of 2`, `2 of 2`) were self-certified -- a reader
scanning a single block could not verify that "2" came from a declared count rather
than the author's assumption. V-05 embeds the authorization citation inline within the
ordinal label parenthetical: `AUDIT-BLOCK (1 of 2 -- cardinality declared at
CHARTER-AUDIT-ORDER above)`. The ordinal claim and its source are a single token; the
`above` directional anchor makes the citation navigation-complete. A separate
`CARDINALITY-SOURCE:` line is a deletable satellite; an inline annotation is not.

### 3. DRI-COVERAGE-GATE nested inside the enforcement block (A-48)
V-01 declared DRI violations in RHYTHM-DRI-ASSIGNMENT-COVERAGE but left the gate
consequence outside the enforcement block (only in the rubric). V-05 adds a
DRI-COVERAGE-GATE sub-block nested inside RHYTHM-DRI-ASSIGNMENT-COVERAGE that
explicitly names the gated section (Operating Rhythm Table section completion) and
the blocking condition (DRI-MISSING on any DRI-REQUIRED row). Gate consequence is
declared at the enforcement site itself, following the A-40 HEADCOUNT-SUM-GATE
structural pattern applied within the coverage block domain.

### 4. Dual-site gate closure pattern (A-40 + A-43/A-46 together)
The three R20 additions are not independent improvements -- they complete a dual-site
gate closure pattern that runs through the whole document. Every structural gate in the
full prompt now has both a departure declaration (announcing what must be true before
proceeding) and an arrival acknowledgment (confirming it was met with labeled evidence).
V-01 through V-04 each closed two of the three A-46/A-47/A-48 gaps; only V-05 closes
all three simultaneously, achieving the first fully consistent dual-site pattern across
all gate transitions.

### 5. Self-documenting constraint blocks (cumulative pattern)
The progression from R01 to R20 repeatedly applied one meta-pattern: move constraint
knowledge from rubric annotations into the prompt body itself. Each aspirational
criterion in bands A-36 through A-48 captures a specific instance of this: the
REGISTER-CATEGORY-CONTRACT declares categories before the register (A-36), the
RHYTHM-DRI-COLUMN-CONTRACT declares DRI requirements before the table (A-42), the
CHARTER-AUDIT-ORDER declares sequencing before the audit blocks (A-41), the
DRI-COVERAGE-GATE declares the gate consequence inside the coverage block (A-48).
V-05 is the first variation where every structural constraint in the document is
co-located with the block it governs -- no constraint is lookup-only in the rubric.
