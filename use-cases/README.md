# Signal Use Cases

Real-world accounts of Signal being used in specific domains and workflows.
Each use case documents: what was tried, what worked, what didn't, gaps found, and lessons.

## How to Add a Use Case

1. Create a new `.md` file in this directory
2. Follow the structure: What / Methodology / What Worked / What Didn't / Gaps / Metrics
3. Add an entry to the index above
4. The use case becomes reference material for future skill development and onboarding

## What Makes a Good Use Case

- **Specific, not generic.** "We used discover-hypothesis" is less useful than "discover-hypothesis surfaced a confirmation bias risk in our ANES normalization that we wouldn't have caught until submission."
- **Honest about failures.** Skills that underperformed, gaps that were painful, workflows that needed to be run twice — these are the most valuable data.
- **Actionable.** Each use case should tell the next user what to do differently, what to expect, and what to watch out for.
